## User Story

**Como** utilizador autenticado da Expressia,
**Quero** abrir `/visao` e ver, num só ecrã, os widgets que escolhi (briefing, tarefas de hoje, gastos do mês, próximas recorrências, tarefas atrasadas — e, se activados, saldo de contas e calendário da semana) com os meus dados reais,
**Para que** ao entrar na aplicação veja imediatamente o *contexto* do meu dia em vez de uma lista vazia — com layout que funciona em desktop, tablet e mobile, e cada widget a carregar independentemente sem bloquear os outros.

## Dev Notes

### Contexto arquitectural crítico

**Onde a `/visao` renderiza:** dentro de `<main className="flex-1 overflow-x-hidden p-4 lg:p-6">{children}</main>` do `AppShell` (`apps/web/src/components/shell/AppShell.tsx:57`). A story **não toca** `AppShell`, `Sidebar`, `TopBar`, `ChatPanelSlot` nem `layout.tsx`. O `overflow-x-hidden` já protege contra overflow lateral.

**REQ-INLINE-1 (regra load-bearing do Epic 5 backend):** imports de **tipos** de `@meu-jarvis/db` são permitidos (precedente `apps/web/src/lib/api-schemas/preferences.ts:18` → `import type { WidgetsEnabled } from '@meu-jarvis/db'`). A regra REQ-INLINE-1 restringe o import do **cliente DB runtime** (`getDb`) — em route handlers/RSC usa-se `@/lib/agent/db-shim` (ver `apps/web/src/lib/agent/db-shim.ts:5-17` — `@meu-jarvis/db` não resolve `@/schema` cross-package no tsc do apps/web). **Logo:** se a `/visao` ou widgets lerem a DB directamente, importam `getDb` de `@/lib/agent/db-shim`, **nunca** de `@meu-jarvis/db`.

**RLS (NFR5 — INVIOLÁVEL):** qualquer leitura de DB usa `getDb()` (role `authenticated`) — **nunca** `getServiceDb()` em código de utilizador. RLS injecta `is_household_member(household_id)` via JWT. Esta story não cria/altera tabelas nem policies → `check:rls` INALTERADO.

**Valores monetários (CON9):** os agregados 5.5 devolvem cêntimos (`*Cents`/`incomeTotal`/etc.). Conversão para `€1.234,56` (PT-PT) é do `<MoneyDisplay>` de `@meu-jarvis/ui`. Nunca formatar à mão.

### Leitura de `widgets_enabled` (DP-5.6.B)

- `GET /api/conta/preferencias` **só devolve `{ always_preview }`** (verificado em `apps/web/src/app/api/conta/preferencias/route.ts:118-124`) — **não** devolve `widgets_enabled` nem `theme`. Não estender esse endpoint nesta story (Story 5.7 fá-lo-á se precisar).
- A `/visao` RSC lê `widgets_enabled` directamente via `getDb()` de `@/lib/agent/db-shim`:
  ```sql
  select widgets_enabled from public.user_prefs where user_id = ${user.id}::uuid limit 1
  ```
  Se a row não existir (utilizador novo cujo lazy-init de prefs ainda não correu), fallback para `DEFAULT_WIDGETS_ENABLED` (importado de `@meu-jarvis/db` como **valor** — confirmar: é uma const exportada, ver `prefs.ts:69`; se o import de valor cross-package falhar no tsc, redeclarar a const localmente com nota, mas **preferir** o import para single-source-of-truth).
