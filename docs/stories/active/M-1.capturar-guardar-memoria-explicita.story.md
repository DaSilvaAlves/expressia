## Story

**As a** Eurico (único utilizador do Jarvis),
**I want** poder dizer ao Jarvis, via Telegram, "lembra-te que X" e ter a certeza de que essa preferência/facto pessoal fica guardado de forma privada e persistente,
**so that** o Jarvis comece a construir memória sobre mim sem eu ter de repetir a mesma informação em cada conversa (esta story só captura e guarda — usar a memória num prompt posterior é a M-2, e no brief diário é a M-3).

## Contexto de âmbito

**Fonte:** `docs/epics/v2-memoria-rica-brief.md` (@pm, 04/07/2026) — defaults D1-D5 todos aprovados pelo Eurico. Esta story implementa exclusivamente D1+D2 (captura explícita + tabela nova). D3 (injectar no motor) e D4 (usar no brief) ficam para M-2/M-3. D5 (esquecer) fica para M-4.

**Esta story entrega:**
- Novo intent `memorizar` no motor: o Eurico dita um facto/preferência em linguagem natural ("lembra-te que odeio reuniões antes das 10h", "não te esqueças que a minha mãe faz anos a 3 de março") → classifier reconhece `memorizar` → tool grava o texto tal-e-qual (sem parsing/estruturação) na tabela nova `jarvis_memories`.
- Migration nova `jarvis_memories` (household-scoped, 4 RLS policies) — tabela de texto livre, distinta de `jarvis_facts` (key-value, existente desde a J-3, fica reservada para settings estruturados como `timezone`/`brief_tone` — não se toca nela).
- Nova tool `memorizar` (domínio novo `memory`) — escrita interna simples e reversível: INSERT atómico + `reverse_op` real (`delete_row`) para undo honesto de 30s, exactamente como `criar_tarefa` (Story 3.8). **NÃO** é escrita externa (não chama nenhuma API fora do Postgres do próprio projecto) — por isso **não** entra nos 4 conjuntos generalizados pela J-7/J-8 (`EXTERNAL_WRITE_INTENTS`, `PREVIEW_RENDER_INTENTS`, `REUSE_PERSISTED_PLAN_INTENTS`, `IRREVERSIBLE_WRITE_TOOLS`) — ver Dev Notes.
- Confirmação ao utilizador via Telegram reaproveitando o pipeline genérico já existente (mesma frase-tipo de `criar_tarefa`: "Executei 1 operação(ões) com sucesso. Tens 30 segundos para reverter.") — sem necessidade de nenhum summary customizado.

**NÃO inclui (fora de âmbito — fica para stories seguintes da epic v2):**
- **Usar a memória** — injectar as memórias guardadas como prefixo da user message do Planner (`serializeMemoryContextForPlanner`, análogo a `serializeAccountContextForPlanner`) é a **M-2**. Nesta story, uma memória guardada fica na tabela mas **não é lida por nenhum código** ainda (mesmo estado "frio" que `jarvis_facts` teve entre a J-3 e agora — intencional, prova a espinha em passos pequenos).
- **Usar no brief diário** — `generate-daily-brief.ts` **não é tocado** nesta story. É a **M-3**.
- **Esquecer** — intent `esquecer`/tool `forget_memory` (delete por pedido) é a **M-4**. Sem ele, uma memória errada só pode ser apagada directamente na DB (aceitável para uma única story de captura).
- **Inferência automática** de factos a partir de conversas normais — explicitamente adiada para v2.x (D1 do brief). Esta story é 100% captura explícita e consentida.
- **Categorização/tags, listagem/edição via UI, retrieval (keyword ou embeddings)** — rejeitados do MVP da epic inteira (brief §1/§3), não só desta story.
- Cap de N=50 memórias injectadas (brief D3) — só relevante quando existir injecção (M-2); esta story não implementa nenhum limite de leitura (não há leitura).

**Depende de:** nada (base já existe — motor multi-intent, `toolRegistry`, `executeAtomic`, padrão RLS, todos provados nas Stories 3.8/2.14/J-5..J-8).
