## User Story

**Como** utilizador da Expressia que gere as finanças do agregado familiar,
**Quero** uma vista "Variáveis" que liste as minhas transacções pontuais com filtros (período, categoria, conta/cartão, tipo), e uma vista "Recorrentes" que liste as despesas/receitas recorrentes configuradas (renda, salário, internet) com a respectiva frequência e próxima ocorrência,
**Para que** eu possa rever, filtrar e — quando necessário — apagar movimentos sem ter de passar pela vista mensal agregada, com todos os valores em formato PT-PT (`€1.234,56`).

### [AUTO-DECISION D-4.7.1] Story 4.7 entrega list + filtros + DELETE; CREATE/UPDATE deferidos ao Jarvis

**Decisão:** As vistas "Variáveis" e "Recorrentes" entregam **listagem + filtros + eliminação**. **Criar** e **editar** uma transacção/recorrência **não** são scope desta story — o caminho é o cérebro Jarvis (Story 4.10). Os botões "+ Nova" são renderizados desactivados com tooltip a apontar para o Jarvis.

**Justificação:** (1) O Epic 4 §1 posiciona a captura via cérebro como o diferenciador central — `"paguei €78,70 no supermercado"` cria a transacção; duplicar isso em formulários UI é trabalho redundante; (2) precedente directo do Epic 3 — `tarefas/page.tsx` tem "+ Nova" desactivado com `title="usa o Jarvis"`; (3) mantém a Story 4.7 num M honesto (duas páginas); um modal de edição completo por entidade empurraria para L; (4) DELETE é incluído porque a API 4.3/4.4 já o suporta com semântica clara e uma lista sem eliminar é frustrante. **@po deve confirmar este recorte** — se o negócio exigir edição UI no MVP, 4.7 passa a L ou cria-se uma story 4.7b.

### [AUTO-DECISION D-4.7.2] "Variáveis" = transacções manuais (`recurrence_id` e `installment_id` NULL)

**Decisão:** A vista "Variáveis" lista apenas transacções com `recurrence_id IS NULL AND installment_id IS NULL` (`origin=manual`). Transacções geradas pelo cron de recorrências ou por prestações **não** aparecem aqui.

**Justificação:** Coerência com a API 4.3 (`origin='manual'`) e com a semântica variable-only do `PATCH`/`DELETE` de `transacoes/[id]` (transacção gerada → 409). As transacções recorrentes vêem-se via a recorrência-mãe na vista "Recorrentes"; as prestações na vista "Cartões" (Story 4.8). Misturar tudo numa lista única confundiria o que é editável/apagável.

### [AUTO-DECISION D-4.7.3] DELETE reflecte a semântica do backend — hard (variável) vs soft (recorrência)

**Decisão:** Eliminar uma transacção variável é hard delete; eliminar uma recorrência é soft delete (`active=false`). A UI **reflecte** a semântica que a API 4.3/4.4 já definiu (DP-4.3.4 / DP-4.4.3) — não a inventa nem a contorna.

**Justificação:** `transactions` não tem `archived_at` (hard delete). `recurrences` tem `active boolean` e o soft delete preserva a proveniência das transacções já geradas (`transactions.recurrence_id`). A copy da confirmação difere conforme a semântica: para a recorrência, informar que "as transacções já geradas são mantidas".

### [AUTO-DECISION D-4.7.4] "Recorrentes" sem paginação; "Variáveis" com paginação cursor

**Decisão:** A vista "Recorrentes" carrega tudo (sem paginação); a vista "Variáveis" usa paginação cursor.

**Justificação:** A API `recorrencias` tem hard cap 200 e sem cursor (volume baixo — uma família tem dezenas de recorrências, não centenas). A API `transacoes` tem cursor (`limit` 50) — as transacções variáveis acumulam-se e exigem paginação. A UI segue o que cada API oferece.

### [AUTO-DECISION D-4.7.5] Helpers de leitura separados e `db`-injectáveis

**Decisão:** `list-variable-transactions.ts` e `list-recurrences.ts` — funções `db`-injectáveis, testáveis sem Postgres.

**Justificação:** Padrão estabelecido na Story 4.6 (D-4.6.7) e em `listTasksHelper`. Mantém as queries fora do componente, testáveis, e reutilizáveis (ex: tool `query_finance_summary` da Story 4.10).

### [AUTO-DECISION D-4.7.6] `RecurrenceFrequencyLabel` co-localizado em `financas/_components/`

**Decisão:** Pequeno componente que mapeia `recurrence_freq_finance` enum → label PT-PT (`monthly`→"Mensal", `weekly`→"Semanal", `biweekly`→"Quinzenal", `daily`→"Diária", `quarterly`→"Trimestral", `yearly`→"Anual", `custom`→"Personalizada"). Em `apps/web/src/app/(app)/financas/_components/`.

**Justificação:** Coerente com D-4.6.1 (componentes de Finanças co-localizados, sem `packages/ui`). O `architecture.md §8.4` previa `RecurrenceDisplay` em `packages/ui` — mesma discrepância documental já registada na Story 4.6 C2; segue o precedente real.
