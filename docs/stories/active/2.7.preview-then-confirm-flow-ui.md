## User Story

**Como** utilizador autenticado de um household no Expressia,
**Quero** abrir uma página `/jarvis`, escrever um prompt em PT-PT (ex: "amanhã reunião 15h e paguei 78,70 no supermercado"), ver o cérebro AI a classificar as intents, e — quando a confiança for baixa OU eu tiver activado *sempre confirmar* nas preferências — receber um cartão de pré-visualização com a lista de operações que vão ser persistidas, com botões **Confirmar** e **Cancelar**,
**Para que** eu controle o que o agente regista no meu household antes de qualquer escrita irreversível e tenha visibilidade real do pipeline AI multi-intent (FR4).

### [AUTO-DECISION D29] Schema `user_prefs` 1:1 user (não 1:1 user×household)

**Decisão:** `user_prefs.user_id` é PK único (1 row por user). `household_id` é FK obrigatória (cascade) para suportar RLS via `current_household_id()`, mas user com múltiplos households partilha a mesma `always_preview`.

**Justificação:**
1. Preferência UX (always_preview) é cognitiva-pessoal, não household-específica.
2. Simplificação: sem composite PK `(user_id, household_id)`.
3. Architecture §5.2 prevê endpoint `/api/auth/switch-household` que actualiza JWT — ao trocar household, RLS aplica `current_household_id()` mas `user_prefs.always_preview` mantém-se.
4. Edge case: user em 2 households com prefs diferentes — adiado (DP futuro). MVP single-pref-per-user.

**Trace:** EPIC-2 DP2 recomendação C "toggle com default 0.70 (FR4)" — não especifica cardinalidade.

### [AUTO-DECISION D30] PreviewToken via run_id (não HMAC adicional)

**Decisão:** Story 2.6 D21 ratificada por DOC-FIX-001 estabeleceu nested REST `/api/agent/prompt/[runId]/confirm` com `confirm_expires_at` row-level em `agent_runs`. Story 2.7 NÃO introduz HMAC previewToken adicional — usa `run_id` directo (URL-safe UUID) + RLS para ownership.

**Justificação:**
1. EPIC plan original (escrito antes de 2.6) sugeria "previewToken HMAC 5min". Esta proposta foi superseded por D21+DOC-FIX-001 (run_id + RLS).
2. RLS no `getDb()` garante que apenas o owner (via JWT `household_id`) pode confirmar a run — equivalent security a HMAC sem complexidade adicional.
3. KISS: menos um secret a gerir (HMAC key), menos uma dependência crypto.
4. Defense-in-depth: ownership via RLS + status check (`pending_preview`) + TTL check (`confirm_expires_at > now()`).

**Trace:** Story 2.6 D21, DOC-FIX-001, Architecture §4.4 + §4.5.

### [AUTO-DECISION D31] UI components inline (sem `components/` directory)

**Decisão:** Components do chat (ChatInput, PreviewCard, ResultMessage, IntentList) ficam em `app/(app)/jarvis/_components/*.tsx` (ou inline no `page.tsx` se ≤ 200 linhas). Não criar `apps/web/src/components/` directory ou design system geral.

**Justificação:**
1. Codebase actual não tem `components/` directory — toda a UI até hoje (visao/, entrar/, registar/) é inline em `page.tsx`.
2. KISS para MVP — generalização prematura é dívida técnica.
3. Story 5.x (futura, UI Phase 2 quando @ux-design-expert engajar) extrairá design system se padrões repetirem-se.
4. Convenção Next.js 15 App Router: `_components/` (underscore prefix) não é route — é colocation idiomática.

**Risco:** se Story 2.8 (Undo UI) precisar dos mesmos components, refactor para `components/` será necessário. Mitigação: documentar trade-off no Change Log; @architect ratifica no gate.

### [AUTO-DECISION D32] Lazy-init de `user_prefs` row no GET

**Decisão:** `GET /api/conta/preferencias` cria a row em `user_prefs` com defaults (`always_preview=false`) se não existir, em vez de exigir migration de seed para cada user existente.

**Justificação:**
1. Nenhum user existente tem row em `user_prefs` (tabela nova).
2. Seed migration 0007 forçaria backfill de auth.users em produção — complexo + slow + race conditions com novos signups.
3. Lazy-init no GET é idempotente (UPSERT com `ON CONFLICT DO NOTHING`), zero overhead na maioria dos requests (1 INSERT no primeiro acesso por user, depois SELECT).
4. PATCH faz UPSERT também (ON CONFLICT UPDATE) para suportar caso edge "user nunca abriu /conta/preferencias mas activou via API directa".

**Trace:** Padrão Supabase `auth.users` lazy-init em domínio (Story 1.3 fez isto para `households` via trigger `on_auth_user_created`). 2.7 simplifica: lazy-init no endpoint, sem trigger.

### [AUTO-DECISION D33] CodeRabbit pre-commit/push skipped

**Decisão:** Consistente com Stories 2.2 (D14) / 2.3 / 2.4 (D12) / 2.5 (D14) / 2.6 (D22+D26). Architect gate substitui review automatizado.

**Justificação:**
1. 2.7 é primariamente UI + endpoint refactor + nova tabela com RLS — todos padrões já validados em 2.6 (RLS DO block + endpoint + auth + tests).
2. Architect gate cobre cross-domain (backend + frontend) com profundidade superior ao CodeRabbit em context PT-PT.
3. CodeRabbit re-baseline schedulada para Story 2.10 (benchmark E2E + credentials reais — onde brilha em auth/secrets).

### [AUTO-DECISION D34] Tests UI com Testing Library + jsdom (não Playwright E2E)

**Decisão:** Tests UI usam Vitest + `@testing-library/react` + `jsdom` (já configurado em `apps/web/vitest.config.ts`). Sem Playwright E2E nesta story.

**Justificação:**
1. Vitest + jsdom já é o setup de Story 1.5+1.6 (login/register tests) — zero novo setup.
2. Playwright E2E adicionaria dependência + CI infra + flakiness em runners GitHub Actions.
3. Mocks Vitest cobrem `fetch /api/agent/prompt` deterministicamente (precedente 2.6 com 64 mocks).
4. Story futura (2.10 ou 5.x) pode adicionar Playwright para E2E user journey real.

### Schema canónico user_prefs (D29 single-row per user) — [PO_FIX_INLINE v1.1] split 0007/0001

**Em `packages/db/migrations/0001_rls_policies.sql`** (anexar antes do "FIM DA MIGRAÇÃO 0001", pattern espelhado de `agent_rate_limit_counters` em :692-708):

```sql
-- =====================================================================
-- user_prefs (Story 2.7 — tabela criada via 0007)
-- =====================================================================
-- create policy "user_prefs_select_self" on public.user_prefs for select to authenticated
-- create policy "user_prefs_insert_self" on public.user_prefs for insert to authenticated
-- create policy "user_prefs_update_self" on public.user_prefs for update to authenticated
-- create policy "user_prefs_delete_self" on public.user_prefs for delete to authenticated

do $rls_user_prefs$
begin
  if exists (select 1 from pg_tables where schemaname = 'public' and tablename = 'user_prefs') then
    execute 'alter table public.user_prefs enable row level security';
    execute 'alter table public.user_prefs force row level security';

    execute 'drop policy if exists "user_prefs_select_self" on public.user_prefs';
    execute 'drop policy if exists "user_prefs_insert_self" on public.user_prefs';
    execute 'drop policy if exists "user_prefs_update_self" on public.user_prefs';
    execute 'drop policy if exists "user_prefs_delete_self" on public.user_prefs';

    execute $POLICY$create policy "user_prefs_select_self" on public.user_prefs for select to authenticated using (public.is_household_member(household_id) and auth.uid() = user_id)$POLICY$;
    execute $POLICY$create policy "user_prefs_insert_self" on public.user_prefs for insert to authenticated with check (public.is_household_member(household_id) and auth.uid() = user_id)$POLICY$;
    execute $POLICY$create policy "user_prefs_update_self" on public.user_prefs for update to authenticated using (public.is_household_member(household_id) and auth.uid() = user_id) with check (public.is_household_member(household_id) and auth.uid() = user_id)$POLICY$;
    execute $POLICY$create policy "user_prefs_delete_self" on public.user_prefs for delete to authenticated using (public.is_household_member(household_id) and auth.uid() = user_id)$POLICY$;
  end if;
end$rls_user_prefs$;
```

**Razão da split:** `scripts/check-rls-coverage.ts:33` lê APENAS `0001_rls_policies.sql` como fonte de verdade do gate NFR5; policies em 0007 não são detectadas. Predicate combina cross-tenancy isolation (`is_household_member`) com user-scoped constraint (`auth.uid() = user_id`) — owner do household NÃO pode ler prefs cognitivas de outros membros.
