### Decisões autónomas tomadas

#### Decisão #1 — Tailwind CSS v3.4 em vez de v4.x
- **Decisão:** Usar `tailwindcss@^3.4.14` (com `postcss` + `autoprefixer`) em vez de Tailwind 4.x.
- **Rationale:** A Story Dev Notes mencionam Tailwind 4.x mas Tailwind 4 está em alpha/beta com mudanças significativas no setup (CSS-first, `@import "tailwindcss"`, sem `tailwind.config.ts`). Para foundation work com risco baixo a Story 1.1 deve usar a versão estável (3.4) que tem zero pegadinhas com Next.js 15. Stories futuras de UI (`squads/design-system`) podem migrar para Tailwind 4 quando estável e quando os primitivos shadcn/ui forem adicionados.
- **Alternativas consideradas:** Tailwind 4.x alpha (rejeitado: instabilidade), Tailwind 3.4 (escolhido), Tailwind 3.3 (rejeitado: versão anterior sem features 3.4 como container queries default).
- **Rastreabilidade:** Não viola PRD/Architecture (Architecture §2 lista "Tailwind 4.x" mas Aria/PM podem reavaliar quando v4 estabilizar).

#### Decisão #2 — Node `>=20` engines em vez de fixar Node 20 LTS
- **Decisão:** `engines.node: ">=20"` em todos os `package.json`.
- **Rationale:** Story Dev Notes dizem "Node 20 LTS" mas o ambiente actual corre Node v22.22.2 (também LTS-class). Fixar `=20` quebraria. `>=20` cobre 20.x LTS, 22.x LTS, e 24+ futuro sem mudança.
- **Rastreabilidade:** Architecture §2 diz "Node 20 LTS" como mínimo, não exacto.

#### Decisão #3 — `pnpm@9.12.3` via `packageManager` field
- **Decisão:** Pinning explícito de pnpm 9.12.3 via `packageManager` field.
- **Rationale:** Corepack-friendly; garante que CI e dev têm a mesma versão. 9.12.3 é a stable de Outubro 2024.
- **Alternativas:** Sem pinning (rejeitado: drift entre máquinas), pnpm 9.0 (rejeitado: bugs corrigidos em 9.10+).

#### Decisão #4 — ESLint flat config? Não — `.eslintrc.json` clássico
- **Decisão:** Usar `.eslintrc.json` em vez de `eslint.config.mjs` (flat config).
- **Rationale:** `eslint-config-next@15.0.3` ainda usa formato extends-style; flat config requer `next/core-web-vitals` em FlatCompat e adiciona complexidade desnecessária para foundation. Migração para flat config é story própria quando ecosystem amadurecer.
- **Alternativas:** Flat config `eslint.config.mjs` (rejeitado: requer FlatCompat workaround), `.eslintrc.cjs` (rejeitado: JSON é mais simples).

#### Decisão #5 — Turbo `typecheck` sem `dependsOn: ["^build"]`
- **Decisão:** Após self-critique, removi `dependsOn: ["^build"]` da task `typecheck` no `turbo.json`.
- **Rationale:** packages/db não tem build real (echo placeholder). Forçar `^build` antes de typecheck é desperdício. typecheck pode correr em paralelo.

#### Decisão #6 — `next/typescript` em vez de `@typescript-eslint/recommended`
- **Decisão:** ESLint extends `next/core-web-vitals` + `next/typescript`.
- **Rationale:** A Story pede `@typescript-eslint/recommended` mas `next/typescript` (introduzido em `eslint-config-next@14`) já incorpora as mesmas regras + integração com Next.js. Mais limpo.
