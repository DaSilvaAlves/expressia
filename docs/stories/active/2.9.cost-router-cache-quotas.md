## User Story

**Como** utilizador autenticado de um household Família que submete prompts recorrentes ao `/jarvis` ao longo do mês,
**Quero** que prompts repetidos sejam respondidos mais depressa (sem re-invocar o classificador LLM), que perguntas simples como "quantas tarefas tenho?" sejam resolvidas directamente na base de dados sem chamar o executor Sonnet, e que o sistema me avise claramente em PT-PT quando atingir o limite mensal de prompts antes de cobrar overrun,
**Para que** a Expressia mantenha custo LLM abaixo de 30% MRR (NFR20 / R3 mitigação) e eu possa usar o agente com confiança de que não haverá cobranças surpresa nem performance degradada por abuso.

## Acceptance Criteria

### 1. Cache Upstash — lookup antes do classifier

- **AC1:** Criar `apps/web/src/lib/agent/cache.ts` com interface `UpstashClientLike` mockable-only e implementação `UpstashCache` que usa `@upstash/redis`:
  - Método `get(key: string): Promise<string | null>` — lookup de ClassificationResult cacheado.
  - Método `set(key: string, value: string, ttlSeconds: number): Promise<void>` — cache SET com TTL.
  - Método `del(key: string): Promise<void>` — invalidação manual (para testes).
  - Cliente inicializado lazy via `UPSTASH_REDIS_REST_URL` + `UPSTASH_REDIS_REST_TOKEN`; se variáveis ausentes, `UpstashCache` retorna sempre `null` em `get` (modo degradado — sem crash em dev/CI). Trace: Architecture §4.6, `.env.example:61-62`.

- **AC2:** Criar função `buildCacheKey(prompt: string, householdPlan: string): string` em `apps/web/src/lib/agent/cache.ts`:
  - Implementação: `sha256(normalize(prompt) + householdPlan)` usando `node:crypto.createHash('sha256').update(...).digest('hex')`.
  - `normalize(prompt)`: trim + lowercase + collapse whitespace (`prompt.trim().toLowerCase().replace(/\s+/g, ' ')`).
  - **NOTA anti-hallucination:** key usa `householdPlan` (ex: `'familia'`), NÃO `householdId` — conforme Architecture §4.6 literal (`sha256(normalize(prompt) + household_plan)`). Dois utilizadores do mesmo plano partilham cache de classificação (segurança: ClassificationResult não contém PII — apenas intents detectadas).
  - NUNCA logar o prompt em cleartext nos logs OTel (NFR12 PII redaction — precedente DN5 Story 2.6 redactPii). Cache key (hash) é seguro para logging.
  - Trace: Architecture §4.6 algoritmo de routing — linha `sha256(normalize(prompt) + household_plan)`.

- **AC3:** Integrar cache lookup em `apps/web/src/app/api/agent/prompt/route.ts` **antes** do classifier (step 6 actual):
  - Após rate limit + quota check (step 4) e INSERT initial agent_run (step 5), tentar `cacheClient.get(cacheKey)`.
  - **HIT:** parsear ClassificationResult cacheado, pular classifier LLM, registar `cache_hit: true` em OTel span (`annotateAgentPromptSpan`). Continuar para cost router (AC6) usando ClassificationResult do cache.
  - **MISS:** executar classifier normalmente. Após classificação bem-sucedida, fazer `cacheClient.set(cacheKey, JSON.stringify(result), CACHE_TTL_SECONDS)`.
  - `CACHE_TTL_SECONDS`: lido de `process.env.CACHE_TTL_SECONDS`; default `300` (5 minutos) se ausente ou inválido.
  - Cache NÃO é invocado no path idempotent replay (step 3) — idempotency check tem prioridade.
  - Trace: Architecture §4.6 fluxo "HIT (5min TTL) → return cached response [classifier-only path]".

## [AUTO-DECISIONS] @sm

### [AUTO-DECISION D44] UpstashClientLike mockable — zero call real em tests (precedente 2.2-2.8)

**Decisão:** Cliente Upstash Redis implementado via interface `UpstashClientLike` injectável (padrão D8/D13/D22 stories anteriores). Em tests, `vi.mock('@upstash/redis')` é a estratégia. Em modo degradado (env vars ausentes), `UpstashCache` retorna `null` em `get` sem throw.

**Justificação:** EB3 pending. Zero CI infra extra. Padrão estabelecido em 6 stories Done APPROVED 9.4/10. `UpstashClientLike` interface permite swap futuro para outros redis clients sem alterar o caller.

**Trace:** Stories 2.2-2.8 precedente mockable-only, Architecture §4.6 "Redis cache 5min".

### [AUTO-DECISION D45] SHA256 cache key via `node:crypto` (Edge-compatible)

**Decisão:** `node:crypto.createHash('sha256')` — não `crypto.subtle.digest` (async). `node:crypto` está disponível em Next.js Edge Runtime (Vercel `fra1`) e em Vitest `node` environment sem polyfill. `crypto.subtle` requer await + Buffer conversion adicional sem vantagem.

**Justificação:** Consistência com precedente `hashPrompt` em `apps/web/src/lib/agent/redaction.ts` que usa `node:crypto` (Story 2.6 D25). Re-usar mesmo módulo no mesmo codebase.

**Trace:** `apps/web/src/lib/agent/redaction.ts:hashPrompt` (Story 2.6), Next.js Edge Runtime docs `node:crypto` support.

### [AUTO-DECISION D46] Cache TTL 300s default — configurável via `CACHE_TTL_SECONDS`

**Decisão:** TTL de 5 minutos (`300s`) conforme Architecture §4.6. Configurável via env `CACHE_TTL_SECONDS` (fallback `300`) para facilitar override em staging/dev sem deploy.

**Justificação:** Architecture §4.6 literal "cache Upstash 5min". Env var override é padrão AIOX para configuração não-estrutural (precedente `AGENT_RATE_LIMIT_PER_MINUTE` em rate-limiter.ts).

**Trace:** Architecture §4.6, `rate-limiter.ts:28-31` (padrão env var override).

### [AUTO-DECISION D47] Cost router bypass APENAS para intent singleton `consultar_dados`

**Decisão:** `isSingleConsultarDados` retorna `true` APENAS quando `intents.length === 1 AND intent === 'consultar_dados'`. Multi-intent com `consultar_dados` mantém pipeline completo. "Consultar + criar" num mesmo prompt usa executor Sonnet.

**Justificação:** Architecture §4.6 literal "if intents == [consultar_dados] AND read-only → query DB direct, no executor". O "==" implica singleton — multi-intent não foi explicitamente incluído no design e o executor é necessário para os side-effects das outras intents.

**Trace:** Architecture §4.6 algoritmo de routing, ADR-009 §14.9.

### [AUTO-DECISION D48] Correcção `QUOTA_BY_PLAN` — valores actuais vs Architecture §4.6

**Decisão:** Corrigir `QUOTA_BY_PLAN` em `rate-limiter.ts` de `{free:100, pessoal:500, familia:2000, pro:10000}` para `{free:50, pessoal:1500, familia:3000, pro:10000}`.

**Justificação anti-hallucination:** Leitura directa de `apps/web/src/lib/agent/rate-limiter.ts:40-45` confirma valores actuais. Architecture §4.6 tabela "Quotas por plano (NFR20)" especifica `{Free:50, Pessoal:1500, Família:3000, Pro:10000}`. Divergência é real — Story 2.9 corrige. Esta é uma correcção de alinhamento com spec, não uma feature nova.

**Trace:** `rate-limiter.ts:40-45` (estado actual codebase), Architecture §4.6 (spec correcta).

### [AUTO-DECISION D49] Hard-stop 110% — `Math.floor(limit * 1.1)`

**Decisão:** Condição hard-stop: `used >= Math.floor(limit * 1.1)`. Exemplo família: `3000 * 1.1 = 3300`.

**Justificação:** Architecture §4.6 literal "Hard-stop a 110% para evitar abuso". `Math.floor` garante inteiro (prompts são inteiros). Actualmente `checkQuota` usa `>= limit` (100%) — correcção alinha com spec.

**Trace:** Architecture §4.6, `rate-limiter.ts:149` (condição actual `if (used >= limit)`).

### [AUTO-DECISION D53] Cache key usa `householdPlan` (não `householdId`) — correcção vs spawn prompt

**Decisão:** Cache key = `sha256(normalize(prompt) + householdPlan)` conforme Architecture §4.6. O spawn prompt desta story alegou `sha256(prompt + household_id)` — discrepância corrigida aqui.

**Justificação anti-hallucination:** Architecture §4.6 literal: `sha256(normalize(prompt) + household_plan)`. Usar `household_id` criaria uma cache per-household (muito granular, hit rate baixo). Usar `household_plan` partilha cache entre users do mesmo plano (classificação de intents não é PII-sensitive — é apenas a lista de intents detectadas). Correcto para maximizar hit rate (R7 mitigação).

**Trace:** Architecture §4.6 linha "sha256(normalize(prompt) + household_plan)".

### [AUTO-DECISION D54] `consultar_dados` direct-DB usa `getDb()` (authenticated) com RLS

**Decisão:** `executeDirectQuery` usa `getDb()` (authenticated, RLS aplicada) — NUNCA `getServiceDb()`. Queries são read-only SELECT com `household_id = $1` onde `$1` vem do JWT claim `current_household_id()`.

**Justificação:** `consultar_dados` é operação de leitura de dados do utilizador — RLS via authenticated é o mecanismo correcto (garante que utilizador só vê os próprios dados). `getServiceDb()` seria bypass de segurança inaceitável para reads de dados do utilizador. Análogo a `checkQuota` que usa `getDb()` para SELECT em `agent_quotas` (mas UPDATE/INSERT precisam de service_role — AC9 D50).

**Trace:** Architecture NFR5, CLAUDE.md "getDb() em todas as rotas/RSC/Server Actions com utilizador final", D50 distingue claramente reads (getDb) vs writes (getServiceDb) em agent_quotas.

## Critical Dev Notes

**DN1 — Mockable-only ZERO calls reais:** `UpstashClientLike` interface injectável em todos os testes. `vi.mock('@upstash/redis')` cobre o cliente real. Em CI (sem env vars Upstash), `UpstashCache` cai em modo degradado (sempre MISS) sem throw. Zero chamadas reais a Upstash em qualquer test.

**DN2 — Cache key é `household_plan`, NÃO `household_id`:** Architecture §4.6 literal. Dois utilizadores do mesmo plano partilham cache de ClassificationResult (intents — não PII). Maximiza hit rate. Ver D53.

**DN3 — SHA256 via `node:crypto` (precedente redaction.ts):** `import { createHash } from 'node:crypto'; createHash('sha256').update(input).digest('hex')`. Mesmo módulo usado em `redaction.ts` hashPrompt. NÃO usar `crypto.subtle` (async desnecessário).

**DN4 — NUNCA logar prompt cleartext em cache logs:** O `cacheKey` (hash) pode ser logado. O `body.prompt` original nunca aparece em logs Pino (NFR12 PII). Ver precedente DN5 Story 2.6.

**DN5 — `agent_quotas` INSERT/UPDATE requer `getServiceDb()`:** RLS bloqueia authenticated. Crítico fix em `incrementQuota`. Ver D50 + AC9. `getDb()` pode ser mantido para SELECT em `checkQuota` (read-only é permitido per `agent_quotas_select_member` policy).

**DN6 — `consultar_dados` direct-DB usa `getDb()` (authenticated + RLS):** Leitura de dados do utilizador DEVE usar authenticated com RLS. Nunca `getServiceDb()` para reads de domínio. Ver D54.

**DN7 — Hard-stop 110%:** `Math.floor(limit * 1.1)` — família: 3300. Testar limite exacto (3299 passa, 3300 bloqueia). Ver D49.

**DN8 — `QUOTA_BY_PLAN` correcto pós-fix:** `{free:50, pessoal:1500, familia:3000, pro:10000}`. Os valores actuais no codebase estão errados vs Architecture §4.6. Ver D48.

**DN9 — Cost router path sem `undo_url`/`undo_expires_at`:** Operação `consultar_dados` é read-only — não tem reverse_op, não tem undo. Response shape: `{ mode: 'executed', run_id, results: directResult, summary: '...' }` sem campos undo. Client (ResultMessage) já lida com `undoUrl` ausente (D40 Story 2.8).

**DN10 — `incrementQuota` chamado no path direct-DB:** Quota é por prompt processado (inclui direct-DB). O contador `prompts_used` deve incrementar independentemente de LLM ter sido chamado ou não.

**DN11 — `householdPlan` necessário para `cacheKey`:** Para calcular `cacheKey`, `route.ts` precisa do `plan` do household. Pode ser lido de `agent_quotas.plan` (se row existe) ou de `households.plan`. Preferir `households.plan` via uma query simples (fallback: `'free'` se não encontrado, para ser conservador).

**DN12 — `period_end` em `QuotaExceededError`:** Para mensagem "Próxima janela em N min", `checkQuota` precisa de ler `period_end` da `agent_quotas`. Se row não existe, usar `date_trunc('month', now()) + interval '1 month'` como estimativa. N min = `Math.ceil((period_end - now()) / 60000)`.

**DN13 — PT-PT estrito:** "Limite de prompts atingido" (não "Quota excedida"). "Próxima janela em" (não "Próximo período"). "Cache hit — a reutilizar classificação" (não "Cache hit — reusing"). Consultar vocabulário Stories 2.6/2.7/2.8 para consistência.

**DN14 — Templates `consultar_dados` são MVP mínimo:** 3 templates (count_tasks, count_finances, balance_summary) cobrem os casos mais frequentes. Catálogo expandido em Epic 3/4 quando tools de domínio completas estiverem implementadas. Story 2.9 NÃO inventa templates além destes 3.
