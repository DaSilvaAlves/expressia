### AC5 — Script de benchmark executável

Criar `packages/db-test/src/benchmark/run-benchmark.ts` (ou `scripts/benchmark/run.ts`):

- Lê os 200 prompts de `prompts-pt-pt.ts`.
- Para cada prompt, invoca `Classifier.classify(prompt, ctx)` com provider OpenAI real (gated por `RUN_BENCHMARK_REAL=true`).
- Regista: `prompt_hash` (SHA256 — NFR12 sem PII), `expected_intents`, `actual_intents`, `actual_confidence`, `latencyMs`, `costEur`, `correct: boolean`.
- `correct = true` quando: todos os `expected_intents` estão em `actual_intents` E nenhuma intent extra de alta confiança (>0.70) está presente.
- Calcula e imprime: precisão global (`correct_count / 200 * 100`), precisão por intent, p50/p95 latência, custo total EUR.
- **Passa/falha o script** com exit code 1 se precisão < 90% OU p95 latência > 6000ms (NFR1).
- Modo mock (sem `RUN_BENCHMARK_REAL=true`): usa fixtures determinísticos de `__fixtures__/mock-classifications.ts` — roda em CI sem LLM call real.

**Trace:** PRD OKR O2 KR2 (precisão ≥90%); NFR1 (latência p95 < 6s); NFR12 (prompt hash só — sem cleartext); Story 2.5 nota "Story 2.10 (benchmark) — testa Planner+Executor com 200 prompts PT-PT reais".

### [AUTO-DECISION D56] DPA UE Anthropic — risco aceite + mitigação documental

**Decisão:** Anthropic não oferece endpoint regional EU em mai 2026 (verificação via docs públicas Anthropic). DPA contratual disponível para Enterprise. Estratégia MVP: usar endpoint padrão `api.anthropic.com` + DPA contratual (se disponível) + documentar como Questão Aberta QA1. Não bloquear MVP por ausência de endpoint EU se DPA contratual for aceitável pelo @po.

**Justificação:** NFR11 (data residency UE) é inegociável para dados ARMAZENADOS (Postgres em Frankfurt, Supabase EU). O PROCESSAMENTO de prompts por LLM externo não é dados pessoais armazenados — é dado transiente de inferência. DPA contratual é o mecanismo legal standard para GDPR Art. 46 transferência internacional. Questão para @po (QA1) confirmar se DPA contratual é suficiente para mercado PT.

**Trace:** NFR11 (data residency UE); NFR10 (GDPR compliance); CON4 (mercado PT exclusivo); GDPR Art. 46 (transferências para países terceiros via DPA).

### [AUTO-DECISION D57] Modo degradado Upstash no benchmark — cache miss always

**Decisão:** Benchmark E2E funciona com ou sem Upstash disponível. Se `UPSTASH_REDIS_REST_URL` ausente, `UpstashCache` retorna `null` sempre (comportamento existente Story 2.9 AC1 — sem nova lógica). Cache hit rate no benchmark será 0% neste modo — regista correctamente no JSON output.

**Justificação:** Benchmark deve ser executável em staging antes de Upstash estar provisionado (EB3 pending). Correctness do pipeline não depende de cache (cache é optimização). Story 2.9 já implementou modo degradado.

**Trace:** Story 2.9 AC1 (UpstashCache modo degradado); Architecture §4.6 "HIT (5min TTL)".

**Decisão:** Criada config Vitest dedicada (`packages/db-test/vitest.benchmark.config.ts`) com `include: ['src/benchmark/__tests__/**/*.test.ts']` e sem `globalSetup` (que arranca Testcontainers). Script `test:benchmark` invoca esta config.

**Razão:** A config principal (`vitest.config.ts`) precisa de Postgres 16 ephemeral para os RLS tests. Quando Docker está offline, todos os testes do package falham incluindo os benchmark (mockable). Config separada permite gate parcial (`test:benchmark` PASS) mesmo em ambiente local sem Docker. CI continua a usar a config principal (gate rls-gate cobre).

**Alternativa rejeitada:** Mover benchmark tests para outro package. Rejeitado por overhead de novo package.json/tsconfig sem benefício prático.

**Confidence:** high. **Scope-risk:** narrow.

### [DEV-DECISION D64] `forceP95Ms` aplica aos últimos 10% — garante captura p95

**Decisão:** Em `runBenchmarkMock`, quando `opts.forceP95Ms` é fornecido, a latência alta é aplicada aos últimos 10% de fixtures (IDs ≥ 181 em 200) em vez de só ao último. Calculado via `Math.floor(fixtures.length * 0.9) + 1 = 181`.

**Razão:** Cálculo de percentile com `sorted.length=200` e `pct=95`: `Math.floor((95/100) * 200) = 190` → `sorted[190]`. Aplicar a latência alta só ao último fixture (ID=200) ficaria em sorted[199] e o p95 ainda seria baseLatency. Aplicar a ≥10% garante captura no idx 190.

**Alternativa rejeitada:** Aplicar a 1 fixture e ajustar fórmula percentile para usar idx 199. Rejeitado por divergir de práticas standard de percentil (numpy/scipy floor convention).

**Confidence:** high. **Scope-risk:** narrow.

### [DEV-DECISION D65] 3 fixtures ajustados de 0.7 para <0.7 — validar caminho FR4

**Decisão:** Fixtures #38 (`'regista tarefa: passar a noite no quarto da minha filha'`), #164 (`'anula isso'`), #175 (`'Netflix 13,99 por mês e Spotify 6,99 por mês'`) tinham `expected_confidence_min = 0.7`. Ajustados para 0.65, 0.6, 0.65 respectivamente — explicitamente abaixo do threshold `CLASSIFIER_CONFIDENCE_THRESHOLD = 0.7`.

**Razão:** AC9 da Story 2.10 exige "≥3 fixtures com `expected_confidence_min < 0.70`" para validar preview-then-confirm (FR4). Teste `preview_blocked_count corresponde a fixtures com expected_confidence_min < 0.70` falhava com `expectedPreviewCount=0` porque os 3 fixtures ambíguos estavam exactamente em 0.7 (strict less-than `<` exclui).

**Alternativa rejeitada:** Mudar threshold do test para `<=0.70`. Rejeitado por divergir do contrato real do classifier (`needs_confirmation: minConfidence < CLASSIFIER_CONFIDENCE_THRESHOLD` — strict less-than).

**Confidence:** high. **Scope-risk:** narrow.
