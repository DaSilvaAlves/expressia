## User Story

**As a** developer da equipa Expressia,
**I want** uma suite de testes automatizados que prove que o RLS Postgres bloqueia efectivamente o acesso cross-household em todas as tabelas,
**so that** a NFR5 está garantida mecanicamente e qualquer regressão de segurança multi-tenant é detectada automaticamente antes de merge.

## Acceptance Criteria

- [ ] **AC1:** Package `packages/db-test/` existe com harness de testes RLS usando Vitest + Testcontainers (Postgres 16 efémero).
- [ ] **AC2:** Para cada tabela com `household_id` (mínimo 22 tabelas de domínio), existem testes que validam:
  - (a) User A do household A não consegue fazer SELECT de dados do household B.
  - (b) User A não consegue fazer INSERT com `household_id` do household B.
  - (c) User A consegue fazer SELECT dos seus próprios dados.
- [ ] **AC3:** Os testes usam utilizadores Postgres reais (não service role) para que o RLS seja activado — o harness faz `SET ROLE authenticated; SET request.jwt.claims TO '...'` antes de cada query.
- [ ] **AC4:** O harness `rls-harness.ts` exporta helpers reutilizáveis: `seedTwoHouseholds()`, `asUser(userId, async (db) => {...})`.
- [ ] **AC5:** `pnpm test` em `packages/db-test/` corre todos os testes e passa (exit 0) contra Postgres 16 efémero via Testcontainers.
- [ ] **AC6:** Os testes RLS estão integrados no job `rls-gate` do CI (Story 1.2) — o job falha se qualquer teste RLS falhar.
- [ ] **AC7:** Tempo total dos testes RLS < 60 segundos em máquina CI (GitHub Actions ubuntu-latest).
- [ ] **AC8:** Testes cobrem as tabelas de maior risco: `tasks`, `transactions`, `accounts`, `cards`, `agent_runs`, `audit_log`.
- [ ] **AC9:** `pnpm typecheck` em `packages/db-test/` passa sem erros (TypeScript strict).
- [ ] **AC10:** Testes documentam o padrão de teste RLS em comentários para que futuros developers saibam como adicionar testes para novas tabelas.

## Dev Notes

### Padrão de teste RLS (Architecture §10.2)

```typescript
// packages/db-test/src/tests/tasks.rls.test.ts

import { describe, test, expect, beforeAll, afterAll } from 'vitest'
import { startTestPostgres, stopTestPostgres, seedTwoHouseholds, asUser } from '../rls-harness'

describe('RLS isolation: tasks', () => {
  let container: TestPg

  beforeAll(async () => {
    container = await startTestPostgres()
    await applyMigrations(container)
  })

  afterAll(async () => {
    await stopTestPostgres(container)
  })

  test('cross-household SELECT bloqueado', async () => {
    const { householdA, userA, householdB, userB } = await seedTwoHouseholds(container)

    // userA cria uma tarefa no householdA
    await asUser(userA, householdA, container, async (db) => {
      await db.insert(tasks).values({
        household_id: householdA.id,
        title: 'Tarefa A',
        created_by: userA.id,
      })
    })

    // userB tenta ver as tarefas do householdA — deve ver 0 rows (RLS)
    await asUser(userB, householdB, container, async (db) => {
      const rows = await db.select().from(tasks)
        .where(eq(tasks.household_id, householdA.id))
      expect(rows).toHaveLength(0) // (1) cross-household SELECT bloqueado
    })
  })

  test('cross-household INSERT bloqueado', async () => {
    const { householdA, userB, householdB } = await seedTwoHouseholds(container)

    // userB tenta criar tarefa com household_id do householdA — deve falhar
    await asUser(userB, householdB, container, async (db) => {
      await expect(
        db.insert(tasks).values({
          household_id: householdA.id, // CROSS-HOUSEHOLD!
          title: 'Tarefa inválida',
          created_by: userB.id,
        })
      ).rejects.toThrow(/row-level security/i)  // (2) RLS rejeita
    })
  })
})
```

### JWT claims simulados para RLS (Architecture §3.2, §5.1)

O RLS usa `current_setting('request.jwt.claims')::json->>'household_id'`. Para simular:

```typescript
// No helper asUser:
async function asUser(userId: string, householdId: string, container: TestPg, fn: (db: Drizzle) => Promise<void>) {
  const client = await container.connect()
  await client.query(`SET role authenticated`)
  await client.query(`
    SELECT set_config(
      'request.jwt.claims',
      '{"sub": "${userId}", "household_id": "${householdId}"}',
      true
    )
  `)
  const db = drizzle(client, { schema })
  await fn(db)
  await client.release()
}
```

### Testcontainers com Postgres 16 (Architecture §10.1)

```typescript
import { PostgreSqlContainer } from '@testcontainers/postgresql'

async function startTestPostgres() {
  const container = await new PostgreSqlContainer('postgres:16')
    .withDatabase('testdb')
    .withUsername('test')
    .withPassword('test')
    .start()
  return container
}
```
