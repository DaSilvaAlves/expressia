## User Story

**Como** utilizador autenticado da Expressia,
**Quero** alternar entre modo claro, escuro e automático (sistema) através de um toggle em `/conta/preferências` ao lado do `always_preview`,
**Para que** a aplicação respeite a minha preferência visual consistentemente em todos os dispositivos — sem flash de tema errado ao carregar a página, e sem ter de redefinir o tema cada vez que inicio sessão.

### AC1 — Tailwind `darkMode: 'class'` + anti-FOUC

**AC1:** `tailwind.config.ts` passa a ter `darkMode: 'class'` (ou `['class']`). O `globals.css` migra de `@media (prefers-color-scheme: dark) { :root { ... } }` para `.dark { ... }`. Um script anti-FOUC (inline no `<head>` antes de qualquer CSS) lê o cookie `expressia-theme` e aplica a classe `dark` ou remove-a do elemento `<html>` antes do primeiro paint, eliminando o flash de tema errado.

**(a)** `tailwind.config.ts` — `darkMode: 'class'` adicionado (`apps/web/tailwind.config.ts`).
**(b)** `globals.css` — seletor `.dark { ... }` substitui `@media (prefers-color-scheme: dark) { :root { ... } }`. CSS vars de tema ficam em `:root` (claro) e `.dark` (escuro), alinhadas com tokens hex do front-end-spec §3.1-3.2 (DP-5.8.C).
**(c)** Script anti-FOUC: `<script>` inline em `apps/web/src/app/layout.tsx` (RootLayout, antes de `<body>`) que lê `document.cookie` ou equivalente server-side e adiciona a classe `dark` no `<html>` se `theme === 'dark'` ou (`theme === 'system'` e `window.matchMedia('(prefers-color-scheme: dark)').matches`). Script minificado, sem dependências externas. Deve estar no HTML antes de qualquer folha de estilos.
**(d)** `ThemeProvider` Client Component (em `apps/web/src/app/(app)/layout.tsx` ou `apps/web/src/components/ThemeProvider.tsx`) que sincroniza a classe `dark` no `<html>` após hidratação e quando o utilizador muda de tema via toggle. Expõe um hook `useTheme()` para o toggle UI.

**AC1 PASS:** Teste: script anti-FOUC presente no HTML (grep/snapshot ao HTML renderizado); `tailwind.config.ts` tem `darkMode: 'class'`; `globals.css` não tem `@media (prefers-color-scheme: dark)` (substituído por `.dark`); `ThemeProvider` renderiza sem erros em jsdom; snapshot do `<html class="dark">` quando tema = 'dark'.

### AC5 — Reconciliação dos tokens (DP-5.8.C) — FUP-5.3.B resolvido

**AC5:** Os dois sistemas de tokens conflituantes são reconciliados numa fonte única: `packages/ui/src/tokens.ts` hex (front-end-spec §3.1-3.2) é a fonte canónica; `globals.css` passa a usar CSS vars mapeadas aos hex exactos desse ficheiro.

**(a)** `globals.css` `:root { ... }` usa hex exactos do `colorsLight` de `packages/ui/src/tokens.ts`: `--bg-canvas: #FAFAF7`, `--bg-surface: #FFFFFF`, `--text-primary: #1A1A1A`, etc. **[PO-FIX-2 v1.1 — mapeamento Tailwind obrigatório:** as CSS vars actuais de `globals.css` chamam-se `--background`/`--foreground`/`--muted`/`--muted-foreground` (HSL genérico), e as classes `text-muted-foreground`/`bg-background`/`text-foreground` são usadas em 10+ ficheiros (`page.tsx`, `error.tsx`, `ChatPanel.tsx`, `prefs-toggle.tsx`, etc.) — mas o `tailwind.config.ts` actual **NÃO tem bloco `theme.extend.colors`** que mapeie esses nomes. A reconciliação (DP-5.8.C) tem de cobrir **três** superfícies coerentemente, não só `globals.css`: (1) CSS vars em `:root`/`.dark`; (2) o bloco `colors` em `tailwind.config.ts` que mapeia os nomes semânticos às vars; (3) os componentes que referenciam as classes. O @dev decide se mantém os nomes legacy (`--background`/`--foreground`/`--muted`) como aliases das vars semânticas (menos sweep) ou se renomeia tudo para os nomes do front-end-spec (`--bg-canvas`/`--text-primary` — mais sweep). Sem decidir o mapeamento Tailwind, as classes existentes ficam silenciosamente sem estilo após a migração.]**

## Dev Notes

**Núcleo arquitectural: mudar de `media` para `class` (DP-5.8.A).** Actualmente `tailwind.config.ts` não tem a key `darkMode` (default `'media'` — dark mode via `@media (prefers-color-scheme: dark)` no `globals.css`). Para toggle manual por utilizador, Tailwind precisa de `darkMode: 'class'` — as classes `dark:` só são emitidas quando existe a classe `dark` no elemento ancestral (tipicamente `<html>`). A mudança em si é de 1 linha no config, mas exige a cadeia completa: script anti-FOUC no `<head>` + `ThemeProvider` Client + cookie `expressia-theme`.

**Script anti-FOUC (padrão canónico).** O script deve ser `dangerouslySetInnerHTML` num `<script>` em `apps/web/src/app/layout.tsx` (RootLayout), **antes do `<body>`**. Lê o cookie `expressia-theme` via `document.cookie` (ou o valor emitido server-side pelo RootLayout), e aplica `document.documentElement.classList.toggle('dark', isDark)` antes de qualquer folha de estilos ser aplicada. Isto elimina o FOUC. O script corre no navegador **antes** da hidratação React — é o único ponto onde podemos garantir zero flash. Ver front-end-spec §11.3: "SSR detecta cookie de preferência antes de hydratação".

**`ThemeProvider` e `useTheme()`.** O `ThemeProvider` é um Client Component que:
1. Recebe `theme` inicial (do RSC via prop).
2. Aplica a classe `dark` no `document.documentElement` via `useEffect` (client-only — lição do FIX-1 da Story 5.7: **nunca mutar o DOM no corpo do render Client Component**).
3. Expõe `useTheme()` com `{ theme, setTheme }`. O `setTheme` actualiza a classe imediatamente (optimistic), depois o `<ThemeToggle>` faz o PATCH + escreve o cookie.
4. Para `'system'`: adiciona listener `window.matchMedia('(prefers-color-scheme: dark)')` com cleanup no unmount.

**Cookie `expressia-theme` — detalhes.** O cookie é `SameSite=Lax; Path=/; Max-Age=31536000` (1 ano). Não contém dados sensíveis — apenas `'light'`, `'dark'`, ou `'system'`. O servidor pode lê-lo nos headers para sincronização server-side no RootLayout SSR. No PATCH success, o cliente escreve `document.cookie = 'expressia-theme=dark; ...'` — simples, sem biblioteca de cookies.

**GET estendido (AC4).** O SELECT em `route.ts` actualmente retorna `always_preview, widgets_enabled` (linha 134). Adicionar `theme` ao SELECT e ao JSON de resposta. Usar `ThemeSchema.safeParse(rows[0]?.theme).data ?? 'system'` como fallback (análogo ao `WidgetsEnabledSchema.safeParse` da Story 5.7). O comentário do JsDoc do GET (linha 77-82) está desactualizado (diz `{ always_preview: boolean }` — é residual da Story 2.7 pre-5.7); actualizar ao estender.

**Reconciliação tokens (DP-5.8.C).** Os valores HSL actuais em `globals.css` (`--background: 222 47% 6%`) foram introduzidos pela Story 5.3 como placeholder. Não correspondem aos hex do front-end-spec (§3.2, `#0F1311` para `bg-canvas` dark). A migração é: para cada CSS var em `:root`/`.dark`, substituir o valor HSL pelo hex exacto do `packages/ui/src/tokens.ts`. Nomes das vars a manter ou renomear (ex.: `--bg-canvas` vs `--background`) — decisão ao @dev, alinhada com o que os componentes existentes já usam.

**Audit-script (AC3).** Padrões problemáticos a detectar (não exaustiva — script configurável):
- `bg-white` sem `dark:bg-*` na mesma linha/elemento
- `text-black` sem `dark:text-*`
- `bg-gray-\d+` / `bg-zinc-\d+` / `bg-slate-\d+` sem `dark:` counterpart
- `text-gray-\d+` sem `dark:text-*`
Falsos positivos válidos: classes com opacidade (ex.: `bg-white/5`) geralmente são semi-transparentes e podem ser deixados; whitelist configurável.

**REQ-INLINE-1 (precedente 5.5/5.6/5.7).** Em apps/web, `ThemeSchema` é importado de `@/lib/api-schemas/preferences` (local) — NÃO de `@meu-jarvis/db`. O `ThemeProvider` e o `<ThemeToggle>` não importam valores de `@meu-jarvis/db`.

**Precedente anti-FOUC da Story 5.7.** O FIX-1 da Story 5.7 ensinou que hidratar stores Zustand deve ser em `useEffect` (nunca no corpo do render de Client Components — correm no SSR). O mesmo princípio aplica-se ao `ThemeProvider`: a aplicação da classe `dark` no DOM deve estar num `useEffect`. O script anti-FOUC no `<head>` é a única excepção — e não é React, é JavaScript puro.

**Impacto em testes existentes.** A mudança de `'media'` para `'class'` pode afectar testes que fazem snapshot de classes CSS ou que dependem do comportamento `@media`. Verificar `page.test.tsx` (visao) e outros snapshots — possivelmente necessário actualizar.
