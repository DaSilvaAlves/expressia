## Story

**Como** sistema Expressia (job Inngest cron),
**quero** detectar diariamente as subscrições em trial cuja `trial_ends_at` expirou e regredir o household a Free,
**para que** utilizadores que não fizeram upgrade deixem de ter acesso ao tier `familia` após os 14 dias de trial, sem intervenção manual (FR33 / Epic 6 AC2).

## Análise da Dependência 6.3 / Billing-Freeze

**Veredicto: EXECUTÁVEL SEM STRIPE. Dependência 6.3 é de sequenciação narrativa, NÃO técnica.**

Evidência recolhida do código real:

1. `packages/db/src/schema/billing.ts:79` — `stripeCustomerId` é explicitamente nullable:
   ```
   /** Pode ser null em trials internos sem Stripe customer (FR33). */
   stripeCustomerId: text('stripe_customer_id'),
   ```
   Trials internos criados pelo trigger `handle_new_user` (migration `0003_auth_user_trigger.sql`) NÃO têm Stripe customer.

2. `packages/db/migrations/0003_auth_user_trigger.sql:92-96` — o trigger insere a subscription:
   ```sql
   insert into public.subscriptions (
     household_id, plan, status, currency,
     trial_ends_at, current_period_start, current_period_end
   ) values (
     new_household_id, 'familia', 'trialing', 'EUR',
     now() + interval '14 days', now(), now() + interval '14 days'
   );
   ```
   Campos `stripe_customer_id` e `stripe_subscription_id` ficam NULL — estado `trialing` existe na DB sem qualquer SDK Stripe.

3. A Story 6.4 só precisa de:
   - SELECT em `subscriptions` (leitura simples, zero Stripe)
   - UPDATE `subscriptions.status` → `'canceled'` (muda para `'canceled'` que representa "regredido a Free por expiração")
   - UPDATE `subscriptions.plan` → `'free'` (coluna Drizzle confirmada)
   - UPDATE `households.plan` → `'free'` (coluna `plan planTierEnum` em `tenancy.ts:57` confirmada; denormalizado para fast-path RLS/quotas)
   - Ambos os UPDATEs no mesmo commit da transação (padrão R-6.8 do epic)

4. Zero chamadas à API Stripe, zero webhook, zero checkout, zero `STRIPE_SECRET_KEY` necessária.

**Conclusão:** A dependência de 6.3 no epic era de ordem narrativa (faz sentido que 6.3 esteja done antes de 6.4 em produção para que os trials se convertam em subscriptions pagas). Tecnicamente, o job opera sobre dados DB-only. **Story 6.4 pode ser implementada e testada sem violar o billing-freeze.**

**Nota sobre estado após expiração:** O job define `subscriptions.status = 'canceled'` (valor do enum `subscription_status` existente em `billing.ts:35-44`; outros candidatos: `'incomplete_expired'` — mas `'canceled'` é semanticamente correcto para "trial terminou sem upgrade"). O `plan` em ambas as tabelas regride para `'free'`.

## Acceptance Criteria

**AC1 — Query correcta:** O job lê `subscriptions WHERE status = 'trialing' AND trial_ends_at <= now()` via `getServiceDb()`. Apenas rows com trial expirado (inclusive o momento exacto) são processadas.

**AC2 — Regressão atómica:** Para cada subscription expirada, o job executa num único commit:
- `UPDATE subscriptions SET status = 'canceled', plan = 'free', updated_at = now() WHERE id = $id`
- `UPDATE households SET plan = 'free', updated_at = now() WHERE id = $household_id`

Ambos os writes numa transação — zero divergência entre `subscriptions.plan` e `households.plan` (padrão R-6.8 epic).

**AC3 — Idempotência:** Executar o cron mais de uma vez no mesmo dia é inofensivo. Uma subscription já com `status != 'trialing'` (ex.: `'canceled'`, `'active'`) não é tocada. Um household já com `plan = 'free'` recebe um UPDATE no-op (SET plan = 'free' quando já é 'free') — aceitável; idempotência garantida pela query WHERE.

**AC4 — Registo Inngest:** A função é registada em `apps/web/src/app/api/inngest/route.ts` com `id: 'expire-trials'` e trigger `{ cron: '0 3 * * *' }` (03:00 UTC diário, alinhado com os outros jobs cron do projecto).

**AC5 — OTel / Observability:** A função envolve o trabalho em `withSpan('billing.trials.expire', {}, ...)` e emite `span.setAttribute` com contadores sem PII:
- `billing.trials.expired_count` — número de trials regredidos
- `billing.trials.skipped_count` — trials processados mas já inactivos (no-op)
- `billing.cron.duration_ms` — duração total da execução
- `inngest.run_id` — `runId` da invocação Inngest

**AC6 — Log estruturado:** `childLogger({ job: 'expire-trials' })` emite:
- `info` por trial processado com `{ subscription_id, household_id, trial_ends_at }` (sem PII de utilizador)
- `info` de sumário agregado no final
- `error` em caso de excepção, com `captureException` para Sentry

**AC7 — Teste unitário mockável:** Suite Vitest em `apps/web/src/lib/inngest/functions/__tests__/expire-trials.test.ts` com padrão `vi.hoisted` (idêntico a `generate-finance-recurrences.test.ts`), cobrindo:
- (1) zero trials expirados → zero updates
- (2) 1 trial expirado → 2 UPDATEs (subscriptions + households) + sumário correcto
- (3) múltiplos trials → todos processados sequencialmente
- (4) trial `status != 'trialing'` → não processado (idempotência)
- (5) erro DB → exception re-thrown para Inngest retry
- (6) span attrs correcto e sem PII

**AC8 — Sem migração nova:** O job não cria tabelas. Opera sobre `subscriptions` e `households` que existem (schema `0000_initial_schema.sql` + RLS `0001_rls_policies.sql`). `pnpm check:rls` não é afectado.

> **[PO-MUST-FIX-1 — Audit action (T1.7) usa valor de enum EXISTENTE]** `audit_log.action` é o ENUM tipado `audit_action` (`packages/db/src/schema/audit.ts:28-112`). NÃO existe valor `trials_expired` / `trial_expired`. **Não criar novo valor de enum** — isso exigiria `ALTER TYPE audit_action ADD VALUE` (migração nova), violando o AC8 e o "zero migrações". O `gdpr-purge.ts:135` prova o padrão: usa `'account_deletion_executed'::audit_action` (valor existente, com cast explícito). **Decisão:** o INSERT agregado de auditoria do T1.7 usa `action = 'plan_changed'::audit_action` (valor de Billing já presente, linha 36) com `entity_table = 'subscriptions'`, `household_id = null`, `after_state` com `{ expired_count, run_id, duration_ms }`. Em alternativa, **omitir T1.7 por completo** (o audit log agregado é explicitamente não-fatal/opcional). Não deixar o `action` por especificar — um valor inválido parte o INSERT em runtime (constraint do enum).

### Padrão de Implementação

Espelhar `generate-finance-recurrences.ts` em estrutura e observability. Diferenças chave:

1. **Sem helpers externos** — não há equivalente a `calcNextRunDate`; a lógica é apenas SELECT + 2 UPDATEs.
2. **Transação por subscription vs batch:** O job de finance-recurrences itera por recorrência individualmente. Para expire-trials, o padrão mais simples é um único `step.run('expire-trialing-subscriptions', async () => { ... })` que itera todas as subscriptions expiradas — batch atómico. Se alguma falhar, Inngest retenta o step inteiro (idempotência garantida pelo WHERE `status = 'trialing'`).
3. **Dois UPDATEs por subscription**, não um INSERT. Usar `db.execute(sql\`...\`)` raw como nos outros jobs.

> **[PO-MUST-FIX-2 — Atomicidade real dos 2 UPDATEs (AC2 / R-6.4.2)]** O modelo `generate-finance-recurrences.ts` corre `db.execute()` em sequência — cada statement **auto-commita independentemente** (não há `BEGIN/COMMIT` à volta). Copiar esse padrão à letra NÃO satisfaz a promessa do AC2 ("ambos os writes no mesmo commit") nem mitiga R-6.4.2 (divergência `subscriptions.plan` ↔ `households.plan`). O `DbShim` expõe `transaction()` (`db-shim.ts:31`). **Decisão:** envolver os 2 UPDATEs de cada subscription num `db.transaction(async (tx) => { await tx.execute(updateSubscription); await tx.execute(updateHousehold); })` para garantir atomicidade por-subscription (se o UPDATE de households falhar, o de subscriptions faz rollback). O `step.run` continua a iterar o batch; a transacção é **por-subscription**, não por-batch (evita que 1 falha tardia reverta os trials já correctamente expirados). Actualizar AC2/Dev Notes/T1.3 em conformidade. **Nota:** os mocks de teste (`vi.mock('@/lib/agent/db-shim')`) terão de expor `transaction: vi.fn(async (cb) => cb({ execute: executeMock }))` além de `execute` — o mock actual de `generate-finance-recurrences.test.ts` só mocka `execute`.

### Query canónica (orientação, não código normativo)

```sql
-- SELECT subscriptions expiradas
SELECT id, household_id, trial_ends_at
FROM subscriptions
WHERE status = 'trialing'
  AND trial_ends_at <= now()
ORDER BY id;

-- UPDATE subscriptions (por id)
UPDATE subscriptions
SET status = 'canceled', plan = 'free', updated_at = now()
WHERE id = $id AND status = 'trialing';

-- UPDATE households (denormalização)
UPDATE households
SET plan = 'free', updated_at = now()
WHERE id = $household_id;
```

A condição `AND status = 'trialing'` no UPDATE é uma segunda linha de defesa de idempotência (UPDATE no-op se já foi processado por outro worker em corrida).

### Implementação dos PO-MUST-FIX

**PO-MUST-FIX-1 — Audit action usa enum EXISTENTE.** Confirmado por leitura directa de `packages/db/src/schema/audit.ts:28-112`: o enum `audit_action` contém `'plan_changed'` (linha 36, categoria Billing) e NÃO contém `'trials_expired'` nem `'trial_expired'`. O INSERT de auditoria agregado em `expire-trials.ts` usa `'plan_changed'::audit_action` com `entity_table = 'subscriptions'`, `household_id = null`, `after_state` com `{ expired_count, skipped_count, duration_ms, run_id }`. O cast `::audit_action` espelha byte-a-byte o padrão de `gdpr-purge.ts:135` (`'account_deletion_executed'::audit_action`). Teste TC7 prova que a query INSERT contém `'plan_changed'::audit_action` e NÃO contém `trials_expired`/`trial_expired`. Zero migrações → AC8 satisfeito.

**PO-MUST-FIX-2 — Atomicidade real dos 2 UPDATEs.** Confirmado que `DbShim.transaction()` está exposto em `apps/web/src/lib/agent/db-shim.ts:31` com assinatura `transaction<T>(fn: (tx: DbShim) => Promise<T>): Promise<T>`. Cada subscription expirada corre os 2 UPDATEs dentro de `db.transaction(async (tx) => { ... })` — atomicidade POR-SUBSCRIPTION (não por-batch): se o UPDATE de `households` falhar, o UPDATE de `subscriptions` faz rollback (zero divergência `subscriptions.plan` ↔ `households.plan`, mitiga R-6.4.2). A iteração do batch continua dentro do único `step.run` para que uma falha tardia não reverta os trials já correctamente expirados. Decisão de implementação adicional: se o UPDATE de subscriptions (com a segunda linha de defesa `AND status = 'trialing'`) tocar 0 rows (race com upgrade concorrente), salta-se o UPDATE de households e conta-se como `skipped` — evita um write no-op desnecessário e a divergência. O mock de teste expõe `transaction:` (callback recebe `tx` cujo `.execute` partilha o `executeMock`) além de `execute:`, conforme exigido pelo @po.

### Decisões autónomas (YOLO)

- **[DEV-DECISION] D-6.4.1 — `skipped_count` semântica.** A story (AC5) prevê `billing.trials.skipped_count` = "trials processados mas já inactivos (no-op)". Como a SELECT já filtra `status = 'trialing'`, um trial "já inactivo" só aparece numa race condition (outro worker mudou o status entre a SELECT e o UPDATE). O `skipped_count` incrementa exactamente nesse caso (UPDATE subscriptions tocou 0 rows). Semântica coerente com a definição da story e com R-6.4.1.
- **[DEV-DECISION] D-6.4.2 — UPDATE households fora da condição de status.** O UPDATE de `households` não tem `WHERE plan = 'free'`-guard porque é idempotente por natureza (SET plan='free' quando já é 'free' = no-op aceitável, AC3) e só corre se o UPDATE de subscriptions confirmou a transição (`subUpdate.length > 0`). Evita uma leitura extra do plan actual do household.
- **[DEV-DECISION] D-6.4.3 — Sem helper de data.** Ao contrário de `generate-finance-recurrences.ts` (que usa `calcNextRunDate`), `expire-trials` delega o filtro temporal ao Postgres (`trial_ends_at <= now()`). Zero aritmética de calendário em JS → zero risco de drift de timezone entre o processo Node e a DB.
