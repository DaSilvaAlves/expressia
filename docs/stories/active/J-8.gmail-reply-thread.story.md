## Story

**As a** Eurico (único utilizador do Jarvis),
**I want** pedir ao Jarvis, via Telegram, que responda a um email que já recebi — mantendo a resposta na mesma thread do Gmail (não como email novo solto) — sempre com pré-visualização e confirmação antes do envio,
**so that** consigo despachar correspondência que já li sem abrir o Gmail, com a conversa a ficar coerente para quem a recebe (o destinatário vê a resposta agrupada com o email original, não um email novo desligado de contexto).

## Contexto de âmbito

**Esta story entrega:**
- Novo intent `responder_email` no motor: responder a um email já existente, mantendo a thread do Gmail (`threadId` + cabeçalhos `In-Reply-To`/`References`).
- Nova tool `responder_email` (Gmail API `users.messages.send` com `threadId` + cabeçalhos de thread — mesma família de escrita externa irreversível que `enviar_email`, J-7).
- O mecanismo de **resolução do email-alvo** ("responde ao Pedro", "responde a esse email da Ana sobre o jantar") — o problema de design central desta story, deliberadamente descoped da J-7 (ver Change Log da J-7, v1.1-draft e Contexto de âmbito). Ver secção "Problema de design" abaixo.
- Reutiliza integralmente a base J-6/J-7: OAuth (SEM novo scope — ver Restrições), `gmail-api.ts`, `toolRegistry`, guard multi-intent, padrão SEND-PREVIEW-1 (rascunho real no preview) e binding preview==envio, `irreversible.ts` (undo honesto).

**NÃO inclui (fora de âmbito):**
- Anexos, HTML rico, CC/BCC múltiplos (mantém-se texto simples, 1 destinatário — mesmo âmbito da J-7).
- Encaminhar (forward) um email — só responder (reply) ao remetente original.
- Responder a >1 destinatário de uma thread com múltiplos participantes (reply-all) — v1 = reply simples ao remetente.
- Composição de email novo — já entregue pela J-7 (`enviar_email`); esta story só acrescenta a variante "responder".
- Desambiguação multi-turn (o Jarvis pedir "qual dos 3 emails do Pedro?" e esperar resposta numa segunda mensagem) — o pipeline actual é single-shot por prompt (ver Problema de design); a desambiguação é feita pelo LLM do Planner a partir de uma shortlist, não por diálogo.
- Cache de emails em DB (mantém-se a política J-6/J-7: email nunca persistido).

**Depende de:** J-6 Done (Gmail readonly: `gmail-api.ts`, `getGmailAccessToken`, `ToolDomain 'email'`) + J-7 Done (Gmail send: `enviar_email`, `EXTERNAL_WRITE_INTENTS`, `PREVIEW_RENDER_INTENTS`, `REUSE_PERSISTED_PLAN_INTENTS`, `irreversible.ts`/`IRREVERSIBLE_WRITE_TOOLS` — todos os 4 conjuntos generalizados/criados na J-7 precisamente a pensar numa v2 de reply, ver J-7 Dev Notes + Change Log v1.2-DEV).

### Problema de design — resolução do email-alvo vs. `preview()` puro

Verificado no draft desta story (fonte: `packages/tools/src/contracts.ts:342`): `preview(input: I, ctx: ToolExecutionContext): string` é **síncrono e sem I/O** — não pode chamar a Gmail API para descobrir "qual é o último email do Pedro" no momento em que o preview é renderizado. Este foi exactamente o bloqueio que descopou o reply da J-7 (ver `docs/stories/completed/J-7...story.md`, Change Log v1.1-draft).

**O que mudou desde a J-7 que torna isto resolúvel agora:** a correcção **SEND-PREVIEW-1** (J-7, v1.2-DEV) introduziu o padrão de **correr o Planner no branch de preview** (`renderExternalWritePreview` em `apps/web/src/lib/agent/run-agent.ts:793-859`) para intents com rascunho real (hoje só `enviar_email`, via `PREVIEW_RENDER_INTENTS`). O Planner corre UMA VEZ (não duas — o plano é persistido e reutilizado no confirm, `REUSE_PERSISTED_PLAN_INTENTS` em `confirm/route.ts:99`). Isto muda o terreno: já existe um ponto no pipeline, ANTES do preview ser mostrado, onde código da aplicação (não uma tool) pode preparar contexto adicional para o Planner — exactamente como `buildAccountContext` (`run-agent.ts:742-776`) já faz para contas/cartões (Story 2.13 AC6): resolve entidades por SELECT/API FORA do LLM e injecta o resultado como **prefixo da user message** do Planner (nunca no `system`/`tools` — preserva o cache Anthropic/prompt caching), para o LLM escolher entre candidatos concretos em vez de inventar IDs.

**Mecanismo recomendado (Opção A — menor blast radius, reaproveita padrão provado):**

1. Quando o plano tem `responder_email` (o mesmo ponto onde `planHasPreviewRenderIntent` já detecta `enviar_email`), ANTES de chamar `planner.plan()`, correr um novo passo de resolução (paralelo a `buildAccountContext`, não dentro de `tool.preview()`/`tool.execute()`): pesquisar os **N emails mais recentes do inbox** (reaproveitando `GMAIL_API_ENDPOINT`, `extractListItems`, `isGmailListItem` de `gmail-api.ts` — mesmo padrão de `consultar_emails`/`list-emails.ts`, mas query `in:inbox` SEM o filtro `is:unread` — uma resposta pode ser a um email já lido) e ler os metadados de cada um (`Subject`, `From`, `Date`, **`Message-ID`** — novo cabeçalho a pedir, `list-emails.ts` hoje só pede `Subject`/`From`/`Date` — mais o `threadId`, que já vem no corpo da resposta `GET /messages/{id}` mas não é lido hoje por `isGmailMessageDetail`).
2. Construir uma shortlist limitada (ex.: 10 candidatos) `{ threadId, messageId (valor do cabeçalho Message-ID, formato `<...>`), from, subject, receivedAt }` — SEM conteúdo do email (só metadados, mesma política de privacidade da J-6/J-7).
3. Injectar a shortlist no `Planner.plan()` como um novo campo opcional de contexto (mesma forma que `accountContext` — `PlannerInputSchema` em `packages/planner-executor/src/schemas.ts:103-141`, serializado como prefixo da user message via uma função análoga a `serializeAccountContextForPlanner`, `planner.ts:474`), para o LLM escolher o candidato certo a partir da referência em linguagem natural do utilizador ("o Pedro", "aquele email da Ana sobre o jantar") e devolver `{ threadId, messageId, to, subject, body }` como input estruturado da tool `responder_email`.
4. `tool.preview()` e `tool.execute()` da `responder_email` permanecem **síncronos/puros no preview e sem resolução no execute** — recebem `threadId`/`messageId` JÁ concretos (escolhidos pelo Planner), exactamente como `enviar_email` recebe `to`/`subject`/`body` já concretos. **O contrato `ToolDefinition.preview` NÃO muda** (zero blast radius nas outras tools).
5. Zero-match (a shortlist não tem nada com o `to` resolvido, ou o Planner não conseguiu popular `threadId`) → falha honesta: `ToolExecutionError` PT-PT ("Não encontrei esse email para responder.") — NÃO adivinhar um `threadId` arbitrário.
6. Ambiguidade genuína (2+ candidatos plausíveis) é resolvida pelo PRÓPRIO Planner (LLM, chamada única) a partir do texto do utilizador + shortlist — não há loop de diálogo multi-turn no pipeline actual (confirmado: `planner-executor` não tem chaining output→input, ver J-7 Dev Notes). Isto é uma simplificação de âmbito v1 aceite nesta story — não um bug a corrigir aqui.

**Alternativas consideradas (para o @architect/@dev decidirem se Opção A não for suficiente na prática):**
- **Opção B — query Gmail dirigida por entidade** (ex.: extrair "Pedro" do `raw_span` do intent e construir `q=from:pedro`): mais precisa mas exige um passo extra de extracção de entidade (NLP) antes da pesquisa; maior complexidade sem benefício claro para uma caixa de correio de um único utilizador com volume moderado — **não recomendada para v1**, mas viável como refinamento futuro se a shortlist "top N recentes" se mostrar insuficiente em uso real.
- **Opção C — chaining output→input no `planner-executor`** (o Planner emite uma tool call de pesquisa que o Executor corre ANTES de gerar o plano final): resolveria o problema de forma mais geral (reutilizável para outras resoluções de entidade futuras), mas exige adicionar uma capacidade nova ao `planner-executor` (hoje inexistente, grep `previousResults|toolOutputs|chain` = 0) — investimento maior, fora de âmbito de uma story de reply. Não recomendada para v1; registar como possível débito arquitectural se Opção A limitar demasiado o produto.
- **Opção D — tornar `preview()` assíncrono/com I/O**: rejeitada (mesma conclusão da J-7) — mudaria o contrato `ToolDefinition` para TODAS as tools existentes.

O @architect/@dev têm liberdade para ajustar o desenho exacto (nomes de funções, exact shape do contexto) desde que preserve: (i) `preview()` continua puro/síncrono; (ii) a resolução ocorre FORA da tool, no pipeline (`run-agent.ts`), nunca dentro de `execute()`/`preview()`; (iii) falha honesta em zero-match; (iv) sem I/O extra no `execute()` além do já existente (`getGmailAccessToken` + `sendGmailMessage`).

### Threading Gmail — mecânica RFC 2822 + Gmail API

Um "reply" correcto no Gmail exige TRÊS coisas alinhadas (falhar uma produz um email novo desligado, não uma resposta agrupada):
1. Cabeçalho MIME `In-Reply-To: <Message-ID-do-original>` (o `Message-ID` é um cabeçalho RFC 2822 do email ORIGINAL — DIFERENTE do `id` da Gmail API, que é um hash interno da Gmail, não usável em cabeçalhos de threading).
2. Cabeçalho MIME `References: <Message-ID-do-original>` (numa cadeia de 1 salto, igual ao `In-Reply-To`; threads mais longas concatenariam vários `Message-ID` — fora de âmbito v1).
3. Campo `threadId` no corpo do `POST /messages/send` (`{ raw, threadId }`) — reforça a associação à mesma thread do lado do Gmail, independente dos cabeçalhos MIME.

`list-emails.ts` hoje pede `metadataHeaders: ['Subject', 'From', 'Date']` (linhas 159-162) — falta `Message-ID`. O `threadId` já vem no JSON de resposta de `GET /messages/{id}` (campo top-level, ao lado de `id`/`snippet`/`payload`), mas `isGmailMessageDetail` (`gmail-api.ts:112-127`) não o valida/expõe hoje — precisa de ser estendido (ou um novo type guard) para capturar `threadId` também.

### Decisões de implementação [DEV-DECISION]

- **[D-J8.1] Resolução do email-alvo em `renderExternalWritePreview` (Opção A da story).** A shortlist do inbox resolve-se ANTES do `Planner.plan()`, no MESMO ponto onde o SEND-PREVIEW-1 (J-7) já corre o Planner no branch de preview — não numa etapa separada. Justificação: `responder_email` força `needs_confirmation:true`, logo cai sempre no branch de preview; resolver ali (e persistir o plano) garante que o confirm reutiliza o rascunho (binding preview==envio, `REUSE_PERSISTED_PLAN_INTENTS`) sem re-resolver nem re-planear. O contrato `ToolDefinition.preview` (síncrono/puro) fica intacto — zero blast radius nas outras tools.

- **[D-J8.2] Zero-match honesto reconciliado com AC13 vs AC17d.** AC13 diz "a tool falha com `ToolExecutionError`"; AC17d exige que o utilizador VEJA a mensagem honesta "não encontrei". Como o `execute()` recebe `threadId`/`messageId` já concretos (Zod `.min(1)`), nunca é alcançável com valores vazios — e um `ToolExecutionError` lançado seria engolido pela mensagem neutra do catch do webhook ("Não consegui tratar disso agora."), NÃO a mensagem honesta. Decisão: o zero-match é detectado no passo de resolução (`resolveReplyCandidates` devolve `[]`) e o pipeline devolve uma resposta INFORMATIVA — reutilizando a forma `executed/kind:'direct_query'` (o webhook mostra só o `summary`, sem botões "(Cancelar)"/"sim/não"). Nem o Planner nem o Executor correm; nunca se inventa um `threadId` (o objectivo de segurança de AC13 é cumprido). A shortlist injecta na user message a instrução "Se NENHUM candidato corresponder, NÃO emitas responder_email" (defesa contra hallucination do LLM).

- **[D-J8.3] `sendGmailMessage` ganhou `toolName?` opcional (default `enviar_email`).** O `responder_email` reutiliza `sendGmailMessage`, mas os `ToolExecutionError` internos precisam do nome de tool correcto (`responder_email`) para observabilidade/audit. Alternativa rejeitada: re-embrulhar no `reply-email.execute` (produziria `ToolExecutionError` aninhado com `cause` a apontar para um `enviar_email` enganador).

- **[D-J8.4] `isGmailMessageDetail` estendido (threadId opcional) em vez de novo type guard.** A story permitia "estender OU novo type guard". Escolhi estender com `threadId?: string` validado (rejeita se presente e não-string) — backward-compatible com as leituras J-6 (`consultar_emails`) que não pedem threadId, sem duplicar lógica. O `Message-ID` é lido via `extractEmailHeader` (já existente).

- **[D-J8.5] Cast `db as unknown as DrizzleDbClient` em `renderExternalWritePreview`.** `getDb()` é passado como `ctx.db` a `resolveReplyCandidates` (que só usa `ctx.db.execute` para ler o token OAuth, RLS-scoped). É o mesmo padrão de runtime documentado em `contracts.ts` ("o caller passa o resultado de `getDb()` directamente"); o cast é necessário porque o tipo estrutural de `getDb()` não é nominalmente `DrizzleDbClient`. Não é `any` (cast explícito duplo, não supressão de tipos).
