## User Story

**As a** developer da equipa Expressia construindo o pipeline 3-stage do Cérebro AI,
**I want** um package `@meu-jarvis/tools` com a interface tipada `ToolDefinition<I,O>`, o `ToolRegistry` que regista e recupera tools, o helper `executeAtomic` que envolve múltiplas execuções de tools numa única transacção Drizzle com persistência de `ReverseOp` em `agent_reverse_ops`, e a taxonomia de erros específica de tools,
**so that** as Stories 2.5 (Planner+Executor) e 2.6 (endpoint agent/prompt) possam orquestrar tool calls multi-intent em transacção atómica com suporte a undo de 30s (FR6), sem duplicar lógica de atomicidade, sem fugas de PII para logs, e com OTel tracing consistente com o package `@meu-jarvis/agent` entregue em 2.2.

### AC2 — Contrato `ToolDefinition<I, O>` tipado com Zod

`packages/tools/src/contracts.ts` exporta:

```ts
export interface ToolDefinition<I, O> {
  readonly name: string;               // ex: 'create_task' — snake_case, lowercase
  readonly domain: ToolDomain;         // 'tasks' | 'finance' | 'query' | 'system'
  readonly description: string;        // PT-PT; descreve QUANDO usar (passado ao LLM)
  readonly inputSchema: z.ZodType<I>;  // valida args vindos do LLM
  readonly outputSchema: z.ZodType<O>; // valida resultado de execute()
  readonly requiredPlan?: PlanTier;    // 'pessoal' | 'familia' | 'pro' — optional; ausência = free+
  readonly estimatedTokens?: number;   // dica para cost router (Story 2.9)
  preview(input: I, ctx: ToolExecutionContext): string;                         // texto PT-PT para preview card (FR4)
  execute(input: I, ctx: ToolExecutionContext): Promise<O>;                     // execução dentro de transacção (ver executeAtomic)
  reverse(output: O, ctx: ToolExecutionContext): Promise<ReverseOpPayload>;     // produz reverse_op declarativo para undo (FR6)
}
```

Exporta também:
- `ToolDomain` — `z.enum(['tasks', 'finance', 'query', 'system'])`, tipo derivado `ToolDomain`.
- `PlanTier` — `z.enum(['free', 'pessoal', 'familia', 'pro'])`, tipo derivado `PlanTier`.
- `ToolExecutionContext` — interface com `{ householdId: string; userId: string; db: DrizzleDbClient; traceId: string; runId: string }`. `db` é o cliente Drizzle com RLS activa (retornado por `getDb()` de `@meu-jarvis/db`). `DrizzleDbClient` é o tipo inferido de `getDb()`.
- `ReverseOpPayload` — `z.discriminatedUnion('kind', [...])` com variantes:
  - `{ kind: 'delete_row'; table: string; id: string }` — insert revertido com delete
  - `{ kind: 'restore_row'; table: string; id: string; snapshot: Record<string, unknown> }` — update revertido com snapshot prévio
  - `{ kind: 'composite'; ops: ReverseOpPayload[] }` — lista de ops (máximo 10 — guard para prevenir payloads descontrolados)

Restrições:
- Nenhum campo de `ToolExecutionContext` inclui prompt text ou content de mensagens (PII).
- `ToolDefinition` é interface, não classe — a implementação concreta fica nas stories de domínio (2.6, 2.7).

### AC3 — `ToolRegistry` com API idempotente

`packages/tools/src/registry.ts` exporta `ToolRegistry` (class ou factory singleton):

```ts
class ToolRegistry {
  register<I, O>(tool: ToolDefinition<I, O>): void;            // lança DuplicateToolError se name+domain já registado
  get(name: string): ToolDefinition<unknown, unknown>;          // lança ToolNotFoundError se ausente
  list(): ToolDefinition<unknown, unknown>[];                   // todos os tools registados
  getByDomain(domain: ToolDomain): ToolDefinition<unknown, unknown>[];  // filtra por domain
  getAnthropicToolDefinitions(): AnthropicToolDefinition[];     // formato { name, description, input_schema: {} } para passar ao Sonnet
  clear(): void;                                                // apenas para uso em testes (marcado @internal)
}

export const toolRegistry: ToolRegistry;  // singleton exportado
```

- `register()` é **idempotente se chamado com exactamente o mesmo objecto** (identidade por referência); lança `DuplicateToolError` se `name` colidir com tool diferente já registado.
- `get()` tem complexidade **O(1)** via Map interno.
- `getAnthropicToolDefinitions()` serializa `inputSchema` para JSON Schema via `zodToJsonSchema` de `zod-to-json-schema` (a adicionar como dep); retorna array no formato esperado por Anthropic SDK.

### AC4 — `serializeReverseOp` / `deserializeReverseOp`

`packages/tools/src/contracts.ts` exporta:
- `serializeReverseOp(op: ReverseOpPayload): string` — JSON.stringify com validação Zod prévia. Lança `ToolValidationError` se payload inválido.
- `deserializeReverseOp(raw: string): ReverseOpPayload` — JSON.parse + validação Zod. Lança `ToolValidationError` se inválido ou não parseável.
- Round-trip garantido: `deserializeReverseOp(serializeReverseOp(op))` produz objecto deep-equal ao original para todas as variantes (`delete_row`, `restore_row`, `composite`).

### AC5 — `executeAtomic` — atomicidade transaccional Drizzle

`packages/tools/src/atomic.ts` exporta:

```ts
export async function executeAtomic(
  tools: Array<{ definition: ToolDefinition<unknown, unknown>; input: unknown }>,
  ctx: ToolExecutionContext,
): Promise<AtomicResult>
```

Comportamento obrigatório:
- Abre uma transacção Drizzle (`ctx.db.transaction(async (tx) => { ... })`).
- Dentro da transacção, executa **cada tool sequencialmente** (não em paralelo — evitar race conditions em estado partilhado Postgres na mesma transacção).
- Para cada tool: valida `input` contra `inputSchema` (Zod parse), executa `tool.execute(input, ctxWithTx)`, obtém `output`, chama `tool.reverse(output, ctxWithTx)` para obter `ReverseOpPayload`.
- Persiste cada `ReverseOpPayload` em `agent_reverse_ops` com `expires_at = now() + interval '30 seconds'`, `household_id = ctx.householdId`, `agent_run_id = ctx.runId` — tudo dentro da mesma transacção.
- **Se qualquer tool falhar** (input inválido, execute lança excepção, output não valida): a transacção faz **ROLLBACK completo** — nenhuma entidade é criada, nenhuma `agent_reverse_ops` row persiste.
- Retorna `AtomicResult`:
  ```ts
  interface AtomicResult {
    success: true;
    results: Array<{ toolName: string; output: unknown; reverseOpId: string }>;
  }
  // ou em caso de falha parcial (não lançada — apenas quando o rollback ocorreu de forma controlada):
  interface AtomicFailure {
    success: false;
    failedToolName: string;
    error: ToolError;
    rolledBack: true;
  }
  ```
- `ctxWithTx` é um `ToolExecutionContext` com `db` substituído pelo cliente da transacção Drizzle (para que as tools escrevam dentro da mesma transacção).

### AC6 — Persistência `agent_reverse_ops` com `expires_at = now() + 30s`

Dentro de `executeAtomic`, cada `ReverseOpPayload` é persistido na tabela `agent_reverse_ops` (schema real de `@meu-jarvis/db`, migration `0005`) com:
- `id`: `gen_random_uuid()` (gerado antes do insert, incluído em `reverseOpId` do resultado).
- `agent_run_id`: `ctx.runId`.
- `household_id`: `ctx.householdId`.
- `reverse_op`: `serializeReverseOp(payload)` (JSONB).
- `expires_at`: `sql\`now() + interval '30 seconds'\`` via Drizzle raw SQL — não calculado em JavaScript para evitar drift de clock.
- `reverted_at`: `null`.

Insert usa o cliente Drizzle da transacção (`tx`) para garantir atomicidade com as tools.

### AC7 — RLS: `executeAtomic` respeita `household_id` (NFR5)

- `ToolExecutionContext.db` é sempre o cliente `getDb()` (com RLS activa via JWT) — **nunca** `getServiceDb()`.
- A transacção Drizzle corre com o mesmo role `authenticated` com claims do household.
- O campo `household_id` em `agent_reverse_ops` é obrigatório e populado com `ctx.householdId` — Postgres rejeitará qualquer tentativa de insert sem este campo.
- Teste unitário verifica que `executeAtomic` usa o `db` do contexto (nunca cria o seu próprio cliente).

### AC8 — Taxonomia de erros específica de tools

`packages/tools/src/errors.ts` exporta hierarquia:

```
ToolError (abstract base)
 ├─ ToolValidationError(toolName, field, message)     — input/output Zod inválido; NÃO retryable
 ├─ ToolExecutionError(toolName, cause)               — execute() lançou excepção; NÃO retryable (causa determinística)
 ├─ ToolTransactionError(cause)                       — falha da transacção Drizzle/Postgres; retryable
 ├─ ToolNotFoundError(name)                           — tool não registado; NÃO retryable
 ├─ DuplicateToolError(name, domain)                  — conflito no register(); NÃO retryable
 └─ ToolPlanGateError(toolName, requiredPlan, actualPlan)  — plano insuficiente; NÃO retryable
```

- Todos os erros têm `message` técnico (logs/Sentry) e `userMessage` PT-PT (mostrável ao utilizador em Story 2.6).
- `message` **nunca inclui** tool input content (PII).
- `ToolExecutionError.cause` guarda a excepção original mas é redactada de PII antes de qualquer log.

### AC9 — OTel tracing whitelist para tool execution (zero payload em spans)

`packages/tools/src/tracing.ts` exporta:
- `withToolSpan(toolName, domain, fn)` — cria span `agent.tool.call` com atributos whitelist:
  - `tool.name` (string)
  - `tool.domain` (string)
  - `tool.duration_ms` (number, post-call)
  - `tool.success` (boolean)
  - `tool.household_hash` (sha256 do householdId — via `hashForCorrelation`)
  - `tool.trace_id` (string)
- `withAtomicSpan(runId, toolCount, fn)` — span `agent.tool.atomic` para o `executeAtomic` completo com:
  - `tool.atomic.tool_count` (number)
  - `tool.atomic.run_id` (string)
  - `tool.atomic.success` (boolean)
  - `tool.atomic.rolled_back` (boolean — true em falha parcial)
- **NUNCA** incluir: tool inputs, tool outputs, prompt content, snapshot data de `restore_row`.

### AC10 — PII redaction nos tool inputs antes de log

`packages/tools/src/atomic.ts` e `src/registry.ts` utilizam `redactToolInputForLog(toolName, input)`:
- Helper interno que retorna `{ toolName, inputRedacted: true }` — nunca serializa o input em logs.
- Logs de debug usam apenas: `toolName`, `domain`, `householdHash`, `traceId`, `durationMs`, `success`.
- Compatible com `PII_REDACT_PATHS` de `@meu-jarvis/observability` (não redefinir, reutilizar onde aplicável).
