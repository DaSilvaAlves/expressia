## User Story

**As a** developer da equipa Expressia construindo o pipeline 3-stage do Cérebro AI,
**I want** um package `@meu-jarvis/classifier` com `IntentSchema`/`ClassificationSchema` Zod-tipados, a classe `Classifier` que encapsula a chamada GPT-4o-mini via OpenAI structured outputs, um language gate PT-PT baseado em regex, e a taxonomia de erros específica do classifier,
**so that** a Story 2.5 (Planner+Executor) e a Story 2.6 (endpoint `/api/agent/prompt`) consomem intents classificadas num contrato tipado estável — sem repetir lógica de detecção de língua, sem fugas de PII para logs, com OTel tracing consistente com os packages `@meu-jarvis/agent` (2.2) e `@meu-jarvis/tools` (2.3), e com preview-then-confirm habilitado via flag `needs_confirmation` (FR4).

### AC3 — `ClassificationSchema` com guard max 5 e campo `needs_confirmation`

`packages/classifier/src/schemas.ts` exporta:

```ts
export const ClassifiedIntentSchema = z.object({
  intent: IntentSchema,
  confidence: z.number().min(0).max(1),
  raw_span: z.string(),     // sub-string do prompt que originou esta intent
});

export const ClassificationSchema = z.object({
  intents: z.array(ClassifiedIntentSchema).min(1).max(5),
  language: z.literal('pt-PT'),
  needs_confirmation: z.boolean(),  // true se algum intent.confidence < 0.70
  overall_confidence: z.number().min(0).max(1),
});

export type ClassificationResult = z.infer<typeof ClassificationSchema>;
```

- `intents.min(1)` está literal em Architecture §4.2 — pelo menos uma intent (eventualmente `unknown`).
- `intents.max(5)` é **guardrail anti-hallucination [AUTO-DECISION D8 — adicionada por @po na validação]**: arquitectura §4.2 não impõe limite superior, mas Epic 2 AC1 do PRD ilustra cenário canónico com 2 intents ("amanhã reunião às 15h, paguei €78,70 no supermercado"); um limite de 5 protege contra deriva LLM gerar arrays excessivos sem rejeitar prompts realistas multi-intent. PRD não especifica limite explícito — decisão arquitectural defensável validável em @architect gate. Se architect rejeitar limite, alternativa é `max(10)` ou remoção (`min(1)` apenas).
- `needs_confirmation` implementa a flag de trigger para preview-then-confirm (FR4). É derivado pelo Classifier da lista de confidences — `true` se `Math.min(...intents.map(i => i.confidence)) < 0.70`.
- `overall_confidence` é o mínimo das confidences individuais (estratégia conservadora).

**Trace:** Architecture §4.2 `ClassificationSchema` (`min(1)` literal); FR4 (`needs_confirmation` < 0.70 threshold); Architecture §4.4 (preview-then-confirm); `[AUTO-DECISION D8]` para `max(5)` anti-hallucination guardrail (sem fonte explícita FR — validação @architect).

### AC6 — Language gate PT-PT (regex, pré-LLM)

`packages/classifier/src/language-gate.ts` exporta:

```ts
export interface LanguageGateResult {
  isPortugueseEuropean: boolean;
  detectedPatterns: string[];  // padrões PT-BR/EN/ES encontrados, para debug
}

export function detectNonPtPt(text: string): LanguageGateResult
```

- Implementa detecção heurística leve via lista de palavras-chave inequivocamente não-PT-PT (PT-BR: `você`, `seu `, `voce`, `deletar`, `checar`, `printar`; EN: sequências `the `, `is `, `are `, ` to `; ES: `usted`, `tú`, `¿`, `¡`).
- Se `isPortugueseEuropean === false`, o `Classifier.classify()` retorna imediatamente sem chamar o LLM: `{ intents: [{ intent: 'unknown', confidence: 1.0, raw_span: text }], language: 'pt-PT', needs_confirmation: false, overall_confidence: 1.0 }`.
- Gate é conservador: falsos negativos (PT-PT com palavra ocasional semelhante a PT-BR) são aceites e enviados ao LLM. A lista de padrões é intencional e pequena (não tenta cobertura 100%).
- Testes incluem: 10+ frases PT-PT legítimas que NÃO devem ser rejeitadas (palavras como "você" em PT-PT são raras mas existem em contexto formal — a lista usa `você ` com espaço e outras formas para minimizar falsos positivos).

**Trace:** `language-standards.md` vocabulário PT-BR proibido; Architecture §4.2 "instruction-tuned para PT-PT"; NFR12 (conteúdo de prompts NÃO é logado).

[AUTO-DECISION] Boundary do language gate → regex conservadora (sem ML). Razão: (1) poupa tokens OpenAI para input claramente não-PT-PT; (2) mantém o código determinístico e testável sem dependência de API; (3) PRD não pede 100% accuracy na detecção de língua — apenas que o sistema "domine PT-PT".

### AC7 — Classe `Classifier` com método `classify()` injectável

`packages/classifier/src/classifier.ts` exporta:

```ts
export interface ClassifierInput {
  text: string;         // prompt do utilizador (max 1000 chars — validado)
  householdId: string;  // UUID — para hash OTel, NUNCA logado raw
  userId: string;       // UUID — para hash OTel, NUNCA logado raw
  traceId: string;      // trace ID OTel
}

export class Classifier {
  constructor(client: OpenAIClientLike, opts?: ClassifierOpts)
  async classify(input: ClassifierInput): Promise<ClassificationResult>
}
```

- `OpenAIClientLike` é importado de `@meu-jarvis/agent` (interface da Story 2.2, AC4) — sem acoplamento ao SDK OpenAI directamente.
- `ClassifierOpts` opcional: `{ maxInputLength?: number /* default 1000 */, timeoutMs?: number /* default 10000 */ }`.
- Fluxo interno do `classify()`:
  1. Validar `input.text` não-vazio e `length <= maxInputLength` → `ClassifierValidationError` se falhar.
  2. Invocar `detectNonPtPt(input.text)` → se `!isPortugueseEuropean` retornar resultado determinístico `unknown` sem chamar LLM (lança `ClassifierLanguageError` com severity `warn`, não `error`).
  3. Construir payload OpenAI com `response_format: { type: 'json_schema', json_schema: <ClassificationSchema convertida para JSON Schema> }` e system prompt + user message.
  4. Invocar `client.chat.completions.create(...)` com `model: CLASSIFIER_MODEL`, `temperature: 0`, `max_tokens: 256`.
  5. Parse e validação Zod do output → `ClassifierOutputError` se falhar.
  6. Derivar `needs_confirmation` e `overall_confidence`.
  7. Retornar `ClassificationResult`.
- `householdId` e `userId` NUNCA aparecem em logs ou spans sem `hashForCorrelation`.
- Timeout 10s default (alinhado com NFR1 latência p95 < 6s; classifier sozinho deve ser < 2s em regime normal).

**Trace:** Architecture §4.2, FR3, FR4, NFR1, NFR12.

### AC8 — Taxonomia de erros do classifier (4 subclasses)

`packages/classifier/src/errors.ts` exporta:

```ts
export abstract class ClassifierError extends Error {
  public readonly userMessage: string;
  public readonly retryable: boolean;
}

export class ClassifierValidationError extends ClassifierError
  // input inválido (vazio, > maxLength)
  // retryable: false, userMessage PT-PT

export class ClassifierLanguageError extends ClassifierError
  // gate PT-PT detectou input non-PT-PT
  // retryable: false, userMessage PT-PT
  // severity: warn (não é um erro de sistema)

export class ClassifierLLMError extends ClassifierError
  // wrapper de ProviderError da 2.2 (rate limit, network, etc.)
  // retryable: herda de ProviderError.retryable
  // userMessage PT-PT

export class ClassifierOutputError extends ClassifierError
  // LLM devolveu output que não passa ClassificationSchema
  // retryable: true (retry 1× com temperature=0)
  // userMessage PT-PT
```

- Todas as `userMessage` em PT-PT.
- `ClassifierLLMError` wraps `ProviderError` e expõe `cause: ProviderError`.
- `ClassifierOutputError` NÃO inclui o raw output do LLM no `message` (PII potential).
- Sem `ClassifierConfidenceError` — confidence baixa não é erro, é sinalizado via `needs_confirmation: true`.

**Trace:** Architecture §4.2 "qualquer deriva → retry 1× com temperature=0 ou cair em unknown"; padrão de error taxonomy de 2.2.

### AC9 — OTel tracing whitelist 10 atributos, zero payload

`packages/classifier/src/tracing.ts` implementa o wrapper de span para `Classifier.classify()` com whitelist canónica:

| Atributo | Tipo | Descrição |
|----------|------|-----------|
| `classifier.model` | string | sempre `'gpt-4o-mini'` |
| `classifier.input_length` | int | `input.text.length` |
| `classifier.intent_count` | int | `result.intents.length` |
| `classifier.overall_confidence` | float | `result.overall_confidence` |
| `classifier.language_detected` | string | `'pt-PT'` ou `'non-pt-pt'` |
| `classifier.duration_ms` | int | latência da chamada LLM |
| `classifier.tokens_input` | int | tokens input OpenAI |
| `classifier.tokens_output` | int | tokens output OpenAI |
| `classifier.success` | bool | `true` se sem erro |
| `classifier.error_class` | string | nome da classe de erro (se erro) |

**Zero payload:** `input.text`, detalhes dos intents individuais, `raw_span`, `householdId`, `userId` NUNCA aparecem em span attributes. `householdId` pode aparecer como `hashForCorrelation(householdId)` apenas.

**Trace:** NFR12 (prompts não logados em texto claro), Architecture §9.3 OTel, padrão de 2.2/2.3.

### AC10 — PII redaction nos error messages

- `ClassifierValidationError.message` NÃO inclui o conteúdo de `input.text` (pode conter dados pessoais).
- `ClassifierLLMError.message` usa `sanitizeHint` (importado de `@meu-jarvis/agent`) para limpar hints do provider.
- `ClassifierOutputError.message` NÃO inclui raw output do LLM.
- `ClassifierLanguageError.message` NÃO inclui o texto de input.
- Teste dedicado em `__tests__/errors.test.ts` verifica que inputs com NIF PT (9 dígitos), email (`@`), e número de telefone PT (`+351...`) NÃO aparecem em `.message` de nenhuma subclasse.

**Trace:** NFR12, Architecture §9.3, padrão de 2.2 AC9 (`sanitizeHint`).

4. Parse JSON + validação Zod → `ClassifierOutputError` (retry 1× automático com mesmo payload determinístico) | `ClassifierLLMError` (envolvendo `ProviderError` da 2.2) se SDK error.
5. Derivar `needs_confirmation` (any confidence < 0.70) e `overall_confidence` (min). O Classifier RECALCULA estes campos a partir do array de intents — protege contra incoerência LLM.
