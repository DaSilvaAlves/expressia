### 5. Geração atómica de prestações + estratégias de DELETE divergentes

**AC5:**

(a) **`installments` POST — geração atómica (DP-4.4.4 + DP8=A).** Criar uma prestação gera, **na mesma transacção Postgres** (`db.transaction()`), o `installment` + N `transactions` (`is_projected=true`). Cada transaction gerada:
- `installment_id` = id do installment; `installment_index` = k (1..N); `is_projected` = `true`.
- `card_id` = `installment.card_id` (satisfaz o CHECK `transactions_account_or_card`); `account_id` = NULL.
- `category_id` = `installment.category_id` (pode ser NULL).
- `kind` = `'expense'`; `payment_method` = `'card'`.
- `amount_cents` = `per_installment_cents` para k=1..N-1; **`total_amount_cents - (N-1) × per_installment_cents` para k=N** (resto na última — R-4.1).
- `transaction_date` = `first_installment_on + (k-1) meses` (DP-4.4.6 — cadência mensal).
- `description` = `installment.description` (sufixo `(parcela k/N)` opcional — DEV pode incluir para clareza na vista cartões).
Se qualquer INSERT falhar, a transacção inteira faz rollback (zero installment órfão, zero transactions parciais).

(b) **`installments` DELETE — hard delete em cascata (DP-4.4.5).** Numa transacção Postgres: `DELETE FROM transactions WHERE installment_id = X` **antes** de `DELETE FROM installments WHERE id = X`. A ordem inversa violaria o CHECK `transactions_installment_index_coherent` (`finance.ts:425-428` — `installment_id IS NULL` exige `installment_index IS NULL`) via o `ON DELETE set null`.

(c) **`recurrences` DELETE — soft delete (DP-4.4.3).** `UPDATE set active = false` — preserva `transactions.recurrence_id` das transacções já geradas pelo cron e pára geração futura (o cron 4.5 filtra `active = true`).

**AC5 PASS:** POST de prestação gera N transactions atómicas com `per_installment_cents` correcto (resto na última); DELETE de prestação elimina transactions + installment em cascata; DELETE de recorrência faz soft via `active=false`; tests cobrem rollback, resto de divisão e cascata.

### 6. Validações compostas

**AC6:**

(a) **`recurrences` — `account_or_card`.** `FinanceRecurrenceCreateSchema` aplica `.refine()`: pelo menos um de `account_id`/`card_id` → 400 VALIDATION_ERROR "Recorrência requer conta ou cartão." Alinha o CHECK `recurrences_account_or_card` (`finance.ts:274-277`). No PATCH: anular ambos viola o CHECK do DB; o handler detecta `/recurrences_account_or_card/i` na mensagem de erro e converte em 400 (pattern D-4.2.C / D-4.3.B).

(b) **FK cross-household.** `account_id`/`card_id`/`category_id` (recurrences) e `card_id`/`category_id` (installments), quando presentes, têm de pertencer ao household — verificado por SELECT RLS-scoped (`getDb()` não encontra rows cross-household) → 404 NOT_FOUND. (Categorias globais — `household_id NULL` — são válidas como `category_id`.) Nota: `installments.card_id` é **NOT NULL** (`finance.ts:305-307`) — obrigatório no payload.

(c) **`installments` — cálculo de prestação (R-4.1).** `per_installment_cents = floor(total_amount_cents / num_installments)`. As transactions 1..N-1 recebem `per_installment_cents`; a transaction N recebe `total_amount_cents - (N-1) × per_installment_cents` (o resto). O `installment.per_installment_cents` gravado é o `floor`. Exemplo: €1.000,00 (`100000` cents) / 3 → `per = 33333`; parcelas `33333 + 33333 + 33334 = 100000`. `num_installments` validado 1-60 (CHECK `installments_num_positive`).

> **[PO_FIX_INLINE F1] — guarda `total_amount_cents >= num_installments` (gap de validação).** Se `total_amount_cents < num_installments`, `per_installment_cents = floor(total/num) = 0`, e as transactions 1..N-1 geradas teriam `amount_cents = 0` — o que **viola o CHECK `transactions_amount_positive`** (`finance.ts:423` — `amount_cents > 0`). Sem esta guarda, o INSERT falha dentro de `db.transaction()`, faz rollback e o handler devolve `500 INTERNAL_ERROR` em vez de um `400 VALIDATION_ERROR` limpo. Por isso o `InstallmentCreateSchema` aplica `.refine(d => d.total_amount_cents >= d.num_installments)` → `400 VALIDATION_ERROR` "O valor total tem de ser pelo menos o número de parcelas (mínimo 1 cêntimo por parcela)." Cobre também o caso degenerado `num_installments = 1` (válido — `per = total`, zero parcelas a 0). [Source: `@po` cross-confirm `finance.ts:422-423` CHECK `transactions_amount_positive`.]

(d) **`recurrences` — `custom_rrule`.** Se `frequency='custom'`, `custom_rrule` é obrigatório (`.refine()`). O handler 4.4 apenas **persiste** `custom_rrule` — a interpretação RRULE pertence ao cron (Story 4.5). `next_run_on` é inicializado `= starts_on` no POST independentemente da frequência.

**AC6 PASS:** Tests cobrem: recorrência sem conta/cartão → 400; `frequency='custom'` sem `custom_rrule` → 400; FK cross-household → 404; `installments.card_id` ausente → 400; `num_installments` 0 ou 61 → 400; **`total_amount_cents < num_installments` → 400 VALIDATION_ERROR (PO_FIX_INLINE F1 — ex.: `total=5`, `num=10`)**; `per_installment_cents` com resto (€1.000/3, €100/7) verificado nos amounts das transactions geradas.
