## Dev Notes

### Estrutura do endpoint (Architecture §7.1, §7.3)

```typescript
// apps/web/src/app/api/me/route.ts
import { NextResponse } from 'next/server'
import { createServerSupabaseClient } from '@meu-jarvis/auth'
import { db } from '@meu-jarvis/db'
import { householdMembers, households, subscriptions } from '@meu-jarvis/db/schema'
import { eq } from 'drizzle-orm'
import { apiError } from '@/lib/errors'

export async function GET() {
  const supabase = createServerSupabaseClient()
  const { data: { user }, error: authError } = await supabase.auth.getUser()

  if (authError || !user) {
    return apiError('AUTH_REQUIRED', 'Sessão inválida ou expirada. Por favor inicie sessão novamente.', 401)
  }

  // Query com RLS activo (user context via JWT)
  const membership = await db
    .select({
      household: households,
      role: householdMembers.role,
      plan: subscriptions.plan,
    })
    .from(householdMembers)
    .innerJoin(households, eq(householdMembers.household_id, households.id))
    .leftJoin(subscriptions, eq(subscriptions.household_id, households.id))
    .where(eq(householdMembers.user_id, user.id))
    .limit(1)

  if (!membership.length) {
    return apiError('HOUSEHOLD_NOT_FOUND', 'Household não encontrado. Por favor complete o registo.', 404)
  }

  return NextResponse.json({
    user: { id: user.id, email: user.email },
    household: {
      id: membership[0].household.id,
      name: membership[0].household.name,
      plan: membership[0].plan,
    },
    role: membership[0].role,
  })
}
```

### `ApiError` type (Architecture §7.3)

```typescript
// apps/web/src/lib/errors.ts
import { NextResponse } from 'next/server'
import { randomUUID } from 'crypto'

export type ApiError = {
  error: {
    code: string
    message: string        // PT-PT user-facing
    details?: Record<string, unknown>
    timestamp: string      // ISO-8601
    requestId: string
  }
}

export function apiError(
  code: string,
  message: string,
  status: number,
  details?: Record<string, unknown>
): NextResponse<ApiError> {
  return NextResponse.json(
    {
      error: {
        code,
        message,
        details,
        timestamp: new Date().toISOString(),
        requestId: randomUUID(),
      }
    },
    { status }
  )
}
```

**Nota:** Nunca expor stack traces ou mensagens internas ao cliente. Nunca usar `details` para PII.

### RLS no endpoint — CRITICAL (Architecture §5.1, ADR-002)

O endpoint usa Drizzle **com o cliente Supabase do utilizador** (não service role). Isto significa que o RLS está activo automaticamente — a query filtra apenas o household do JWT.

**NÃO usar `SUPABASE_SERVICE_ROLE_KEY`** no endpoint `/api/me` — service role bypassa RLS.

### Teste E2E — RLS cross-household (Architecture §10.4, PRD Epic 1 AC2)

```typescript
// apps/web/e2e/rls-isolation.spec.ts
import { test, expect } from '@playwright/test'

test('RLS bloqueia acesso cross-household', async ({ request }) => {
  // Criar utilizador A + B (via Supabase Admin API em test setup)
  const tokenA = await getTestToken('user-a@test.com')
  const tokenB = await getTestToken('user-b@test.com')

  // User A vê o seu próprio household
  const resA = await request.get('/api/me', {
    headers: { Authorization: `Bearer ${tokenA}` }
  })
  expect(resA.status()).toBe(200)
  const bodyA = await resA.json()
  expect(bodyA.household).toBeDefined()

  // User B vê o seu próprio household (diferente de A)
  const resB = await request.get('/api/me', {
    headers: { Authorization: `Bearer ${tokenB}` }
  })
  expect(resB.status()).toBe(200)
  const bodyB = await resB.json()

  // Os dois households são diferentes
  expect(bodyA.household.id).not.toBe(bodyB.household.id)
})
```
