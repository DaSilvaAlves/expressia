### Drizzle client pattern (Architecture §11.2, db-schema.md §7.3)

```typescript
// packages/db/src/client.ts
import { drizzle } from 'drizzle-orm/postgres-js'
import postgres from 'postgres'
import * as schema from './schema'

// Runtime queries — Supabase pooler (porta 6543)
const queryClient = postgres(process.env.DATABASE_URL!)
export const db = drizzle(queryClient, { schema })

// Para migrations — usar DATABASE_URL_DIRECT (porta 5432)
```

### Connection pooling (Architecture §11.2)

| Variável | URL format | Uso |
|----------|------------|-----|
| `DATABASE_URL` | `postgresql://...@db.xxx.supabase.co:6543/postgres` | Runtime queries (PgBouncer) |
| `DATABASE_URL_DIRECT` | `postgresql://...@db.xxx.supabase.co:5432/postgres` | Migrations (Drizzle Kit) |

Usar `DATABASE_URL_DIRECT` APENAS em `drizzle.config.ts`. Em produção (Vercel), apenas `DATABASE_URL` está disponível via variável de ambiente runtime.

### Supabase CLI (alternativa a Drizzle Kit push)

```bash
# Ligar ao projecto
supabase link --project-ref <REF>

# Aplicar migrations
supabase db push

# Verificar estado
supabase migration list
```

### Decisões tomadas (`[AUTO-DECISION]`)

1. **Naming `DIRECT_URL` vs `DATABASE_URL_DIRECT`** — `[AUTO-DECISION] Story menciona `DATABASE_URL_DIRECT` mas o `.env.local` entregue pelo Orion usa `DIRECT_URL` (convenção Supabase oficial). → Mantive `DIRECT_URL` e `drizzle.config.ts` aceita ambos com fallback. Reason: alinhar com convenção Supabase + não modificar `.env.local`.

2. **Migration runner custom em vez de `drizzle-kit migrate`** — `[AUTO-DECISION] Os ficheiros 0000_initial_schema.sql e 0001_rls_policies.sql são handwritten pela Dara (@data-engineer), fora do journal Drizzle (`migrations/meta/_journal.json` não existe). `drizzle-kit migrate` requer journal. → Criei runner custom `apply-migrations.ts` que aplica SQL files puros em ordem, com tracking idempotente em `public.__schema_migrations`. Reason: respeitar SQL existente, manter idempotência, evitar conflito com possíveis migrations Drizzle-geradas no futuro (que ficam noutro padrão de naming).

3. **Região `eu-west-1` (Ireland) em vez de `eu-central-1` (Frankfurt)** — `[AUTO-DECISION] Architecture §11.2 menciona Frankfurt. Projecto Supabase real está em Ireland. → Documentei a divergência no runbook como equivalente GDPR (ambas UE/EEE) cumprindo NFR11. AC8 marcado parcialmente — região é UE mas não Frankfurt. Recomendar @architect rever Architecture §11.2 ou aceitar Ireland como variante. Reason: não recriar projecto Supabase só para mover entre data centers UE.

4. **`createServerClient` para Story 1.5** — `[AUTO-DECISION] AC7 menciona apenas exportar cliente Drizzle tipado. O helper `createServerClient` (Supabase com cookies/auth) é referenciado nas Tasks (3.4) mas pertence ao scope de Auth (Story 1.5). → Não implementado nesta story. Reason: evitar adicionar features fora dos AC validados pelo @po (Article IV — No Invention).

5. **Smoke test usa `it.skipIf(!HAS_DB)`** — `[AUTO-DECISION] Os smoke tests precisam de DB real. Em CI sem creds não devem falhar. → Usei `it.skipIf` do Vitest em todos os testes do `db-connection.test.ts`. Reason: NFR — CI nunca falha por falta de creds opcionais; o gate real é `check-rls-coverage` que é estático.

### Self-critique (checkpoint pós-código)

- **Risk #1 (alto):** Sem permissão Bash não posso provar que `db:migrate` corre realmente contra o pooler — pode haver bug no driver `postgres.js` com `pooler 5432` que só aparece em runtime. Mitigação: runner usa `prepare: false` defensivo.
- **Risk #2 (médio):** O test `it.skipIf(!HAS_DB)` significa que se as creds não estiverem na shell quando se corre `pnpm test`, todos os testes do DB são skipped silenciosamente. Mitigação: docs no runbook indicam carregar `.env.local`.
- **Risk #3 (baixo):** Tabela `__schema_migrations` é convencional mas não combina com naming Supabase (`supabase_migrations.schema_migrations`). Decidi standalone para evitar acoplamento ao Supabase CLI; trade-off documentado.
