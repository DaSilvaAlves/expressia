## User Story

**As a** utilizador autenticado da Expressia com prompt PT-PT no chat principal,
**I want** que ao enviar `POST /api/agent/prompt` com o meu texto o sistema orquestre o pipeline completo (Classifier → Planner → Executor), execute as minhas intenções atomicamente em Postgres com RLS activa, me mostre um preview de confirmação quando a confiança for inferior a 70%, e me permita reverter a operação dentro de 30 segundos,
**so that** eu tenha um assistente AI que entende frases complexas PT-PT, executa múltiplas acções em segurança dentro do meu household, e me dá controlo sobre o que foi criado sem qualquer dado meu vazar entre sessões, tenants, ou logs do sistema.

## Acceptance Criteria

### AC1 — Route Handler POST /api/agent/prompt

`apps/web/src/app/api/agent/prompt/route.ts` exporta `POST` (Next.js Route Handler, App Router). Aceita body JSON `{ prompt: string }` com validação Zod (`prompt.min(1).max(2000)`). Retorna `NextResponse` com Content-Type `application/json`. HTTP 401 se sem sessão Supabase Auth válida (verificado com `createServerClient` do `@supabase/ssr`).

**Trace:** Architecture §7.1 (Route Handlers para endpoints públicos contractuais), §8.1 (`apps/web/src/app/api/agent/prompt/route.ts`), §1.3 (fluxo Prompt multi-intent), PRD FR1, NFR19.

### AC2 — RLS multi-tenant via getDb() (role authenticated)

Endpoint usa `getDb()` (role `authenticated`, connection via `DATABASE_URL` pooler 6543, `prepare: false`) — **NUNCA** `getServiceDb()`. RLS Postgres activa via JWT Supabase claim `household_id` (helper `current_household_id()`). Cross-household access bloqueado pelo Postgres transparentemente.

**Trace:** Architecture §3.2 (RLS Pattern), §5.1-5.2 (Auth Hook + custom claim `household_id`), CLAUDE.md (getDb vs getServiceDb), PRD NFR5, CON2.

### AC3 — Pipeline orquestrado: Classifier → Planner → Executor

Fluxo interno do handler: (1) `Classifier.classify(prompt, { householdId, userId, traceId, runId })` de `@meu-jarvis/classifier`; (2) `Planner.plan(classification, { householdId, userId, traceId, runId })` de `@meu-jarvis/planner-executor`; (3) `Executor.execute(plan, { householdId, userId, traceId, runId })` de `@meu-jarvis/planner-executor` que chama `executeAtomic` de `@meu-jarvis/tools`. Cada estágio recebe `runId` (UUID da entrada `agent_runs` criada no início do handler).

**Trace:** Architecture §4.1 Pipeline em 3 estágios (sequência Classifier→Planner→Executor), §4.3 (Planner+Executor Sonnet), PRD FR1, FR2.

### AC4 — FR4 Preview-then-confirm: branch confidence < 0.70

Se `classification.needs_confirmation === true` (i.e., algum `intent.confidence < 0.70` no resultado do Classifier, conforme `ClassificationSchema.needs_confirmation` da Story 2.4):

- Endpoint **NÃO executa** Planner+Executor atomicamente.
- Cria entrada `agent_runs` com `status = 'pending_preview'`.
- Retorna HTTP 200 com body:

```ts
{
  mode: 'preview',
  run_id: string,                     // UUID da agent_run criada
  plan_summary: string[],             // textos preview de cada tool (via toolRegistry.preview())
  confidence: number,                 // confidence média do classifier
  confirmation_url: string,           // "/api/agent/prompt/{run_id}/confirm"
  expires_at: string,                 // ISO-8601 — 5 minutos para confirmar (D20)
}
```

Se `classification.needs_confirmation === false` (confidence ≥ 0.70 em todos os intents) → pipeline executa atomicamente (ver AC5).

**Trace:** Architecture §4.4 (Preview-then-Confirm Flow), PRD FR4, §3.2 Key Interaction Paradigms.

### AC5 — Execução atómica e resposta executed

Se confidence ≥ 0.70 (ou após confirmação via `/confirm`): endpoint executa `Executor.execute()` que chama `executeAtomic`. `agent_runs.status` evolui `classifying → executing → success | failed`. Retorna HTTP 200:

```ts
{
  mode: 'executed',
  run_id: string,
  results: AtomicResult,              // de @meu-jarvis/tools — sumário das operações
  summary: string,                    // texto PT-PT agregado do que foi criado/alterado
  undo_url: string,                   // "/api/agent/prompt/{run_id}/undo"
  undo_expires_at: string,            // ISO-8601 — now() + 30s (FR6, D21)
}
```

`agent_runs.status` terminal `success` é imutável após escrita (trigger `trg_agent_runs_immutability` da Story 2.1, mutation apenas via `service_role`).

**Trace:** Architecture §4.1 (fluxo completo), §4.5 (Undo Mechanism), PRD FR2, FR6.

### AC6 — Endpoint POST /api/agent/prompt/[runId]/confirm

`apps/web/src/app/api/agent/prompt/[runId]/confirm/route.ts` exporta `POST`. Valida que:
- `agent_run.household_id === current_household_id()` (RLS — cross-household bloqueado)
- `agent_run.status === 'pending_preview'`
- `now() < confirm_expires_at` (5 minutos D20)

Se válido: executa Planner + Executor com o `classification` guardado em `agent_run.intent_classification` JSONB. Actualiza `agent_runs.status` → `executing` → `success`. Retorna mesmo shape `{mode: 'executed', ...}` do AC5.

Se expirado ou status errado: HTTP 409 com `ApiError` código `CONFIRM_EXPIRED` ou `CONFIRM_INVALID_STATE`.

**Trace:** Architecture §4.4 (preview token / confirm flow), PRD FR4. Decisão D20 abaixo.

### AC7 — Endpoint POST /api/agent/prompt/[runId]/undo

`apps/web/src/app/api/agent/prompt/[runId]/undo/route.ts` exporta `POST`. Valida que:
- `agent_run.household_id === current_household_id()` (RLS)
- `agent_run.status === 'success'`
- Existem rows em `agent_reverse_ops` com `agent_run_id = runId` e `expires_at > now()` e `executed_at IS NULL` <!-- [PO_FIX_INLINE v1.1] real schema column é `executed_at` (não `reverted_at`); ver db-schema.md §4.4 invariantes + packages/db/src/schema/agent.ts:202. Architecture §4.5 tem doc bug stale `reverted_at` (DOC-001-NB do gate 2.3, não corrigido) -->

Se válido: executa transacção inversa (lê `reverse_op` JSONB de cada row, aplica operação inversa), marca `agent_reverse_ops.executed_at = now()` (NÃO `reverted_at` — esse campo não existe na tabela `agent_reverse_ops`), e actualiza `agent_runs.status → 'reverted'` + `agent_runs.reverted_at = now()` via `getServiceDb()` (única operação justificada como service_role — NFR9 mutation de terminal; `agent_runs` tem coluna `reverted_at` real, distinta da semântica do `executed_at` em `agent_reverse_ops`). Retorna HTTP 200 `{reverted: true, run_id}`.

Se janela expirada: HTTP 409 `UNDO_EXPIRED`. Se já revertido (run-level: `agent_runs.status = 'reverted'`; row-level: `agent_reverse_ops.executed_at IS NOT NULL`): HTTP 409 `UNDO_ALREADY_REVERTED`.

**Trace:** Architecture §4.5 (Undo Mechanism — endpoint `POST /api/agent/undo/{runId}`), §1.3 (fluxo Undo 30s), PRD FR6, db-schema.md §4.4 invariantes (canónico para nomes de colunas reais).

### AC8 — NFR9 Idempotency via Idempotency-Key header

Endpoint principal `POST /api/agent/prompt` suporta header `Idempotency-Key` (UUID string):
- Se presente: lookup em `agent_runs` onde `idempotency_key = header_value AND household_id = current_household_id()`.
- Se encontrado com `status` terminal (`success` | `reverted` | `failed`): retorna o mesmo response sem re-executar (replay determinístico).
- Se encontrado com `status` não-terminal: retorna HTTP 409 `IDEMPOTENCY_IN_PROGRESS`.
- Se não encontrado: prossegue normalmente, persiste `idempotency_key` em `agent_runs`.

Janela de idempotency: 24h (D19) — `agent_runs` com `created_at < now() - 24h` não são considerados para replay (criação de nova run).

**Trace:** PRD NFR9 (Audit log imutável — traçado operações sensíveis), Architecture §6.3 (Webhook idempotência via stripe_event_id como precedente de pattern). Decisão D19 abaixo.

### AC9 — NFR13 Rate limiting per household

Endpoint verifica rate limit antes de executar pipeline:
- Limite: `10 req/min burst` por `household_id` (Architecture §7.2 — limite documentado para `/api/agent/prompt`).
- Storage MVP: tabela Postgres `agent_rate_limit_counters` (D18 — EB3 Upstash pending; migração em Story 2.9).
- Se excedido: HTTP 429 com `ApiError` código `RATE_LIMIT_EXCEEDED`, mensagem PT-PT, header `Retry-After` em segundos.
- Quota mensal separada (Architecture §4.6): verificada em `agent_quotas` — se excedida: HTTP 429 `QUOTA_EXCEEDED`.

**Trace:** Architecture §7.2 (Rate Limiting — `/api/agent/prompt` 10 req/min burst), §4.6 (Cost Router e Quotas), PRD NFR13. Decisão D18 abaixo.

## [AUTO-DECISIONS] @sm

### [AUTO-DECISION] D17 — Rate limit threshold
**Decisão:** 10 req/min burst por household — valor exacto documentado em `docs/architecture.md §7.2` para `/api/agent/prompt` (Architecture §7.2: "quota mensal por household + 10 req/min burst").
**Razão:** Usar o valor canónico da architecture evita contradição com source of truth. Valor é conservador para MVP (1000 households activos × 50 prompts/dia = ~0.035 req/min médio por household). Burst de 10/min cobre picos de uso intenso. Revisão pós-launch com dados Grafana `agent.latency.p95` se necessário.

### [AUTO-DECISION] D18 — Rate limit storage para Story 2.6 (MVP)
**Decisão:** Tabela Postgres `agent_rate_limit_counters(household_id PK, window_start timestamptz, count int)` — Postgres counter simples para MVP. Migração para Upstash Redis EU (EB3) fica em **Story 2.9** (Cost Router + Cache + Quotas que depende de EB3 Upstash).
**Razão:** Architecture §7.2 especifica "Upstash Redis EU + sliding window" mas EB3 está PENDING. Para Story 2.6 mockable-only, Postgres é suficiente e elimina dependência externa. Story 2.9 tem EB3 como bloqueador explícito — é o local canónico para a migração. KISS: não adicionar infra nova sem necessidade imediata (Architecture §4.6 já menciona Upstash em Story 2.9 scope).

### [AUTO-DECISION] D19 — Idempotency window
**Decisão:** 24h. `agent_runs.idempotency_key` (nullable text, unique partial index onde `idempotency_key IS NOT NULL`). Lookup: `created_at > now() - interval '24 hours'` para evitar replay de runs muito antigos.
**Razão:** 24h é window standard SaaS para idempotency (Stripe usa 24h). Janelas mais curtas criam risco de double-execution em falhas de rede; mais longas causam acumulação de estado. O campo `idempotency_key` é adicionado em migration 0006 (Task 12).

### [AUTO-DECISION] D20 — Preview confirmation flow: endpoint POST /[runId]/confirm
**Decisão:** REST `POST /api/agent/prompt/[runId]/confirm` (endpoint separado no `[runId]` dynamic segment). A `agent_run` em `status = 'pending_preview'` guarda o `classification` JSONB original. O confirm endpoint relê esse classification, chama Planner + Executor, e retorna o mesmo shape `{mode: 'executed', ...}`.
**Razão:** Endpoint separado é mais idiomático em Next.js App Router (evita overloading do handler principal com `?action=confirm`). Guarda `classification` JSONB evita re-chamar Classifier (economiza tokens GPT-4o-mini + latência). TTL de 5 minutos para confirmar (Architecture §4.4 cita "Preview token assinado HMAC válido 5 min" — adoptado como deadline do pending_preview run). Alternativa `confirm_run_id` no body principal seria menos discoverable para Story 2.7 (UI).

### [AUTO-DECISION] D21 — Undo endpoint shape: REST POST /[runId]/undo
**Decisão:** REST `POST /api/agent/prompt/[runId]/undo`. Architecture §4.5 documenta literalmente `POST /api/agent/undo/{runId}` — path ligeiramente diferente mas semântica idêntica. Adoptado `[runId]` nested sob `/api/agent/prompt/` para coerência com o confirm endpoint e co-localização no mesmo feature directory. Note: Architecture §1.3 usa `/api/agent/undo/{token}` — esta story normaliza para `/api/agent/prompt/{run_id}/undo` (run_id é mais explícito que token genérico; @architect pode ajustar no gate se preferir path plano).
**Razão:** Co-localizar confirm+undo sob `/api/agent/prompt/[runId]/` simplifica middleware auth e facilita testar em conjunto. Path normalizado para `run_id` eliminata a ambiguidade de "token" vs "run_id" que existe entre §1.3 e §4.5 da architecture.

### [AUTO-DECISION] D22 — CodeRabbit pre-push skip
**Decisão:** Consistente com D12 (Story 2.4), D14 (Story 2.5) — CodeRabbit skipped. `@architect` gate exhaustivo substitui review automatizado quando file count ou complexidade de decisões arquitecturais o justifica.
**Razão:** Padrão estabelecido nas 3 stories anteriores Done com gate APPROVED 9.4-9.5/10. Architect gate de Story 2.6 terá escopo ~15 ficheiros novos + modificados, o que está bem dentro da janela de análise manual exhaustiva.

### [AUTO-DECISION] D23 — OTel context propagation: re-usar tracing.ts da 2.5
**Decisão:** Criar `apps/web/src/lib/agent/tracing.ts` que re-usa `withSpan`, `annotateSpan`, `recordSpanError`, `hashForCorrelation` de `@meu-jarvis/observability` (padrão das stories anteriores). ### Debug Incidents Resolved

1. **`Cannot find module '@meu-jarvis/db' has no exported member 'getDb'`** — apps/web typecheck cross-package. **Causa raíz**: package `@meu-jarvis/db` interno usa `@/schema` paths que tsc do consumer não resolve. **Resolução**: refactor `@/*` → `./*` nos packages `db` + `planner-executor` (D27) + criar `db-shim.ts` lazy require para evitar load do schema em testes mocked.

2. **`AtomicFailure.failure?` doesn't exist** — typecheck em route.ts + confirm/route.ts. **Causa raíz**: assumi shape errado de `AtomicFailure` (real shape: `{success: false, failedToolName, error: ToolError, rolledBack: true}`). **Resolução**: corrigido para `outcome.error.message` + `outcome.failedToolName`.

3. **`@typescript-eslint/no-require-imports` em route.ts + db-shim.ts** — lint failure. **Causa raíz**: usei `require()` para lazy load de `openai` SDK e `@meu-jarvis/db/client`. **Resolução**: `eslint-disable-next-line` localizado (consistente com pattern em `packages/planner-executor/src/planner.ts:142`).

4. **`@typescript-eslint/consistent-type-imports` `import()` type annotations forbidden** — lint nos test files. **Causa raíz**: usei `typeof import('...')` para tipar `vi.importActual`. **Resolução**: cast inline `(await vi.importActual('...')) as Record<string, unknown>`.

5. **`Module not found: Can't resolve '@/schema'` em build webpack** — Next.js build falha. **Causa raíz**: webpack do Next.js não resolve paths internos do package via tsconfig do package. **Resolução**: refactor `@/schema` → `./schema` em `packages/db/src/{client,index,schema/index}.ts` + 6 schema files (D27).

6. **`agent_rate_limit_counters: Em falta SELECT/INSERT/UPDATE/DELETE`** — check:rls falha. **Causa raíz**: nova tabela com `household_id` requer 4 policies em `0001_rls_policies.sql` (regex inspection do gate). **Resolução**: adicionado DO block condicional (`if exists ... execute create policy ...`) com 4 policies idempotentes — corre apenas se tabela já existe (evita falha em fresh install pre-0006); regex do gate detecta strings literais `create policy "..." on public.agent_rate_limit_counters for <command>` mesmo dentro de `EXECUTE`.

7. **DB mock fails: `INSERT returning vazio (DB inesperado)`** — tests falhavam. **Causa raíz**: tentei matchar `query.strings.join` mas o sql template do Drizzle não expõe strings dessa forma. **Resolução**: refactor mock para usar `let callIndex = 0` + switch sobre ordem das chamadas (mais robusto, alinhado com pattern de testes 2.4/2.5).

8. **Vitest tests `mockResolvedValueOnce` não interage bem com `mockImplementation` default** — testes específicos (replay, in_progress) precisam sobrescrever só a primeira call. **Resolução**: usar `mockResolvedValueOnce` antes do `setupDbDefault` no beforeEach + queue ordem natural do vitest.
