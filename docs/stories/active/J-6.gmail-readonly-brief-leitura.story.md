## Story

**As a** Eurico (único utilizador do Jarvis),
**I want** receber no brief diário um resumo dos emails não lidos que pedem resposta, e poder pedir ao Jarvis que leia e resuma emails a pedido via Telegram (ex.: "mostra os meus emails" ou "procura emails do Pedro"),
**so that** consigo gerir a minha caixa de entrada sem ter de abrir o Gmail, com o mesmo Telegram que já uso para tarefas, finanças e agenda.

## Contexto de âmbito

**Esta story entrega:**
- Gmail readonly via OAuth scope `gmail.readonly` (cumula com `calendar.events` já em prod).
- Re-consentimento OAuth one-shot [EURICO] (novo scope — Google obriga nova autorização quando scopes aumentam).
- Novo intent `consultar_emails` no motor: leitura de emails a pedido via Telegram.
- Nova tool `consultar_emails` (Gmail API readonly; `reverse` inerte `_noop` — padrão R1b v1.1 das tools read-only existentes).
- Helper `getGmailSummaryForBrief` para o job Inngest cron do brief diário.
- Brief diário actualizado: nova secção de email (emails não lidos do inbox) + fix cosmético "tu" informal herdado de J-4.

**NÃO inclui (fora de âmbito):**
- Envio de email — J-7 (`gmail.send`, scope mais sensível, story separada).
- Responder/compor/arquivar/apagar emails via chat.
- Cache de emails na DB (dados de email são efémeros — processados em memória, nunca persistidos).
- Notificações proactivas de email além do brief diário.
- Suporte a múltiplos labels ou contas Gmail.

**Depende de:** J-5 Done (OAuth `calendar.events` em prod; `google_oauth_tokens` em prod com token cifrado; `calendar-api.ts` como template para `gmail-api.ts`; padrão de registo de tools externas em `apps/web` estabelecido e provado E2E).

7. Existe o módulo `apps/web/src/lib/agent/tools/gmail/list-emails.ts` que implementa `ToolDefinition` com `name: 'consultar_emails'`, `domain: 'email'`:
   - `inputSchema` Zod: `{ query: z.string().optional(), maxResults: z.number().int().min(1).max(10).default(5) }`. Sem `query`, usa `is:unread in:inbox`; com `query`, usa o valor como pesquisa Gmail (ex.: `from:pedro`, `subject:factura`).
   - `preview(input)`: devolve texto PT-PT como "Vou procurar os teus emails recentes no Gmail." (sem query) ou "Vou procurar emails sobre '{query}' no Gmail." (com query).
   - `execute(input, ctx)`:
     1. Obtém `accessToken` via `getGmailAccessToken(ctx, 'consultar_emails')`.
     2. `GET {GMAIL_API_ENDPOINT}/messages?q={q}&maxResults={input.maxResults}` onde `q = input.query ?? 'is:unread in:inbox'`.
     3. Para cada `message.id`: `GET {GMAIL_API_ENDPOINT}/messages/{id}?format=metadata&metadataHeaders=Subject,From,Date`.
     4. Devolve `GmailMessageMetadata[]` (máx. `input.maxResults`). Lista vazia → devolve `[]` sem lançar (resultado vazio válido).
     5. Falha HTTP (status != 200) → lança `ToolExecutionError` PT-PT descritivo.
   - `reverse()`: devolve `{ kind: 'delete_row', table: '_noop', id: randomUUID() }` — padrão R1b v1.1, idêntico a `listar-tarefas.ts` e `query-finance-summary.ts`. Tool read-only; `executeAtomic` persiste a row inerte; endpoint `/undo` responde 410 Gone para `table = '_noop'` (comportamento já existente, sem alteração).

### Padrão `reverse()` para tool read-only — R1b v1.1

`ToolDefinition.reverse` é obrigatório (interface `contracts.ts` linha 358). Para tools read-only, o padrão estabelecido no projecto é:

async reverse(): Promise<ReverseOpPayload> {

`executeAtomic` persiste esta row inerte em `agent_reverse_ops`. O endpoint `/undo` responde 410 Gone quando lê `reverse_op->>'table' = '_noop'` (comportamento já existente). Referência: `packages/tools/src/tasks/listar-tarefas.ts` L187-193 + `packages/tools/src/finance/query-finance-summary.ts`.

### `gmail-api.ts` — diferenças face a `calendar-api.ts`

O `gmail-api.ts` deve modelar-se em `calendar-api.ts` mas adaptar:

```ts
// calendar-api.ts usa:
export const CALENDAR_EVENTS_ENDPOINT = 'https://www.googleapis.com/calendar/v3/calendars/primary/events';
export const CALENDAR_TZ = 'Europe/Lisbon';

// gmail-api.ts usa:
export const GMAIL_API_ENDPOINT = 'https://www.googleapis.com/gmail/v1/users/me';
// Sem CALENDAR_TZ — Gmail não usa fuso para as chamadas da API
```

`getGmailAccessToken(ctx, toolName)` é idêntico a `getCalendarAccessToken` — mesma query SQL a `google_oauth_tokens`, mesmo `refreshAccessToken`, mesmo tratamento de erro. Os type guards são diferentes (respostas JSON da Gmail API vs Calendar API).

### Gmail API — estrutura de resposta relevante

**List messages** (`GET /messages?q=...`):
```json
{ "messages": [{ "id": "...", "threadId": "..." }], "resultSizeEstimate": 3 }
```

**Get message metadata** (`GET /messages/{id}?format=metadata&metadataHeaders=Subject,From,Date`):
```json
{
  "id": "...",
  "snippet": "primeiro texto do email...",
  "payload": {
    "headers": [
      { "name": "Subject", "value": "Reunião amanhã" },
      { "name": "From", "value": "Pedro <pedro@example.com>" },
      { "name": "Date", "value": "Fri, 27 Jun 2026 10:30:00 +0000" }
    ]
  }
}
```

O `extractEmailHeader(headers, name)` deve iterar `payload.headers` e retornar `header.value` onde `header.name.toLowerCase() === name.toLowerCase()` (case-insensitive).

### `getGmailSummaryForBrief` vs `getGmailAccessToken` (tool)
