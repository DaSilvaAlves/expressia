## User Story

**Como** utilizador da Expressia com tarefas recorrentes configuradas no agregado familiar (ex: "Pagar renda" todo dia 1 do mês, "Comprar pão" todas as segundas-feiras, "Limpar fogão" última sexta de cada mês),
**Quero** que o sistema gere automaticamente todos os dias as instâncias futuras dessas recorrências (até 90 dias à frente) sem eu ter de fazer nada,
**Para que** as minhas tarefas recorrentes apareçam sempre nas vistas Lista/Kanban/Calendário com a data correcta — respeitando regras complexas (todos os 30 do mês, último dia útil, primeiro sábado, etc.) e DST Portugal (último domingo de Março/Outubro) — e nunca seja apanhado de surpresa por uma recorrência esquecida (FR8).

         ```sql
           $o.targetDate, $template.dueTime, $template.priority, 'todo',
           NULL /* spawned tasks default sem coluna kanban */, $template.project,
         ```

### AC3 — Helper `rrule-helpers.ts` (timezone-aware Europe/Lisbon DST-safe)

Criar `apps/web/src/lib/recurrences/rrule-helpers.ts` com função pura **testable em isolamento** (sem Inngest, sem Drizzle):

```ts
import { RRule, Frequency } from 'rrule';
import { fromZonedTime, toZonedTime, formatInTimeZone } from 'date-fns-tz';
import { addDays, parseISO } from 'date-fns';

const TZ = 'Europe/Lisbon';

export interface ExpandOptions {
  horizonDays: number;  // 90 per DP-3.7.1 A
  now: Date;            // injectable para tests determinísticos
}

export interface ExpandResult {
  occurrences: Array<{ targetDate: string }>;  // YYYY-MM-DD
  nextRunAfterHorizon: string | null;          // YYYY-MM-DD ou null (RRULE esgotada)
  isExhausted: boolean;                        // true se endsOn passou OU RRULE não retorna mais
}

export function expandRecurrence(
  recurrence: {
    frequency: 'daily'|'weekdays'|'weekends'|'weekly'|'biweekly'|'monthly'|'yearly'|'custom';
    interval: number;
    customRrule: string | null;
    startsOn: string;       // YYYY-MM-DD
    endsOn: string | null;  // YYYY-MM-DD ou null
    nextRunOn: string | null;
  },
  options: ExpandOptions
): ExpandResult {
  // ... lógica:
  // 1. Construir RRULE string baseado em frequency (mapeando para iCal):
  //    - daily → RRULE:FREQ=DAILY;INTERVAL=$interval
  //    - weekly → RRULE:FREQ=WEEKLY;INTERVAL=$interval
  //    - biweekly → RRULE:FREQ=WEEKLY;INTERVAL=2 (ignora $interval)
  //    - monthly → RRULE:FREQ=MONTHLY;INTERVAL=$interval
  //    - yearly → RRULE:FREQ=YEARLY;INTERVAL=$interval
  //    - weekdays → RRULE:FREQ=WEEKLY;BYDAY=MO,TU,WE,TH,FR
  //    - weekends → RRULE:FREQ=WEEKLY;BYDAY=SA,SU
  //    - custom → usa $recurrence.customRrule directamente (assumido RFC 5545 válido)
  // 2. UNTIL = endsOn (se presente) — converter para UTC midnight via fromZonedTime
  // 3. DTSTART = startsOn em Europe/Lisbon noon (12:00 — evita ambiguidade DST 02:00/03:00)
  // 4. rule.between(now, now + horizonDays) → Date[] UTC
  // 5. Para cada Date d: formatInTimeZone(d, TZ, 'yyyy-MM-dd') → targetDate
  // 6. nextRunAfterHorizon: rule.after(now + horizonDays + 1) → próxima após horizon, ou null
  // 7. isExhausted: endsOn !== null && parseISO(endsOn) < now() || nextRunAfterHorizon === null
}
```

**Critical:** usar `formatInTimeZone(date, 'Europe/Lisbon', 'yyyy-MM-dd')` em vez de `date.toISOString().slice(0, 10)` (UTC) — diferença crítica em transição DST. Exemplo: ocorrência computada 2026-03-29T23:30:00Z em UTC corresponde a **2026-03-30** em Europe/Lisbon (DST clock spring forward 02:00→03:00). UTC date slice retornaria errado.

Exportar também `frequencyToRRuleString(frequency, interval)` (helper internal) para reuso em tests + `EXPAND_HORIZON_DAYS = 90 as const` constant.

### AC6 — Audit log INSERT em `audit_log` (NFR9)

Imediatamente antes do `return { total_generated, ... }` no handler de `generate-recurring-tasks.ts`, fazer INSERT em `audit_log` via `getServiceDb()` (job sistémico, sem JWT user):

```ts
await db.execute(sql`
  INSERT INTO audit_log (
    household_id, user_id, action,
    entity_table, entity_id,
    before_state, after_state,
    ip, user_agent, trace_id
  ) VALUES (
    NULL /* job cross-household; agg row */, NULL /* sistémico */, 'recurrences_generated',
    'task_recurrences', NULL /* agg, não single entity */,

      total_generated: totalGenerated,
      total_skipped: totalSkipped,
```

## [DEV-DECISIONS] @dev

> Decisões de implementação documentadas pelo @dev (Dex) durante a execução. @architect ratifica no gate.

### [DEV-DECISION D-3.7.8] `withSpan` usa a assinatura real de 3 argumentos

**Decisão:** O handler usa `withSpan('agent.recurrences.generate', {}, async (span) => { ... })` — 3 argumentos `(name, initialAttrs, fn)`.

**Justificação:** A Story (AC7 / Dev Notes linha 660) mostrava `withSpan(name, async (span) => {...})` (2 argumentos). A assinatura real exportada por `@meu-jarvis/observability` (`packages/observability/src/tracer.ts:124`) é `withSpan<T>(name, initialAttrs: DomainAttributes, fn)`. Aplicado o NIT-PO-3.7.1 (importar de `@meu-jarvis/observability`, não de `@/lib/agent/tracing.ts`) e, em consequência, a assinatura correcta de 3 args. `initialAttrs` é `{}` porque os counts só são conhecidos após o `step.run`; os 7 atributos AC7 são aplicados via `span.setAttribute(...)` antes do return. Comportamento de tracing inalterado.

**Trace:** `packages/observability/src/tracer.ts:124`, NIT-PO-3.7.1, AC7.

### [DEV-DECISION D-3.7.9] `horizonEnd` levado ao fim do dia + `nextRunAfterHorizon` via `after(horizonEnd, false)`

**Decisão:** Em `rrule-helpers.ts`, o fim da janela de expansão é `now + horizonDays` levado a 23:59:59.999 UTC; `nextRunAfterHorizon` é `rule.after(horizonEnd, false)` (fronteira exclusiva).

**Justificação:** As ocorrências são fixadas ao meio-dia Lisboa (≈12:00 UTC) para evitar a janela ambígua DST. Com `horizonEnd` à meia-noite, a ocorrência do último dia (12:00) cairia fora do `between` — daria sempre N−1 ocorrências (ex.: horizon=7 → 7 em vez de 8). Levar `horizonEnd` ao fim do dia garante a contagem `horizonDays+1`. Consequentemente, `nextRunAfterHorizon` usa `after(horizonEnd, false)` em vez de `after(horizonEnd+1day, true)` para não saltar a ocorrência imediatamente seguinte. `isExhausted` passou a ser `nextRunAfterHorizon === null` (a RRULE não tem mais nada para lá do horizonte) — alinhado com AC8 teste (11). Validado via 22 testes incluindo DST spring/fall.

**Trace:** AC3, AC8 testes (1)+(11)+(14), R-3.7.2.

### [DEV-DECISION D-3.7.10] Audit `after_state` via `${JSON.stringify(...)}::jsonb` (padrão do codebase)

**Decisão:** O INSERT em `audit_log` serializa o `after_state` com `${JSON.stringify(summary)}::jsonb` em vez de `sql.json(...)` (mencionado na Story AC6).

**Justificação:** `sql.json` não existe em `drizzle-orm@0.34`. O padrão estabelecido em `apps/web/src/lib/api-helpers/audit.ts:106-107` (Story 3.2) é `${JSON.stringify(state)}::jsonb` — Drizzle parametriza a string com segurança contra SQL injection e o `::jsonb` faz o cast. Story 3.7 adopta o mesmo padrão por consistência.

**Trace:** `apps/web/src/lib/api-helpers/audit.ts:106-107`, AC6.
