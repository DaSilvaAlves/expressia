## Story

**As a** Eurico (único utilizador do Jarvis v1),
**I want** que o Jarvis consiga ler os meus eventos do Google Calendar para hoje, de forma segura, com os meus tokens OAuth armazenados cifrados na base de dados,
**so that** o brief diário de J-4 pode incluir a minha agenda real, e eu tenho a garantia de que as minhas credenciais Google nunca ficam expostas em plaintext.

## Contexto de âmbito

- Esta story entrega a integração Google Calendar até à camada de leitura: OAuth single-user com `calendar.readonly`, cifragem AES-256-GCM do `refresh_token`, tabelas `google_oauth_tokens` e `jarvis_facts` (migration `0029`), e função `getCalendarEventsToday()` que devolve os eventos de hoje.
- A **injecção desses eventos no brief de J-4** é um follow-up posterior — J-3 entrega apenas a integração OAuth + leitura. J-4 já está em prod (sem agenda); o follow-up é um patch curto sobre J-4.
- **Depende de J-1** (webhook seguro, infra Telegram estabelecida).
- NÃO inclui: escrita no Calendar, fluxo de reagendar, botões inline para agenda — tudo v1.1.
- O fluxo OAuth de consentimento é one-shot (feito uma vez, pelo Eurico, assistido pelo @dev). Não há UI de onboarding para terceiros.

## Dev Notes

### Decisões já fechadas (pré-implementação)

- **Cifragem:** AES-256-GCM via `node:crypto` (`createCipheriv`/`createDecipheriv`). IV aleatório de 96 bits (12 bytes) por cifração. `authTag` de 128 bits (16 bytes) verificado na decifração — garante integridade. Chave em `OAUTH_TOKEN_ENCRYPTION_KEY` (32 bytes), apenas em Vercel Env, nunca na DB nem em git. *Alternativa rejeitada:* pgcrypto/Supabase Vault — mistura blast-radius de fuga de DB com o da chave; menos portável. [Fonte: PRD §4.4, §7]
- **Scope OAuth:** `calendar.readonly` apenas. `access_type=offline` + `prompt=consent` para forçar `refresh_token`. Sem escrever na agenda (v1.1). [Fonte: PRD §4.4, FR-J9]
- **Fallback de agenda:** se `getCalendarEventsToday` falhar, devolve `null`; o brief (J-4 follow-up) omite a secção com nota "não consegui ler a tua agenda hoje". O brief nunca falha por causa do Calendar. [Fonte: PRD §4.4]
- **Âmbito de `jarvis_facts`:** tabela de factos simples key-value por household (ex.: `user_name`, `timezone`, `brief_tone`). UNIQUE `(household_id, key)` — upsert por chave. Âmbito mínimo para v1 — cresce na v2 com memória rica. [Fonte: PRD FR-J10, §5]
- **`googleapis` vs `fetch` nativo:** o epic (linha 49) diz "googleapis apenas em J-3 (decisão a confirmar na altura)". O PRD §8 deixa a decisão aberta. O @dev deve decidir e documentar no Dev Agent Record. Se `googleapis` for escolhido, adicionar como dependência de `apps/web` (não do monorepo inteiro). Se `fetch` nativo for escolhido, documentar os endpoints exactos usados.
- **Migration 0029:** número confirmado — 0027=telegram_link e 0028=daily_briefing_cache já existem em produção. [AUTO-DECISION: migration número 0029]
- **Trigger `set_updated_at`:** usar `public.set_updated_at()` (helper canónico do `0000_initial_schema.sql`). NÃO usar `update_updated_at_column()` — gotcha de J-2 (PO-MUST-FIX-1).

```sql
set local check_function_bodies = off;

create table if not exists public.google_oauth_tokens (
  id                      uuid        not null default gen_random_uuid() primary key,
  household_id            uuid        not null references public.households(id) on delete cascade,
  user_id                 uuid        not null references auth.users(id) on delete cascade,
  encrypted_refresh_token text        not null,
  token_iv                text        not null,
  token_auth_tag          text        not null,
  access_token_hint       text,
  token_expiry            timestamptz,
  google_email            text,
  created_at              timestamptz not null default now(),
  updated_at              timestamptz not null default now(),
  unique (household_id, user_id)
);

drop trigger if exists google_oauth_tokens_set_updated_at on public.google_oauth_tokens;
create trigger google_oauth_tokens_set_updated_at
  before update on public.google_oauth_tokens
  for each row execute function public.set_updated_at();

create table if not exists public.jarvis_facts (
  id           uuid        not null default gen_random_uuid() primary key,
  household_id uuid        not null references public.households(id) on delete cascade,
  key          text        not null,
  value        text        not null,
  created_at   timestamptz not null default now(),
  updated_at   timestamptz not null default now(),
  unique (household_id, key)
);

drop trigger if exists jarvis_facts_set_updated_at on public.jarvis_facts;
create trigger jarvis_facts_set_updated_at
  before update on public.jarvis_facts
  for each row execute function public.set_updated_at();
```

```sql
    execute 'alter table public.google_oauth_tokens force row level security';
    execute 'drop policy if exists "google_oauth_tokens_select" on public.google_oauth_tokens';
    execute 'drop policy if exists "google_oauth_tokens_insert" on public.google_oauth_tokens';
    execute 'drop policy if exists "google_oauth_tokens_update" on public.google_oauth_tokens';
    execute 'drop policy if exists "google_oauth_tokens_delete" on public.google_oauth_tokens';
```

```ts
// apps/web/src/lib/crypto/token-cipher.ts
import { createCipheriv, createDecipheriv, randomBytes } from 'node:crypto';

function getKey(): Buffer {
  const raw = process.env.OAUTH_TOKEN_ENCRYPTION_KEY;
  if (!raw) throw new Error('OAUTH_TOKEN_ENCRYPTION_KEY não está definida');
  const key = Buffer.from(raw, raw.length === 64 ? 'hex' : 'base64');
  if (key.length !== 32) throw new Error('OAUTH_TOKEN_ENCRYPTION_KEY tem de ter 32 bytes');
  return key;
}

export function encryptToken(plaintext: string): { ciphertext: string; iv: string; authTag: string } {
  const key = getKey();
  const iv = randomBytes(12); // 96 bits — tamanho recomendado para GCM
  const cipher = createCipheriv('aes-256-gcm', key, iv);
  const encrypted = Buffer.concat([cipher.update(plaintext, 'utf8'), cipher.final()]);
  return {
    ciphertext: encrypted.toString('base64'),
    iv: iv.toString('base64'),
    authTag: cipher.getAuthTag().toString('base64'),
  };
}

export function decryptToken(ciphertext: string, iv: string, authTag: string): string {
  const key = getKey();
  const decipher = createDecipheriv('aes-256-gcm', key, Buffer.from(iv, 'base64'));
  decipher.setAuthTag(Buffer.from(authTag, 'base64'));
  const decrypted = Buffer.concat([
    decipher.update(Buffer.from(ciphertext, 'base64')),
    decipher.final(),
  ]);
  return decrypted.toString('utf8');
}
```

### Decisão de dependência: `fetch` nativo vs `googleapis` (AC6 / Tarefa 3.1)

**Decisão: `fetch` nativo** ao token endpoint do Google (`https://oauth2.googleapis.com/token`) e ao Calendar API (`https://www.googleapis.com/calendar/v3/calendars/primary/events`). **NÃO** foi adicionada a dependência `googleapis`.

**Razão:**
- Alinhamento com J-1/J-2, que usaram `fetch` para a Bot API do Telegram — consistência de padrão no projecto.
- Zero dependência nova pesada: `googleapis` traz centenas de KB de superfície de API (Discovery clients de dezenas de serviços) para usar apenas 3 chamadas HTTP triviais (token exchange, token refresh, userinfo) + 1 chamada ao Calendar.
- Menor superfície de cold-start em serverless (Vercel `fra1`) e menos transitive deps a auditar.
- Os endpoints OAuth do Google são `application/x-www-form-urlencoded` simples — `URLSearchParams` + `fetch` cobrem-nos sem fricção.

*Alternativa rejeitada:* `googleapis` npm em `apps/web` — daria token-refresh/expiry helpers prontos, mas o custo (peso + transitive deps) não se justifica para a superfície mínima desta story.
