## User Story

**As a** developer da equipa Expressia construindo o pipeline 3-stage do Cérebro AI,
**I want** um package `@meu-jarvis/planner-executor` com a classe `Planner` que recebe `ClassificationResult` (Story 2.4) e constrói um tool plan via Claude Sonnet com tool calling sobre `toolRegistry.getAnthropicToolDefinitions()` (Story 2.3) usando prompt-cache `ephemeral`, a classe `Executor` que valida tool names contra o registry e delega execução atómica a `executeAtomic` (Story 2.3) numa única transacção Postgres com rollback completo em caso de falha de qualquer tool, taxonomia de erros específica (`PlannerError` + 4 subclasses; `Executor` delega `ToolError` da 2.3), spans OTel `agent.planner.call` + `agent.executor.run` com whitelist zero-PII, e mocks determinísticos `AnthropicClientLike`,

### AC2 — Schemas Zod: `PlanResultSchema`, `ToolCallSchema`, `PlannerInputSchema`, `ExecutorInputSchema`

`packages/planner-executor/src/schemas.ts` exporta:

```ts
export const PlanToolCallSchema = z.object({
  toolName: z.string().min(1),
  input: z.record(z.string(), z.unknown()),
  intent: IntentSchema,                       // intent originária (de ClassificationResult)
  rawCallId: z.string().optional(),           // tool_use_id devolvido pelo Anthropic SDK (preservar para debug)
});

export const PlanResultSchema = z.object({
  toolCalls: z.array(PlanToolCallSchema).min(0).max(10),  // 0 = LLM não conseguiu construir plan; max 10 [AUTO-DECISION D5]
  planReasoning: z.string().nullable(),       // texto livre do Sonnet entre tool_use blocks (nullable se ausente)
  latencyMs: z.number().int().nonnegative(),
  tokensInput: z.number().int().nonnegative(),
  tokensOutput: z.number().int().nonnegative(),
  costEur: z.number().nonnegative(),
  cacheHit: z.boolean(),
});

export const PlannerInputSchema = z.object({
  classification: ClassificationSchema,        // import de @meu-jarvis/classifier
  householdId: z.string().uuid(),
  userId: z.string().uuid(),
  traceId: z.string().min(1),
  runId: z.string().uuid(),                    // agent_runs.id — passado pelo endpoint (Story 2.6)
});

export const ExecutorInputSchema = z.object({
  plan: PlanResultSchema,
  householdId: z.string().uuid(),
  userId: z.string().uuid(),
  traceId: z.string().min(1),
  runId: z.string().uuid(),
});

export type PlanToolCall = z.infer<typeof PlanToolCallSchema>;
export type PlanResult = z.infer<typeof PlanResultSchema>;
export type PlannerInput = z.infer<typeof PlannerInputSchema>;
export type ExecutorInput = z.infer<typeof ExecutorInputSchema>;
```

- `toolCalls.min(0)`: o LLM pode legitimamente devolver array vazio se `classification.intents` for `[{intent: 'unknown'}]` — caso permitido como degradação graceful (sem tools = nada a executar).
- `toolCalls.max(10)` é guardrail anti-hallucination análogo a `ClassificationSchema.intents.max(5)` da Story 2.4 AC3 D8 — ver `[AUTO-DECISION D5]` abaixo.
- `planReasoning` é o texto livre que o Sonnet eventualmente produz entre `tool_use` blocks (ex: "Vou criar 1 tarefa e registar 1 transacção..."). É preservado para debug/audit mas **não exibido ao utilizador final** (UI de preview na 2.7 usa apenas `toolCalls`).

[AUTO-DECISION D5] `toolCalls.max(10)` anti-hallucination guardrail. Razão: (1) o cap de 5 intents simultâneas vem da Story 2.4 AC3 [AUTO-DECISION D8] (já em produção/Done, validado por architect gate APPROVED 9.4/10 com nota telemetry Story 2.11) — `ClassificationSchema.intents.max(5)`; (2) 1 intent pode legitimamente expandir-se em 2-3 tools (ex: `criar_parcelada` → `create_card_transaction` + `create_installment_plan` + `create_recurrence`); (3) 5 intents × 2-3 tools/intent = 10-15 tools razoável; (4) limite duro 10 protege contra deriva LLM gerar arrays excessivos sem rejeitar cenários multi-intent realistas. **NOTA [PO FIX inline 2026-05-09]**: PRD não especifica limite explícito de intents simultâneas (FR3 do PRD é sobre audit log — `prd.md:42` — não sobre limite multi-intent). O cap de 5 vem da decisão arquitectural D8 da 2.4; o cap de 10 desta story deriva por expansão. Architecture §4.2 ClassificationSchema literal só impõe `intents.array(...).min(1)`. Validável em @architect gate — alternativas: `max(15)` (mais permissivo, cobre 5 intents × 3 tools) ou `max(5)` (mais restritivo, mas pode rejeitar `criar_parcelada` legítima). Sugestão: monitorizar empiricamente em Story 2.10 benchmark + telemetria 2.11.

**Trace:** Architecture §4.3 (`tool_calls[]` array do Sonnet); Story 2.4 AC3 D8 (`max(5)` intents — fonte real do cap de 5); FR4 (preview); FR2 (atomicidade multi-intent); §4.5 (reverse_op delegado a `executeAtomic` da 2.3). ### AC4 — Classe `Planner` com método `plan(input: PlannerInput): Promise<PlanResult>`

`packages/planner-executor/src/planner.ts` exporta:

```ts
export interface PlannerOpts {
  client?: AnthropicClientLike;             // injecção para testes — em produção usa getProvider() interno
  maxTokens?: number;                        // default 1024
  timeoutMs?: number;                        // default 15000
  cacheControl?: 'ephemeral' | null;         // default 'ephemeral' (D11)
}

export class Planner {
  constructor(opts?: PlannerOpts);
  async plan(input: PlannerInput): Promise<PlanResult>;
}
```

Fluxo interno de `plan()`:
1. Validar `PlannerInputSchema.parse(input)` → `PlannerValidationError` se falhar.
2. Early-return: se `input.classification.intents` for `[{intent: 'unknown', ...}]` (case `unknown` único), retornar `PlanResult` com `toolCalls: []`, `planReasoning: 'Intent unknown — sem tools a executar.'`, custo zero. Não invoca LLM.
3. Construir `ProviderCompleteInput` com:
   - `system`: `PLANNER_SYSTEM_PROMPT` (constante de `prompts/planner-system.ts`)
   - `messages`: `[{ role: 'user', content: <texto serializado de classification.intents + raw_spans> }]`
   - `tools`: `toolRegistry.getAnthropicToolDefinitions()` — single source of truth da 2.3
   - `cacheControl`: do opts (default `'ephemeral'`)
   - `temperature`: `0.2` (mais conservador que classifier; permitir alguma variação para tool selection mas evitar deriva)
   - `maxTokens`: do opts (default `1024`)
   - `traceId`, `householdId` propagados
4. Invocar `provider.complete(input)` via `getProvider({ preferredProvider: 'anthropic' })` (ou `client` injectado nos testes).
5. Validar output: `output.toolCalls` deve ser array; cada `toolCall.name` deve existir em `toolRegistry.get(name)` → `PlannerToolNotFoundError` se Sonnet alucinou tool name.
6. Mapear `output.toolCalls` → `PlanToolCall[]` enriquecendo cada call com a `intent` originária (heurística: fazer match `toolName` → `intent` via mapping declarativo `TOOL_TO_INTENT_MAP` em `schemas.ts` ou — alternativamente — pedir ao LLM para anotar a intent no `input` da tool call. **AUTO-DECISION D6** — escolher o mapping mais simples).
7. Construir `PlanResult` com `latencyMs`, `tokensInput`, `tokensOutput`, `costEur`, `cacheHit` propagados de `output`.
8. Retornar.

`AnthropicClientLike` da 2.2 isola SDK Anthropic — sem acoplamento directo. Timeout 15s default (alinhado com NFR1 latência p95 < 6s; planner sozinho ~3-5s em regime estacionário).

[AUTO-DECISION D6] Mapping `toolName → intent`. Opção A: tabela declarativa estática `TOOL_TO_INTENT_MAP` em `schemas.ts` (ex: `{ create_task: 'criar_tarefa', create_finance_variable: 'criar_financa_variavel', ... }`). Opção B: LLM anota a intent no schema da tool call. **Decisão: Opção A** — mapping estático. Razão: (1) determinístico, testável; (2) evita custo LLM extra; (3) tools são single source of truth da 2.3 e não mudam entre runs; (4) se um tool não tiver mapping, retornar `'unknown'` como intent associada (falha graciosa). Validável @architect.

**Trace:** Architecture §4.3 (Planner—Sonnet, prompt cache ephemeral, tool calling); FR3 (multi-intent); FR4 (preview hooks); NFR1 (latência); NFR12 (PII).

### AC6 — Classe `Executor` com método `execute(input: ExecutorInput): Promise<AtomicResult | AtomicFailure>`

`packages/planner-executor/src/executor.ts` exporta:

```ts
export interface ExecutorOpts {
  registry?: ToolRegistry;                   // injecção para testes — em produção usa singleton toolRegistry
}

export class Executor {
  constructor(opts?: ExecutorOpts);
  async execute(input: ExecutorInput): Promise<AtomicResult | AtomicFailure>;
}
```

Fluxo interno de `execute()`:
1. Validar `ExecutorInputSchema.parse(input)` → `ExecutorValidationError` se falhar.
2. Early-return: se `input.plan.toolCalls.length === 0`, retornar `AtomicResult { success: true, results: [] }` directamente — não abre transacção, não chama `executeAtomic`. Caso degenerado de `unknown` intent ou plan vazio.
3. **Defense-in-depth tool name validation** [AUTO-DECISION D8]: para cada `toolCall` em `input.plan.toolCalls`, invocar `registry.get(toolCall.toolName)` — se algum lança `ToolNotFoundError`, retornar `AtomicFailure { success: false, failedToolName, error: ToolNotFoundError, rolledBack: false }` SEM abrir transacção. Razão: (1) Planner já validou em AC4 step 5, mas o Executor pode receber `PlanResult` de outras origens em stories futuras (preview-then-confirm, undo redo); (2) early-fail antes de transacção poupa Postgres round-trip; (3) defense-in-depth é padrão obrigatório em código transaccional.
4. Construir array `tools` no formato esperado por `executeAtomic`:
   ```ts
   const tools = input.plan.toolCalls.map(tc => ({
     definition: registry.get(tc.toolName),
     input: tc.input,
   }));
   ```
5. Construir `ToolExecutionContext`:
   ```ts
   const ctx: ToolExecutionContext = {
     householdId: input.householdId,
     userId: input.userId,
     db: getDb(),                              // cliente Drizzle com RLS activa (NÃO getServiceDb!)
     traceId: input.traceId,
     runId: input.runId,
   };
   ```
6. Invocar `executeAtomic(tools, ctx)` da Story 2.3 — toda a lógica transaccional, RLS enforcement, persistência de `agent_reverse_ops` com `expires_at = now() + interval '30 seconds'`, e rollback em caso de falha está delegada à 2.3.
7. Envolver tudo em span OTel `agent.executor.run` (AC9).
8. Retornar resultado de `executeAtomic` directamente — sem transformar/wrap (preserva `AtomicResult.results[].reverseOpId` que a Story 2.8 vai consumir para o endpoint undo).

**Atomicidade Postgres preservada:** A transacção é responsabilidade da Story 2.3 (`executeAtomic`). Se qualquer tool falhar, rollback completo automático — nenhuma row em entidades de domínio, nenhuma row em `agent_reverse_ops`. NFR5 (RLS) preservada porque `ctx.db = getDb()` (role `authenticated` com claims `household_id`).

**Trace:** Architecture §4.3 ("Executor — Sonnet + Tools"); §4.1 (BEGIN tx → loop tools → COMMIT); FR2 (atomicidade); FR6 (reverse_op preparado para undo); NFR5 (RLS); Story 2.3 AC5/AC6/AC7 (`executeAtomic`).

[AUTO-DECISION D8] Defense-in-depth tool name validation no Executor mesmo já validado no Planner. Razão acima (3 motivos). Custo: 1 lookup Map O(1) por tool call × max 10 = trivial. Validável @architect.

### AC7 — Atomicidade Postgres — rollback completo em falha de qualquer tool

Comportamento herdado de `executeAtomic` da 2.3 mas explicitamente reafirmado nesta story:
- Se 1 das N tool calls do plan falhar (input inválido, `execute()` lança, output schema falha, ou erro de transacção Postgres), a transacção faz **ROLLBACK completo**.
- **Nenhuma entidade de domínio é criada** (zero rows em `tasks`, `transactions`, `cards`, etc.).
- **Nenhuma row de `agent_reverse_ops` persiste** — não há "undo parcial" possível.
- O Executor retorna `AtomicFailure { success: false, failedToolName, error, rolledBack: true }`.
- O endpoint `/api/agent/prompt` (Story 2.6) é responsável por mapear `AtomicFailure` para HTTP response com `userMessage` PT-PT do `error`.

Teste obrigatório em `__tests__/executor.test.ts`: cenário com 3 tool calls onde a 2ª falha → verificar que (a) `executeAtomic` foi chamada uma vez, (b) o resultado é `AtomicFailure` com `failedToolName === 'tool 2'`, (c) `rolledBack === true`. Mock Drizzle tx confirma que `tx.rollback` foi invocado.

**Trace:** FR2 ("transacções atómicas multi-intent — all-or-nothing"); Architecture §4.1 ("BEGIN ... COMMIT" com rollback implícito); Story 2.3 AC5 (`executeAtomic` rollback semantics).

### AC8 — Reverse_op persistence delegada a `executeAtomic` (preparação Story 2.8)

Esta story **NÃO implementa** o endpoint undo (Story 2.8) nem job de cleanup (Story 2.8 + Inngest). Limita-se a:
- Garantir que `Executor.execute()` invoca `executeAtomic` sem suprimir a persistência de `agent_reverse_ops`.
- Cada `AtomicResult.results[]` retornado expõe `{ toolName, output, reverseOpId }` — o `reverseOpId` é o UUID da row em `agent_reverse_ops` criada durante `executeAtomic` (Story 2.3 AC5/AC6).
- Estes IDs ficam disponíveis para o handler do endpoint /api/agent/prompt (Story 2.6) gerar o `undo_token` (HMAC-assinado) que volta ao utilizador.
- Esta story **não** persiste `agent_runs` — é responsabilidade do endpoint (2.6) fazer o INSERT inicial (status `classifying`) e UPDATE final (status `success`/`failed`).

Teste obrigatório: snapshot de `AtomicResult.results[i].reverseOpId` é um UUID válido em todos os cenários happy-path.

**Trace:** Architecture §4.5 (`agent_reverse_ops.expires_at = now() + interval '30 seconds'`); FR6 (undo 30s — implementação completa em Story 2.8); Story 2.3 AC6 (persistência delegada).

### AC10 — Taxonomia de erros do package (5 subclasses Planner; Executor delega ToolError)

`packages/planner-executor/src/errors.ts` exporta:

// Planner errors — específicos desta story
export abstract class PlannerError extends Error {
  public readonly userMessage: string;     // PT-PT
  public readonly retryable: boolean;
}

export class PlannerValidationError extends PlannerError
  // PlannerInputSchema.parse falhou (input inválido, classification malformada)
  // retryable: false; userMessage PT-PT

export class PlannerLLMError extends PlannerError
  // wrapper de ProviderError da 2.2 (rate limit, timeout, network, server)
  // retryable: herda de ProviderError.retryable
  // userMessage PT-PT

export class PlannerToolNotFoundError extends PlannerError
  // Sonnet alucinou um tool name que não existe em toolRegistry
  // retryable: false (não vale retry — modelo provavelmente persiste no erro)
  // userMessage PT-PT

export class PlannerOutputError extends PlannerError
  // output do Sonnet não passa PlanResultSchema (incoerência estrutural)
  // retryable: true (1× retry com temperature=0)
  // userMessage PT-PT

export class PlannerEmptyPlanError extends PlannerError
  // intents != [unknown] mas Sonnet retornou toolCalls=[] — caso degenerado warning
  // retryable: false; severity: warn; userMessage PT-PT
```

**Executor errors:** **NÃO criar nova hierarquia.** Reutilizar `ToolError` + 6 subclasses já exportados de `@meu-jarvis/tools` (Story 2.3 AC8). Razão: o Executor é **thin wrapper** sobre `executeAtomic` que já produz `ToolValidationError`/`ToolExecutionError`/`ToolTransactionError`/`ToolNotFoundError`/`ToolPlanGateError`/`DuplicateToolError`. Reinventar errors específicos do Executor causaria duplicação e quebraria error mapping no endpoint (Story 2.6). [AUTO-DECISION D13]

Excepção: **`ExecutorValidationError`** — única classe nova do Executor, para falha de `ExecutorInputSchema.parse(input)` (input inválido antes de chegar a `executeAtomic`). Análoga a `PlannerValidationError`. Definida em `errors.ts` ao lado das `PlannerError` para coesão de package.

- Todas as `userMessage` em PT-PT.
- `PlannerLLMError` wraps `ProviderError` e expõe `cause: ProviderError`.
- `PlannerOutputError` NÃO inclui o raw output do LLM no `message` (PII potential).
- `PlannerEmptyPlanError` NÃO inclui o conteúdo de `classification.intents[].raw_span` no `message`.

**Trace:** Architecture §4.3 (error handling implícito no tool calling Sonnet); padrão de error taxonomy das Stories 2.2 AC7, 2.3 AC8, 2.4 AC8.

[AUTO-DECISION D13] Executor delega `ToolError` da 2.3 — não cria hierarquia paralela. Razão: thin wrapper, single source of truth na 2.3. Excepção `ExecutorValidationError` para input parsing apenas.

    { role: 'user', content: serializeClassificationForPlanner(input.classification) }
    // serializeClassificationForPlanner: helper interno que produz texto PT-PT
    // descrevendo as intents detectadas + raw_spans (NÃO o prompt original do utilizador)
  ],
  tools: toolRegistry.getAnthropicToolDefinitions(),  // single source of truth da 2.3
  cacheControl: 'ephemeral',                         // cache hit ~90% saving (Architecture §4.3)
  temperature: 0.2,
  maxTokens: 1024,
  traceId: input.traceId,
  householdId: input.householdId,
};

const output = await provider.complete(providerInput);
// output.toolCalls = [{ name: 'create_task', input: { title: '...', ... } }, ...]
// output.cacheHit = true (após primeiro request por 5min)
// output.costEur = 0.001-0.005 típico
```

### Multi-intent → multi-tool mapping (D6 detalhe)

`TOOL_TO_INTENT_MAP` declarativo:

```ts
// packages/planner-executor/src/schemas.ts
export const TOOL_TO_INTENT_MAP: Record<string, Intent> = {
  create_task: 'criar_tarefa',
  create_finance_variable: 'criar_financa_variavel',
  create_finance_recurrence: 'criar_financa_recorrente',
  create_card: 'criar_cartao',
  create_installment: 'criar_parcelada',
  query_finance_summary: 'consultar_dados',
  query_tasks: 'consultar_dados',
  cancel_last_run: 'cancelar_ultima',
  // tool name desconhecido → 'unknown' (fallback)
} as const;
```

Ao mapear `output.toolCalls[i] → PlanToolCall`:
```ts
const intent = TOOL_TO_INTENT_MAP[toolCall.name] ?? 'unknown';
```

Caso 1 intent `criar_parcelada` produzir 3 tool calls (`create_card_transaction`, `create_installment_plan`, `create_recurrence`), o `intent` de cada `PlanToolCall` será `'criar_parcelada'` para os primeiros 2 e `'unknown'` para `create_recurrence` (não está no MAP) — útil para debug mas não bloqueante (Executor não usa `intent` para validação).
