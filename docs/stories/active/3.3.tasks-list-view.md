## User Story

**Como** utilizadora da Expressia (`@household-owner` ou `@household-member`) que tem tarefas geradas via cérebro AI Jarvis (Epic 2) e via API CRUD (Story 3.2),
**Quero** uma vista lista densa em `/tarefas` (com tabs Lista/Kanban/Calendário visíveis), com filtros por estado/tag/prazo/projecto/prioridade/atribuído, ordenação configurável (prazo crescente / criação descendente / prioridade descendente / título A-Z), agrupamento natural (Hoje · Amanhã · Esta semana · Atrasadas · Concluídas hoje), bulk actions (marcar concluído / eliminar / mudar prioridade) e edição inline do título com row actions menu (Editar / Eliminar / Adiar 1 dia / Mudar prioridade),
**Para que** posso operar dezenas de tarefas em fluxo (sem abrir cada uma), encontrar rapidamente o que está atrasado (FR11), e ter a UX padrão moderna alinhada com o front-end-spec §5.6 (wireframe ratificado) — preparando o terreno para Stories 3.4 (Kanban) e 3.5 (calendário) que reutilizam os mesmos endpoints + state pattern URL.

### 2. Server-rendered initial fetch (RSC)

**AC2:** RSC `page.tsx` consome o endpoint Story 3.2 com pattern KISS:

(a) Import direct via helper extraído `listTasksHelper` (DP4-3.3: A) — chamada server-side sem rota HTTP loopback (DP3-3.3: A). Reuse helpers `requireAuth` + `getDb` que já estão no handler.

  **PO-FIX-2 v1.1 (framing precision):** O @sm draft v1.0 cita "Story 2.7 = pattern canónico para import direct" — mas verificação directa de `apps/web/src/app/(app)/conta/preferencias/page.tsx:42-54` revela que Story 2.7 **usa fetch HTTP loopback** (DP3 variant B com cookie pass-through), NÃO import direct. Story 2.7 é canónico para a **estrutura RSC + client islands** (separação `page.tsx` server / `_components/*-toggle.tsx` client), mas **não** para a estratégia de fetch. A recomendação DP3-3.3: A (import direct) permanece sólida pelos seus próprios méritos (KISS + zero loopback latency + zero `NEXT_PUBLIC_APP_URL` dep) — Story 3.3 é o **primeiro precedente** de RSC com import direct neste codebase. Se @architect rejeitar DP4 extract (fallback path), reverter a fetch loopback Story 2.7 é aceitável como alternativa segura.

  **Alternativa rejeitada** (DP3-3.3: B): `fetch(\`${process.env.NEXT_PUBLIC_APP_URL}/api/tasks\`)` loopback HTTP — adiciona latência (~5-20ms) + complexidade env var management + duplica auth context (cookie pass-through). KISS prevalece.

(b) Parâmetros do request: ler `URLSearchParams` from `searchParams` prop da page (Next.js 15 App Router idiomatic — `searchParams` é async Promise em Next 15) e passar para `GET` handler como `NextRequest` mock OR reuse `listTasksRSC(filters, userId, householdId)` helper extraído (DP4-3.3 abaixo recomendação A — extrair helper compartilhado entre RSC e route handler).

(c) Response shape consumido: `{ tasks: TaskRow[], next_cursor: string | null }` (alinhado com `apps/web/src/app/api/tasks/route.ts:145`).

(d) Error handling: se RSC fetch falha (DB down, etc.), renderiza `<EmptyState variant="error">` com mensagem PT-PT "Não foi possível carregar as tarefas. Tenta novamente." + retry button. Captura Sentry via `captureException` from `@meu-jarvis/observability`.

**AC2 PASS:** Initial page load em `/tarefas` renderiza tarefas reais (até 50 items) sem JavaScript client-side activo (RSC HTML stream). Lighthouse FCP ≤ 1.5s em rede 4G simulada (NFR1 sub-budget).

### 4. Sort dropdown + URL-state

**AC4:** Criar `_components/TaskSort.tsx` (client) com:

(a) `<Select>` Ordenar por com 4 opções:
  - `Prazo (crescente)` → `?sort=due_date_asc` (mapping default backend Story 3.2: `due_date asc nulls last, id asc` — alinhado).
  - `Criação (mais recentes)` → `?sort=created_at_desc`.
  - `Prioridade (alta primeiro)` → `?sort=priority_desc`.
  - `Título (A-Z)` → `?sort=title_asc`.

(b) **Backend extension Story 3.2 endpoint:** O endpoint actual `/api/tasks/route.ts:126` tem `ORDER BY due_date asc nulls last, id asc` hardcoded. Story 3.3 estende o handler para aceitar `?sort=` query param + extender `TaskFiltersSchema` com `sort: z.enum(['due_date_asc', 'created_at_desc', 'priority_desc', 'title_asc']).default('due_date_asc')`.

  **DP5-3.3 (NEW):** Onde extender o ORDER BY?
  - **A** (recommended): Backend `/api/tasks/route.ts` — server-side ORDER BY ajustado per param. Pro: cursor pagination continua válida (cursor encoding atualizado para reflectir sort field); índices DB existentes cobrem (`tasks_due_date_idx`, `tasks_priority_idx`, `tasks_created_at_idx` se existir).
  - **B**: Client-side sort após fetch. Contra: quebra cursor pagination, performance ruim em 100+ items, perde RSC benefit.
  - **Recomendação A.** Risk: cursor encoding precisa mudar se sort field não é `due_date`. Mitigação: cursor stays `{ last_due_date, last_id }` mas WHERE clause adapta-se. Aceitar limitation: ordenação por priority/title/created_at significa "Mostrar mais" pode retornar cursor sub-óptimo no boundary — KISS para M scope, refactor cursor multi-field em Epic 4+ housekeeping.

(c) URL-state pattern idêntico a AC3 — `?sort=due_date_asc` parsed por RSC, passado a handler GET.

**AC4 PASS:** 4 opções de sort funcionais. Endpoint `/api/tasks` aceita `?sort=` e devolve ordering correcto. Cursor pagination preservada para `sort=due_date_asc` (default). Outros sorts: cursor pagination pode degrade mas não quebra (UX: até 50 items per página é suficiente — limitation documentada em Project Structure Notes).

### 7. Bulk actions bar — DP1-3.3

**AC7:** Criar `_components/BulkActionsBar.tsx` (client) que:

(a) Estado multi-select: cada `<TaskRow>` tem `<Checkbox>` (visível em hover OU sempre em mobile). Seleccionar 1+ → barra aparece sticky-bottom com:
  - Count badge "N tarefas selecionadas".
  - Button "✓ Marcar como concluídas".
  - Button "🗑 Eliminar" (com confirm dialog PT-PT "Tens a certeza que queres eliminar N tarefas? Esta acção é irreversível." — atenção: o backend faz soft delete `status='archived'`, mas UX label "eliminar" é OK porque user não vê archived sem filter explícito; Project Structure Notes documenta semantic mismatch como acceptable per DP2-3.2 ratificado @architect 9.3/10).
  - Button "Mudar prioridade ▾" → dropdown com Alta / Média / Baixa.
  - Button "Cancelar selecção".

(b) **DP1-3.3 — Bulk action execution strategy:**
  - **A (recommended):** Parallel `Promise.all(taskIds.map(id => fetch(\`/api/tasks/\${id}\`, { method: 'PATCH', body: ... })))` client-side. KISS. Sem novo endpoint server. Aproveita audit_log per-row (Story 3.2 AC10 — cada PATCH gera 1 audit row). Partial failure: contar 200s vs erros, toast PT-PT "8 de 10 tarefas atualizadas. Reverter as 2 falhadas?". Não há all-or-nothing semantic (acceptable para "marcar concluído" / "mudar prioridade"; para "eliminar" usar mesmo pattern — risk: 8 archived + 2 falham. UX: toast retry).
  - **B (alternative):** Novo endpoint `POST /api/tasks/bulk` com `{ action: 'complete'|'delete'|'priority', task_ids: uuid[], new_priority?: enum }` em transacção atómica. Pro: all-or-nothing + 1 audit row aggregate. Contra: scope creep ~+15% Story 3.3 (novo route handler + Zod schema + tests + audit_action enum new value `task.bulk_*` — migration 0011 obrigatória).
  - **Recommendation A.** Justificação: M scope tight, audit_log per-row é mais granular para FR21 user transparency, "marcar concluído 10 tarefas" tipicamente tem 0 falhas (RLS já garantida — só user vê próprio household). Eurico/Pax podem ratificar A silently OR pedir B se quiserem atomicidade.

(c) Loading state: durante bulk action, BulkActionsBar mostra `<Spinner>` + count progress "3 de 10..." (Promise.all incremental progress via individual promise.then). Reactivar bar ao final.

(d) Optimistic UI: mark concluído → row anima fade + strikethrough imediato + toast "Concluída." com botão "Anular" 5s. Se PATCH falha em 1 row, revert visual dessa row + toast erro.

**AC7 PASS:** Selecionar 3 tarefas → BulkActionsBar aparece → click "Marcar concluídas" → 3 paralelos PATCH → 3 audit rows novas → toast PT-PT success/partial. Tests unit BulkActionsBar com mocks `fetch` (cobrindo full success, partial failure, all-failure).

```tsx
// apps/web/src/app/(app)/tarefas/page.tsx (RSC esqueleto pseudo-code)
import { createServerSupabaseClient } from '@meu-jarvis/auth/server';
import { captureException } from '@meu-jarvis/observability';
import { getDb } from '@/lib/agent/db-shim';
import { TaskFiltersSchema } from '@/lib/api-schemas/tasks';
import { listTasksHelper } from '@/lib/api-helpers/list-tasks';  // T2.1
import { requireAuth } from '@/lib/api-helpers/auth';

import { ViewTabs } from './_components/ViewTabs';
import { TaskFilters } from './_components/TaskFilters';
import { TaskSort } from './_components/TaskSort';
import { TaskList } from './_components/TaskList';
import { EmptyState } from './_components/EmptyState';

interface PageProps {
  searchParams: Promise<Record<string, string | undefined>>;
}

export default async function TarefasPage({ searchParams }: PageProps) {
  // Auth (middleware já garante user, mas resolveHouseholdId pode falhar)
  const supabase = await createServerSupabaseClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) return <EmptyState variant="error" />;  // defensive

  // Parse query params via Zod
  const rawParams = await searchParams;
  const parsed = TaskFiltersSchema.safeParse(rawParams);
  if (!parsed.success) return <EmptyState variant="error" />;

  // Fetch tasks server-side via helper extraído (DP4-3.3: A)
  let result;
  try {
    result = await listTasksHelper({
      filters: parsed.data,
      userId: user.id,
      db: getDb(),
    });
  } catch (err) {
    captureException(err instanceof Error ? err : new Error(String(err)), {
      userId: user.id,
      route: '/tarefas',
    });
    return <EmptyState variant="error" />;
  }

  const isEmpty = result.tasks.length === 0;
  const hasFilters = Object.keys(parsed.data).some(k => parsed.data[k] != null && k !== 'limit' && k !== 'sort');

  return (
    <div className="space-y-6">
      <header className="flex items-center justify-between">
        <h1 className="text-2xl font-bold">Tarefas</h1>
        <button className="rounded-md bg-blue-600 px-3 py-1.5 text-white">+ Nova</button>
      </header>
      <ViewTabs current="lista" />
      <div className="flex gap-4">
        <TaskFilters />
        <TaskSort />
      </div>
      {isEmpty ? (
        <EmptyState variant={hasFilters ? 'filtered-empty' : 'no-tasks'} />
      ) : (
        <TaskList tasks={result.tasks} cursor={result.next_cursor} />
      )}
    </div>
  );
}
```

### Sections Logic — `_lib/task-sections.ts`

```typescript
// apps/web/src/app/(app)/tarefas/_lib/task-sections.ts (pseudo)
import { startOfDay, addDays, differenceInCalendarDays } from 'date-fns';
import { utcToZonedTime } from 'date-fns-tz';

import type { TaskRow } from '@/app/api/tasks/route';

export type SectionKey =
  | 'overdue'
  | 'today'
  | 'tomorrow'
  | 'this_week'
  | 'later'
  | 'no_due_date'
  | 'completed_today';

export interface SectionGroup {
  key: SectionKey;
  label: string;        // PT-PT
  count: number;
  tasks: TaskRow[];
  variant?: 'danger' | 'success' | 'default';
}

const TZ = 'Europe/Lisbon';

export function groupTasksBySections(
  tasks: readonly TaskRow[],
  now: Date = new Date(),
): SectionGroup[] {
  const todayLisbon = startOfDay(utcToZonedTime(now, TZ));
  const tomorrowLisbon = addDays(todayLisbon, 1);
  const weekEnd = addDays(todayLisbon, 6);

  const buckets: Record<SectionKey, TaskRow[]> = {
    overdue: [],
    today: [],
    tomorrow: [],
    this_week: [],
    later: [],
    no_due_date: [],
    completed_today: [],
  };

  for (const task of tasks) {
    if (task.status === 'archived') continue;  // Arquivadas escondidas

    // Concluídas hoje (status==='done' + completed_at >= todayLisbon)
    if (task.status === 'done') {
      if (task.completed_at) {
        const completedLisbon = utcToZonedTime(new Date(task.completed_at), TZ);
        if (completedLisbon >= todayLisbon) {
          buckets.completed_today.push(task);
        }
      }
      continue;
    }

    if (!task.due_date) {
      buckets.no_due_date.push(task);
      continue;
    }

    const dueLisbon = utcToZonedTime(new Date(task.due_date), TZ);
    const diffDays = differenceInCalendarDays(dueLisbon, todayLisbon);

    if (diffDays < 0) buckets.overdue.push(task);
    else if (diffDays === 0) buckets.today.push(task);
    else if (diffDays === 1) buckets.tomorrow.push(task);
    else if (diffDays <= 6) buckets.this_week.push(task);
    else buckets.later.push(task);
  }

  const sections: SectionGroup[] = [];

  // Atrasadas SEMPRE primeiro se non-empty (FR11)
  if (buckets.overdue.length) {
    sections.push({
      key: 'overdue',
      label: `Atrasadas (${buckets.overdue.length})`,
      count: buckets.overdue.length,
      tasks: buckets.overdue,
      variant: 'danger',
    });
  }

  // Demais sections em ordem fixa
  const orderedBuckets: Array<{ key: SectionKey; labelFn: () => string }> = [
    { key: 'today', labelFn: () => `Hoje · ${formatPT(todayLisbon)}` },
    { key: 'tomorrow', labelFn: () => `Amanhã · ${formatPT(tomorrowLisbon)}` },
    { key: 'this_week', labelFn: () => 'Esta semana' },
    { key: 'later', labelFn: () => 'Mais tarde' },
    { key: 'no_due_date', labelFn: () => 'Sem prazo' },
  ];

  for (const { key, labelFn } of orderedBuckets) {
    if (buckets[key].length) {
      sections.push({
        key,
        label: labelFn(),
        count: buckets[key].length,
        tasks: buckets[key],
      });
    }
  }

  // Concluídas hoje em último (collapsed por defeito no UI)
  if (buckets.completed_today.length) {
    sections.push({
      key: 'completed_today',
      label: `Concluídas hoje (${buckets.completed_today.length})`,
      count: buckets.completed_today.length,
      tasks: buckets.completed_today,
      variant: 'success',
    });
  }

  return sections;
}

function formatPT(date: Date): string {
  // PT-PT format DD/MM/YYYY
  return `${String(date.getDate()).padStart(2, '0')}/${String(date.getMonth() + 1).padStart(2, '0')}/${date.getFullYear()}`;
}
```
