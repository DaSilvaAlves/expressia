## User Story

**Como** utilizador autenticado da Expressia,
**Quero** poder remover widgets que não quero ver na `/visao` (com um `×` no canto de cada um) e voltar a adicionar widgets disponíveis (com um botão `[+ Adicionar widget]`), sem sair do dashboard,
**Para que** a minha Visão mostre só o que me interessa — e que essa escolha fique guardada e seja respeitada quando volto a entrar, em qualquer dispositivo.

### AC1 — Endpoint `/api/conta/preferencias` estendido para `widgets_enabled`

**AC1:** Refactor de `apps/web/src/app/api/conta/preferencias/route.ts` para suportar `widgets_enabled` em **GET** e **PATCH**, reutilizando os schemas partilhados da Story 5.1 (sem duplicação).

**(a) PATCH** passa a validar o body com `PreferencesPatchSchema` (de `@/lib/api-schemas/preferences`) em vez do `PatchBodySchema` local. Aceita `{ always_preview?, theme?, widgets_enabled? }` — todos opcionais, `.strict()`. O UPSERT escreve apenas os campos presentes no body (não sobrescrever `always_preview` quando só vem `widgets_enabled`, e vice-versa). Retorna o estado actualizado dos campos relevantes.
**(b) GET** passa a devolver `{ always_preview, widgets_enabled }` (lazy-init UPSERT mantido; `widgets_enabled` cai para `DEFAULT_WIDGETS_ENABLED` da DB se a row foi recém-criada — já é o default da coluna). Validar o JSONB lido com `WidgetsEnabledSchema.safeParse` e fallback `DEFAULT_WIDGETS_ENABLED` (precedente `page.tsx:53-54`).
**(c)** RLS: **`getDb()`** (role `authenticated`, RLS via JWT) — **NUNCA `getServiceDb()`** (NFR5). 401 sem auth, 404 sem household, 400 body inválido, 500 erro — mantidos do handler actual.
**(d)** Retrocompatibilidade: o `prefs-toggle.tsx` (Story 2.7) que envia `{ always_preview }` continua a funcionar sem alteração.

**AC1 PASS:** Testes de route handler (extensão de `route.test.ts`): PATCH `{ widgets_enabled }` válido → 200 + persiste; PATCH `{ always_preview }` (legacy) → 200 inalterado; PATCH `{ widgets_enabled }` com chave inválida/em falta → 400 (`.strict()` + 7 chaves obrigatórias); GET → 200 com `{ always_preview, widgets_enabled }`; 401 sem auth. RLS via `getDb()` confirmado por inspecção (grep negativo `getServiceDb`).

### AC4 — Persistência debounced + sobrevive a refresh + RLS isolada

**AC4:** As mudanças persistem em `user_prefs.widgets_enabled` via PATCH debounced (DP-5.7.D) e são respeitadas no recarregamento da página; o isolamento multi-tenant é garantido pela RLS (utilizador A não vê nem altera widgets de B).

**(a)** Debounce: toggles consecutivos (< janela) agrupam-se num único PATCH com `widgets_enabled` completo (7 chaves). Um `flush` no `beforeunload`/`visibilitychange` **opcional** [DEV-DECISION] para não perder a última mudança se o utilizador sai cedo (recomendado, não bloqueante).
**(b)** Refresh: recarregar `/visao` reflecte o estado persistido (o RSC `readWidgetsEnabled` já lê da DB — sem alteração necessária; AC1 garante consistência GET/PATCH).
**(c)** RLS: PATCH/GET via `getDb()`; o predicado RLS de `user_prefs` (via JWT `household_id` + `user_id` PK) garante que cada utilizador só lê/escreve a sua row.

**AC4 PASS:** Teste de debounce (fake timers): 3 toggles em < janela → 1 PATCH com o estado final. Teste/assert RLS: o handler usa `getDb()` (não `getServiceDb()`); a query escreve `where user_id = <auth user>` (UPSERT por PK `user_id`). ### AC6 — Estado "todos OFF" mantém `[+ Adicionar widget]` acessível

> **[PO-FIX-1 — correcção factual contra o comportamento real da 5.6]** A versão original desta AC afirmava que "todos OFF mostra o `<VisaoEmptyState>`". **Isto é incorrecto:** a heurística `isVisaoEmpty()` da 5.6 (`page.tsx:82-120`) faz `if (checks.length === 0) return false` — com **todos os widgets OFF não há `checks`**, logo `empty = false` e a `/visao` renderiza `<WidgetGrid widgetsEnabled={todos-false}>` que produz uma **grid vazia** (`enabledIds = []`), **não** o `<VisaoEmptyState>`. O `<VisaoEmptyState>` é o estado "vazio **por dados**" (widgets ON mas agregados a zero). A AC abaixo foi reformulada para reflectir o comportamento real.

**AC6:** No estado "todos os widgets OFF" (grid vazia — distinto de "vazio por dados"), o utilizador **tem de conseguir voltar a adicionar widgets**: o `[+ Adicionar widget]` (AC3) **permanece sempre acessível**, incluindo quando a grid está vazia (caso contrário o utilizador fica sem saída).

**(a)** O `[+ Adicionar widget]` renderiza **independentemente** do estado da grid (não está condicionado a haver widgets visíveis). Sempre que houver ≥ 1 widget OFF, está activo e lista os widgets OFF (AC3.b).
**(b)** **[DEV-DECISION]** mensagem opcional no estado "grid vazia por config" (ex.: um aviso curto PT-PT "O teu painel está vazio. Adiciona widgets abaixo." junto do `[+ Adicionar widget]`). Recomendado para não deixar a área de conteúdo completamente em branco, mas **não-bloqueante** (o acesso ao `[+ Adicionar]` é o requisito duro). **NÃO** reutilizar o `<VisaoEmptyState>` (5.6) aqui — esse é semanticamente "vazio por dados" e a sua copy ("Carrega no chat…") não se aplica a "removi todos os widgets".
**(c)** Não regredir a heurística `isVisaoEmpty` da 5.6 (`page.tsx:82-120`) — esta story não altera a semântica de "vazio por dados".

**AC6 PASS:** Teste: com `widgets_enabled` todos `false`, a `/visao` renderiza um caminho visível e funcional para `[+ Adicionar widget]` que lista os 7 widgets; o `VisaoEmptyState` (vazio por dados) permanece inalterado para o seu caso original.

## Dev Notes

**Fronteira RSC↔Client (núcleo da story).** A `/visao` (`page.tsx`) é RSC e renderiza `<WidgetGrid>` (RSC) com widgets RSC em `<Suspense>`. Os controlos de config são interactivos → **Client**. RSC pode renderizar Client Components, mas o estado partilhado (todos os `×` + o `[+ Adicionar]` + debounce) vive no client (DP-5.7.B = Zustand store, precedente `shellStore`/`chatStore`). Padrão recomendado (DP-5.7.A = A):
- **Remover (ON→OFF):** optimistic — o store esconde o widget já; PATCH debounced persiste. (Não requer `router.refresh()`.)
- **Adicionar (OFF→ON):** o widget RSC não veio no HTML → após PATCH success, `router.refresh()` re-renderiza a grid server-side com o widget novo.

**Endpoint — cuidado com o UPSERT parcial (AC1.a).** O handler actual escreve sempre `always_preview`. Com `PreferencesPatchSchema` (campos opcionais), o UPSERT tem de escrever **só** os campos presentes no body — senão um PATCH de `widgets_enabled` zeraria `always_preview`. Construir o `set` do `ON CONFLICT DO UPDATE` dinamicamente (ou 1 statement por campo presente). `widgets_enabled` é JSONB — passar via `sql` param com cast `::jsonb` e `JSON.stringify`. Lazy-init INSERT mantém os defaults da coluna (`DEFAULT_WIDGETS_ENABLED` + `always_preview=false`).

**REQ-INLINE-1 (precedente 5.5/5.6).** Em `apps/web`, importar o **cliente** DB de `@meu-jarvis/db` parte a resolução `@/schema` no tsc. Só **imports de tipo** são seguros (`import type { WidgetId, WidgetsEnabled }`). Valores (`DEFAULT_WIDGETS_ENABLED`, `WIDGET_ORDER`, `WIDGET_LABELS`) vivem em `_lib/widgets.ts` (local, com parity guard de tipo contra `WidgetsEnabled`/`WidgetId`).

**Precedente de padrão optimistic + PATCH + banner:** `apps/web/src/app/(app)/conta/preferencias/_components/prefs-toggle.tsx` (optimistic, revert em erro, banner "Guardado."/"Erro" auto-clear 3s, `role`/`aria-label`). Reutilizar a microcopy e a estrutura de banner.

**Schemas já prontos (5.1):** `WidgetsEnabledSchema.strict()` (7 chaves obrigatórias) + `PreferencesPatchSchema.strict()` (`always_preview?`, `theme?`, `widgets_enabled?`) em `apps/web/src/lib/api-schemas/preferences.ts`. Reutilizar — **não** redeclarar.

**Slot já reservado (5.6):** `WidgetCard.headerActions?: React.ReactNode` (`WidgetCard.tsx:30`) — renderizado no header (`:48`). Esta story preenche-o.

**RLS / `getDb()`:** `user_prefs` tem PK `user_id` + `household_id` FK + 4 RLS policies (Story 5.1). PATCH/GET via `getDb()` (authenticated) — o predicado RLS via JWT garante isolamento. **Nunca `getServiceDb()`** nesta story (NFR5).

**Audit log gotcha (D36, herdado 2.7):** o enum `audit_action` não tem `user_prefs.updated`; o handler actual loga via Pino (não insere audit row). Manter esse padrão — não adicionar enum value (exigiria migration, fora de scope).

**Não inventar (Article IV):** labels = títulos actuais byte-a-byte; widgets/ordem/defaults = fontes existentes; nenhum widget novo; nenhuma opção de config além de on/off.
