## User Story

**Como** utilizador autenticado de um household que acabou de submeter um prompt no `/jarvis` e viu uma mensagem "Feito ✓" no chat,
**Quero** clicar num botão **Anular** visível durante 30 segundos após a execução (com contador decrescente claro), receber confirmação imediata de que a operação foi revertida (ou mensagem PT-PT clara se já expirou) e ver o sistema limpar automaticamente registos antigos de reverse_ops para garantir que a base de dados não cresce indefinidamente,
**Para que** eu tenha controlo total sobre escritas irreversíveis do agente AI (FR6) e confie em deixá-lo executar prompts sem confirmação intermédia (NFR12 user trust + R5 mitigação "undo race condition").

## [AUTO-DECISIONS] @sm

### [AUTO-DECISION D38] Inngest cleanup cron diário 03:00 UTC alinhado com `/api/cron/daily`

**Decisão:** Inngest function `cleanupExpiredReverseOps` corre via cron Inngest nativo `0 3 * * *` (diário 03:00 UTC), NÃO via webhook chamado por Vercel Cron `/api/cron/daily`.

**Justificação:**

1. Arquitectura ADR-005 (§14.5) decide: "Vercel Cron mantém-se para o tick diário simples que dispara todos os jobs Inngest (`/api/cron/daily`)". Implementação MVP segue convenção mais simples — cron Inngest directo per Inngest docs Next.js quickstart.
2. Vercel Cron alternativa requer endpoint adicional `/api/cron/daily` (já mencionado em CLAUDE.md `apps/web/vercel.json`) que despacha `inngest.send({ name: 'jobs/daily-tick' })` — adiciona indireccão sem ganho funcional para MVP de 1 job.
3. Quando Story 2.9 (cost router quota reset mensal) e Story GDPR purge (mensal) entrarem, fan-out via `/api/cron/daily` faz sentido. Story 2.8 estabelece o pattern de cron Inngest nativo; refactor posterior é trivial.
4. **Risco:** Inngest cron pricing — free tier permite 30 functions, MVP usa 1. Sem risco financeiro.

**Trace:** Architecture §11.3, ADR-005 §14.5, CLAUDE.md menção cron 03:00 UTC.

### [AUTO-DECISION D39] Inngest client/functions/serve em `apps/web/src/lib/inngest/` (não package separado)

**Decoração:** Inngest setup inline em `apps/web/src/lib/inngest/` (client.ts + functions/*.ts) em vez de criar `packages/jobs/` package separado.

**Justificação:**

1. Convenção monorepo actual: `apps/web/` consume packages internos via `@meu-jarvis/*`; mas Inngest functions tendem a ser específicas de app (não lógica partilhada).
2. KISS para MVP — 1 função actualmente; package overhead (package.json, tsconfig, build) não justificado.
3. Stories futuras (GDPR purge, recurrence generation, Stripe webhook retry) podem juntar-se a `apps/web/src/lib/inngest/functions/` sem refactor.
4. Eventual extracção para `packages/jobs/` quando ≥5 functions OU partilha cross-app (improvável MVP).
5. Pattern análogo a `apps/web/src/lib/agent/*` (Story 2.6 — db-shim, redaction, tracing inline em app sem package).

**Trace:** Codebase pattern actual `apps/web/src/lib/agent/*`, Story 2.6 D27 cross-package imports.

### [AUTO-DECISION D40] UI Undo button retrocompatibilidade — render sem button se `undoUrl` undefined

**Decisão:** ResultMessage SEM props `undoUrl`/`undoExpiresAt` (retrocompatibilidade pre-2.8 — chamadas Story 2.7 ou tests legacy) renderiza o card de resultado **sem** botão Undo (zero render, não placeholder disabled). Tests Story 2.7 existentes (`result-message.test.tsx` 5 testes) DEVEM continuar a passar sem alteração.

**Justificação:**

1. Tests Story 2.7 não passam `undoUrl` — se botão disabled placeholder persistir, asserts negativos (ex: "botão Anular não está visível") quebram.
2. Alternativa: manter placeholder mas com texto neutro tipo "Sem undo disponível" — mais confuso para utilizador real.
3. Production caller principal (`jarvis-chat.tsx`) SEMPRE passa `undoUrl`/`undoExpiresAt` da response do endpoint executed branch. Edge case "undo_url ausente" deve ser visível em telemetria, não fallback silencioso.
4. Tests novos Story 2.8 (AC12 testes (iii)) explicitamente cobrem este caso retrocompat.

**Trace:** Story 2.7 result-message.test.tsx (5 testes preservados), AC7 button states.

## Dev Notes

### DN1 — Inngest dev workflow

Sem keys: Inngest functions registadas mas `inngest.send()` falham silenciosamente em local OU geram erro `INNGEST_EVENT_KEY missing`. Para dev sem keys, alternativa é `npx inngest-cli dev -u http://localhost:3000/api/inngest` que cria local Inngest engine + dashboard em `localhost:8288` sem cloud. **Tests Vitest devem mockar completamente — nunca chamar Inngest engine real em CI.**

### DN2 — Mais um setIInterval em produção vs requestAnimationFrame

Countdown 30s pode ter precision drift ±100ms com setInterval(1000) — aceitável para UX (user vê "29s, 28s, ...; 1s; Expirou"). Não usar requestAnimationFrame para economia bateria/CPU (overkill para countdown coarse).

### DN3 — `app/(app)/` route group middleware

### DN5 — `getServiceDb()` vs `getDb()` no audit_log INSERT
