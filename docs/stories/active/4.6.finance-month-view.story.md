## User Story

**Como** utilizador da Expressia que gere as finanças do agregado familiar (transacções variáveis, despesas recorrentes, compras parceladas),
**Quero** uma vista "Este mês" que me mostre, para o mês escolhido, quanto entrou e quanto saiu, a divisão das despesas por categoria e por dia, e uma projecção do que ainda vou pagar/receber nos próximos 30 dias incluindo recorrentes e prestações,
**Para que** eu consiga responder à pergunta "quanto me sobra no fim do mês" — o diferenciador que o Néctar não resolve — sem ter de somar nada à mão, com todos os valores em formato PT-PT correcto (`€1.234,56`).

### AC3 — Helper `month-summary.ts` — agregação real do mês

Criar `apps/web/src/lib/finance/month-summary.ts` — função pura de leitura, testável (recebe `db` injectável, como `listTasksHelper` da Story 3.3):

```ts
export interface MonthSummaryInput {
  readonly db: DbShim; // getDb() injectado pelo RSC
  readonly monthStart: string; // YYYY-MM-DD — primeiro dia do mês visualizado
  readonly monthEnd: string;   // YYYY-MM-DD — último dia do mês visualizado
}
```

A função `getMonthSummary` retorna `totalIncomeCents`, `totalExpenseCents`, `netCents`, `byCategory` (ordenado desc), `byDay`.

Regras das queries (via `getDb()` — RLS authenticated, household scoping automático):

- **Apenas transacções reais do mês:** `WHERE transaction_date >= monthStart AND transaction_date <= monthEnd AND is_projected = false`. A projecção é tratada à parte (AC4).
- **Totais por `kind` (R-4.10 — invariante crítico):** `SUM(amount_cents) FILTER (WHERE kind='income')` e `FILTER (WHERE kind='expense')`. **`transfer` NUNCA entra** em `totalIncomeCents`/`totalExpenseCents`/`netCents` (D-4.6.5 — movimento interno entre contas do household, não é dinheiro que entrou/saiu do agregado).
- **`byCategory`:** `GROUP BY category_id, kind` com `LEFT JOIN categories` para o nome; `category_id NULL` → `'Sem categoria'`. Ordenar por `totalCents DESC`.
- **`byDay`:** `GROUP BY transaction_date` com `expenseCents`/`incomeCents` via `FILTER`. Só dias com movimento.
- Índices a explorar: `transactions_date_range_idx`, `transactions_category_idx`, `transactions_kind_idx` (todos já existem — verificação #2).

**AC3 PASS:** `month-summary.ts` criado · `getMonthSummary` retorna totais por `kind` + breakdown categoria/dia · `transfer` excluído dos totais · `db` injectável.

### AC4 — Projecção 30 dias (DP2=C híbrido)

Criar `apps/web/src/lib/finance/month-projection.ts` — helper que calcula a projecção dos **próximos 30 dias a contar de hoje** (FR18; janela `today` .. `today + 30d` inclusive). `getMonthProjection` retorna `windowEnd`, `items` (ordenado por data asc, cada um com `source: 'recurrence' | 'installment'`), `projectedIncomeCents`, `projectedExpenseCents`.

- **Prestações (materializadas — DP8=A):** `SELECT` de `transactions WHERE is_projected = true AND installment_id IS NOT NULL AND transaction_date >= today AND transaction_date <= windowEnd`. Ler directamente — as prestações já foram geradas como rows pela Story 4.4. `source = 'installment'`. Índice: `transactions_projected_idx`.
- **Recorrências (on-the-fly — DP2=C):** `SELECT` de `recurrences WHERE active = true`. Para cada recorrência, iterar ocorrências a partir de `next_run_on` (ou `starts_on` se `next_run_on` NULL) usando `calcNextRunDate` (Story 4.5), acumulando cada data que caia dentro de `[today, windowEnd]`. Parar quando a data ultrapassa `windowEnd` ou `calcNextRunDate` retorna `null` (recorrência esgotada). `source = 'recurrence'`. **Limite defensivo:** máximo 60 iterações por recorrência — D-4.6.6.
- **Sem dupla contagem:** a transacção do dia corrente já materializada pelo cron (Story 4.5, `is_projected=false`) entra no **real** (AC3), não na projecção. A projecção começa estritamente nas ocorrências **futuras** (`date > today` para recorrências — critério estrito; prestações usam as rows `is_projected=true`). Edge case: se o cron ainda não correu hoje (corre às 03:00 UTC), a ocorrência de **hoje** de uma recorrência não aparece nem no real (o cron ainda não a materializou) nem na projecção (`date > today` exclui o dia corrente) — fica **momentaneamente invisível** até ao run do cron. É aceitável no MVP e **não há risco de dupla contagem** — o critério estrito `> today` garante-o. Documentar este comportamento em Dev Notes.

  > **[PO_FIX_INLINE @po 2026-05-22]** A versão v1.0 da AC4 afirmava "a ocorrência de hoje aparece na projecção", o que contradiz o critério estrito `date > today` da própria AC4 e o pseudocódigo de Dev Notes (`if (cursor > today)`). Resolvido: com `> today` estrito, a ocorrência de hoje **nunca** entra na projecção; se o cron ainda não correu, fica invisível até ~03:00 UTC. Comportamento correcto e sem dupla contagem — apenas a redacção estava errada. @dev: implementar `cursor > today` estrito conforme o pseudocódigo de Dev Notes (fonte de verdade).

- `transfer` é incluído na lista `items` mas **não** em `projectedIncomeCents`/`projectedExpenseCents` (coerência com D-4.6.5).

**AC4 PASS:** `month-projection.ts` criado · prestações lidas materializadas · recorrências projectadas via `calcNextRunDate` · janela 30d · sem dupla contagem com o real.

### AC5 — Navegação de meses (`?mes=YYYY-MM` — DP3=B)

- A página aceita o search param `?mes=YYYY-MM`. Sem param → mês corrente. Param malformado → fallback ao mês corrente (defensivo, sem 400 — é uma página, não API).
- Validar `mes` com regex `^\d{4}-(0[1-9]|1[0-2])$`. `monthStart = startOfMonth(parseISO(mes-01))`, `monthEnd = endOfMonth(...)` via `date-fns`.
- **Navegação livre (DP3=B):** componente client `MonthNavigation` com setas anterior/seguinte que actualizam `?mes=` via `useRouter` + `useSearchParams` (padrão `WeekNavigation` da Story 3.5). Passado e futuro sem limite.
- **Projecção só no contexto relevante (D-4.6.4):** a secção de projecção 30d (AC4) é renderizada **apenas quando o mês visualizado é o mês corrente**. Para meses passados ou futuros, mostrar só as agregações reais (AC3). Quando escondida, mostrar nota discreta.
- Label do mês em PT-PT: `format(monthStart, 'MMMM yyyy', { locale: pt })` → `"maio 2026"`.

**AC5 PASS:** `?mes=YYYY-MM` navegável passado/futuro · fallback defensivo · projecção só no mês corrente · label PT-PT.

### [AUTO-DECISION D-4.6.1] `MoneyDisplay` + `_components` co-localizados no route-group `financas/`, sem `packages/ui`

**Decisão:** `MoneyDisplay` e todos os componentes da vista vivem em `apps/web/src/app/(app)/financas/_components/`. NÃO se cria `packages/ui`.

**Justificação:** (1) `architecture.md §8.4` prevê `packages/ui/components/MoneyDisplay.tsx` mas o package nunca foi materializado; (2) o Epic 3 inteiro co-localizou componentes em `app/(app)/tarefas/_components/` — precedente real e consistente; (3) criar um package partilhado é decisão de arquitectura transversal, scope creep para uma story de vista. As vistas 4.7-4.9 reusam `MoneyDisplay` por import absoluto. Discrepância registada em Project Structure Notes.

### [AUTO-DECISION D-4.6.2] Fetch direct via `getDb()` no RSC — sem rota HTTP loopback

**Decisão:** A página faz fetch server-side directo via helpers com `getDb()`, não via `fetch('/api/financas/...')`.

**Justificação:** Precedente ratificado HIGH na Story 3.3 (DP4-3.3 A). Zero latência de loopback HTTP. RLS aplicada por `getDb()` (role `authenticated` + JWT).

### [AUTO-DECISION D-4.6.3] Projecção de recorrências reusa `calcNextRunDate` da Story 4.5

**Decisão:** `getMonthProjection` itera ocorrências chamando `calcNextRunDate` (Story 4.5).

**Justificação:** Função pura já testada (16 testes); reimplementar arriscaria divergência entre o que o cron materializa e o que a vista projecta.

### [AUTO-DECISION D-4.6.4] Secção de projecção 30d só no mês corrente

**Decisão:** A projecção dos próximos 30 dias só é renderizada no mês corrente.

**Justificação:** FR18 define a projecção ancorada em `hoje`; sobrepô-la a um mês passado/futuro é incoerente.

### [AUTO-DECISION D-4.6.5] `transfer` excluído dos totais entrado/saído e do saldo

**Decisão:** `kind='transfer'` não entra em `totalIncomeCents`/`totalExpenseCents`/`netCents` nem nos subtotais projectados.

**Justificação:** Transferência é movimento interno do household; incluí-la inflaccionaria "entrou" e "saiu" em simultâneo.

### [AUTO-DECISION D-4.6.6] Limite defensivo de 60 iterações por recorrência na projecção

**Decisão:** O loop de `calcNextRunDate` por recorrência tem teto de 60 iterações.

**Justificação:** Janela 30d; `daily` produz ≤31 ocorrências. 60 é folga 2× contra loop infinito. Atingir o teto → `childLogger.warn`.

### [AUTO-DECISION D-4.6.7] Helpers `month-summary`/`month-projection` separados e `db`-injectáveis

**Decisão:** Duas funções em ficheiros separados, ambas recebendo `db` por argumento.

**Justificação:** Separação por responsabilidade; `db` injectável torna-as testáveis sem mock de módulo. `month-summary.ts` reutilizável pela tool `query_finance_summary` (Story 4.10).

### [AUTO-DECISION D-4.6.8] `FinanceViewTabs` rende 5 tabs; 4 desactivadas até 4.7-4.9

**Decisão:** `FinanceViewTabs` mostra as 5 vistas; só "Este mês" é `<Link>`, as outras 4 são `<span>` desactivados.

**Justificação:** Linkar para rotas inexistentes daria 404. Mostrar as 5 labels comunica o mapa do módulo sem rotas mortas.

## [DEV-DECISIONS] @dev

### [DEV-DECISION D-4.6.9] `MoneyDisplay` — `Intl.NumberFormat('pt-PT')` para a magnitude + prefixo `€` manual

**Decisão:** `MoneyDisplay`/`formatEuroCents` usam `new Intl.NumberFormat('pt-PT', { minimumFractionDigits: 2, maximumFractionDigits: 2 })` para formatar a magnitude e prefixam `€` manualmente — produzindo `€1.234,56`.

**Justificação:** A AC2 (v1.0) sugeria `Intl.NumberFormat('pt-PT', { style: 'currency', currency: 'EUR' })`. Esse estilo em `pt-PT` emite o símbolo **à direita** (`1.234,56 €`), o que contradiz o formato de marca mandatado pelo `CLAUDE.md` e pela memória de design (`€8,88`, `€1.234,56` — símbolo à esquerda). A solução usa o engine `Intl.NumberFormat('pt-PT')` (separadores de milhar/decimal locale-correctos — espírito da arquitectura §8.4) e compõe o símbolo à esquerda (binding do `CLAUDE.md`). `@architect` ratifica no gate.

### Projecção on-the-fly de recorrências (AC4, D-4.6.3)

```ts
let cursor = recurrence.nextRunOn ?? recurrence.startsOn;
for (let i = 0; i < 60; i++) {            // teto D-4.6.6
  if (cursor > windowEnd) break;
  if (cursor > today) items.push({ date: cursor, ... source: 'recurrence' });
  const next = calcNextRunDate(recurrence, cursor);
  if (next === null) break;               // esgotada (endsOn)
  cursor = next;
}
```

`cursor > today` (estrito) evita dupla contagem com a transacção do dia já materializada pelo cron (R-4.6.3).
