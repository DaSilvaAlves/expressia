| Endpoint | Métodos | Comportamento |

### 2. Endpoint drag-and-drop atómico — `/api/tasks/[id]/move`

**AC2:** Criar route handler `PATCH /api/tasks/[id]/move` que:

(a) Aceita body Zod: `{ kanban_column_id: uuid | null, kanban_position: integer >= 0 }`.

(b) Executa em **transacção atómica** (BEGIN/COMMIT/ROLLBACK):
  1. Verifica que `kanban_column_id` (se não-null) pertence ao household do user (RLS via `kanban_columns_household_idx` query).
  2. Se `kanban_column_id` mudou (cross-column move): rebalanceia `kanban_position` da coluna alvo via UPDATE sibling rows `WHERE kanban_position >= new_position AND id != taskId` SET `kanban_position = kanban_position + 1`. Pattern "shift right".
  3. Se `kanban_column_id` igual ao actual (intra-column reorder): rebalanceia siblings na mesma coluna.
  4. UPDATE task com `kanban_column_id` + `kanban_position` + `updated_at = now()`.
  5. Span OTel `agent.tasks.move` com attrs `task_id_hash` + `from_column_hash` + `to_column_hash` + `position_delta` (sem PII per NFR12).

(c) Response 200 com task atualizada. 400 se body inválido. 404 se task não existe ou cross-household. 409 (Conflict) se constraint unique violada (race condition entre 2 users — toast UI revert).

(d) Optimistic UI compatibility (DP7 ED11 ratificado): endpoint é idempotente em re-tentativas — mesma move 2x produz mesmo estado final.

**AC2 PASS:** Endpoint atómico + Zod strict + RLS protegida + tests integration cobrem race condition entre 2 users.

### 6. Pagination cursor-based + filters compostos

**AC6:** `GET /api/tasks` implementa pagination cursor-based:

- Query param `cursor` (opcional): opaque string base64-encoded de `{ last_due_date: ISO_date | null, last_id: uuid }`.
- Query param `limit` (opcional, default 50, max 100).
- Response inclui `next_cursor: string | null` (null se sem mais resultados).
- Query SQL: `WHERE (due_date, id) > (cursor.last_due_date, cursor.last_id) ORDER BY due_date asc nulls last, id asc LIMIT N+1`.
- Filters concatenam via WHERE AND: `status IN (...)`, `tag_id IN (SELECT task_id FROM task_tags WHERE tag_id = X)`, `due_date BETWEEN from AND to`, `kanban_column_id = X`, `assigned_to_user_id = X`, `project ILIKE %X%`, `priority IN (...)`.

**AC6 PASS:** Pagination correctness + filters compostos verificados em integration tests (10+ cenários).

### 8. Zod strict validation — `household_id` NUNCA em payload

**AC8:** Todos os Zod schemas para POST/PATCH têm:
- `.strict()` mode (rejeita campos extra com 400 VALIDATION_ERROR).
- **NÃO** campo `household_id` no schema (derived de JWT internamente).
- **NÃO** campo `created_by_user_id` em PATCH (derived de JWT em POST, immutable depois).
- Validações específicas: `title` min 1 max 200, `description` max 5000, `color` hex regex `^#[0-9A-Fa-f]{6}$`, `due_time` regex `^[0-2][0-9]:[0-5][0-9]$` (alinhado com schema check constraint `tasks.ts:108-111`).

**Anti-R4 enforcement:** Se cliente malicioso enviar `{ household_id: '<OTHER_HOUSEHOLD_UUID>' }`, Zod rejeita 400 VALIDATION_ERROR ANTES de chegar ao DB. RLS é segunda camada de defesa (defense-in-depth).
