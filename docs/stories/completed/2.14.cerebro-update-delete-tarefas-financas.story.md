## Decisões de Design (documentadas para validação @po)

| ID | Decisão | Recomendação @sm | Razão |
|----|---------|-----------------|-------|
| **DP-2.14.A** | Scope de finanças MVP: apenas `transactions` variáveis (não-projectadas, não-parceladas) ou incluir recorrências? | **A — Apenas `transactions` com `is_projected=false` AND `installment_id IS NULL`** | Recorrências têm semântica complexa (afectam transacções futuras geradas pelo cron Inngest 4.5). Prestações têm composite reverse_op com até 60 ops. Ambas exigem design dedicado. UPDATE/DELETE de transacções variáveis cobre ~80% dos casos de uso reais ("errei o valor", "apaga aquela despesa do café"). |
| **DP-2.14.B** | Operações `eliminar_*` requerem preview-then-confirm SEMPRE (independentemente da confiança do classifier)? | **B — Sim, sempre preview em eliminações** | Uma eliminação irreversível (mesmo com undo 30s) é mais destrutiva do que uma criação. O utilizador deve sempre confirmar "apaga a despesa do café de ontem" antes de o agente agir. Implementado via: a tool retorna `needsConfirmation: true` no output preview, que o endpoint `/api/agent/prompt` (Story 2.6) deve respeitar. Coerente com Epic 2 §1 "conservador na destruição". |
| **DP-2.14.C** | Resolução de referência por linguagem natural: fuzzy match por `title` (tarefas) / `description` (transacções) ou também por data/valor? | **C — Título/descrição MAIS data/valor como desambiguadores** | "apaga a despesa do café" pode ter 10 matches. Estratégia: fuzzy por `description` ILIKE como primary, depois desambiguar por `transactionDate` (mais recente por defeito) OU por `amountCents` se fornecido no prompt. Zero matches → `ToolExecutionError` PT-PT. Múltiplos matches sem desambiguador → preview com lista dos primeiros 3 para o utilizador confirmar qual. |
| **DP-2.14.D** | `atualizar_tarefa` scope: quais campos são editáveis via chat? | **D — `title`, `dueDate`, `priority`, `status`, `description`** | Excluir `householdId` (identidade RLS), `createdByUserId` (auditoria), `completedAt` (gerido por `completar_tarefa` — evitar conflito). `assignedToUserId` excluído (feature família removida — single-user). |
| **DP-2.14.E** | Hard delete vs soft delete em `eliminar_*`? | **E — Hard delete** com `reverse_op` tipo `reinsert_row` (snapshot completo, id original) para undo 30s | Sem coluna `deleted_at` no schema actual — adicionar não é scope desta story. Hard delete com snapshot no `reverse_op` é o padrão mais simples. `restore_row` faz apenas UPDATE (`undo/route.ts:353`): para uma row hard-deleted, o UPDATE não atinge nenhuma linha e o undo é no-op silencioso — não funciona. O novo `kind: 'reinsert_row'` (FIX-1, AC6+AC7) executa `INSERT INTO table (...cols incluindo id) VALUES (...)` com o snapshot completo, restaurando a row com o `id` original. |

## User Story

**Como** utilizador da Expressia que gere tarefas e finanças via chat,
**Quero** poder dizer ao Expressia coisas como "muda a tarefa do dentista para sexta", "actualiza a prioridade do relatório para urgente", "apaga a tarefa de compras", "corrige a despesa do café — foi €3,50 não €5,00" e "elimina a transacção do almoço de ontem",
**Para que** possa editar e eliminar registos directamente via chat sem sair do Jarvis, com o mesmo preview-then-confirm, undo 30s e isolamento RLS que tenho nas operações de criação — especialmente em eliminações, onde o agente é "conservador na destruição".

## Acceptance Criteria

### AC1 — Tool `atualizar_tarefa` (UPDATE campos de uma tarefa)

Tool implementada em `packages/tools/src/tasks/atualizar-tarefa.ts` conforme contrato `ToolDefinition<I, O>`:

- **Input Zod schema:**
  ```ts
  z.object({
    taskId:    z.string().uuid().optional(),
    taskTitle: z.string().min(1).max(200).optional(),
    // Campos a actualizar (pelo menos um obrigatório no segundo refine):
    newTitle:       z.string().min(1).max(200).optional(),
    newDueDate:     z.string().regex(/^\d{4}-\d{2}-\d{2}$/).optional(),
    newPriority:    z.enum(['low', 'medium', 'high']).optional(),
    newStatus:      z.enum(['todo', 'doing', 'done', 'archived']).optional(),
    newDescription: z.string().max(2000).optional(),
  })
    .refine(d => d.taskId !== undefined || d.taskTitle !== undefined, {
      message: 'Fornecer taskId ou taskTitle para identificar a tarefa',
    })
    .refine(d => [d.newTitle, d.newDueDate, d.newPriority, d.newStatus, d.newDescription]
      .some(v => v !== undefined), {
      message: 'Fornecer pelo menos um campo para actualizar',
    })
  ```
  [Trace: `packages/db/src/schema/tasks.ts` — `title text NOT NULL`, `dueDate date`, `priority taskPriorityEnum`, `status taskStatusEnum`, `description text nullable`. DP-2.14.D ratificada: `assignedToUserId`, `householdId`, `createdByUserId`, `completedAt` excluídos.]

- **Output Zod schema:**
  ```ts
  z.object({
    taskId:       z.string().uuid(),
    updatedFields: z.array(z.string()),  // lista de campos efectivamente actualizados
    snapshot:     z.record(z.unknown()), // estado anterior dos campos alterados (para reverse_op)
  })
  ```

- **`execute(input, ctx)`:**
  1. Resolver tarefa: se `taskId` presente, SELECT por `id = input.taskId AND household_id = ctx.householdId`. Se `taskTitle`, fuzzy match `ILIKE '%${input.taskTitle}%'` ORDER BY `created_at DESC` LIMIT 1 (padrão Story 3.8 `completar_tarefa`). Zero matches → `ToolExecutionError` com `userMessage` PT-PT: `"Não encontrei nenhuma tarefa com o nome '${taskTitle}'. Verifica o nome e tenta novamente."`. Múltiplos matches (só válido quando por título) → usa o mais recente + adiciona `warnings` no output.
  2. Capturar snapshot dos campos a alterar: `{ title: row.title, dueDate: row.dueDate, priority: row.priority, status: row.status, description: row.description }` — apenas os que têm valor novo no input.
  3. UPDATE em `tasks` via `ctx.db` dos campos fornecidos (SET `updated_at = now()` sempre). RLS activa — Postgres bloqueia automaticamente updates cross-household.
  4. Retorna `{ taskId, updatedFields: [...], snapshot }`.
  - **NUNCA usa `getServiceDb()`** — usa `ctx.db` (RLS activa via JWT).

- **`reverse(output, ctx)`:**
  ```ts
  { kind: 'restore_row', table: 'tasks', id: output.taskId, snapshot: output.snapshot }
  ```
  O endpoint `/undo` (Story 2.8) aplica o snapshot como UPDATE SET dos campos guardados.

- **`preview(input, ctx)`:** string PT-PT descritiva das alterações, ex:
  `"Actualizar tarefa '${resolvedTitle}': dueDate → sexta-feira, priority → high"`

- **Metadados:** `name: 'atualizar_tarefa'`, `domain: 'tasks'`, `description` PT-PT para o LLM (ex: "Usa esta tool quando o utilizador quer editar, alterar ou modificar uma tarefa existente — mudar a data, prioridade, título ou estado."), `estimatedTokens: 90`.

### AC2 — Tool `eliminar_tarefa` (DELETE hard de uma tarefa + preview obrigatório)

Tool implementada em `packages/tools/src/tasks/eliminar-tarefa.ts`:

- **Input Zod schema:**
  ```ts
  z.object({
    taskId:    z.string().uuid().optional(),
    taskTitle: z.string().min(1).max(200).optional(),
    confirmed: z.boolean().optional().default(false),  // ver DP-2.14.B
  }).refine(d => d.taskId !== undefined || d.taskTitle !== undefined, {
    message: 'Fornecer taskId ou taskTitle para identificar a tarefa a eliminar',
  })
  ```

- **Output Zod schema:**
  ```ts
  z.object({
    taskId:           z.string().uuid(),
    title:            z.string(),
    needsConfirmation: z.boolean(),   // true quando `confirmed=false` — forçar preview
    snapshot:         z.record(z.unknown()).optional(), // preenchido após delete efectivo
  })
  ```

- **`execute(input, ctx)`:**
  1. Resolver tarefa (mesmo padrão AC1 — fuzzy ILIKE + mais recente + `ToolExecutionError` PT-PT em zero matches). Guardar snapshot completo: `{ title, description, dueDate, priority, status, completedAt, createdByUserId, createdAt }`.
  2. **Se `input.confirmed !== true`:** retornar imediatamente `{ taskId, title, needsConfirmation: true }` sem executar o DELETE. O endpoint `/api/agent/prompt` deve detectar `needsConfirmation: true` e tratar como preview obrigatório (DP-2.14.B). **Não persiste `reverse_op`** neste caso.
  3. **Se `input.confirmed === true`:** executar DELETE em `tasks` WHERE `id = taskId AND household_id = ctx.householdId`. Retornar `{ taskId, title, needsConfirmation: false, snapshot }`.
  - **NUNCA usa `getServiceDb()`** — usa `ctx.db` (RLS activa).

- **`reverse(output, ctx)`:** `reinsert_row` com snapshot completo (novo kind — FIX-1):
  ```ts
  {
    kind: 'reinsert_row',
    table: 'tasks',
    id: output.taskId,
    snapshot: output.snapshot ?? {},  // snapshot é undefined se não confirmado (nunca chegou a delete)
  }
  ```
  `restore_row` faz apenas UPDATE — não funciona para rows hard-deleted (confirmado em `undo/route.ts:353`). O novo `kind: 'reinsert_row'` é adicionado ao contrato (`packages/tools/src/contracts.ts`) e ao engine (`applyReverseOp` em `undo/route.ts`) como branch que executa `INSERT INTO ${table} (...colunas do snapshot incluindo id...) VALUES (...)`, reutilizando `ALLOWED_REVERSE_TABLES` e validação de tipos do branch `restore_row`.

- **`preview(input, ctx)`:** `"Eliminar tarefa '${resolvedTitle}'" — CONFIRMAR`

- **Metadados:** `name: 'eliminar_tarefa'`, `domain: 'tasks'`, `description` PT-PT para o LLM (ex: "Usa esta tool quando o utilizador quer apagar, eliminar ou remover permanentemente uma tarefa. Requer confirmação explícita — sempre gera preview antes de executar."), `estimatedTokens: 90`.

### Pattern `reverse_op` relevante para esta story

[Source: `packages/tools/src/contracts.ts:195-200` + `undo/route.ts:331-355`]

```ts
// Para UPDATE (atualizar_tarefa, update_finance_variable):
{ kind: 'restore_row', table: 'tasks'|'transactions', id: string, snapshot: Record<string, unknown> }
// snapshot = campos pre-update (ex: { title: 'Dentista', dueDate: '2026-06-20' })
// O engine executa: UPDATE tasks SET title='Dentista', dueDate='2026-06-20' WHERE id='...'

// Para DELETE (eliminar_tarefa, delete_finance_variable) — FIX-1 (AC6+AC7):
// USAR `reinsert_row` com snapshot completo (todos os campos da row eliminada).
// restore_row faz apenas UPDATE (confirmado undo/route.ts:353): se a row foi hard-deleted,
// o UPDATE não atinge nenhuma linha → undo silencioso → FR6 violado.
{ kind: 'reinsert_row', table: 'tasks'|'transactions', id: string, snapshot: Record<string, unknown> }
// O engine executa: INSERT INTO tasks (id, title, ...) VALUES ('...', '...', ...)
// O id original é preservado (id incluído no snapshot e passado explicitamente).
```
