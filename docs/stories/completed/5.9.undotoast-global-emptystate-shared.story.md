**Como** utilizador autenticado da Expressia,

**Para que** possa reverter acções de qualquer rota sem ter de navegar para o chat, e entenda imediatamente o que fazer quando uma lista está vazia — com copy PT-PT coerente em toda a aplicação.

### AC1 — `undoStore` Zustand SSR-safe

**AC1:** Criar `apps/web/src/lib/stores/undoStore.ts` (Zustand, **sem persist**) que mantém o estado do último token de undo disponível.

**(a)** Shape do store:
- `undoUrl: string | null` — endpoint do último undo activo (`/api/agent/prompt/{runId}/undo`).
- `expiresAt: string | null` — ISO 8601 da expiração (30s, da response do agente).
- `setUndo(url: string, expiresAt: string): void` — substitui o token anterior (R-5.9: T2 substitui T1 silenciosamente).
- `clearUndo(): void` — limpa após undo executado ou expiração.
**(b)** Regra R-5.9 implementada: `setUndo` sobrescreve sempre — não há stack de tokens; o token anterior é descartado sem notificação ao utilizador (front-end-spec §1.2 P4 confirma UX; Epic 5 §6 R-5.9 define o comportamento esperado).
**(c)** **SSR-safety (lição FIX-1 da 5.7):** o `undoStore` é singleton de módulo; **nunca é mutado no servidor**. O subscriber que alimenta o store (DP-5.9.B) vive num `useEffect` Client-only. Selectores usados no `<UndoToast>` com `'use client'` — sem problema de SSR.
**(d)** Sem `persist` (os tokens expiram em 30s; persistência em `localStorage` seria risco de segurança com tokens de undo expirados + desnecessária dado o TTL curto).

**AC1 PASS:** Teste `undoStore.test.ts`: `setUndo` guarda url+expiresAt; segundo `setUndo` substitui o primeiro (R-5.9); `clearUndo` limpa. Store inicializa com `null/null`. Sem imports do tipo `import { useEffect } from 'react'` no ficheiro do store (a lógica de hidratação fica fora do store).

### AC3 — `<UndoToast>` global com countdown 30s

**AC3:** Criar `apps/web/src/components/shell/UndoToast.tsx` (Client Component) que renderiza o toast de undo global quando `undoStore.undoUrl !== null`, com countdown em tempo real e botão "Anular (Xs)".

**(a)** Toast visível em qualquer rota: posição `fixed` fora dos painéis (DP-5.9.E — bottom-right ou bottom-center; não colide com FAB do chat mobile — z-index adequado).
**(b)** Countdown de `expiresAt` até 0 (1s interval em `useEffect` — precedente `ResultMessage.tsx:119-129`); ao expirar: toast desaparece automaticamente (`clearUndo()`).
**(c)** Clicar "Anular" → `POST undoUrl` → 200: toast muda para "Anulado ✓" por 2s e desaparece; 409 UNDO_EXPIRED: toast desaparece; outro erro: toast muda para "Erro ao anular" por 2s. Comportamento reutiliza a lógica de `ResultMessage.tsx` (D40 retrocompat — sem `undoUrl` = sem botão).

**(e)** Acessível: `role="status"` ou `aria-live="polite"` (não `alert` — não é erro); botão com `aria-label="Anular última acção do agente"`, foco visível, navegável por teclado.
**(f)** Montado no `AppShell.tsx` junto do `<UndoToastBridge>`.

**AC3 PASS:** Teste `UndoToast.test.tsx`: sem `undoUrl` → nada renderiza; com `undoUrl` + `expiresAt` futuro → renderiza toast com countdown; simular expiração → desaparece; simular clique Anular (mock fetch 200) → "Anulado ✓"; mock fetch 409 → toast desaparece. `role="status"` ou `aria-live` presente.

## Dev Notes

**Arquitectura do `undoStore` — o padrão correcto (lição FIX-1 da 5.7).**

O `undoStore` é um Zustand store de módulo singleton. A lição crítica capturada no gate da 5.7: **nunca mutar um store Zustand no corpo do render de um Client Component** — Client Components renderizam no servidor durante SSR, mutando o singleton, o que cria cross-request state leak em lambdas warm. A solução canónica: toda a hidratação/alimentação do store acontece em `useEffect` (client-only). Para o `undoStore`:
- O `<UndoToastBridge>` é o único ponto que escreve no `undoStore`; ele faz isso exclusivamente em `useEffect`.
- O `<UndoToast>` apenas lê do `undoStore` (selectores) e executa `clearUndo` via handlers de evento — nunca no render.

**Bridge `chatStore` → `undoStore` (DP-5.9.B).** O `chatStore` já tem `AgentResultMessage.undoUrl?`+`undoExpiresAt?` (`chatStore.ts:50,52`). O `<UndoToastBridge>` subscreve `useChatMessages()` e detecta quando a última mensagem é `kind='result'` com `undoUrl` presente. Chamar `undoStore.setUndo(msg.undoUrl, msg.undoExpiresAt)` nesse efeito — dependências `[messages]` (re-corre quando o array de mensagens muda). Atenção ao shape: `AgentResultMessage.undoUrl` e `undoExpiresAt` são `string | undefined` — guardar `null` se não-presentes.

**R-5.9 — race condition T1 → T2 (Epic 5 §6 R-5.9).** O `undoStore.setUndo` substitui sempre o token anterior. Quando T2 chega em < 30s de T1, T1 é descartado. O utilizador não recebe alerta explícito de que T1 ficou inacessível (Epic 5 §6 R-5.9 define que a microcopy `"A acção anterior já não pode ser anulada por aqui — usa o chat"` é opcional). Se o @dev quiser implementar esse aviso: é uma mensagem breve na substituição do toast. Se não, silencioso (T1 fica no histórico do agente, acessível via chat). [DEV-DECISION].

**`ResultMessage.tsx` já tem lógica undo inline.** `apps/web/src/components/chat/ResultMessage.tsx` tem countdown + POST `/undo` + banner PT-PT. O `<UndoToast>` global reutiliza o mesmo padrão (não é extracção — são componentes distintos com propósito diferente: `ResultMessage` é o registo histórico da acção no chat; `UndoToast` é o feedback temporário global fora do chat). Reutilizar a mesma lógica de countdown (`useEffect` com `setInterval` + `expiresAtMs`), o mesmo padrão de fetch, e a mesma microcopy PT-PT (`ResultMessage.tsx:49-75` — `bannerSuccess`, `BANNER_EXPIRED`, etc.).

**`packages/ui` — adicionar `<EmptyState>`.** A estrutura existe: `packages/ui/src/components/MoneyDisplay/` e `DateDisplay/` com os seus barrels. Seguir exactamente o mesmo padrão: criar `packages/ui/src/components/EmptyState/EmptyState.tsx` + `index.ts` (barrel) + `__tests__/EmptyState.test.tsx`. Adicionar `export * from './EmptyState'` ao `packages/ui/src/components/index.ts` (já tem `export * from './MoneyDisplay'` e `export * from './DateDisplay'`). O `packages/ui/src/index.ts` já faz `export * from './components'` — sem alteração necessária.

**`VisaoEmptyState` — consolidação.** O componente actual (`VisaoEmptyState.tsx:19-38`) renderiza exactamente os defaults da variante `visao` do `<EmptyState>`: título "Ainda não há nada para mostrar.", body "Carrega no chat e diz 'criar tarefa de comprar pão amanhã' para começar.", CTA "Abrir o chat" → `/jarvis`. A consolidação é byte-a-byte sem mudança visual. O `page.tsx:109` chama `<VisaoEmptyState />` — substituir por `<EmptyState variant="visao" />` de `@meu-jarvis/ui`. O `<EmptyState variant="chat">` preenche esse espaço com "Olá. Em que posso ajudar?" (front-end-spec §7 l.1239). Não confundir com o histórico do `/jarvis` — o `chatStore` é in-memory (Story 5.4 DP-5.4.F), por isso o empty-state aparece sempre que o store é fresco (página carregada de novo).
