### 1. Migration `0016_user_prefs_theme_widgets.sql` — extensão de colunas

**AC1:** Criar `packages/db/migrations/0016_user_prefs_theme_widgets.sql` que estende a tabela `user_prefs` com 2 colunas idempotentes:

(a) **Coluna `theme`** — `text NOT NULL` com CHECK constraint e default `'system'`:
```sql
alter table public.user_prefs
  add column if not exists theme text not null default 'system';

alter table public.user_prefs
  drop constraint if exists user_prefs_theme_check;
alter table public.user_prefs
  add constraint user_prefs_theme_check
  check (theme in ('light', 'dark', 'system'));
```

(b) **Coluna `widgets_enabled`** — `jsonb NOT NULL` com default contendo todos os 7 widgets identificados em front-end-spec §5.4 (5 default-ON + 2 default-OFF):
```sql
alter table public.user_prefs
  add column if not exists widgets_enabled jsonb not null default
  '{"briefing":true,"tasks_today":true,"finance_month":true,"recurrences_next":true,"tasks_overdue":true,"accounts_balance":false,"calendar_week":false}'::jsonb;
```

(c) Adicionar `COMMENT ON COLUMN` PT-PT explicando cada coluna + trace (DP2 e DP3 do Epic 5 §8).

(d) **Idempotência:** `ADD COLUMN IF NOT EXISTS` (Postgres 9.6+) — re-run não falha. CHECK constraint usa `DROP IF EXISTS` + `ADD` pattern para permitir update sem erro em re-run.

(e) **Decisão de naming dos widget IDs (snake_case):** 7 IDs decididos para alinhar com convenção JSONB key + DB naming (não kebab-case, evita escape de hyphens em queries). Trace: front-end-spec §5.4. Lista final (mapeada do wireframe):

- `tasks_today` (ON) — "Tarefas hoje"
- `finance_month` (ON) — "Balanço financeiro do mês"
- `recurrences_next` (ON) — "Próximas recorrências"
- `tasks_overdue` (ON) — "Tarefas atrasadas"
- `accounts_balance` (OFF) — "Saldo por conta"
- `calendar_week` (OFF) — "Calendário semana"

**AC1 PASS:** Migration aplica em CI rls-gate Postgres efémero sem erros + `pnpm --filter @meu-jarvis/db db:migrate` idempotente verificado em re-run local (`[skip]` na 2ª execução).

### 8. Validation findings documentados

**AC8:** Documentar na "Validation Report" 2 achados que requerem decisão consciente (não bloqueiam, mas têm de ser explicitados — Constitution Article IV):

(a) **CHECK constraint vs Postgres enum nativo (theme):** Decisão: CHECK constraint `theme IN ('light','dark','system')` em vez de criar enum Postgres `theme_enum`. Racional: apenas 3 valores estáveis; CHECK é mais flexível para evolução (ALTER ADD value via Postgres enum exige `ALTER TYPE` que tem caveats em transactions). Precedent: Story 4.1 evitou novo enum para audit actions de Finanças que poderiam evoluir; aplicou ALTER TYPE ADD VALUE para enums já existentes. Aqui, enum novo seria over-engineering. Re-avaliar se theme passar a ter 5+ valores (high-contrast, sepia, etc.).

(b) **`updated_at` automático após ALTER:** o trigger `user_prefs_set_updated_at` (criado em `0007:53-57`) chama `set_updated_at()` que actualiza a coluna `updated_at`. ALTER TABLE com defaults preenche `theme`/`widgets_enabled` em rows existentes via DEFAULT (sem disparar UPDATE trigger). Comportamento esperado e documentado — rows existentes mantêm o `updated_at` antigo enquanto não houver PATCH explícito. Decisão: PASS — coerente com convenção Postgres (DEFAULT no ALTER não conta como UPDATE).

**AC8 PASS:** Os 2 achados documentados com decisão clara; nenhum gap silencioso. Trace: Epic 5 §8 DP3 (JSONB com Zod), DP2 (theme híbrido).
