## Story

**As a** Eurico (único utilizador do Jarvis),
**I want** pedir ao Jarvis, via Telegram, que componha e envie um email novo — sempre com uma pré-visualização e confirmação antes do envio (resposta em thread fica para v2 — ver Change Log v1.1-draft),
**so that** consigo tratar a correspondência sem abrir o Gmail, com a mesma conversa que já uso para ler emails, tarefas, finanças e agenda — e com a segurança de rever antes de enviar.

## Contexto de âmbito

**Esta story entrega:**
- Gmail send via OAuth scope `gmail.send` (cumula com `calendar.events` + `gmail.readonly` já em prod).
- Re-consentimento OAuth one-shot [EURICO] (novo scope sensível — `gmail.send` é *restricted/sensitive*; pode exigir passo extra no OAuth consent screen do GCP, lição J-6).
- Novo intent `enviar_email` no motor: composição/resposta de email via Telegram.
- Nova tool `enviar_email` (Gmail API `users.messages.send`; escrita com efeito externo IRREVERSÍVEL → força `needs_confirmation: true`; `reverse` inerte `_noop` — não é possível "des-enviar").
- Email novo (compose): destinatário, assunto e corpo indicados pelo utilizador em linguagem natural.

**NÃO inclui (fora de âmbito):**
- **Resposta em thread (reply, `In-Reply-To`/`References`/`threadId`) — descoped para v2.** Verificado no draft desta story: `preview(input, ctx): string` (`packages/tools/src/contracts.ts`) é **síncrono, sem I/O** — não pode resolver "o último email do Pedro" via chamada à Gmail API antes do utilizador ver o preview, e o `planner-executor` não tem chaining output→input (grep `previousResults|toolOutputs|chain` = 0) para popular um `replyToMessageId` a partir de linguagem natural. Resolver isto exigiria alterar o contrato `ToolDefinition.preview` (síncrono → assíncrono) em `packages/tools/src/contracts.ts`, o que teria de propagar a TODAS as tools existentes — fora de âmbito de uma story de Gmail send. v1 entrega compose sólido; reply fica para v2 com desenho próprio.
- Anexos, HTML rico, CC/BCC múltiplos (v1 = texto simples, 1 destinatário).
- Apagar, arquivar ou marcar emails.
- Rascunhos (`drafts`) — o fluxo é preview→confirm→send directo.
- Cache de emails em DB (mantém-se a política J-6: email nunca persistido).
- Reenvio automático / agendamento de envio.

**Depende de:** J-6 Done (Gmail readonly em prod: `gmail-api.ts`, `getGmailAccessToken`, ToolDomain `'email'`, registo side-effect da pasta `gmail/` em `run-agent.ts` E `confirm/route.ts`, padrão de tool externa provado E2E). ### Diferença chave face a J-6 (leitura) — `needs_confirmation`

`consultar_emails` (J-6) é leitura → saltou o preview→confirm (follow-up `182f760`) e não tem undo. `enviar_email` (J-7) é ESCRITA com efeito externo IRREVERSÍVEL → tem de estar na lista `needs_confirmation: true` do classifier (AC3), passando SEMPRE por preview→confirm antes de enviar. NÃO regredir a distinção read-vs-write introduzida no follow-up de J-6 (`READ_ONLY_INTENTS`): `enviar_email` NÃO é read-only.

### `reverse()` — porquê `_noop` numa escrita

Ao contrário de tarefas/finanças (undo por `reverse_op` real em 30s), um email enviado **não é revertível** — não há "des-enviar". A rede de segurança é o preview→confirm obrigatório (o Eurico vê o rascunho e confirma). O `reverse()` devolve `{ kind: 'delete_row', table: '_noop' }` (padrão R1b v1.1) e o `/undo` responde 410 Gone. Documentar isto na resposta do bot se fizer sentido ("Emails enviados não podem ser recuperados.").

### MIME + base64url (Gmail `users.messages.send`)

A Gmail API espera `{ raw: <RFC 2822 message em base64url> }`. base64url = base64 com `+`→`-`, `/`→`_` e SEM padding `=` (RFC 4648 §5). Mensagem mínima texto simples:
```
To: dest@example.com
From: euricojsalves@gmail.com
Subject: Assunto
Content-Type: text/plain; charset="UTF-8"
MIME-Version: 1.0

Corpo do email.
```
Sem dependências novas (`Buffer.from(mime).toString('base64url')` do Node cobre a codificação). **Reply** (`In-Reply-To`/`References`/`threadId`) fica para v2 — descoped neste draft porque `preview()` é síncrono e o planner-executor não tem chaining output→input para resolver "o último email de X" antes do preview (ver Contexto de âmbito e Change Log v1.1-draft).

  outro plano seria decorativo (a segurança de uma acção irreversível só é real se o que se confirma
