## Acceptance Criteria

### 1. Endpoints Accounts CRUD — `/api/financas/contas` + `/[id]`

**AC1:** Criar 2 route handlers em `apps/web/src/app/api/financas/contas/` (namespace per DP-4.2.1):

| Endpoint | Métodos | Comportamento |
|----------|---------|---------------|
| `/api/financas/contas` | GET | List per household (RLS filtra). Filter query param `archived` (boolean, default `false` = só contas activas — `archived_at IS NULL`). Order: `name asc`. Hard cap limit 200 (sem paginação cursor — volume baixo por household). |
| `/api/financas/contas` | POST | Create — Zod body `.strict()`: `name` obrigatório, `account_type` enum (default `corrente`), `bank_name` opcional, `iban_last4` opcional (4 dígitos), `initial_balance_cents` integer (default 0). RLS injecta `household_id` via JWT (NÃO aceito em payload). `currency` fixo `'EUR'` (não em payload — default DB). Na criação, `balance_cents` é inicializado igual a `initial_balance_cents` (snapshot — ver DP-4.2.4). |
| `/api/financas/contas/[id]` | GET | Single account. RLS filtra. 404 se não existe ou cross-household. |
| `/api/financas/contas/[id]` | PATCH | Update parcial. Zod body `.strict()` (todos opcionais). `household_id` + `currency` IMMUTABLE (rejeitar 400 se presentes). |
| `/api/financas/contas/[id]` | DELETE | Soft delete via `archived_at = now()` (DP-4.2.2). Variant `accounts_delete_owner_admin` (`0001:454`) — só owner/admin (403 FORBIDDEN para member). |

**AC1 PASS:** 5 métodos de endpoint com tests integration mockable-friendly + Zod `.strict()` + RLS injection via JWT.

### 5. Soft delete + variant `delete_owner_admin`

**AC5:** Os DELETE handlers de `accounts` e `cards`:

(a) **Soft delete** — `UPDATE ... SET archived_at = now(), updated_at = now() WHERE id = X` (RLS filtra). NÃO hard delete (DP-4.2.2): a coluna `archived_at` existe no schema; hard delete partiria a FK `cards.account_id ON DELETE restrict` e anularia `transactions.account_id`/`card_id` (perda de histórico financeiro).

(b) **Variant `delete_owner_admin`** — as policies `accounts_delete_owner_admin` (`0001_rls_policies.sql:454`) e `cards_delete_owner_admin` (`0001:471`) restringem DELETE a owner/admin. Como o soft-delete é tecnicamente um `UPDATE` (não `DELETE`), a policy de UPDATE (`*_update_member`) aplica-se — qualquer member conseguiria. **Por isso o handler verifica `resolveHouseholdRole` explicitamente:** se o role for `member`, retornar **403 FORBIDDEN** ("Apenas owner ou admin do household pode arquivar contas/cartões.") ANTES do UPDATE. Isto alinha a semântica de "eliminar" com a intenção das policies `delete_owner_admin`.

**AC5 PASS:** DELETE soft via `archived_at`; member recebe 403; owner/admin arquiva com sucesso; teste cobre ambos.

### 6. Validação composta de cartão de crédito

**AC6:** O `POST /api/financas/cartoes` valida:
- Se `card_type='credit'` ⇒ `credit_limit_cents` obrigatório (Zod `.refine()` → 400 VALIDATION_ERROR "Cartão de crédito requer limite de crédito."). Alinha com o CHECK constraint `cards_credit_needs_limit` (`finance.ts:163-166`) — defesa em profundidade (Zod antes do DB).
- `account_id` tem de existir, pertencer ao household do user **e não estar arquivada** — verificado por SELECT RLS-scoped com filtro `archived_at IS NULL` (`getDb()` não encontra contas cross-household; o filtro `archived_at` impede criar um cartão sobre uma conta arquivada — inconsistência lógica) → 404 NOT_FOUND "Conta não encontrada." se inválido ou arquivada. **[PO_FIX_INLINE F1]**
- `closing_day`/`due_day` no intervalo 1-28 (Zod) — alinha com CHECK `cards_closing_day_range`/`cards_due_day_range`.

**AC6 PASS:** Testes cobrem: credit sem limite → 400; account_id cross-household → 404; account_id de conta arquivada → 404 (PO_FIX F1); closing_day=31 → 400.
