## Acceptance Criteria

### AC1 — Endpoint `GET /api/visao/tarefas-hoje` retorna tarefas do dia actual

**AC1:** Criar `apps/web/src/app/api/visao/tarefas-hoje/route.ts`. Handler `GET` autenticado que retorna tarefas do utilizador cujo `due_date = hoje (UTC)` e `status NOT IN ('done', 'archived')`.

**(a) Query Drizzle** (via `getDb()` — role `authenticated`, RLS aplicada):
```
SELECT id, title, status, priority, due_time, kanban_column_id
FROM tasks
WHERE household_id = current_household_id()
  AND due_date = CURRENT_DATE
  AND status NOT IN ('done', 'archived')
ORDER BY due_time ASC NULLS LAST, priority DESC, created_at ASC
LIMIT 20
```

**(b) Schema de resposta Zod** (`TasksTodayResponse`):
```typescript
{ count: number; tasks: Array<{ id: string; title: string; status: string; priority: string; dueTime: string | null }> }
```

**(c) RLS inviolável:** NUNCA usar `getServiceDb()`. O `getDb()` aplica automaticamente `is_household_member(household_id)` via JWT. Cross-household access é bloqueado a nível Postgres (NFR5).

**(d) Auth guard:** Se a sessão não existir, retornar `401 Unauthorized`. Padrão idêntico a `apps/web/src/app/api/tasks/route.ts` (Story 3.2).

**(e) Limite defensivo:** `LIMIT 20` na query para evitar payloads gigantes. O widget mostra no máximo 5 itens visíveis; o count total é sempre retornado independente do LIMIT.

**AC1 PASS:** `GET /api/visao/tarefas-hoje` com sessão válida retorna `{ count: number, tasks: [...] }` com `Content-Type: application/json`. Sem sessão retorna `401`. Teste unitário (Vitest + mock `getDb`) cobre: resposta vazia (0 tarefas hoje), resposta com dados, 401 sem sessão.

### AC3 — Endpoint `GET /api/visao/financas-mes` retorna balanço do mês corrente

**AC3:** Criar `apps/web/src/app/api/visao/financas-mes/route.ts`. Handler `GET` autenticado que agrega transacções reais (não projecções) do mês corrente (1.º dia a hoje) agrupadas por `kind`.

**(a) Query Drizzle:**
```
SELECT
  kind,
  SUM(amount_cents) AS total_cents,
  COUNT(*) AS transaction_count
FROM transactions
WHERE household_id = current_household_id()
  AND transaction_date >= date_trunc('month', CURRENT_DATE)
  AND transaction_date <= CURRENT_DATE
  AND is_projected = false
GROUP BY kind
```

**(b) Schema de resposta Zod** (`FinancesMonthResponse`):
```typescript
{
  incomeTotal: number;    // cents
  expenseTotal: number;   // cents
  balance: number;        // incomeTotal - expenseTotal (cents; pode ser negativo)
  transactionCount: number;
  currency: 'EUR';        // sempre EUR (CON9)
}
```

**(c) RLS:** `getDb()` exclusivamente. Valores em cêntimos (`integer`) — conversão para display (`€` format) é responsabilidade do frontend via `MoneyDisplay` de `@meu-jarvis/ui` (Story 5.2). NUNCA converter na API.

**(d) Mês corrente:** usar `date_trunc('month', CURRENT_DATE)` a nível SQL (não em TypeScript) para evitar timezone edge cases.

**AC3 PASS:** `GET /api/visao/financas-mes` retorna `{ incomeTotal, expenseTotal, balance, transactionCount, currency }` com valores em cêntimos. Teste unitário cobre: sem transacções (tudo 0), apenas despesas, apenas receitas, mix.

## Dev Notes

### Contexto arquitectural crítico

**RLS multi-tenant (NFR5 — INVIOLÁVEL):**

- **SEMPRE `getDb()`** (role `authenticated`) em todos os route handlers desta story. O `getDb()` está em `packages/db/src/client.ts` e aplica RLS via JWT do Supabase Auth que contém `household_id` no claim top-level (custom_access_token_hook, migration 0002).
- **NUNCA `getServiceDb()`** em response handlers de utilizador. `getServiceDb()` ignora RLS e é exclusivo de jobs Inngest/admin scripts. Regra absoluta: `packages/db/src/client.ts` + `CLAUDE.md §Multi-tenancy via Postgres RLS`.
- Cross-household access é bloqueado a nível Postgres — a aplicação não precisa de filtrar por `householdId` explicitamente nas queries (RLS faz-o automaticamente). Mas para clareza e performance, as queries incluem o predicado de household via helper SQL `current_household_id()`.

**Padrão de auth guard em route handlers:**

Seguir exactamente o padrão de `apps/web/src/app/api/tasks/route.ts` (Story 3.2):
```typescript
import { createServerSupabaseClient } from '@meu-jarvis/auth/server';
import { NextResponse } from 'next/server';

export async function GET() {
  const supabase = await createServerSupabaseClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) return NextResponse.json({ error: 'Não autorizado' }, { status: 401 });
  
  const db = await getDb();
  // ... query
}
```

**Valores monetários em cêntimos (CON9):**

Todas as respostas da API de finanças retornam valores em cêntimos (`integer`). `€78,70` = `7870`. Nunca converter para decimal na API — conversão é responsabilidade do `<MoneyDisplay>` de `@meu-jarvis/ui` (Story 5.2) no frontend.

### Drizzle ORM — queries agregadoras

Usar operadores Drizzle para agregação:
```typescript
import { eq, lt, gte, lte, and, not, inArray, sql, count, sum } from 'drizzle-orm';
import { tasks } from '@meu-jarvis/db';

// Exemplo COUNT:
const result = await db.select({ count: count() }).from(tasks).where(
  and(eq(tasks.householdId, householdId), ...)
);

// Exemplo SUM:
const result = await db.select({ total: sum(transactions.amountCents) })
  .from(transactions).where(...);
```

**Atenção ao tipo de retorno de `sum()`:** Drizzle retorna `string | null` para `sum()` (via Postgres `numeric`). Converter com `parseInt(result[0]?.total ?? '0', 10)` antes de colocar na resposta.

### Agrupamento por dia (AC6 — calendario-semana)

O agrupamento por dia é feito em TypeScript após a query SQL (não GROUP BY), para preservar os items individuais:

```typescript
// Construir array de 7 dias iniciando em hoje
const days = Array.from({ length: 7 }, (_, i) => {
  const d = new Date();
  d.setDate(d.getDate() + i);
  return d.toISOString().slice(0, 10); // 'YYYY-MM-DD'
});

const result = days.map((date) => ({
  date,
  tasks: rawTasks.filter((t) => t.dueDate === date),
  taskCount: rawTasks.filter((t) => t.dueDate === date).length,
}));
```
