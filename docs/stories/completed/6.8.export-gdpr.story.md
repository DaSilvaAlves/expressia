## Story

**As a** utilizador autenticado da Expressia,
**I want** solicitar e descarregar um export completo dos meus dados em formato ZIP (JSON + CSV) a partir da página da minha conta,
**so that** exerço o meu direito de portabilidade de dados previsto no GDPR Art. 20, sabendo que toda a minha informação pessoal e do household está disponível para exportar em qualquer momento.

## Contexto e âmbito (ler antes das ACs)

Esta story implementa o **direito de portabilidade** (GDPR Art. 20, FR28, NFR10) — o lado **"obter os meus dados"**. É complementar à RGPD-1 (Art. 17 — eliminação), que documenta o processo de purge e **não deve ser duplicada aqui**. Referência cruzada obrigatória: `docs/runbooks/rgpd-account-deletion.md`.

**Billing está CONGELADO** (directiva 2026-05-29). Esta story exporta dados de domínio do utilizador — não toca em Stripe, planos, faturas nem qualquer mecanismo de billing. Os dados de `subscriptions`, `invoices` e `payment_methods` estão **fora do âmbito do export MVP**.

**Padrão de geração — decisão PO-D1 fixada:** geração síncrona/inline dentro do request HTTP (Opção A). Volume de soft-launch é reduzido; sem infra Inngest de export; KISS. Não existe Opção B nesta story.

**Schema já preparado:** a tabela `data_export_jobs` existe em `packages/db/src/schema/audit.ts` (migração `0000_initial_schema.sql`) com os campos `status` (enum `data_export_status`), `storage_path`, `download_url`, `expires_at`. O status default é `'pending'` (evidência: `audit.ts:186`). Os enums `data_export_requested`/`data_export_completed` existem em `auditActionEnum`. **Esta story não cria tabelas novas** — zero impacto no RLS Coverage Gate.

**RLS de `data_export_jobs` — restrição crítica de design:** a policy `data_export_jobs_update_blocked` (linha 599-601 de `0001_rls_policies.sql`) usa `using(false) with check(false)` para o role `authenticated`. O mesmo para DELETE (linha 602-604). Qualquer UPDATE ou DELETE via `getDb()` (role `authenticated`) falha SEMPRE por design. Ver AC8 e Dev Notes §RLS para o padrão obrigatório.

**RLS inegociável para leitura:** todo o SELECT de dados de domínio usa `getDb()` (role `authenticated`, RLS por JWT). O campo `household_id` no JWT garante que só os dados do household do utilizador são exportados.

## Acceptance Criteria

> Rastreabilidade: FR28 (portabilidade), NFR10 (GDPR compliance), NFR9 (audit log), AC5 de Epic 6 ("Export devolve ZIP com JSON + CSV de toda a data do utilizador"), CON3 (PT-PT), NFR5 (RLS), NFR19 (imports absolutos).

1. **`POST /api/conta/export`** — iniciar pedido de export e devolver o ficheiro.
   - Requer autenticação (401 sem sessão). Apenas o próprio utilizador pode iniciar export do seu household.
   - Cria registo em `data_export_jobs` com `status='pending'`, `household_id` e `requested_by_user_id` a partir do JWT, via `getDb()` (INSERT permitido pela policy `data_export_jobs_insert_member`).
   - Imediatamente após o INSERT, `getServiceDb()` actualiza o job para `status='generating'` — UPDATE via role `authenticated` é bloqueado por design (policy `data_export_jobs_update_blocked: using(false)`); a escrita service-role exige verificação app-level de pertença (`job.household_id == household_id do JWT`) antes de qualquer escrita (ver AC8).
   - Geração síncrona/inline (PO-D1): recolhe dados via `getDb()`, gera ZIP em memória, persiste em Supabase Storage (`exports/{household_id}/{job_id}.zip`), `getServiceDb()` actualiza job para `status='ready'` com `download_url` (signed URL 24h) + `expires_at`. Devolve o URL imediatamente na resposta: `{ jobId, downloadUrl, expiresAt }`.
   - Insere `audit_log` com `action='data_export_requested'` (enum já existe).
   - Se já existe um job com `status IN ('pending','generating','ready')` e `expires_at > now()` para o household, devolve 409 com mensagem PT-PT ("Já tens um export em curso ou disponível para download.").
   - [Source: `packages/db/src/schema/audit.ts` — `dataExportJobs`, `dataExportStatusEnum`, `auditActionEnum`; `packages/db/migrations/0001_rls_policies.sql:593-604`; PRD FR28; Epic 6 AC5]

2. **`GET /api/conta/export/{jobId}`** — verificar estado e obter link de download.
   - Requer autenticação; SELECT usa `getDb()` (RLS garante isolamento por household); 404 se o job não pertencer ao household do utilizador (validar também na app para defesa em profundidade).
   - Devolve `{ jobId, status, downloadUrl?, expiresAt?, createdAt }`.
   - Se `status='ready'` e `expiresAt < now()`: `getServiceDb()` actualiza `status='expired'` (UPDATE via `authenticated` bloqueado por design — ver AC8); devolver `downloadUrl: null` ao cliente.
   - Se `status='failed'`, inclui mensagem de erro genérica PT-PT.
   - [Source: `dataExportJobs.storagePath`/`downloadUrl`/`expiresAt`/`status`; `dataExportStatusEnum` (`pending`/`generating`/`ready`/`expired`/`failed`)]

3. **Conteúdo do export — tabelas incluídas** (household-scoped via `getDb()` + RLS):
   - **Tarefas:** `tasks`, `task_recurrences`, `tags`, `task_tags` (todas com `household_id` CASCADE).
   - **Finanças:** `accounts`, `cards`, `categories` (per-household: `WHERE household_id = $hid`), `transactions`, `recurrences` (Finanças), `installments`.
   - **Household e membros:** `households` (a row do household), `household_members` (membros + papéis), `household_invites` (histórico de convites — incluir para completude GDPR).
   - **Preferências:** `user_prefs` (preferências do utilizador autenticado — `WHERE user_id = auth.uid()`).
   - **Kanban:** `kanban_columns`.
   - **Audit log:** `audit_log` (entradas onde `household_id = $hid` — omitir campos `ip`/`user_agent` por privacidade, ou anonimizar para `/24`).
   - **Não incluído no MVP** (billing CONGELADO): `subscriptions`, `invoices`, `payment_methods`, `feature_flags`.
   - **Não incluído** (dados externos não geridos pela app): entradas `auth.users` (gerido pelo Supabase Auth — só exportar campos públicos como `email`, `created_at` a partir do JWT/contexto disponível), logs Sentry/Grafana (não PII directo).
   - [Source: `packages/db/src/schema/*.ts` — tabelas auditadas em RGPD-1 Dev Notes; CLAUDE.md §Multi-tenancy; `finance.ts` nota: categorias globais (`is_default=true`) NÃO exportadas — só as per-household]

4. **Formato do ZIP:** `{household_id}-export-{YYYYMMDD}.zip` contém:
   - Um ficheiro JSON por entidade: `tasks.json`, `accounts.json`, `transactions.json`, etc. (array de objectos com todos os campos; valores monetários em cêntimos — nomes de campo snake_case técnicos para interoperabilidade Art. 20).
   - Um ficheiro CSV por entidade: mesmos dados em formato tabular (`tasks.csv`, `accounts.csv`, `transactions.csv`, etc.) com as seguintes convenções (PO-D2 + PO-D3):
     - **Headers CSV em PT-PT** (legibilidade humana; o JSON preserva snake_case técnico).
     - **Separador de colunas: `;` (ponto-e-vírgula)** — obrigatório porque os valores decimais usam vírgula (PO-D3; separador `,` criaria ambiguidade).
     - **Valores monetários:** coluna `amount_cents` (inteiro — fonte de verdade, CON9) **E** coluna `amount_eur` em formato decimal PT-PT (vírgula decimal, ex.: `78,70`). Ambas as colunas obrigatórias nos CSVs de entidades financeiras.
     - Encoding UTF-8 BOM (`\xEF\xBB\xBF`) para compatibilidade Excel PT.
   - Ficheiro `README.txt` em PT-PT com: data do export, household exportado, descrição das entidades, nota sobre valores monetários (cêntimos + decimal PT-PT), nota sobre dados não incluídos (billing, logs externos), nota sobre separador `;` nos CSVs.
   - [Source: PRD FR28, Epic 6 AC5 "ZIP com JSON + CSV"; CON3 PT-PT; CON9 EUR cêntimos; PO-D2; PO-D3]

   - O upload e a geração do signed URL requerem service role (bucket privado, só Admin API pode escrever). Implementar como `[DEV-DECISION]` com o precedente D-12C (ver AC8).
   - Após `expires_at`, o signed URL deixa de funcionar; a linha do job fica `status='expired'`.
   - [Source: `dataExportJobs.storagePath`/`downloadUrl`/`expiresAt`; NFR11 data residency UE; BLOQUEADOR EXTERNO §topo]
