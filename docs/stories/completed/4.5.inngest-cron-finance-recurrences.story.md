## User Story

**Como** utilizador da Expressia com finanças recorrentes configuradas no agregado familiar (ex: "Renda" todo dia 8 de cada mês, "Internet NOS" mensal, "Salário" quinzenal),
**Quero** que o sistema gere automaticamente todos os dias as transacções de finanças recorrentes cujo `next_run_on <= hoje`, sem eu ter de fazer nada,
**Para que** as minhas finanças recorrentes apareçam automaticamente nas vistas "Este mês", "Variáveis" (filtro recorrentes) e na projecção dos próximos 30 dias sem intervenção manual — garantindo que nenhuma transacção recorrente é gerada em duplicado mesmo que o cron corra duas vezes (at-least-once delivery do Inngest).

import { addDays, addWeeks, addMonths, addQuarters, addYears, parseISO, formatISO } from 'date-fns';

/**
 * Calcula a próxima data de execução de uma recorrência financeira,
 * dado o `currentRunDate` (data que acabou de ser executada).
 *
 * Retorna a próxima data como string YYYY-MM-DD, ou `null` se a

 *
 * Nota: usa aritmética de calendário pura (date-fns) — sem RRULE.
 * Finanças recorrentes têm frequências simples (daily/weekly/biweekly/
 * monthly/quarterly/yearly/custom). 'custom' usa `customRrule` como
 * expressão RRULE-like futura — no MVP, trata como `monthly` se
 * `customRrule` não for interpretável.
 */
export function calcNextRunDate(
  recurrence: {
    frequency: 'daily' | 'weekly' | 'biweekly' | 'monthly' | 'quarterly' | 'yearly' | 'custom';
    interval: number;
    customRrule: string | null;
    endsOn: string | null; // YYYY-MM-DD
  },
  currentRunDate: string, // YYYY-MM-DD — data que acabou de ser executada
): string | null {
  // ...
  // 1. Calcular nextDate a partir de currentRunDate + frequency/interval:
  //    - daily   → addDays(currentRunDate, interval)
  //    - weekly  → addWeeks(currentRunDate, interval)
  //    - biweekly → addWeeks(currentRunDate, 2) (ignora interval)
  //    - monthly  → addMonths(currentRunDate, interval)
  //    - quarterly → addQuarters(currentRunDate, interval)
  //    - yearly   → addYears(currentRunDate, interval)
  //    - custom   → addMonths(currentRunDate, interval) // fallback MVP KISS
  // 2. Se endsOn NOT NULL e nextDate > endsOn → return null (esgotada)
  // 3. Retornar nextDate em formato YYYY-MM-DD via formatISO(nextDate, { representation: 'date' })
}

/**
 * Verifica se uma recorrência está devida hoje ou passou:
 *   nextRunOn <= today (DP4=A)
 */
export function isRecurrenceDue(nextRunOn: string | null, today: string): boolean {
  if (nextRunOn === null) return false; // nunca processada + starts_on no futuro gerido pelo SELECT
  return nextRunOn <= today;
}

### AC3 — Inngest function `generate-finance-recurrences.ts`

Criar `apps/web/src/lib/inngest/functions/generate-finance-recurrences.ts` seguindo o padrão de `generate-recurring-tasks.ts` (Story 3.7 v1.4):

**Trigger:**

```ts
inngest.createFunction(
  { id: FINANCE_CRON_ID, name: 'Generate finance recurrences' },
  { cron: '0 3 * * *' }, // alinhado Story 2.8 D38 + Story 3.7 D-3.7.4
  handler,
)
```

**Handler `step.run('generate-finance-transactions')`:**

1. Calcular `today = formatISO(new Date(), { representation: 'date' })` (YYYY-MM-DD).
2. `SELECT * FROM recurrences WHERE active = true AND (next_run_on IS NULL OR next_run_on <= $today)` via `getServiceDb()` (RLS-bypass — job sistémico sem JWT user, padrão Story 3.7 D-3.7.5 / Story 2.8 D38).
3. Para cada recurrence `r`:
   - **Determinar `transactionDate`:** se `r.nextRunOn IS NULL` (nunca correu), usar `r.startsOn`; senão usar `r.nextRunOn`.
   - **INSERT em `transactions` ON CONFLICT DO NOTHING** com chave idempotente `(recurrence_id, transaction_date)` (AC1 índice unique):
     ```sql
     INSERT INTO transactions (
       household_id, created_by_user_id,
       account_id, card_id, category_id,
       amount_cents, currency, kind, description,
       transaction_date, payment_method,
       recurrence_id, is_projected,
       agent_run_id, notes
     ) VALUES (
       $r.householdId, $r.createdByUserId,
       $r.accountId, $r.cardId, $r.categoryId,
       $r.amountCents, 'EUR', $r.kind, $r.description,
       $transactionDate, $r.paymentMethod,
       $r.id, false,  -- is_projected=false (transacção do dia corrente)
       NULL, NULL
     )
     ON CONFLICT (recurrence_id, transaction_date)
       WHERE recurrence_id IS NOT NULL
     DO NOTHING
     RETURNING id
     ```
   - Contar `insertedId` (retornou `id` → `count_generated++`) vs vazio (conflito → `count_skipped++`).
   - **Calcular próxima data:** `nextDate = calcNextRunDate(r, transactionDate)` (AC2 helper).
   - **UPDATE `recurrences`:**
     - Se `nextDate !== null`: `UPDATE recurrences SET next_run_on = $nextDate, updated_at = now() WHERE id = $r.id`.
     - Se `nextDate === null` (esgotada — `endsOn` passou): `UPDATE recurrences SET active = false, next_run_on = NULL, updated_at = now() WHERE id = $r.id`.
4. Acumular `totalGenerated`, `totalSkipped`, `processedRecurrences`, `inactivatedRecurrences`.

**Observability:**
- `childLogger({ job: 'generate-finance-recurrences' })` → Pino structured log com `recurrence_id`, `transaction_date`, `count_generated`, `count_skipped` por recorrência + summary final.
- Span OTel `finance.recurrences.generate` via `withSpan` (AC5).
- `captureException` em catch — Inngest retry ADR-005 max 4 attempts.

**Retorno:** `{ total_generated: number, total_skipped: number, processed_recurrences: number, inactivated_recurrences: number }`.

**AC3 PASS:** ficheiro criado + INSERT ON CONFLICT DO NOTHING + UPDATE next_run_on + Pino logs + retorno tipado.

### [AUTO-DECISION D-4.5.1] `finance_recurrences_generated` enum distinto de `recurrences_generated` (Story 3.7)

**Decisão:** Migration 0015 adiciona `'finance_recurrences_generated'` ao enum `audit_action`. NÃO reusa `'recurrences_generated'` da Story 3.7.

**Justificação:**

1. **Desambiguação:** `recurrences_generated` (Story 3.7) é emitido pelo cron de TAREFAS (`generate-recurring-tasks`). `finance_recurrences_generated` é emitido pelo cron de FINANÇAS (`generate-finance-recurrences`). São jobs distintos, filtrável independentemente em Grafana/logs.
2. **Consistência com prefixo `finance_`:** Story 4.1 estabeleceu a convenção `finance_recurrence.*` (vs `recurrence.*` de Tasks). O job sistémico segue a mesma convenção.
3. **Trace:** `audit.ts:77-107` — secção `// Finanças` prefixada com `finance_`; Story 4.1 precedente explícito.

### [AUTO-DECISION D-4.5.2] Sem reuso do `rrule-helpers.ts` de Tarefas — aritmética de calendário pura

**Decisão:** Story 4.5 cria `finance-recurrence-helpers.ts` com `calcNextRunDate` via `date-fns` (`addDays/Weeks/Months/Quarters/Years`), sem usar o `rrule-helpers.ts` da Story 3.7.

**Justificação:**

1. **Enum distinto:** `recurrenceFreqFinanceEnum` (`daily`, `weekly`, `biweekly`, `monthly`, `quarterly`, `yearly`, `custom`) NÃO tem `weekdays`/`weekends` (que exigem RRULE). Tem `quarterly` (não existente em Tasks).
2. **DP4=A escopo mínimo:** cron materializa apenas o dia corrente — não um horizonte de 90 dias. Expandir N ocorrências futuras via RRULE seria over-engineering para este scope.
3. **KISS:** para frequências simples (daily/weekly/biweekly/monthly/quarterly/yearly), `addMonths(date, n)` de `date-fns` é mais directo e legível que RRULE strings. `date-fns@^4` já é dependência instalada.
4. **`custom` no MVP:** custom RRULE é aceite na criação mas o cron trata como `monthly` (fallback) até um helper RRULE dedicado ser implementado futuramente. Documentado explicitamente no JSDoc do helper.

**Trace:** `finance.ts:68-76` (enum sem `weekdays`/`weekends`), DP4=A (Epic §8).

### [AUTO-DECISION D-4.5.3] `is_projected=false` para transacções do cron (DP4=A)

**Decisão:** Transacções criadas pelo cron têm `is_projected=false`.

**Justificação:**

1. **Semântica:** `is_projected=true` é para transacções futuras materializadas (prestações — Story 4.4 DP8=A). O cron de recorrências materializa a transacção do **dia corrente** (`next_run_on <= today`) — é uma transacção que "aconteceu" hoje, não futura.
2. **Correctness das agregações:** a vista "Este mês" (Story 4.6) filtra `is_projected=false` para totais reais vs projecção. Marcar como `false` garante que as transacções recorrentes entram no total real, não na projecção.
3. **Trace:** `transactions.ts:394` (`isProjected boolean default false`), DP2=C (prestações `is_projected=true`, recorrências on-the-fly — confirmado que o cron apenas materializa no dia corrente).

### [AUTO-DECISION D-4.5.4] `customRrule` tratado como fallback `monthly` no cron MVP

**Decisão:** Se `recurrence.frequency = 'custom'`, o `calcNextRunDate` usa `addMonths(date, interval)` (monthly fallback). O campo `customRrule` é ignorado pelo cron no MVP.

**Justificação:**

1. **KISS MVP:** implementar um parser RRULE completo para o cron de Finanças no MVP seria scope-creep. A API `POST /api/financas/recorrencias` aceita o campo mas o cron não o processa.
2. **Transparência:** documentado explicitamente no JSDoc de `calcNextRunDate` e no runbook AC8 como limitação conhecida do MVP.
3. **Re-avaliar:** se em produção algum utilizador usar `custom`, o fallback `monthly` é safe (não causa divergência de dados, apenas a frequência pode ser incorrecta). Rastreável via `recurrences.custom_rrule` + `recurrences.frequency = 'custom'` para implementar RRULE futuro.

### [AUTO-DECISION D-4.5.5] `transactionDate = nextRunOn` (ou `startsOn` se NULL) — sem ajuste de timezone

**Decisão:** O cron não aplica conversão de timezone para `transactionDate`. Usa o valor de `nextRunOn` (ou `startsOn`) directamente como string YYYY-MM-DD.

**Justificação:**

1. **Diferença de domínio vs Tarefas:** Story 3.7 precisou de DST mitigation porque as datas de tasks precisam de corresponder ao dia percepcionado pelo utilizador PT (timezone Europe/Lisbon). Finanças `transactionDate` é uma **data financeira** (quando a operação ocorreu), não um instante de hora-a-hora — a data YYYY-MM-DD armazenada em `recurrences.starts_on` e `recurrences.next_run_on` já é a data financeira correcta.
2. **Schema `date` type:** Postgres `date` não tem timezone — armazena apenas YYYY-MM-DD. Não há ambiguidade de UTC vs local.
3. **Simplificação:** eliminação de `date-fns-tz` nesta story (sem nova dependência).
