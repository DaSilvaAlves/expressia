## Acceptance Criteria

### AC1 — Tool `criar_tarefa` (scope minimal — DP5: A)

Tool implementada em `packages/tools/src/tasks/criar-tarefa.ts` e conforme contrato `ToolDefinition<I, O>`:

- **Input Zod schema** (`inputSchema`):
  ```ts
  z.object({
    title: z.string().min(1).max(200),
    dueDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).optional(),  // ISO 8601 date
    priority: z.enum(['low', 'medium', 'high']).optional().default('medium'),
  })
  ```
  [Trace: `packages/db/src/schema/tasks.ts` — `title text NOT NULL`, `dueDate date`, `priority taskPriorityEnum default 'medium'`]

- **Output Zod schema** (`outputSchema`):
  ```ts
  z.object({
    taskId: z.string().uuid(),
    title: z.string(),
    dueDate: z.string().nullable(),
    priority: z.enum(['low', 'medium', 'high']),
  })
  ```

- **`execute(input, ctx)`:** INSERT em `tasks` via Drizzle (`ctx.db`) com `householdId = ctx.householdId`, `createdByUserId = ctx.userId`, `status = 'todo'`. Retorna `{ taskId, title, dueDate, priority }`. **Nunca usa `getServiceDb()`** — usa `ctx.db` (RLS activa).

- **`reverse(output, ctx)`:** retorna `ReverseOpPayload` do tipo `{ kind: 'delete_row', table: 'tasks', id: output.taskId }` — permite undo via `executeAtomic` + endpoint `POST /api/agent/prompt/[runId]/undo` (Story 2.8).

- **`preview(input, ctx)`:** string PT-PT para card preview-then-confirm: `"Criar tarefa '${input.title}'"` (+ `" para ${formatDate(input.dueDate)}"` se `dueDate` presente). [Trace: PRD FR4 — preview-then-confirm quando confiança < 0.70]

- **Metadados obrigatórios:**
  - `name: 'criar_tarefa'`
  - `domain: 'tasks'` (`ToolDomain`)
  - `description`: string PT-PT para o LLM — descreve quando usar (ex: "Usa esta tool quando o utilizador quer criar uma nova tarefa ou to-do.")
  - `estimatedTokens: 50` (dica cost router — Story 2.9)

### AC2 — Tool `completar_tarefa`

Tool implementada em `packages/tools/src/tasks/completar-tarefa.ts`:

- **Input Zod schema:**
  ```ts
  z.object({
    taskId: z.string().uuid().optional(),
    taskTitle: z.string().min(1).optional(),
  }).refine(d => d.taskId !== undefined || d.taskTitle !== undefined, {
    message: 'Fornecer taskId ou taskTitle',
  })
  ```

- **Resolução fuzzy por título:** se `taskId` ausente, pesquisa em `tasks` WHERE `householdId = ctx.householdId` AND `status != 'done'` AND `LOWER(title) LIKE LOWER('%${taskTitle}%')` com LIMIT 1. Se zero matches → lança `ToolExecutionError` com `userMessage` PT-PT: `"Não encontrei nenhuma tarefa com o nome '${taskTitle}'. Verifica o nome e tenta novamente."`. Se múltiplos matches → usa o mais recente por `createdAt DESC` e inclui aviso no output. [Trace: EPIC-3-EXECUTION §stories[3.8] "fuzzy match by title"]

- **`execute(input, ctx)`:** UPDATE `tasks` SET `status = 'done'`, `completedAt = now()` WHERE `id = resolvedTaskId AND householdId = ctx.householdId` (RLS double-check). Retorna `{ taskId: resolvedTaskId, title, prevStatus, completedAt }`.

- **`reverse(output, ctx)`:** retorna `{ kind: 'restore_row', table: 'tasks', id: output.taskId, snapshot: { status: output.prevStatus, completedAt: null } }` para undo. [Trace: Story 2.8 pattern + EPIC-3-EXECUTION §risks[R6]]

- **`preview(input, ctx)`:** `"Marcar '${resolvedTitle}' como concluída"` (resolvedTitle calculado em preview via query leve se `taskTitle` fornecido).

- **Metadados:** `name: 'completar_tarefa'`, `domain: 'tasks'`, `estimatedTokens: 80` (inclui query fuzzy).

### AC3 — Tool `listar_tarefas` (read-only)

Tool implementada em `packages/tools/src/tasks/listar-tarefas.ts`:

- **Input Zod schema:**
  ```ts
  z.object({
    status: z.enum(['todo', 'doing', 'done', 'archived']).optional(),
    dueDateFrom: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).optional(),
    dueDateTo: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).optional(),
    limit: z.number().int().min(1).max(50).optional().default(10),
  })
  ```

- **`execute(input, ctx)`:** SELECT em `tasks` WHERE `householdId = ctx.householdId` com filtros aplicados. ORDER BY `dueDate ASC NULLS LAST, createdAt DESC`. LIMIT `input.limit`. Retorna `{ tasks: Array<{ id, title, dueDate, priority, status }>, count: number }`.

- **Read-only — `reverse()` retorna sentinela inerte (R1b v1.1):** método `reverse` retorna `{ kind: 'delete_row', table: '_noop', id: crypto.randomUUID() }`. **Porquê:** `executeAtomic` (`packages/tools/src/atomic.ts:217-261`) **sempre** chama `reverse()` e **sempre** persiste em `agent_reverse_ops` — não há condicional sobre se `execute()` fez writes. `ReverseOpDeleteRowSchema` (`contracts.ts:158`) exige `id: z.string().uuid()` — `'noop'` literal falha Zod. Solução: UUID válido + table sentinela `_noop` (não existe fisicamente). Row persiste mas é **inerte** — filtrada por job cleanup Inngest (Risk R-3.8.5). **Documentar no JSDoc:** "Tool read-only — produz row inerte em `agent_reverse_ops` com `table='_noop'`. FR6 undo conceptualmente não-aplicável; endpoint `/undo` deve responder 410 Gone para `table='_noop'`." [Trace: EPIC-3-EXECUTION §stories[3.8] "read-only, sem reverse_op (FR6 não-aplicável)" — reinterpretado como "reverse_op inerte" para satisfazer contract obrigatório Story 2.3 Done]

- **`preview(input, ctx)`:** `"Listar tarefas"` (+ resumo dos filtros se presentes).

- **Metadados:** `name: 'listar_tarefas'`, `domain: 'tasks'`, `estimatedTokens: 100`.

### AC4 — Tool `listar_atrasadas` (read-only)

Tool implementada em `packages/tools/src/tasks/listar-atrasadas.ts`:

- **Input Zod schema:**
  ```ts
  z.object({
    limit: z.number().int().min(1).max(20).optional().default(10),
  })
  ```

- **`execute(input, ctx)`:** SELECT em `tasks` WHERE `householdId = ctx.householdId` AND `status NOT IN ('done', 'archived')` AND `dueDate < CURRENT_DATE` ORDER BY `dueDate ASC` LIMIT `input.limit`. Usa index `tasks_overdue_idx` (confirmado em `tasks.ts:106`). Retorna `{ tasks: Array<{ id, title, dueDate, priority, daysOverdue }>, count: number }`. Campo `daysOverdue` calculado em SELECT: `CURRENT_DATE - due_date::date`.

- **Read-only — mesmo padrão AC3 sentinela inerte (R1b v1.1):** `reverse()` retorna `{ kind: 'delete_row', table: '_noop', id: crypto.randomUUID() }`. Mesma justificação que AC3 — `executeAtomic` sempre persiste, sentinela `_noop` torna row inerte. JSDoc igual a `listar_tarefas`. [Trace: EPIC-3-EXECUTION §stories[3.8] + Risk R-3.8.5]

- **`preview(input, ctx)`:** `"Listar tarefas em atraso"`.

- **Metadados:** `name: 'listar_atrasadas'`, `domain: 'tasks'`, `estimatedTokens: 80`.

### AC6 — Migration enum `agentIntentEnum` (nova migration — R2 v1.1)

Nova migration `packages/db/migrations/0012_agent_intent_tasks_tools.sql` adiciona **3 novos valores** ao enum `agent_intent` (`criar_tarefa` **já existe** em `0000_initial_schema.sql:125` — confirmado por PO cross-confirm). Postgres suporta `ALTER TYPE ... ADD VALUE` sem downtime em 16+:

```sql
-- criar_tarefa já existe desde 0000_initial_schema.sql:125 — mantido com IF NOT EXISTS
-- por defensive programming (no-op idempotente, sem risco)
ALTER TYPE agent_intent ADD VALUE IF NOT EXISTS 'criar_tarefa';
-- 3 valores genuinamente novos:
ALTER TYPE agent_intent ADD VALUE IF NOT EXISTS 'completar_tarefa';
ALTER TYPE agent_intent ADD VALUE IF NOT EXISTS 'listar_tarefas';
ALTER TYPE agent_intent ADD VALUE IF NOT EXISTS 'listar_atrasadas';
```

**Sem migration Drizzle** para o schema Drizzle `agentIntentEnum` em `packages/db/src/schema/agent.ts` — actualizar manualmente o array enum **adicionando apenas os 3 valores novos** (`criar_tarefa` já está no array literal — verificar antes de adicionar, pode existir já). [Trace: anti-hallucination verificação #9 v1.1 — `criar_tarefa` confirmado existente em `agent.ts:47` + `0000_initial_schema.sql:125`. Pattern idempotente `IF NOT EXISTS` garante safety em reruns.]

**Sem breaking change:** `ALTER TYPE ... ADD VALUE` é não-destrutivo; queries existentes sobre `agent_intent` não são afectadas. Manter as 4 linhas `ALTER TYPE` no SQL (uma é no-op) é preferível a remover por: (a) defensive programming, (b) consistência semântica com lista da story, (c) zero overhead em reruns.

### Contrato `ToolDefinition<I, O>` — padrão a seguir

[Source: `packages/tools/src/contracts.ts` + Story 2.3 AC2]

```ts
interface ToolDefinition<I, O> {
  readonly name: string;              // snake_case, lowercase (e.g., 'criar_tarefa')
  readonly domain: ToolDomain;        // 'tasks' para esta story
  readonly description: string;       // PT-PT — descreve QUANDO usar (passado ao LLM)
  readonly inputSchema: z.ZodType<I>;
  readonly outputSchema: z.ZodType<O>;
  readonly requiredPlan?: PlanTier;   // ausência = free+
  readonly estimatedTokens?: number;  // dica cost router (Story 2.9)
  preview(input: I, ctx: ToolExecutionContext): string;
  execute(input: I, ctx: ToolExecutionContext): Promise<O>;
  reverse(output: O, ctx: ToolExecutionContext): Promise<ReverseOpPayload>;
}
```

`ToolExecutionContext` tem: `{ householdId: string; userId: string; db: DrizzleDbClient; traceId: string; runId: string }`.
