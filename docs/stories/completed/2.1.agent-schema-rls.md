### Padrão de teste RLS (Story 1.4 AC10)

```typescript
// Estrutura padrão de cada test file (ver agent_runs.rls.test.ts como modelo):
import { afterAll, beforeEach, describe, expect, test } from 'vitest';
import { admin, insertAgentRun } from '@/helpers/fixtures';
import { asUser, closeRlsHarness, expectRlsBlocks, resetData, seedTwoHouseholds } from '@/rls-harness';

afterAll(async () => { await closeRlsHarness(); });

describe('RLS isolation: <tabela>', () => {
  beforeEach(async () => { await resetData(); });

  test('cross-household SELECT bloqueado', async () => {
    const { householdA, householdB, userA, userB } = await seedTwoHouseholds();
    // admin() = service_role (bypassa RLS para preparar dados)
    await insertAlgo(admin(), householdA.id, ...);

    // asUser() activa SET ROLE authenticated + JWT claims com household_id
    await asUser(userB.id, householdB.id, async (sql) => {
      const rows = await sql`select id from public.<tabela>`;
      expect(rows).toHaveLength(0); // não vê dados de household A
    });
  });
});
```

**`expectRlsBlocks(userId, householdId, fn)`** → retorna `true` se a função lançou erro de permissão. Usar para testes de INSERT/UPDATE/DELETE bloqueados.

### Imutabilidade — análise do estado actual

A policy `agent_runs_update_self` (linha 288-294 de `0001_rls_policies.sql`) PERMITE UPDATE por `authenticated` sem restrição de estado. O comentário diz *"imutabilidade reforçada por convenção aplicacional"* — esta story formaliza o enforcement via trigger.

**Decisão a tomar pelo @dev (Task 3.2-3.3):**
- **Opção A (trigger):** `BEFORE UPDATE ON agent_runs FOR EACH ROW WHEN (OLD.status IN ('success', 'reverted', 'failed'))` → raise exception excepto se `current_role = 'service_role'`. Mais robusto, independente de convenção.
- **Opção B (column-level):** Não existe column-level RLS em Postgres nativo — esta opção não é viável sem trigger ou application-layer whitelist.

Recomendação: **Opção A**. Submeter decisão ao @architect no quality gate se houver dúvida.

### TTL dos reverse_ops (FR6)

O schema Drizzle tem `expiresAt: timestamp('expires_at', { withTimezone: true }).notNull()` mas **sem `.default()`** na linha 201 do `agent.ts`. Isto significa que o caller (pipeline executor em Story 2.5) é responsável por passar `expires_at = now() + interval '30 seconds'`. Esta story verifica se há DEFAULT na migration (`0000_initial_schema.sql`) — se não houver, adicionar em `0005_agent_immutability.sql` como camada de segurança extra.

### Completion Notes

**Decisões arquitecturais tomadas durante implementação:**

1. **Trigger semantics — quem bypassa?** Initial design: `current_user = 'service_role'` only. Refinado para: bloquear apenas roles aplicacionais (`authenticated`, `anon`); permitir tudo o resto (service_role, postgres, supabase_auth_admin). Razão: o migration runner (`apply-migrations.ts`) conecta como `postgres` superuser; bloquear esse role partia migrations e DBA ops. A semântica correcta é "o utilizador final via JWT não pode mutar audit log", não "apenas Inngest pode".

2. **Bootstrap do test harness — grant service_role.** Bootstrap em `packages/db-test/src/harness/bootstrap.ts` criava `service_role` com `bypassrls` mas sem `grant select/insert/update/delete on tables`. Em produção (Supabase) esses grants vêm por default. Adicionei `alter default privileges in schema public grant ... to service_role` no bootstrap (test-only — L4) para o test environment espelhar produção. Sem isso, o teste "service_role bypassa trigger" falhava com `permission denied for table agent_runs` (não pelo trigger, mas pelo grants ausentes).

3. **NÃO tocar `agent_runs_update_self` policy existente.** A policy continua a aplicar (RLS layer 1: only own rows + own household). O trigger é layer 2: nem sequer permite UPDATE quando OLD.status terminal. Defense-in-depth: RLS bloqueia cross-tenant; trigger bloqueia mutação post-terminal.
