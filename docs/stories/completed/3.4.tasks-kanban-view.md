## User Story

**Como** utilizadora da Expressia (`@household-owner` ou `@household-member`) que tem tarefas nas colunas Kanban configuradas per household (seed default: "A fazer" / "Em curso" / "Concluído" aplicado em Story 3.1),
**Quero** uma vista quadro visual em `/tarefas/kanban` com colunas configuráveis (3 a 6 por household), drag-and-drop atómico entre colunas e intra-coluna, configuração de colunas via Sheet lateral (adicionar/renomear/reordenar/eliminar), filtro leve (search + tag), e acessibilidade keyboard-first com anúncios screen reader PT-PT,
**Para que** possa ver o estado das minhas tarefas em formato visual familiar (Kanban), reorganizar o fluxo de trabalho arrastando cards, e gerir as colunas do quadro sem sair da página — com paridade mobile (tap-and-hold 250ms + scroll horizontal snap) e performance garantida (< 200ms render inicial, ≥ 50fps drag desktop).

(e) Haptic feedback mobile: `navigator.vibrate(10)` no `onDragStart` se disponível (DP-3.4.5 recomendação A, 1 linha).

### 8. Endpoint POST /api/kanban-columns (criar coluna)

**AC8:** Handler `POST` no mesmo `route.ts`:

- Body Zod: `{ name: z.string().min(1).max(40), color?: z.string().regex(/^#[0-9A-Fa-f]{6}$/), sort_order?: z.number().int() }` com `.strict()`.
- Server valida `count(*) WHERE household_id < 6` antes de INSERT (constraint app-layer; migration 0010 adiciona CHECK DB — defesa dupla).
- `sort_order` omitido: `MAX(sort_order) + 1` per household.
- Nome único per household: `INSERT ... ON CONFLICT (household_id, name)` → `409 Conflict`.
- `insertAuditLog({ entityTable: 'kanban_columns', entityId: newId, action: 'kanban_column.created', ... })` — **PO-FIX-1+2 v1.1:** schema usa `entity_table` (não `entity_type`) e enum `audit_action` exige formato `entity.verb` (ver migration 0011 T10.1).
- Response: coluna criada `201 Created`.

**AC8 PASS:** `POST /api/kanban-columns` cria coluna, valida count < 6, nome único, audit_log entry, resposta 201. `409` se nome duplicado ou count = 6.

### 9. Endpoint PATCH /api/kanban-columns/[id] (actualizar coluna)

**AC9 PASS:** `PATCH /api/kanban-columns/[id]` actualiza parcialmente. `is_done_column=true` desliga outros em transaction. `sort_order` rebalanceia siblings. Audit_log com diff. `403` se household mismatch.

### 10. Endpoint DELETE /api/kanban-columns/[id] (eliminar coluna)

**AC10:** Handler `DELETE` no mesmo `[id]/route.ts`:

- Auth: `requireAuth` + `resolveHouseholdId` + `resolveHouseholdRole` → só `owner` ou `admin` pode eliminar (RLS `kanban_columns_delete_owner_admin` reforça server-side).
- Se coluna tem tasks (`COUNT(*) FROM tasks WHERE kanban_column_id = id > 0`): query param `?move_to=<column_id>` obrigatório; validar destino ∈ household e destino ≠ id.
  - Executar em transaction: `UPDATE tasks SET kanban_column_id = move_to WHERE kanban_column_id = id` + `DELETE FROM kanban_columns WHERE id = id`.
- Se coluna sem tasks: `DELETE` directo.
- Sem `move_to` quando tasks existem: `409 Conflict { tasks_count: N }`.
- `insertAuditLog({ entityTable: 'kanban_columns', entityId: id, action: 'kanban_column.deleted', beforeState: { column, tasks_moved_to: move_to ?? null } })` — **PO-FIX-1+2 v1.1:** schema usa `before_state` para snapshot (não `snapshot`); enum exige `kanban_column.deleted`.
- `403` PT-PT se membro sem role owner/admin.

**AC10 PASS:** `DELETE /api/kanban-columns/[id]` funciona com e sem move_to. Transaction atómica garante integridade referencial. RLS + resolveHouseholdRole em dupla defesa. Audit_log com snapshot. `409` se tasks existem sem move_to.

### 11. Endpoint PATCH /api/kanban-columns/batch (DP-3.4.6 Opção A)

**AC11:** Criar `apps/web/src/app/api/kanban-columns/batch/route.ts` — handler `PATCH`:

- Body Zod: `{ columns: [{id: uuid, sort_order: int, name?: string, is_done_column?: boolean}], creates?: [{name, color?, sort_order?}], deletes?: [{id, move_to?: uuid}] }`.
- **Tudo em `db.transaction()`** — rollback total se qualquer operação falhar (atomicidade DP-3.4.6 A; evita estado inconsistente se 1 update falhar).
- Valida invariants pós-batch: `3 ≤ count ≤ 6`, exactly `1 is_done_column=true`, `names únicos per household`.
- Returns: `{ columns: KanbanColumn[] }` estado final.
- `insertAuditLog({ entityTable: 'kanban_columns', entityId: null, action: 'kanban_column.batch_updated', afterState: { changes_summary } })` — **PO-FIX-1+2 v1.1:** schema usa `entity_table` + `after_state`. `entityId: null` aceitável (NFR9 não obriga 1 entity por batch). Enum value `kanban_column.batch_updated` (formato `entity.verb_pp` consistente com `task.completed`/`recurrence.updated`).
- Usado por `ColumnConfigSheet` ao "Guardar" — single request vs múltiplos PATCHs (DP-3.4.6 A).

**AC11 PASS:** `PATCH /api/kanban-columns/batch` executa todas as operações em transaction. Rollback total em falha parcial. Invariants validados. Audit_log com summary. Response com estado final.

### Estrutura DndContext mínima

```tsx
// KanbanBoardClient.tsx (ver pre-flight §14.2 para versão completa)
'use client';

import { DndContext, DragOverlay, closestCorners } from '@dnd-kit/core';
import { SortableContext, verticalListSortingStrategy } from '@dnd-kit/sortable';

export function KanbanBoardClient({ initialTasks, columns }: Props) {
  const [tasks, setTasks] = useOptimistic(initialTasks, optimisticReducer);
  const sensors = useSensors(
    useSensor(PointerSensor, { activationConstraint: { distance: 8 } }),
    useSensor(TouchSensor, { activationConstraint: { delay: 250, tolerance: 5 } }),
    useSensor(KeyboardSensor, { coordinateGetter: sortableKeyboardCoordinates }),
  );

  return (
    <DndContext
      sensors={sensors}
      collisionDetection={closestCorners}
      onDragStart={handleDragStart}
      onDragOver={handleDragOver}
      onDragEnd={handleDragEnd}
      onDragCancel={handleDragCancel}
      accessibility={{
        announcements: announcementsPt,        // §8.3 PT-PT
        screenReaderInstructions: {
          draggable: 'Premir espaço para mover. Setas para reposicionar. Enter para soltar. Esc para cancelar.',
        },
      }}
    >
      <KanbanBoard columns={columns} tasks={tasks} />
      <DragOverlay>
        {activeTask ? <KanbanCardOverlay task={activeTask} /> : null}
      </DragOverlay>
    </DndContext>
  );
}
```

### Pattern optimistic + revert (ver pre-flight §14.3 para versão completa)

```typescript
// handleDragEnd
async function handleDragEnd(event: DragEndEvent) {
  const previousTasks = tasks;
  setTasks(prev => moveTask(prev, taskId, targetColumnId, targetPosition));  // optimistic

  try {
    const res = await fetch(`/api/tasks/${taskId}/move`, {
      method: 'PATCH',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ kanbanColumnId: targetColumnId, kanbanPosition: targetPosition }),
    });
    if (!res.ok) throw new ApiError(res.status, await res.text());
  } catch (err) {
    setTasks(previousTasks);       // revert
    toast.error(mapErrorToCopy(err));   // §5.4 PT-PT mapping
    captureException(err);              // Sentry
  }
}
```
