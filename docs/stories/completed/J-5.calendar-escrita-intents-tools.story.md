## Story

**As a** Eurico (único utilizador do Jarvis),
**I want** poder marcar e reagendar eventos no Google Calendar directamente a partir do Telegram (ex.: "marca reunião com a Ana sexta às 15h" ou "reagenda a reunião de amanhã para segunda às 10h"),
**so that** consigo gerir a minha agenda em conversa, com o mesmo fluxo de confirmação e botão de cancelar que já uso para tarefas e finanças.

## Contexto de âmbito

**Esta story entrega:**
- Dois novos intents no motor: `criar_evento_calendario` e `reagendar_evento_calendario`.
- Novas tools de escrita no Google Calendar com `execute()` + reverse_op de undo.
- Actualização do scope OAuth de `calendar.readonly` para `calendar.events` + re-consentimento one-shot do Eurico.
- Extensão do motor de undo para suportar reversão de chamadas externas (Calendar API).

**NÃO inclui (fora de âmbito):**
- Eliminar evento via chat (pode vir em J-5.x ou v1.2 futura).
- Listar/consultar agenda via chat (já tratado no brief de J-4; `consultar_dados` pode responder parcialmente a pedidos de leitura).
- Gmail (J-6 e J-7 — epic v1.1, histórias separadas).
- Notificações/lembretes proactivos sobre eventos.
- Gestão de convidados, links de videoconferência, ou outros campos avançados do evento.
- UI web para calendário.

**Depende de:** J-3 Done (OAuth readonly, `google_oauth_tokens` em prod, `refreshAccessToken` implementado, `token-cipher.ts` implementado).

6. Existe o módulo `apps/web/src/lib/agent/tools/calendar/create-calendar-event.ts` que implementa a tool `ToolDefinition` com `name: 'criar_evento_calendario'`, `domain: 'calendar'`. A tool:
   - `inputSchema`: inclui `title: string` (obrigatório), `start: string` (ISO-8601 datetime, obrigatório), `end?: string` (ISO-8601 datetime; se omitido, o @dev decide o comportamento padrão, ex.: 1 hora após `start`), `description?: string`.
   - `preview(input)`: devolve texto PT-PT como "Vou criar o evento '{title}' no dia {data formatada} às {hora}."
   - `execute(input, ctx)`: obtém o `accessToken` via `refreshAccessToken` (lendo `google_oauth_tokens` do `ctx.db` com RLS); chama `POST /calendars/primary/events` da Google Calendar API; devolve o `eventId` (string) e dados do evento criado. **JUSTIFICAÇÃO da excepção à regra "no HTTP in execute":** a Google Calendar API é externa ao Postgres — não pode participar numa transacção Postgres. A chamada HTTP ocorre dentro de `execute()` por necessidade arquitectural. Para minimizar inconsistência: todos os writes Postgres (audit) ocorrem ANTES da chamada HTTP; se a chamada HTTP falhar, a transacção Postgres faz rollback. A operação não é 100% atómica (se a chamada HTTP tiver sucesso mas o Postgres falhar depois, o evento fica criado no Calendar sem registo local), mas este trade-off é aceitável para v1.1 dada a baixa frequência do caso e a possibilidade de undo manual.
   - `reverse(output, ctx)`: devolve um `ReverseOpPayload` com `kind: 'external_call'`, `provider: 'google_calendar'`, `operation: 'delete_event'`, `eventId: <string>` (novo kind definido no AC7).

7. Existe o módulo `apps/web/src/lib/agent/tools/calendar/update-calendar-event.ts` que implementa a tool `ToolDefinition` com `name: 'reagendar_evento_calendario'`, `domain: 'calendar'`. A tool:
   - `inputSchema`: inclui `query: string` (descrição do evento a reagendar, ex.: "reunião de amanhã de manhã"), `newStart: string` (ISO-8601 datetime, novo horário de início), `newEnd?: string` (novo horário de fim; se omitido, mantém a duração original).
   - `preview(input)`: devolve texto PT-PT como "Vou reagendar '{query}' para {data+hora}."
   - `execute(input, ctx)`: obtém `accessToken`; pesquisa o evento mais provável em `GET /calendars/primary/events` por correspondência ao `query` (strategy simples: eventos futuros próximos de hoje, ordenados por proximidade temporal, primeiro resultado); guarda os dados originais (start, end) para o reverse_op; chama `PATCH /calendars/primary/events/{eventId}` com os novos horários; devolve `eventId`, `originalStart`, `originalEnd`, `newStart`, `newEnd`. Se nenhum evento for encontrado ou se mais de 5 candidatos existirem sem match inequívoco, lança erro PT-PT que o motor traduz em preview/clarificação para o utilizador.
   - `reverse(output, ctx)`: devolve `{ kind: 'external_call', provider: 'google_calendar', operation: 'restore_event', eventId: <string>, originalStart: <string>, originalEnd: <string> }`.

8. `ReverseOpPayload` em `packages/tools/src/contracts.ts` ganha um novo kind `external_call`:
   ```ts
   export const ReverseOpExternalCallSchema = z.object({
     kind: z.literal('external_call'),
     provider: z.literal('google_calendar'),
     operation: z.union([z.literal('delete_event'), z.literal('restore_event')]),
     eventId: z.string().min(1),
     originalStart: z.string().optional(),  // ISO-8601; para restore_event
     originalEnd:   z.string().optional(),  // ISO-8601; para restore_event
   });
   ```
   O `ReverseOpPayloadSchema` Zod (discriminatedUnion) inclui este novo kind. Os schemas `ReverseOpDeleteRowSchema`, `ReverseOpRestoreRowSchema`, `ReverseOpReinsertRowSchema` e `composite` são preservados sem alteração. `serializeReverseOp` e `deserializeReverseOp` funcionam correctamente com o novo kind.

9. A função `executeUndo` (exportada de `apps/web/src/app/api/agent/prompt/[runId]/undo/route.ts`) é extendida para reconhecer e tratar o kind `external_call` com `provider: 'google_calendar'`:
   - **CRÍTICO — tipo local:** o tipo `ReverseOp` em `undo/route.ts` (~linha 69-73) é uma CÓPIA LOCAL (não importa de `@meu-jarvis/tools`). Este tipo tem de ser extendido com o union do `ReverseOpExternalCallSchema`, caso contrário `applyReverseOp` cai no branch `throw new Error('Kind desconhecido')` (~linha 422) sem nunca tratar o undo de Calendar. Ver Tarefa 7.3.0.
   - `operation: 'delete_event'`: lê `google_oauth_tokens` inline via `getServiceDb()` por `(household_id = householdId, user_id = userId)` — `userId` já está em scope em `executeUndo` (`params.userId`); chama `refreshAccessToken` para obter `accessToken`; chama `DELETE /calendars/primary/events/{eventId}` da Calendar API. Sucesso (204) → `{ ok: true }`. Falha (ex.: evento já apagado, 404) → `{ ok: false, reason: 'not_found' }`.
   - `operation: 'restore_event'`: obtém `accessToken` com o mesmo padrão; chama `PATCH /calendars/primary/events/{eventId}` com os `originalStart`/`originalEnd` para repor o horário anterior. Sucesso → `{ ok: true }`. Falha → `{ ok: false, reason: 'external_error' }`.
   - O fallback para kinds `delete_row`/`restore_row`/`reinsert_row`/`composite` permanece sem alteração.
   - A janela de 30 s e toda a lógica existente de undo (transição de estado, audit log) são preservadas.

### Localização das calendar tools — justificação de direcção de dependência

As calendar tools vivem em `apps/web/src/lib/agent/tools/calendar/`, **NÃO** em `packages/tools/src/`. Esta não é "a localização padrão dos tools existentes" — é uma excepção justificada:

**Motivo:** `packages/tools` só pode depender de `@meu-jarvis/db` e `@meu-jarvis/observability`. Não pode importar de `apps/web`. As calendar tools precisam de:
- `@/lib/google/oauth.ts` → `refreshAccessToken` (decifra token + chama Google API)
- `@/lib/crypto/token-cipher.ts` → `decryptToken` (AES-256-GCM)

Ambos vivem em `apps/web`. Colocar as calendar tools em `packages/tools` criaria uma dependência circular (`packages/tools` → `apps/web` → `packages/tools`).

**Registo:** as tools de tasks/finance registam-se pelo barrel `packages/tools/src/index.ts` (importado em toda a parte que usa `@meu-jarvis/tools`). As calendar tools registam-se via `apps/web/src/lib/agent/tools/calendar/index.ts`, importado como side-effect no `run-agent.ts`. O `toolRegistry` é o mesmo objecto singleton de `@meu-jarvis/tools` — só o ponto de registo é diferente.

**FOOTGUN de tree-shaking:** imports de side-effect (`import 'módulo'` sem desestruturar) podem ser eliminados por bundlers agressivos (Next.js/turbopack em modo produção). A Tarefa 7.6 adiciona um teste que detecta esta regressão.

### Constraint crítica — chamadas HTTP em execute()

O contrato `ToolDefinition.execute()` em `contracts.ts` diz: "NUNCA fazer chamadas externas (HTTP, LLM, email) — quebra atomicidade."

As tools de Calendar **quebram esta regra por necessidade arquitectural**: o Google Calendar é uma API externa que não participa em transacções Postgres. A excepção é justificada e documentada aqui para o @dev:

1. **Ordem de operações dentro de execute():** quaisquer writes Postgres (audit) fazem-se ANTES da chamada HTTP. Se o Postgres falhar, a chamada HTTP nunca ocorre → sem inconsistência.
2. **Inconsistência residual:** se a chamada HTTP ao Calendar tiver sucesso mas o Postgres falhar depois (ex.: inserir reverse_op), o evento fica criado no Calendar sem registo local. Este caso é improvável (Postgres é o nosso dado de maior confiança; o evento pode ser apagado manualmente). Para v1.1 este trade-off é aceitável.
3. **Alternativa rejeitada:** pré-chamar a Calendar API antes de `executeAtomic` e passar o `eventId` à tool (two-phase execution) — adiciona complexidade ao pipeline que não está justificada para v1.1.

O @dev deve incluir este trade-off no Dev Agent Record.

### executeUndo — extensão para external_call

O branch novo em `executeUndo` (simplificado):
```ts
if (op.kind === 'external_call' && op.provider === 'google_calendar') {
  // Obter token via getServiceDb() (legítimo — undo corre fora de sessão HTTP).
  // householdId e userId vêm de params (executeUndo já os tem no scope).
  // NÃO usar getGoogleTokenForHousehold — helper inexistente; ler inline por (household_id, user_id).
  const serviceDb = getServiceDb();
  const rows = await serviceDb.execute(
    sql`SELECT encrypted_refresh_token, token_iv, token_auth_tag
        FROM public.google_oauth_tokens
        WHERE household_id = ${householdId} AND user_id = ${userId}
        LIMIT 1`
  );
  if (!rows[0]) return { ok: false, reason: 'not_found' as const };
  const row = rows[0] as { encrypted_refresh_token: string; token_iv: string; token_auth_tag: string };
  const { accessToken } = await refreshAccessToken(
    row.encrypted_refresh_token,
    row.token_iv,
    row.token_auth_tag,
  );

  if (op.operation === 'delete_event') {
    const res = await fetch(
      `https://www.googleapis.com/calendar/v3/calendars/primary/events/${op.eventId}`,
      { method: 'DELETE', headers: { Authorization: `Bearer ${accessToken}` } }
    );
    if (res.status === 404) return { ok: false, reason: 'not_found' as const };
    return { ok: res.ok };
  }
  if (op.operation === 'restore_event') {
    const res = await fetch(
      `https://www.googleapis.com/calendar/v3/calendars/primary/events/${op.eventId}`,
      { method: 'PATCH',
        headers: { Authorization: `Bearer ${accessToken}`, 'Content-Type': 'application/json' },
        body: JSON.stringify({
          start: { dateTime: op.originalStart, timeZone: 'Europe/Lisbon' },
          end:   { dateTime: op.originalEnd,   timeZone: 'Europe/Lisbon' },
        })
      }
    );
    return { ok: res.ok };
  }
}
```
