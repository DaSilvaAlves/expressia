### 5. Estratégias de DELETE divergentes + scope variable-only

**AC5:** Os DELETE/PATCH handlers respeitam as características de cada tabela:

(a) **`transactions` — hard delete (DP-4.3.4).** A tabela `transactions` **não tem coluna `archived_at`** (`finance.ts:358-431` — confirmado). Nenhuma FK aponta para `transactions.id`. Sob DP1=A (recompute on-read — Epic §8), o saldo é `initial_balance_cents + SUM transactions`; remover a row remove-a correctamente do agregado. DELETE faz `DELETE FROM public.transactions WHERE id = X` (RLS filtra). Resposta `{ deleted: true, id }`.

(b) **`categories` — soft delete (DP-4.3.5).** A tabela `categories` **tem coluna `archived_at`** (`finance.ts:201`). Hard delete dispararia `transactions.category_id ON DELETE set null` (`finance.ts:370`) — perda da categorização do histórico. DELETE faz `UPDATE public.categories SET archived_at = now(), updated_at = now() WHERE id = X`. Resposta `{ archived: true, id }`.

(c) **Scope variable-only (DP-4.3.3).** PATCH e DELETE de `transactions` aplicam-se **só a transacções variáveis (manuais)** — `recurrence_id IS NULL AND installment_id IS NULL`. Editar/eliminar directamente uma transacção gerada por recorrência ou prestação causaria drift face à fonte (`recurrences`/`installments`). O handler detecta `recurrence_id` ou `installment_id NOT NULL` no SELECT prévio e devolve **409 CONFLICT** ("Transacção gerada por recorrência ou prestação não pode ser editada directamente.") ANTES da mutação.

**AC5 PASS:** transactions DELETE hard + categories DELETE soft; transacção gerada → 409 em PATCH/DELETE; tests cobrem os 3 casos.

### 6. Validações compostas

**AC6:**

(a) **`transactions` — `account_or_card`.** `TransactionCreateSchema` aplica `.refine()`: pelo menos um de `account_id`/`card_id` presente → 400 VALIDATION_ERROR "Transacção requer conta ou cartão." Alinha o CHECK `transactions_account_or_card` (`finance.ts:418-421`) — defesa em profundidade. No PATCH (`TransactionUpdateSchema` não tem refinamento composto — não conhece o estado existente): pôr ambos a `null` viola o CHECK do DB; o handler PATCH detecta `/transactions_account_or_card/` na mensagem de erro e converte em 400 VALIDATION_ERROR (não 500). Pattern idêntico ao D-4.2.C da Story 4.2.

(b) **`transactions` — FK cross-household.** `account_id`/`card_id`/`category_id` (quando presentes) têm de pertencer ao household — verificado por SELECT RLS-scoped (`getDb()` não encontra rows cross-household) → 404 NOT_FOUND "Conta/Cartão/Categoria não encontrada." se inválido. (Categorias globais — `household_id NULL` — são válidas como `category_id`: a verificação aceita-as.)

(c) **`categories` — `parent_id` 1-nível.** Se `parent_id` presente: tem de resolver a uma categoria visível ao household (própria ou global), **não-arquivada**, e essa categoria-pai tem de ter `parent_id IS NULL` (hierarquia máxima 1 nível — `finance.ts:182` comment) → 404 NOT_FOUND "Categoria-pai não encontrada." ou 400 VALIDATION_ERROR "Categorias suportam apenas 1 nível de hierarquia." se a pai já for sub-categoria. **NOTA:** a coluna `categories.parent_id` NÃO tem FK no schema (`finance.ts:196` — `uuid('parent_id')` sem `.references()`) — a verificação aplicacional é a única defesa, obrigatória.

(d) **`categories` — nome único `[PO_FIX_INLINE F1]`.** O schema tem a constraint `categories_unique_global_name` = `unique(household_id, name)` (`finance.ts:210`) — dentro de um household (e entre globais) o `name` é único. Um POST com nome duplicado, ou um PATCH que renomeie para um nome colidente, dispara um `unique_violation` (SQLSTATE `23505`) do Postgres. O handler detecta `/categories_unique_global_name/i` na mensagem de erro e converte em **409 CONFLICT** ("Já existe uma categoria com esse nome.") — defesa em profundidade, pattern idêntico ao D-4.2.C da Story 4.2. Aplica-se a POST e PATCH de `categorias`.

**AC6 PASS:** Tests cobrem: transacção sem conta/cartão → 400; `account_id` cross-household → 404; PATCH transacção a anular ambos → 400; `parent_id` inexistente → 404; `parent_id` apontando a sub-categoria → 400; POST categoria com nome duplicado → 409 (PO_FIX F1).
