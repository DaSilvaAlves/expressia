## Story

**As a** Eurico (único utilizador do Jarvis v1),
**I want** que o bot de Telegram deixe de ecoar e passe a agir — criar tarefas e registar finanças a partir das minhas mensagens, com confirmação para intenções ambíguas e botão de cancelar —,
**so that** posso conversar com o Jarvis no Telegram para tratar das minhas tarefas e finanças do dia-a-dia, sem ter de abrir a web app, com o motor cognitivo já provado a 95% a trabalhar por baixo.

## Acceptance Criteria

1. A tabela `telegram_link` existe no schema Postgres com colunas `id uuid PK`, `household_id uuid NOT NULL FK households(id) ON DELETE CASCADE`, `user_id uuid NOT NULL FK auth.users(id) ON DELETE CASCADE`, `chat_id bigint NOT NULL UNIQUE`, `created_at timestamptz NOT NULL DEFAULT now()`, `updated_at timestamptz NOT NULL DEFAULT now()`. Migration com número `0027` aplicada via `pnpm db:migrate`.
2. A tabela `telegram_link` tem exactamente 4 RLS policies (SELECT, INSERT, UPDATE, DELETE) e `pnpm check:rls` passa com exit 0 (sem regressão das 28+ tabelas existentes).
3. A tabela `telegram_link` tem um índice único em `chat_id` (garantir unicidade mesmo sem constraint duplicado). Existe também uma linha seed para o Eurico inserida via script de seed ou instrução no Dev Notes (não é criada no código de produção — é configuração de arranque).
4. A função `runAgentForHousehold({ userId, householdId, prompt }): Promise<AgentResult>` existe em `apps/web/src/lib/agent/run-agent.ts` e executa o pipeline completo (classifier → cache → cost-router → preview-gate → executeAtomic → reverse-op) **sem** chamar `supabase.auth.getUser()` nem `resolveHouseholdId()`. O tipo `AgentResult` distingue `{ mode: 'executed' }` de `{ mode: 'preview' }` idêntico ao contrato actual de `POST /api/agent/prompt`.
5. O route handler `apps/web/src/app/api/agent/prompt/route.ts` passa a ser um wrapper fino: resolve `{userId, householdId}` via Supabase Auth (sem alterar a lógica de auth actual) e delega em `runAgentForHousehold`. O comportamento externo do endpoint é idêntico ao actual — todos os testes existentes da suite passam sem alteração. Nota: `AgentResult` usa camelCase internamente (`runId`, `undoUrl`, `undoExpiresAt`); o wrapper HTTP serializa para o snake_case actual da resposta JSON (`run_id`, `undo_url`, `undo_expires_at`, `results`) sem alterar o contrato de API público.
6. O webhook do Telegram (`apps/web/src/app/api/telegram/webhook/route.ts`) resolve `chat_id` → `{ userId, householdId }` via query à tabela `telegram_link` usando `getServiceDb()` (sem sessão Supabase) e chama `runAgentForHousehold`. Se o `chat_id` não existir em `telegram_link`, o handler devolve `200` silencioso (comportamento igual ao da allowlist de J-1 para chat desconhecido).
7. Para mensagens de texto com resultado `mode: 'executed'`, o webhook envia ao `chat_id` do Eurico uma mensagem de confirmação da acção executada (ex.: "Feito. Criei a tarefa '…' com prazo …") mais um botão inline `(Cancelar)` com `callback_data = "undo:{run_id}"`, activo durante 30 s.
8. Para resultados `mode: 'preview'` (confiança < 0,70), o webhook envia uma pergunta de confirmação (ex.: "Não tenho a certeza. Quiseste dizer: …? Responde **sim** para confirmar ou diz-me o que é.") com botões inline `sim` e `não`.
9. O handler trata `callback_query` do tipo `undo:{run_id}`: se dentro do janela de 30 s, chama a lógica de undo existente (`agent_reverse_ops`) e confirma ao utilizador; após 30 s, responde "Cancelar já não é possível (expirou)." — reutiliza o `POST /api/agent/prompt/[runId]/undo` como função reutilizável, sem duplicar lógica.
10. O handler trata `callback_query` de confirmação `confirm:{run_id}`: chama a lógica de confirmação existente (`POST /api/agent/prompt/[runId]/confirm`) como função reutilizável e executa a acção. Resposta negativa (`não`) cancela e informa o utilizador.
11. `sendChatAction('typing')` é chamado imediatamente após receber a mensagem do Eurico, antes de invocar `runAgentForHousehold`, para sinalizar processamento ao Telegram.
12. Conteúdo de mensagens e dados do utilizador são redigidos nos logs. O `user_id` é hashado com `hashForCorrelation` importado de `@meu-jarvis/observability` (re-exportado em `index.ts:12`, já usado em `route.ts:38-40`). Logs contêm apenas `update_id`, hash de correlação do `user_id`, e tipo de resultado — nunca texto das mensagens nem dados financeiros/de tarefa em claro.
13. `pnpm lint` (--max-warnings=0), `pnpm typecheck`, `pnpm --filter @meu-jarvis/web test` (incluindo novos testes unitários da story), `pnpm build` e `pnpm check:rls` passam sem erros.

- O `callback_query` de confirmação (`sim`/`não`) e undo (`(Cancelar)`) **reutiliza** a lógica já existente nos sub-handlers `[runId]/confirm/route.ts` e `[runId]/undo/route.ts` — não duplica. O objectivo é extrair essas funções como chamáveis directamente, sem passar por HTTP interno.

```sql

create table if not exists public.telegram_link (
  id           uuid        not null default gen_random_uuid() primary key,
  household_id uuid        not null references public.households(id) on delete cascade,
  user_id      uuid        not null references auth.users(id) on delete cascade,
  chat_id      bigint      not null unique,
  created_at   timestamptz not null default now(),
  updated_at   timestamptz not null default now()
);

create index if not exists telegram_link_household_id_idx on public.telegram_link (household_id);

create trigger telegram_link_set_updated_at
  before update on public.telegram_link
  for each row execute function public.set_updated_at();

-- RLS
alter table public.telegram_link enable row level security;

create policy "telegram_link_select" on public.telegram_link
  for select using (household_id = current_household_id());

create policy "telegram_link_insert" on public.telegram_link
  for insert with check (household_id = current_household_id());

create policy "telegram_link_update" on public.telegram_link
  for update using (household_id = current_household_id());

create policy "telegram_link_delete" on public.telegram_link
  for delete using (household_id = current_household_id());
```

### Resolução de identidade no webhook (sem sessão Supabase)

O webhook do Telegram não tem JWT de utilizador. A resolução usa `getServiceDb()` para ler `telegram_link`:

```ts
// Dentro do webhook handler — após verificar secret + extrair chat_id
const serviceDb = getServiceDb();
const rows = await serviceDb.execute<{ user_id: string; household_id: string }>(
  sql`select user_id, household_id from public.telegram_link where chat_id = ${chatId} limit 1`
);
if (!rows[0]) {
  // chat_id não registado → 200 silencioso
  return new Response(null, { status: 200 });
}
const { user_id: userId, household_id: householdId } = rows[0];
```

`getServiceDb()` é **legítimo** aqui (está na lista de usos legítimos de SEC-10: jobs/handlers que resolvem identidade fora de sessão HTTP): lê a própria tabela de mapeamento de identidade, não dados do utilizador. A partir daqui, `runAgentForHousehold` usa `withHousehold({userId, householdId})` — RLS fica viva.

```ts
export async function sendChatAction(chatId: number, action: 'typing'): Promise<void> {
  const token = process.env.TELEGRAM_BOT_TOKEN;
  if (!token) throw new Error('[telegram] TELEGRAM_BOT_TOKEN não definido');
  await fetch(`https://api.telegram.org/bot${token}/sendChatAction`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ chat_id: chatId, action }),
  });
  // Falha silenciosa — é apenas UX indicator; não afecta o resultado
}

export async function answerCallbackQuery(
  callbackQueryId: string,
  text?: string,
): Promise<void> {
  const token = process.env.TELEGRAM_BOT_TOKEN;
  if (!token) throw new Error('[telegram] TELEGRAM_BOT_TOKEN não definido');
  const res = await fetch(
    `https://api.telegram.org/bot${token}/answerCallbackQuery`,
    {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ callback_query_id: callbackQueryId, text }),
    }
  );
  if (!res.ok) {
    console.error(`[telegram] answerCallbackQuery falhou: ${res.status}`);
  }
}
```

- **[DEV-DECISION D-J2.1]** As 4 RLS policies de `telegram_link` vivem em `0001_rls_policies.sql` (DO-block condicional `if exists`), NÃO em `0027`. Pattern idêntico a `user_prefs` (0007+0001) e `agent_rate_limit_counters` (0006+0001). A `0027` cria tabela + index + trigger; a `0001` cria as policies. Predicate: `household_id = public.current_household_id()` (as 4). A DDL de referência da story (que punha as policies na 0027) foi adaptada a este pattern canónico do projecto.
- **[DEV-DECISION D-J2.2]** `runAgentForHousehold` devolve `AgentRunOutcome` (superset discriminado) em vez de só `AgentResult`. O superset inclui os casos de controlo de fluxo que o route handler tem de traduzir em HTTP específicos (replay/idempotency_in_progress/rate_limited/quota_exceeded) sem `NextResponse` na lib. `AgentResult` (executed|preview, camelCase, conforme AC4) é o subconjunto público consumido pelo webhook. A função NUNCA devolve `NextResponse`; lança os erros do pipeline (ClassifierError/PlannerError/ToolError/AtomicExecutionError) para cada caller mapear.
- **[DEV-DECISION D-J2.3 / D-J2.4]** `executeUndo`/`executeConfirm` devolvem resultados discriminados (`UndoResult`/`ConfirmResult` com `ok`+`reason`) em vez de lançar — comunicam o estado de erro (not_found/expired/invalid_state/...) tanto ao route handler (→ HTTP status + error code histórico, regressão zero) como ao webhook (→ mensagem PT-PT), sem HTTP interno nem duplicação. O caso de sucesso de `executeConfirm` embute o equivalente a `AgentResult` executed.
- **[DEV-DECISION D-J2.5]** O audit-log de `failed` (Classifier/Planner/Executor) passou a ocorrer DENTRO de `runAgentForHousehold` (try/catch que faz `updateAfterExecutor(failed)` + re-throw com `runId` anexado), em vez de nos handlers de erro do route. Razão: o registo de `failed` pertence ao pipeline, não ao transporte HTTP — assim qualquer caller (route ou webhook) regista o `failed` consistentemente.
