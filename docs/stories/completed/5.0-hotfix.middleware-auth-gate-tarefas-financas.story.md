## Contexto e motivação

Durante a sessão da Story 5.8, o @po (Pax) **confirmou byte-a-byte** um achado de segurança que já tinha sido registado **duas vezes** sem resolução:

- **CO-2 da Story 5.4** — "middleware APP_PATH_PREFIXES folding 5.10" (carry-over preservado, nunca executado).
- **FUP-5.3.C da Story 5.3** — "middleware APP_PATH_PREFIXES fix Story 5.10 MEDIUM".
- **AC6 reportado pela Story 5.10** (draft por validar).

O auth gate do middleware (`apps/web/src/middleware.ts:36`) define:

```typescript
const APP_PATH_PREFIXES = ['/visao', '/jarvis', '/conta'] as const;
```

A docstring do próprio ficheiro (`middleware.ts:8`) afirma que `/(app)/**` "(visão, tarefas, finanças, perfil…)" requer sessão válida — **mas o array omite `/tarefas` e `/financas`**, ambas adicionadas pelos Epics 3 (tarefas) e 4 (finanças) sem actualizar o array. A linha 70-76 (`isAppPath = APP_PATH_PREFIXES.some(p => pathname.startsWith(p))`) nunca dá match nessas rotas, logo o redirect para `/entrar` (linha 71-76) nunca dispara para utilizadores não autenticados.

O `(app)/layout.tsx:28-30` delega ao `<AppShell>` **sem guarda server-side própria** (a linha 23 até afirma "auth gate intacto") — **não há segunda linha de defesa** ao nível da rota.

**Padrão de reincidência:** o `PO_FIX_INLINE 4` da Story 2.7 (`middleware.ts:30-32`) criou este array precisamente para evitar o bug, depois de `/jarvis` e `/conta/preferencias` terem ficado públicas. O array voltou a falhar. A causa raiz é a **ausência de um teste que force cobertura** — esta story corrige isso (DP-HF.A), não só os dois prefixos em falta.

### AC3 — Teste de cobertura auto-mantido (anti-reincidência, DP-HF.A)

**AC3:** Um teste verifica que **todos** os segmentos de topo de `apps/web/src/app/(app)/*/` estão cobertos por `APP_PATH_PREFIXES`, de modo a que uma rota `(app)/` futura sem cobertura **parta o build**.

**(a)** O teste (ambiente `node`) enumera os subdirectórios de topo de `apps/web/src/app/(app)/` (ex.: `visao`, `jarvis`, `conta`, `tarefas`, `financas`) — excluindo ficheiros não-rota (ex.: `layout.tsx`, `_components/`, directórios prefixados por `_` ou `(`).
**(b)** Para cada segmento, assert que existe um prefixo em `APP_PATH_PREFIXES` tal que o auth gate o cobre (`'/' + segmento` começa por algum prefixo do array).
**(c)** O teste documenta no nome/descrição que serve para impedir a reincidência do CO-2 (rota nova sem auth gate).

**AC3 PASS:** o teste passa com o estado actual (5 segmentos, 5 prefixos); um teste de meta-verificação (ou inspecção manual documentada) confirma que adicionar um segmento fake não coberto fá-lo falhar. [@qa ratifica o mecanismo no gate.]

## Dev Notes

**Núcleo do fix (AC1).** A correcção funcional é literalmente adicionar dois elementos ao array `APP_PATH_PREFIXES` (`middleware.ts:36`). O `matcher` (`middleware.ts:90-102`) **já** apanha `/tarefas` e `/financas` (a regex só exclui `_next/static`, `_next/image`, `favicon.ico`, `api/billing`, `api/cron`, e estáticos com extensão) — portanto o middleware **já corre** nessas rotas; o que falha é a condição `isAppPath` (linha 70) que nunca dá match. Não tocar no matcher.

**Molde de teste (AC2).** O `__tests__/middleware.test.ts` (Story 2.7) já tem o padrão exacto: `vi.mock('@supabase/ssr', ...)` com `mocks.getUserMock` hoisted; `makeRequest(pathname)` cria `NextRequest`; redirect espera `status === 307` + `location` contém `/entrar` + `next=%2F...` (path URL-encoded). Os casos novos copiam este padrão. O ambiente é `node` (`@vitest-environment node` na linha 1) — logo `fs` está disponível para o teste de cobertura (AC3).

**Teste de cobertura (AC3, DP-HF.A).** É o coração desta story — sem ele, o array reincide. Enumerar `app/(app)/*/` por `fs.readdirSync(..., { withFileTypes: true })`, filtrar directórios cujo nome não começa por `_` nem `(` (route groups e privados), comparar cada segmento contra `APP_PATH_PREFIXES`. Cuidado: o array é `as const` e não exportado — pode ser preciso exportá-lo do `middleware.ts` (ex.: `export const APP_PATH_PREFIXES`) para o teste o importar, OU duplicar a fonte de verdade (evitar — preferir export). [DEV-DECISION: exportar `APP_PATH_PREFIXES` para teste é a opção limpa; confirmar que o export não quebra o bundle Edge do middleware.]

**Defesa em profundidade (DP-HF.B).** O `(app)/layout.tsx:28-30` é hoje um Server Component fino que devolve `<AppShell>{children}</AppShell>` sem `getUser()`. Adicionar guarda aqui seria belt-and-suspenders. Decisão deferida ao @architect — recomendação @sm: manter mínimo (B) e registar follow-up, mas aceitável incluir (A) se trivial. Se incluída, usar o server client de `@meu-jarvis/auth` (verificar API real — `middleware.ts` usa `createServerClient` de `@supabase/ssr` inline porque não tem `next/headers`; o layout TEM `next/headers`, logo pode usar o helper canónico do package auth). **Não inventar** a API.

**Verificação runtime (AC4 — @qa).** O @po foi explícito: confirmou a condição estrutural (gate quebrado) mas **não** testou o que um GET anónimo de `/financas/este-mes` devolve. Provavelmente a RLS (NFR5 — `getDb()` role `authenticated` sem `household_id` claim) impede leitura, mas isto **não foi confirmado runtime**. O @qa deve fechar esta lacuna no security gate — não dar a fuga por nula sem evidência.
