### 2. Migration `0009_kanban_seed.sql` — defaults PT-PT per household

**AC2:** Criar `packages/db/migrations/0009_kanban_seed.sql` que:

(a) Cria SQL function `seed_household_kanban_defaults(p_household_id uuid) returns void` que insere 3 `kanban_columns` defaults per household:
- 'A fazer' (sort_order=0, color='#6B7280', is_done_column='false')
- 'Em curso' (sort_order=1, color='#3B82F6', is_done_column='false')
- 'Concluído' (sort_order=2, color='#10B981', is_done_column='true')

(b) Função é **idempotente** — usa `INSERT ... ON CONFLICT (household_id, sort_order) DO NOTHING` (constraint unique já existe per `tenancy.ts:171`). Re-executar a função em household que já tem colunas não duplica nem actualiza.

(c) Trigger `trigger_seed_kanban_after_household_insert` AFTER INSERT em `households` chama a função automaticamente. Households criados pelo signup/invite recebem 3 colunas sem código aplicação.

(d) Migration backfill: aplica `seed_household_kanban_defaults(h.id)` para TODOS os households existentes em produção (idempotente — safe re-run). SELECT que itera linhas + chamada da função.

(e) **PT-PT estrito:** Os 3 nomes em PT-PT europeu — não 'To do/Doing/Done' (EN), não 'A fazer/Em progresso/Concluído' (PT-BR). 'Em curso' é o termo PT-PT correcto (não 'Em progresso' que é PT-BR/anglicismo).

**AC2 PASS:** Migration aplica em CI rls-gate Postgres efémero sem erros + idempotente verified em re-run.

### 5. Suite Testcontainers RLS — 20 cenários cross-household

**AC5:** Criar `packages/db/test/rls-tasks.test.ts` (ou `packages/db-test/src/rls/tasks-isolation.test.ts` se Story 1.4 harness Done) com 20 testes (5 tabelas × 4 operações):

| Tabela | SELECT | INSERT | UPDATE | DELETE |
|--------|--------|--------|--------|--------|
| `tasks` | T1 | T2 | T3 | T4 |
| `task_recurrences` | T5 | T6 | T7 | T8 |
| `tags` | T9 | T10 | T11 | T12 (delete restrito a owner/admin per `0001:416-418` — variant `tags_delete_owner_admin`) |
| `task_tags` | T13 | T14 | T15 | T16 |
| `kanban_columns` | T17 | T18 | T19 | T20 (delete restrito a owner/admin per `0001:166-168` — variant `kanban_columns_delete_owner_admin`) |

**Variants nota (PO_FIX_INLINE v1.1):** Duas tabelas (`tags` + `kanban_columns`) usam policy DELETE `owner_admin` em vez do padrão `member`. Para os testes T12 e T20, o setup precisa de garantir que U1 (criador) é owner/admin do H1 e que U2 (cross-household) é owner/admin do H2 — caso contrário a assertion "U2 NÃO consegue DELETE" pode dar falso-positivo (falha por privilégio insuficiente em vez de RLS). Restantes 18 testes usam `member` standard.

**Setup:** 2 households (`H1`, `H2`) com 1 user cada (`U1@H1`, `U2@H2`). Cada teste:
1. Authenticate as `U1` (JWT com `household_id=H1`).
2. Criar row em `H1` para a tabela em test.
3. Authenticate as `U2` (JWT com `household_id=H2`).
4. Tentar a operação (SELECT/INSERT/UPDATE/DELETE) sobre row de `H1`.
5. **Assert:** RLS bloqueia — SELECT retorna 0 rows; INSERT/UPDATE/DELETE retorna 0 rows affected (ou erro permission denied conforme policy).

**AC5 PASS:** 20/20 testes PASS — RLS isolation provada cross-household para todas as tabelas Tarefas.
