**Como** utilizador da Expressia que gere as finanças do agregado familiar via Jarvis,
**Quero** poder dizer ao Expressia coisas como "paguei €78,70 no supermercado com o cartão Millennium", "renda 600 euros todo o dia 1", "adiciona o cartão Activobank com fecho no dia 25 e vencimento no dia 5", "comprei o portátil de €1.200 em 12 prestações no Activobank" e "quanto gastei este mês?",

### [AUTO-DECISION D-4.10.1] Tool naming convention — EN (Epic literal) em vez de PT (precedente 3.8)

**Decisão:** As 5 tools desta story usam nomes em **inglês** (`create_finance_variable`, `create_finance_recurrence`, `create_card`, `create_installment`, `query_finance_summary`) conforme Epic §5 literal — em vez de mimetizarem o PT da Story 3.8 (`criar_tarefa`).

**Justificação:** Epic 4 §5 lista os 5 nomes em EN explicitamente. O `toolRegistry` é case-sensitive mas indiferente à língua do nome. Refactor de 3.8 para EN é scope creep, parte o snapshot test do prompt já APROVADO pelo @architect na 3.8 gate, e quebra a invariante "uma story = uma alteração coerente". Convenção a estabilizar pós-Epic 4: **EN para tool names cross-domain**. Drift entre `criar_tarefa` (3.8) e `create_*` (4.10) é aceite como tech-debt LOW — pode ser harmonizado no Epic 5/6 ou no primeiro refactor que tocar o `toolRegistry`. **@po deve confirmar.**

### [AUTO-DECISION D-4.10.2] Zero migration — `agent_intent` enum já cobre as 5 intents de Finanças

**Decisão:** Esta story **NÃO inclui migration** ao enum `agent_intent`. Os 5 intents de Finanças (`criar_financa_variavel`, `criar_financa_recorrente`, `criar_cartao`, `criar_parcelada`, `consultar_dados`) já existem desde a migration `0000_initial_schema.sql:125` (verificação #6).

**Justificação:** Diferentemente da Story 3.8 (que adicionou 3 intents novos via migration `0012`), aqui o enum já contempla todos os intents alvo. Adicionar uma migration "no-op" (apenas `ALTER TYPE ADD VALUE IF NOT EXISTS` redundante) seria ruído. Story 4.10 fica zero-DB-change.

### [AUTO-DECISION D-4.10.3] Tools de Finanças seguem a sentinela `_noop` da 3.8 para read-only

**Decisão:** `query_finance_summary` retorna `reverse_op = { kind: 'delete_row', table: '_noop', id: crypto.randomUUID() }` — pattern R1b da Story 3.8.

**Justificação:** `executeAtomic` (`atomic.ts:217-261`) **sempre** persiste em `agent_reverse_ops`. `ReverseOpDeleteRowSchema` exige `id` UUID válido. `_noop` é a sentinela acordada na 3.8 — row inerte filtrada por job cleanup; endpoint `/undo` responde 410 Gone. Reuse total do pattern, zero invenção.

### [AUTO-DECISION D-4.10.4] `create_installment` usa composite reverse_op **aninhado** — sem migração FK

**Decisão:** A `reverse_op` de `create_installment` é um `composite` de até 7 ops top-level: `[{ delete_row installments }, sub-composite[10 tx deletes], sub-composite[10 tx deletes], ...]`. Para N=60 (máximo schema), 6 sub-composites de 10 transactions cada cobrem com folga; para N=12 (típico), 1 sub-composite de 10 + 1 de 2.

**Justificação:** `COMPOSITE_REVERSE_OP_MAX_OPS=10` é **per-level** (verificação #7) — top-level com 13 ops directos é rejeitado pelo Zod. Aninhar respeita o limite sem alterar `contracts.ts` ou o schema FK. Alternativas consideradas e rejeitadas:
- **Alt-A** (mudar `transactions.installment_id` ON DELETE SET NULL → CASCADE via nova migration): cleaner mas adiciona uma migration e altera invariantes documentadas — scope creep.
- **Alt-B** (elevar `COMPOSITE_REVERSE_OP_MAX_OPS` para 100): mudança de contrato global, requer @architect signoff em separado.
- **Alt-C** (introduzir `kind: 'cascade_delete'`): novo tipo de reverse_op — out-of-scope.

A alt escolhida (aninhar) é zero-schema-change e reusa o composite existente. **Ordem de execução é crítica** (T11 verifica): no payload, sub-composites de transactions vêm ANTES do `delete_row installments` para evitar CHECK violation `transactions_installment_index_coherent` durante o undo (verificação #5).

### [AUTO-DECISION D-4.10.5] `query_finance_summary` reusa helpers existentes via `ctx.db`

**Decisão:** `query_finance_summary.execute()` chama `getMonthSummary({ db: ctx.db, monthAnchor })` (Story 4.6) e `getAccountBalances({ db: ctx.db })` (Story 4.9) — em vez de re-implementar as queries.

**Justificação:** Helpers são `db`-injectáveis (D-4.6.7 / D-4.9.4) e foram precisamente desenhados para reuse. Garante coerência: o número que o utilizador vê na UI (`/financas/este-mes`, `/financas/patrimonio`) é o mesmo que o Jarvis devolve via chat. Zero divergência calculatória. R-4.10 (consistência de agregações) automaticamente herdada.

### [AUTO-DECISION D-4.10.6] PII whitelist OTel — finance domain tem regras adicionais

**Decisão:** Spans `agent.tool.call` para tools de Finanças seguem a whitelist da Story 3.8 (NFR12) + **regras adicionais**: NUNCA incluir `amount_cents`, `description`, `last4`, `iban_last4`, `card_id`, `account_id`, `transaction_id` em span attributes. Apenas: `tool.name`, `tool.domain='finance'`, `tool.success`, `tool.duration_ms`, `household.hash` (SHA-256 trunc 8), `run.id`.

**Justificação:** Valores monetários, descrições de transacções, últimos 4 dígitos de cartão e IBAN são PII financeira sensível (NFR12 + GDPR). O `household_id` raw já é PII (Story 3.8 AC7); o `card_id` / `account_id` são pseudónimos mas identificáveis cruzados com a DB → tratados como PII por defensive default. Métricas operacionais (`duration_ms`, `success`) suficientes para o dashboard Grafana.

### [AUTO-DECISION D-4.10.7] Cálculo de prestações extraído para helper puro

**Decisão:** A fórmula `perInstallmentCents = floor(total/N); lastInstallmentCents = total - (N-1)*per` é isolada num helper puro `computeInstallmentSplit(totalAmountCents, numInstallments)` em `packages/tools/src/finance/_helpers/installment-split.ts`. Inputs: 2 inteiros. Output: `{ perInstallmentCents, lastInstallmentCents, transactionAmounts: number[] }`.

**Justificação:** Núcleo de risco R-4.1 isolado, testável sem DB, sem Postgres. Espelha o pattern da Story 4.9 (`getAccountBalances` em helper puro `db`-injectável). Se a Story 4.4 já tiver o cálculo num helper exportável, reusar (verificação #13 confirma a fórmula mas não a localização). Se não tiver, esta story cria o helper canónico e a Story 4.4 pode opcionalmente refactor-importar mais tarde — fora de scope.

### [AUTO-DECISION D-4.10.8] `categoryId` resolve "Outros gastos"/"Outros rendimentos" por kind — **[PO_FIX_INLINE F6 aplicado @po]**

**Decisão (revista pelo @po 2026-05-23):** Quando `categoryId` é omitido em `create_finance_variable` / `create_finance_recurrence` / `create_installment` (DP-4.10.D ratificada @po), a tool resolve **dentro do execute()** via helper `resolveDefaultCategory({ db: ctx.db, kind })`:
  - Para `kind='expense'`: `SELECT id FROM categories WHERE household_id IS NULL AND kind='expense' AND name='Outros gastos' LIMIT 1` (seed `0001_default_categories.sql:43` — sort_order 999).
  - Para `kind='income'`: `SELECT id FROM categories WHERE household_id IS NULL AND kind='income' AND name='Outros rendimentos' LIMIT 1` (seed `0001_default_categories.sql:47` — sort_order 990).
  - Para `create_installment` (transactions geradas têm sempre `kind='expense'`): resolução fixa para "Outros gastos".
  - Se a categoria global "Outros gastos"/"Outros rendimentos" não existir (situação patológica — seed deveria garantir): `ToolExecutionError` PT-PT "Categoria por defeito não encontrada. Contacta o suporte."

**Justificação:** O Jarvis raramente recebe `categoryId` UUID do utilizador em linguagem natural. Tornar `categoryId` opcional + fallback determinístico por kind é a UX correcta. **NOTA @po (PO_FIX_INLINE F6):** o draft inicial assumia `slug='outros'` — **`categories.slug` NÃO EXISTE** no schema (verificado contra `finance.ts:184-204` que lista apenas `id`/`householdId`/`name`/`icon`/`color`/`parentId`/`isDefault`/`kind`/`sortOrder`/`archivedAt`/`createdAt`/`updatedAt`). O seed tem **DUAS** categorias por defeito separadas por `kind`, não uma única "Outros" indiscriminada. `transactions.category_id` é nullable no schema (FK `ON DELETE set null`, linha 370) — alternativa "deixar NULL" foi rejeitada porque degrada UX (utilizador vê transacção sem categoria na vista mensal) e quebra agregação em `byCategory` (`month-summary.ts`).

**Helper extraído:** `packages/tools/src/finance/_helpers/resolve-default-category.ts` (puro, testável, `db`-injectável).

### Contrato `ToolDefinition<I, O>` — padrão a seguir

[Source: `packages/tools/src/contracts.ts` + Story 2.3 AC2 + Story 3.8 AC1]

```ts
interface ToolDefinition<I, O> {
  readonly name: string;              // snake_case lowercase (e.g., 'create_finance_variable')
  readonly domain: ToolDomain;        // 'finance' para esta story
  readonly description: string;       // PT-PT — descreve QUANDO usar (passado ao LLM)
  readonly inputSchema: z.ZodType<I>;
  readonly outputSchema: z.ZodType<O>;
  readonly requiredPlan?: PlanTier;   // ausência = free+
  readonly estimatedTokens?: number;  // dica cost router (Story 2.9)
  preview(input: I, ctx: ToolExecutionContext): string;
  execute(input: I, ctx: ToolExecutionContext): Promise<O>;
  reverse(output: O, ctx: ToolExecutionContext): Promise<ReverseOpPayload>;
}
```

`ToolExecutionContext`: `{ householdId, userId, db: DrizzleDbClient (RLS authenticated dentro de tx), traceId, runId }`. **Sempre `ctx.db`** — nunca `getDb()` ou `getServiceDb()` dentro da tool.
