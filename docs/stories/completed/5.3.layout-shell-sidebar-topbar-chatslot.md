## User Story

**Como** utilizador autenticado da Expressia,
**Quero** navegar a aplicação através de uma **sidebar fixa permanente** (240px em desktop) com 5 secções principais (Visão, Jarvis, Tarefas, Finanças, Conta), um **topbar** discreto com o meu avatar e botão Sair, e um **slot de chat panel** preparado à direita (fechado por defeito — a `<ChatPanel>` real chega na Story 5.4), em vez do nav horizontal placeholder actual,
**Para que** o produto deixe de parecer um protótipo e passe a parecer aplicação coerente — alinhada com Linear/Notion/Things que o público-alvo PT-PT já conhece (`front-end-spec §1.1 + §2.3 linha 83`) — sem que isto quebre nenhuma das 11 rotas existentes (visão, jarvis, 3 tarefas, 5 finanças, conta/preferencias) ou regrida o auth/logout flow (Story 1.5 D15). Esta story é a fundação visual sobre a qual a Story 5.4 (montar `<ChatPanel>` extraído), 5.6 (`<WidgetGrid>` em `/visao`), 5.8 (toggle modo escuro) e 5.10 (sweep responsive) constroem.

### 5. `shellStore` Zustand — collapse + chatPanel state com persist

**AC5:** Criar `apps/web/src/lib/stores/shellStore.ts` Zustand store com `persist` middleware em localStorage (`architecture.md §8.3 linha 713`):

(a) **Shape do store:**
```typescript
interface ShellState {
  sidebar: {
    collapsed: boolean; // true em tablet por default; mobile usa drawer separado
    setCollapsed: (collapsed: boolean) => void;
    toggle: () => void;
  };
  chatPanel: {
    open: boolean; // default false
    setOpen: (open: boolean) => void;
    toggle: () => void;
    open: () => void; // alias para setOpen(true)
    close: () => void; // alias para setOpen(false)
  };
}
```

(b) **Persist key** — `expressia-shell-v1` em `localStorage` (versionado para invalidar em mudanças de schema futura). `partialize` para persistir apenas `sidebar.collapsed` e `chatPanel.open` (não funções).

(c) **SSR safety** — Zustand store pode causar hydration mismatch se o estado inicial difere entre server e client. Usar pattern de `useEffect` para hidratar após mount (precedente nas stores existentes do projecto se houver — verificar via grep `zustand` em `apps/web/src` durante implementação; se não houver precedent, seguir docs Zustand v5 SSR section).

(d) **Selectores** — exportar selectores hooks tipados: `useSidebarCollapsed()`, `useChatPanelOpen()`, `useShellActions()` (objecto com `toggleSidebar`, `toggleChatPanel`, etc.) para evitar over-rendering.

(e) **Dependência** — adicionar `zustand@^5` a `apps/web/package.json` se ainda não estiver. Verificar via `grep '"zustand"' apps/web/package.json` durante implementação. **CRÍTICO** — se já houver `zustand` instalado, NÃO duplicar/upgrade sem decisão (DEV-DECISION).

### 9. Validation findings documentados (Constitution Article IV — No Invention)

**AC9:** Documentar na Validation Report 5 achados que requerem decisão consciente:

(a) **DP-5.3.A — Localização do shell: `apps/web/src/components/shell/` vs `packages/ui/`?** Decisão: **apps/web/src/components/shell/** (não em `packages/ui`). Racional: o shell é aplicação-específico (consome `logoutAction` de `apps/web`, `usePathname` de Next.js, store Zustand de `apps/web`, e nav items são definidos pela árvore de rotas do `apps/web/src/app/(app)/`). `packages/ui` é design system genérico (`<MoneyDisplay>`, `<DateDisplay>`, tokens) — shell não cabe aí. Alinhado com `architecture.md §8.4 linha 727` que prevê `apps/web/src/components/chat/` e `apps/web/src/components/visao/` (precedente). Se em algum momento o shell for reutilizado por uma segunda app (mobile? marketing?), promover para `packages/ui/shell/` é trivial — não há lock-in.

(b) **DP-5.3.B — Tokens consumo: directo TypeScript vs Tailwind config vs CSS variables?** Decisão: **directo TypeScript** (`import { tokens } from '@meu-jarvis/ui'`) com Tailwind arbitrary values onde aplicar (`className={`bg-[${tokens.colors.light.bgSurface}]`}` — quando estritamente necessário). Racional: Story 5.8 fará o sweep para CSS variables (`:root` + `.dark` com toggle controlado por `user_prefs.theme`). Antes desse sweep, esta story usa light mode tokens directamente. Tailwind config (`theme.extend.colors`) NÃO é actualizado nesta story — fica para 5.8. Risco mitigado: shell terá hardcoded light colors até 5.8 (já são as light que vão materializar como root vars), zero re-trabalho.

(c) **DP-5.3.C — Theme toggle position: TopBar ou Sidebar bottom?** Decisão: **Slot no TopBar** (`data-slot="theme-toggle"`). Racional: `front-end-spec §11.2 linha 1420` prescreve "atalho rápido no menu do avatar (canto inferior sidebar)" — implica sidebar bottom. MAS a implementação visual é Story 5.8, não esta. Esta story cria o slot vazio onde 5.8 monta. **Alternativa**: criar 2 slots (um em TopBar + um em Sidebar bottom) — over-engineering. Decisão @sm: criar apenas 1 slot no TopBar; se 5.8 decidir mover para sidebar, ajusta lá. Não bloqueia.

(d) **DP-5.3.D — Rota `/jarvis` permanece como vista fullscreen + sidebar label "Chat" não "Jarvis"?** Decisão: **SIM, manter rota `/jarvis`** (DP8 validated em Epic 5 §8: "Panel collapsible em todas as rotas + `/jarvis` mantida como rota dedicada"). Label do nav é "Chat" per `front-end-spec §5.4 linha 500` (wireframe mostra "Chat" no sidebar). Isto cria uma divergência consciente: o link diz "Chat" mas a URL é `/jarvis`. Aceitável porque (a) "Jarvis" é o nome próprio do agente AI internamente, "Chat" é a função user-facing, (b) mudar a URL para `/chat` exigiria refactor de middleware (`APP_PATH_PREFIXES`) + redirects de retro-compatibilidade + actualização de testes — out-of-scope. Documentar discrepância no Sidebar.tsx JSDoc.

(e) **DP-5.3.E — Middleware `APP_PATH_PREFIXES` regression pre-existente** (carry-over para @po + @architect): `apps/web/src/middleware.ts:36` actualmente lista `['/visao', '/jarvis', '/conta']` — **`/tarefas` e `/financas` NÃO estão protegidas pelo auth gate do middleware**. Esta é uma regressão pre-existente das Stories 3.3-3.5 (Tarefas) e 4.6-4.9 (Finanças) que adicionaram as rotas mas não actualizaram `APP_PATH_PREFIXES`. Mitigação actual: as próprias páginas chamam `requireAuth()` que aborta server-side — utilizador deslogado vê erro/empty content em vez de ser redireccionado para `/entrar` (UX degradada, não falha de segurança). **DP-5.3.E é EXPLICITAMENTE OUT-OF-SCOPE para Story 5.3** (`@dev` NÃO toca middleware — preserva contract Story 1.5). Recomenda-se story dedicada (mini-S) ou folding em Story 5.10 (sweep responsive + branding auth) com mudança trivial: `['/visao', '/jarvis', '/tarefas', '/financas', '/conta']`. Flagar ao @po para decidir routing.
