## User Story

**Como** utilizador autenticado da Expressia,
**Quero** ter o chat AI **acessível em qualquer rota** através do panel lateral à direita (collapsible em desktop ≥1024px, FAB+overlay em tablet/mobile <1024px) — incluindo enquanto vejo `/visao`, `/tarefas`, `/financas` ou `/conta` — **e simultaneamente** poder abrir a rota dedicada `/jarvis` para ver o chat em vista *fullscreen* com mais espaço, **partilhando o mesmo histórico de mensagens** entre o panel collapsible e a vista fullscreen (envio uma mensagem no panel da `/visao`, navego para `/jarvis`, vejo a mensagem lá),
**Para que** a tese "chat-first" do PRD §3.2 ("chat sempre disponível, acção primária é escrever ao agente") seja materializada em todas as rotas em vez de só na rota `/jarvis` (DP8 validado em Epic 5 §8: *"Panel collapsible em todas as rotas + `/jarvis` mantida como rota dedicada"*), eliminando a fricção actual de ter que navegar para `/jarvis` para falar ao agente quando estou a ver dados noutra vista. Esta story é o componente UI que **fecha o circuito chat-first do Epic 5** — Story 5.3 entregou o slot vazio (ChatPanelSlot), esta story entrega o conteúdo real (ChatPanel). Story 5.9 acrescentará por cima o `<UndoToast>` global (registado no shell) que substitui o toast inline actual.

### 2. `chatStore` Zustand — estado partilhado panel↔fullscreen (mitiga R-5.6)

**AC2:** Criar `apps/web/src/lib/stores/chatStore.ts` Zustand store **sem `persist` middleware** (mensagens são in-memory per sessão — DP-5.4.F), seguindo o pattern testado do `shellStore` (Story 5.3 — 217 linhas, 7 selectores tipados):

(a) **Shape do store:**
```typescript
interface ChatMessage {
  readonly id: string;
  readonly kind: 'user' | 'result' | 'error';
  // Discriminated union — ver tipos de Story 2.7 jarvis-chat.tsx:16-40
  // (UserMessage | AgentResultMessage | AgentErrorMessage)
  // Forma exacta preservada byte-a-byte.
}

interface PreviewState {
  readonly runId: string;
  readonly planSummary: readonly string[];
  readonly confidence: number;
  readonly expiresAt: string;
}

interface ChatPersistedState {
  // INTENCIONALMENTE VAZIO — chatStore não persiste (DP-5.4.F).
}

interface ChatEphemeralState {
  messages: readonly ChatMessage[];
  preview: PreviewState | null;
  loading: boolean;
}

interface ChatState extends ChatEphemeralState {
  // Message actions
  appendMessage: (msg: ChatMessage) => void;
  clearMessages: () => void;
  // Preview actions
  setPreview: (preview: PreviewState | null) => void;
  // Loading
  setLoading: (loading: boolean) => void;
  // Reset (logout flow defensive)
  resetChat: () => void;
}
```

(b) **Defaults** — `messages: []`, `preview: null`, `loading: false`. Reset via `resetChat()` para usar em logout flow defensivo (não obrigatório nesta story; documentar como hook de extensão futura).

(c) **Sem `persist` middleware** — DP-5.4.F: mensagens NÃO persistem em `localStorage`. Justificação: (i) histórico de chat per-utilizador é dado server-side futuro (não há `chat_history` table no schema actual — confirmar via grep contra `packages/db/src/schema/*.ts`); (ii) persistir em `localStorage` cria lock-in client-side + privacy concern (dados de prompts em plaintext no browser) sem benefício real até existir backend de histórico; (iii) Story 2.7 actual também não persiste (re-load `/jarvis` reinicia o histórico) — esta story preserva comportamento. Re-avaliar em Fase 2 se houver `chat_history` table.

(d) **Selectores tipados** — exportar 4 selectores hooks (pattern `shellStore` Story 5.3):
- `useChatMessages(): readonly ChatMessage[]`
- `useChatPreview(): PreviewState | null`
- `useChatLoading(): boolean`
- `useChatActions(): ChatActions` (bundle de `appendMessage`, `setPreview`, `setLoading`, `clearMessages`, `resetChat`)

(e) **SSR safety** — Como NÃO há `persist`, NÃO há `useChatHydrated()` análogo ao Story 5.3. Server Component pai (`jarvis/page.tsx`) renderiza o shell sem mensagens; client mount inicializa store com defaults vazios; primeiro user prompt populates `messages` array via `appendMessage`. Zero hydration mismatch porque defaults são determinísticos.

(f) **NO duplicação de store** — `chatStore` é o ÚNICO state holder do chat. `<ChatPanel mode="fullscreen">` em `/jarvis` E `<ChatPanel mode="panel">` no slot do shell consomem o mesmo store. R-5.6 do epic ("duplicação de estado entre `/jarvis` e panel") completamente mitigado: enviar prompt no panel, navegar para `/jarvis`, ver mensagem — porque ambos lêem `useChatMessages()` que retorna a mesma referência.

**AC2 PASS:** `pnpm --filter @meu-jarvis/web typecheck` GREEN; `chatStore.test.ts` cobre `appendMessage()` + `setPreview()` + `setLoading()` + `clearMessages()` + `resetChat()` + smoke de selectores; integration test smoke: criar 2 instâncias `<ChatPanel>` em modes diferentes, despachar `appendMessage` no store, ambos renderizam a mesma mensagem.

### 9. Validation findings documentados (Constitution Article IV — No Invention)

**AC9:** Documentar na Validation Report 7 achados que requerem decisão consciente:

(a) **DP-5.4.A — Localização do ChatPanel: `apps/web/src/components/chat/` vs `packages/ui/`?** Decisão: **`apps/web/src/components/chat/`**. Racional: `architecture.md §8.4 linha 727` prescreve EXACTAMENTE: *"`apps/web/src/components/chat/ # ChatPanel, MessageList, PreviewCard, UndoToast`"*. ChatPanel é app-específico — consome `next/navigation` (`useRouter`), `@sentry/nextjs` (Sentry context cliente), endpoint `/api/agent/prompt` (rota interna `apps/web/api/agent/prompt`). `packages/ui` é design system genérico (`MoneyDisplay`, `DateDisplay`, futures `EmptyState`). Promoção para `packages/ui/chat/` é trivial se houver segunda app no futuro — zero lock-in. Alinhado com Story 5.3 DP-5.3.A que justificou o mesmo princípio para o shell.

(b) **DP-5.4.B — Casing dos ficheiros movidos: kebab-case (preserva original) vs PascalCase (precedente Stories 5.2 + 5.3)?** Decisão: **PascalCase para ficheiros tsx que exportam um componente nomeado** (`ChatInput.tsx`, `PreviewCard.tsx`, `ResultMessage.tsx`); **kebab-case para utilities** (`error-messages.ts`). Racional: Story 5.2 produziu `MoneyDisplay.tsx`/`DateDisplay.tsx` (PascalCase) e Story 5.3 produziu `AppShell.tsx`/`Sidebar.tsx`/`TopBar.tsx`/`SidebarNav.tsx`/`ChatPanelSlot.tsx`/`BreadcrumbLabel.tsx`/`HamburgerButton.tsx` (PascalCase) — pattern consolidado em meu-jarvis. Story 2.7 (mais antiga) usou kebab-case porque era convenção anterior; esta story alinha. Renomeação é `git mv` (preserva history). Risco: imports cross-package podem partir em case-insensitive filesystems (Windows/macOS default) — verificar `git config core.ignorecase` durante implementação.

(c) **DP-5.4.C — Estado partilhado: Zustand `chatStore` vs prop drilling vs URL params vs TanStack Query cache?** Decisão: **Zustand `chatStore`** (sem `persist`). Racional: (i) `architecture.md §8.3 linha 711-712` prescreve Zustand para "UI ephemeral (modals, sidebar collapse, theme)"; chat messages embora não sejam "ephemeral" no sentido estrito, são *session-bound* (não persistem cross-session conscientemente — DP-5.4.F); (ii) prop drilling exigiria `<ChatPanel>` ser renderizado no layout root o que cria coupling pesado; (iii) URL params (search params) não cabem para arrays grandes de mensagens; (iv) TanStack Query é para server data — chat é client-side state puro. Pattern Zustand alinhado com `shellStore` Story 5.3 (precedente testado, 7 selectores tipados, 7 testes verdes). Risco residual: dois store separados (`shellStore` para chat open/close + `chatStore` para messages/preview/loading) — não é problema, separation of concerns saudável (shellStore = UI chrome ephemeral; chatStore = chat domain state).

(d) **DP-5.4.D — API design: prop `mode: 'panel' | 'fullscreen'` vs 2 componentes irmãos (`<ChatPanelFullscreen>` + `<ChatPanelCollapsible>`)?** Decisão: **prop `mode`**. Racional: (i) KISS — 95% da lógica é idêntica entre os modos (mesmo state, mesmo endpoint, mesmo branching, mesmo error handling); duplicar em 2 componentes irmãos seria over-engineering; (ii) prop `mode` permite styling adaptativo via conditional Tailwind classes (ex: `className={mode === 'panel' ? 'p-3 text-sm' : 'p-4 text-base'}`); (iii) precedente da indústria — `<Modal>` typically aceita `size: 'sm' | 'md' | 'lg'` em vez de 3 componentes. Risco: se modes divergirem significativamente em Fase 2 (ex: panel ter sugestões inline que fullscreen não tem, ou fullscreen ter right rail de histórico per wireframe §5.5), split em 2 componentes torna-se justificado — re-avaliar nessa altura.

(e) **DP-5.4.E — Hydration safety: chatStore precisa de `useChatHydrated()` análogo a `useShellHydrated()` Story 5.3?** Decisão: **NÃO**. Racional: `chatStore` NÃO usa `persist` middleware (DP-5.4.F), logo NÃO há hidratação assíncrona. Defaults `{ messages: [], preview: null, loading: false }` são determinísticos e idênticos entre server (SSR) e client (CSR). Não há mismatch possível. Pattern hook `useChatHydrated()` é desnecessário e introduziria complexidade sem benefício. Story 5.3 D-5.3.5 (`useShellHydrated`) só é relevante quando há persist.

(f) **DP-5.4.F — Persistência de mensagens em `localStorage`: sim ou não?** Decisão: **NÃO persistir**. Racional: (i) Story 2.7 actual também não persiste (re-load `/jarvis` reinicia o histórico); (ii) persistir em `localStorage` cria privacy concern (dados de prompts em plaintext no browser, sem encryption); (iii) histórico de chat per-utilizador é dado server-side futuro — quando existir schema `chat_history` em DB, fetch via RSC no mount do ChatPanel substituiu localStorage; (iv) sem benefício real até existir esse backend. Esta story preserva comportamento Story 2.7 (re-load reseta) intencionalmente. Re-avaliar em Fase 2 quando houver `chat_history` table.

(g) **DP-5.4.G — Sidebar mostra label "Chat" mas rota é `/jarvis` — divergência consciente?** Decisão: **SIM, manter divergência** (carry-over de Story 5.3 DP-5.3.D). Racional: "Jarvis" é o nome próprio do agente AI internamente (PRD §3.1: *"Jarvis, o cérebro AI..."*), "Chat" é a função user-facing per `front-end-spec §5.4 linha 500` (wireframe Visão mostra "Chat" no nav). Mudar a URL para `/chat` exigiria refactor de middleware (`APP_PATH_PREFIXES` array em `apps/web/src/middleware.ts`), redirects de retro-compatibilidade, actualização de testes — out-of-scope desta story. `metadata.title` mantém *"Jarvis — Expressia"* (DP-5.4.G afirma esta decisão explicitamente). Documentar discrepância em ChatPanel.tsx JSDoc (referência cross-file para Story 5.3 DP-5.3.D).
