## User Story

**Como** utilizador da Expressia que regista despesas via Jarvis,
**Quero** poder escrever "paguei 18,70 euros no pingo doce em compras" e ter a transacção criada sem erro — mesmo sem ter configurado uma conta específica —,
**Para que** a promessa core do Expressia ("uma frase, várias acções") funcione para Finanças com a mesma fluidez que já funciona para Tarefas, com a conta Dinheiro a ser usada como default quando não especifico outra.

## Contexto e Decisão

**Problema (GAP-6):** Teste E2E de 30/05/2026 em `localhost:3000/jarvis` provou que todo o subsistema Finanças via chat está bloqueado. O planner classifica correctamente (`criar_financa_variavel`, 95%) mas o preview-then-confirm falha com:

```
Tool 'create_finance_variable' validation failed on field 'input':
Fornecer accountId ou cardId (CHECK transactions_account_or_card)
```

Causa raiz: **três peças em simultâneo** — o planner não sabe que contas existem (placeholder `<uuid>` no system prompt), as tools rejeitam antes de `execute()` sem fallback, e o utilizador novo tem 0 contas (onboarding não cria nenhuma). Tarefas via chat funcionam E2E; Finanças nunca fecharam a ponte conversacional.

**Decisão (ADR-002, Aria/@architect, 30/05/2026):** Adoptar Opção B + A combinadas, com Opção C diferida para Fase 2:

| Opção | O que faz | Status |
|-------|-----------|--------|
| **B (base)** | Conta `'Dinheiro'` default no onboarding + relaxar `.refine()` + `resolveDefaultAccount` helper | **Nesta story** |
| **A (fidelidade)** | Context injection — lista contas/cartões no `PlannerInput` para o LLM escolher/preencher IDs reais | **Nesta story** |
| C (clarificação) | Máquina de estados multi-turn ("de que conta saiu?") | **Diferida — Fase 2** |

## Acceptance Criteria

> Cada AC referencia explicitamente a peça da causa raiz ou o requisito do ADR-002 / PRD que cobre.

**AC1 — Conta Dinheiro no onboarding (ADR-002 §3, B base, Peça 3)**
O trigger `handle_new_user` cria, após o `INSERT` do household (linha 77 de `0003_auth_user_trigger.sql`), uma conta com `name='Dinheiro'`, `account_type='dinheiro'`, `currency='EUR'`, `balance_cents=0`, `initial_balance_cents=0`, `household_id=new_household_id`. A conta é criada dentro do mesmo bloco fail-hard do trigger (comportamento D2 mantido). O `pnpm check:rls` continua verde após a migration — a tabela `accounts` tem as 4 policies existentes que cobrem a nova linha.

**AC2 — Migration de backfill idempotente para households legacy (ADR-002 §4)**
Uma nova migration SQL insere a conta Dinheiro para todos os households que ainda não têm nenhuma conta (`WHERE NOT EXISTS`). A migration é idempotente (re-run não duplica). Corre com role de migration (`getServiceDb()`/`DIRECT_URL`) — correcto para admin path (não usa RLS). O `pnpm check:rls` continua verde.

**AC3 — `.refine()` relaxado nas tools de finança variável e recorrente (ADR-002 §3, B base, Peça 2)**
Em `create-finance-variable.ts` e `create-finance-recurrence.ts`, o `.refine()` que exigia `accountId` XOR `cardId` é removido ou tornado opcional. `accountId` e `cardId` passam a ser `z.string().uuid().optional()`. A validação estrutural de presença é delegada para dentro de `execute()`, onde `resolveDefaultAccount` preenche o default antes de chegar ao CHECK Postgres.

**AC4 — Helper `resolveDefaultAccount` com lógica de fallback a três níveis (ADR-002 §3 racional)**
Criado `packages/tools/src/finance/_helpers/resolve-default-account.ts` espelho de `resolve-default-category.ts`. Quando `accountId` e `cardId` são ambos ausentes do input:
1. Selecciona a conta com `account_type='dinheiro'` activa do household — filtro `archived_at IS NULL` (a coluna de arquivamento é `archived_at timestamptz`, NÃO `is_archived`; ver Dev Notes T1 ponto 4) — via `ctx.db`, RLS garantida pelo JWT.
2. Se não existir, selecciona a conta activa (`archived_at IS NULL`) mais antiga do household (`created_at ASC`).
3. Se o household não tiver nenhuma conta, lança `ToolExecutionError` com mensagem PT-PT accionável (situação patológica pós-backfill).
O helper é testável independentemente (sem depender de executor real).
**PO-FIX-A (DP-2.13.A):** `resolveDefaultAccount` retorna `{ accountId: string; accountType: AccountType }` (NÃO apenas o UUID), porque T2.4 precisa do tipo da conta resolvida para inferir `paymentMethod` (`'dinheiro'` → `'cash'`, outros → `'transfer'`). Nota de divergência face ao precedente: `resolveDefaultCategory` (`_helpers/resolve-default-category.ts:51`) retorna `Promise<string>` (só o id) — `resolveDefaultAccount` diverge intencionalmente do gémeo neste shape de retorno por causa da inferência de payment method. A Constraint "retorna `Promise<string>`" em Constraints/NFR (TypeScript strict) é actualizada por este PO-FIX.

**AC5 — `create_card` associa à conta default via `resolveDefaultAccount` (ADR-002 §5 nota scope)**
Em `create-card.ts`, quando `accountId` não é fornecido, o helper `resolveDefaultAccount` é chamado dentro de `execute()` para obter a conta default, permitindo que `criar_cartao` funcione para utilizadores novos sem mencionar conta explicitamente.
**PO-FIX-E (verificado byte-a-byte por @po):** o `accountId` em `create-card.ts:34` é actualmente `z.string().uuid()` **obrigatório** (não `.optional()`). Para AC5 funcionar, T5.1 tem de o tornar `z.string().uuid().optional()` ANTES de adicionar a resolução default em `execute()` — caso contrário o Zod rejeita antes de `execute()` (mesma ordem load-bearing que `create-finance-variable`). Usar apenas o `accountId` do retorno de `resolveDefaultAccount` (PO-FIX-A); o `accountType` não é necessário aqui.

**AC6 — Context injection: contas/cartões no payload do planner (ADR-002 §3, A fidelidade, Peça 1; shape e canal ratificados em ADR-002 §9.4 + §9.5)**
`PlannerInput` em `packages/planner-executor/src/schemas.ts` recebe um campo `accountContext` opcional que é um **objecto com duas listas distintas** (ADR-002 §9.4 — cartões NÃO têm `account_type`, logo não podem partilhar o shape das contas):
- `accounts`: array de `{ id: uuid, name: string, type: string }` — `type` tipado como `z.string()` no planner (espelha `accountTypeEnum` sem importar o enum DB, mantendo o package agnóstico de DDL, ADR-002 §9.4 nota de fronteira).
- `cards`: array de `{ id: uuid, name: string }` — sem `type`.

Em `apps/web/src/app/api/agent/prompt/route.ts`, antes de chamar o planner, é executado um SELECT RLS-scoped (`getDb()`, **NUNCA** `getServiceDb()` — ADR-002 §9.3) que obtém as contas e cartões **activos** (`archived_at IS NULL`) do household, mapeia cada row para as duas listas e passa o objecto `accountContext` no `PlannerInput`. O endpoint é quem faz o mapeamento da row de conta para a string `type` (o planner não conhece o `accountTypeEnum`).

Em `planner.ts` (`callProvider`, linhas 267-299), o `accountContext` é injectado como **prefixo da user message (`messages`)** — **NUNCA** no `system` nem nos `tools` (ADR-002 §9.1, §9.5 directiva 2). As `messages` estão fora do prefixo cacheável da Anthropic por construção (`anthropic.ts:159` serializa sem `cache_control`), pelo que injectar contexto volátil do household ali preserva 100% o cache hit rate do `system` + `tools`. Formato denso recomendado (ADR-002 §9.4): uma linha por categoria com ids inline para permitir desambiguação ("cartão Millennium" → id). Casos de borda: `accountContext` ausente ou ambas as listas vazias → não injectar o prefixo (utilizador novo pré-backfill; o fallback `resolveDefaultAccount` da peça B resolve a jusante).
**PII:** os nomes de conta e cartão passam no payload do LLM (necessário para a função) mas NUNCA entram em span attributes raw / logs não-redactados. **PO-FIX-B (precisão de mecanismo PII — verificado byte-a-byte por @po):** a protecção PII já existe estruturalmente em 3 camadas e NÃO requer "adicionar campo a uma lista":
> 1. **Spans do planner usam whitelist, não denylist.** `PLANNER_SPAN_ATTRIBUTE_KEYS` em `packages/planner-executor/src/tracing.ts:36-49` é uma whitelist `as const` de 12 keys — qualquer atributo fora dela (incluindo `accountContext`) é rejeitado pelo test `tracing.test.ts`. O `accountContext` nunca entra num span por construção.
> 2. **Payload do provider redacta `messages` inteiramente.** `redactProviderPayload` (`packages/agent/src/redaction.ts:37-46`) remove `system`, `messages` e `tools` antes de qualquer log. Como DP-2.13.B foi ratificada (ADR-002 §9.1) para injectar o `accountContext` como **prefixo da user message**, este cai dentro de `messages` → já redactado por construção. (Nota de defesa em profundidade: o `system` também está em `REDACTED_FIELD_NAMES`, mas usá-lo contaminaria o cache (R-2.13.4) — por isso o canal é `messages`, decidido.)
> 3. **`redactToolInputForLog`** (`packages/tools/src/errors.ts:233-238`) redacta o input de tools por completo (retorna sempre `{ toolName, inputRedacted: true }`) — NÃO tem "lista de campos" a estender. A redacção é total por construção.
>
> Correcção factual: a frase original "adicionar o campo à lista de redacção" descreve um mecanismo que não existe (não há denylist de campos). T3.4 fica: **verificar** (não adicionar) que (a) `accountContext` não aparece em nenhum span attribute do `agent.planner.call`, e (b) como é injectado em `messages` (canal ratificado, §9.1), é coberto por `redactProviderPayload`. Esta restrição é verificada no gate `@architect`.

**AC7 — Bump do planner system prompt v2 → v3 (ADR-002 §5 impacto testes)**
`packages/planner-executor/src/prompts/planner-system.ts` é bumped de `v2` para `v3`. As linhas 96, 100, 104 que continham `'<uuid>'` placeholder são substituídas por instrução explícita: "usa o `accountId`/`cardId` do `accountContext` fornecido na mensagem; se o utilizador não especificar conta ou cartão, omite o campo (a tool usa a conta por defeito)". O snapshot hash em `prompts.test.ts` é re-gerado para reflectir `v3` (passo conhecido, documentado no header `planner-system.ts:18-21`).

**AC8 — Suite de testes passa limpa (ADR-002 §5 impacto testes)**
`pnpm lint && pnpm typecheck && pnpm test && pnpm build && pnpm check:rls` — todos EXIT 0 após a implementação. Em particular:
- Testes de `create-finance-variable` e `create-finance-recurrence` que assumiam o refine obrigatório são actualizados para o novo comportamento (campo opcional + resolução default).
- Testes de onboarding que validam o trigger `handle_new_user`. **PO-FIX-D (nuance de infra — verificado byte-a-byte por @po):** o teste real é `packages/db/src/__tests__/handle-new-user.trigger.test.ts` e é **estrutural/read-only** (pg_proc/information_schema) — NÃO cria `auth.users` (o smoke funcional exige o SDK Supabase Auth, documentado como manual em `docs/runbooks/supabase-auth-setup.md §6`, dependente da suite Testcontainers da Story 1.4, ainda não entregue). Logo a asserção "1 conta dinheiro criada" NÃO é trivialmente automatizável com a infra actual. Aceitação: T1.5 adiciona uma asserção **estrutural** de que `handle_new_user` contém o INSERT em `accounts` com `account_type='dinheiro'` (via `pg_get_functiondef` regex), e a verificação funcional do row criado fica como **smoke manual** (T6 E2E já cobre via fluxo real do utilizador novo). Esta nuance NÃO bloqueia — apenas reenquadra a expectativa de "teste automatizado da row".
- Snapshot hash do `PLANNER_SYSTEM_PROMPT` em `prompts.test.ts` é re-gerado para `v3`.
- `pnpm check:rls` reporta as tabelas existentes com as mesmas policies — nenhuma regressão de cobertura.

## Dev Notes — T2-T5 (@dev / Dex, para o gate @architect)

> Decisões tomadas autonomamente (modo YOLO) — a ratificar no gate.

**[DEV-DECISION] D-2.13.1 — `AccountType` declarado localmente no helper, não importado de `@meu-jarvis/db`.**
O AC4/NFR-table e a Dev Note T1 ponto 3 sugeriam "importar o tipo de `@meu-jarvis/db`". Verifiquei: `@meu-jarvis/db` **NÃO exporta** um tipo `AccountType` nomeado — só o `accountTypeEnum` (pgEnum runtime) e os tipos `Account`/`NewAccount` (`$inferSelect`). Importar `accountTypeEnum` para derivar o tipo acoplaria o package `@meu-jarvis/tools` ao runtime do schema DB — exactamente a fronteira que `contracts.ts` documenta evitar ("não importamos `Database` de `@meu-jarvis/db`"). Decisão: declarar `export type AccountType = 'corrente' | ... | 'dinheiro' | 'outro'` localmente no helper, espelhando os 6 valores do enum. Razão: mantém a fronteira de package limpa, zero coupling cross-package, mesma filosofia do planner (`z.string()`). Risco: se o enum DB mudar, o tipo local fica dessincronizado — mitigável por um futuro teste de paridade (LOW, fora de scope).

**[DEV-DECISION] D-2.13.2 — Precedência 1+2 numa única query SQL (não duas queries sequenciais).**
O fallback de 3 níveis (dinheiro → conta activa mais antiga → erro) foi implementado com **um SELECT** usando `ORDER BY (account_type = 'dinheiro') DESC, created_at ASC LIMIT 1` em vez de duas queries (primeiro tentar dinheiro, depois fallback). Razão: 1 round-trip DB em vez de 2 (NFR1 latência), determinístico, e o nível 3 (erro) é simplesmente `rows[0]` ausente. Equivalente semântico à precedência da story, menos custo. Pós-backfill 0018, o nível 1 resolve sempre para households reais (Dev Note T1 ponto 1).

**[DEV-DECISION] D-2.13.4 — `paymentMethod` para `accountId` explícito mantém `'transfer'` (não faz lookup do tipo).**
O CodeRabbit (2 findings MINOR) notou que, quando o utilizador passa um `accountId` explícito que por acaso é de uma conta `dinheiro`, o `paymentMethod` resolve para `'transfer'` e não `'cash'` (porque `resolvedAccountType` só é populado quando `resolveDefaultAccount` corre). Decisão: **manter** este comportamento (não adicionar um SELECT do tipo da conta para cada `accountId` explícito). Razão: (1) preserva o comportamento pré-existente documentado (`accountId presente → 'transfer'`, PO_FIX_INLINE F1 da 4.10); (2) DP-2.13.A foca o caso da conta **default resolvida** (caso família-first comum), não o caso de o utilizador escolher explicitamente uma conta — nesse caso pode passar `paymentMethod` se quiser; (3) um lookup extra por request viola o orçamento NFR1 sem ganho prático (raro o utilizador nomear explicitamente a conta "Dinheiro" via chat — quando o faz pela conta default, a inferência cash funciona). MINOR aceite como comportamento intencional — a ratificar no gate. Os 3 findings MINOR do CodeRabbit: 2 são esta decisão (aceites); 1 (dangling-else no helper `captureSqlText` de teste) foi corrigido.

**[DEV-DECISION] D-2.13.3 — Exemplo 3 (parcelada) também teve o `<uuid>` substituído (alargamento mínimo de T4.1).**
T4.1 cita só os exemplos 6/7/8. Mas o Exemplo 3 (`create_installment`, linha 84) também continha `cardId: '<uuid>'` — um placeholder igualmente impossível de resolver pelo LLM. Para o teste negativo `not.toContain('<uuid>')` passar e por coerência (todos os placeholders literais eliminados), substituí-o por `<id do cartão "Activobank" no accountContext>` + nota de que a parcelada exige cartão real (FUP-2.13.A).
