## User Story

**As a** developer da equipa Expressia,
**I want** uma suite de testes automatizados que prove que o RLS Postgres bloqueia efectivamente o acesso cross-household em todas as tabelas,

## Dev Notes

### Padrão de teste RLS (Architecture §10.2)

```typescript
// packages/db-test/src/tests/tasks.rls.test.ts

import { describe, test, expect, beforeAll, afterAll } from 'vitest'
import { startTestPostgres, stopTestPostgres, seedTwoHouseholds, asUser } from '../rls-harness'

describe('RLS isolation: tasks', () => {
  let container: TestPg

  beforeAll(async () => {
    container = await startTestPostgres()
    await applyMigrations(container)
  })

  afterAll(async () => {
    await stopTestPostgres(container)
  })

  test('cross-household SELECT bloqueado', async () => {
    const { householdA, userA, householdB, userB } = await seedTwoHouseholds(container)

    // userA cria uma tarefa no householdA
    await asUser(userA, householdA, container, async (db) => {
      await db.insert(tasks).values({
        household_id: householdA.id,
        title: 'Tarefa A',
        created_by: userA.id,
      })
    })

    // userB tenta ver as tarefas do householdA — deve ver 0 rows (RLS)
    await asUser(userB, householdB, container, async (db) => {
      const rows = await db.select().from(tasks)
        .where(eq(tasks.household_id, householdA.id))
      expect(rows).toHaveLength(0) // (1) cross-household SELECT bloqueado
    })
  })

  test('cross-household INSERT bloqueado', async () => {
    const { householdA, userB, householdB } = await seedTwoHouseholds(container)

    // userB tenta criar tarefa com household_id do householdA — deve falhar
    await asUser(userB, householdB, container, async (db) => {
      await expect(
        db.insert(tasks).values({
          household_id: householdA.id, // CROSS-HOUSEHOLD!
          title: 'Tarefa inválida',
          created_by: userB.id,
        })
      ).rejects.toThrow(/row-level security/i)  // (2) RLS rejeita
    })
  })
})
```

### JWT claims simulados para RLS (Architecture §3.2, §5.1)

O RLS usa `current_setting('request.jwt.claims')::json->>'household_id'`. Para simular:

```typescript
// No helper asUser:
async function asUser(userId: string, householdId: string, container: TestPg, fn: (db: Drizzle) => Promise<void>) {
  const client = await container.connect()
  await client.query(`SET role authenticated`)
  await client.query(`
    SELECT set_config(
      'request.jwt.claims',
      '{"sub": "${userId}", "household_id": "${householdId}"}',
      true
    )
  `)
  const db = drizzle(client, { schema })
  await fn(db)
  await client.release()
}
```

## Dev Agent Record

### Decisões técnicas tomadas durante a implementação

1. **Imagem Docker:** trocou `postgres:16` → `pgvector/pgvector:pg16` porque a migration `0000_initial_schema.sql` cria `extension "vector"` (preparação Fase 3 RAG, architecture §11.2). Service do CI também actualizado para a mesma imagem.
2. **Bootstrap SQL isolado no harness:** decisão (a) do Dev Notes — o ficheiro `harness/bootstrap.ts` cria role `authenticated` + schema `auth` + funções `auth.uid()`/`auth.jwt()` que existem em Supabase mas não em Postgres vanilla. Mantém as migrations de produção limpas e específicas do ambiente Supabase.
3. **Container partilhado:** Vitest `globalSetup` boots o container 1× para toda a suite. Cada teste usa `resetData()` em `beforeEach` (TRUNCATE CASCADE) para isolamento — mantém a suite < 8s wall-clock após cache da imagem.
4. **Fork single-thread:** `pool: 'forks'` + `singleFork: true` em `vitest.config.ts` evita race conditions com o container partilhado.
5. **Tipos `Sql | TransactionSql`:** criado type alias `QuerySql` em `rls-harness.ts` para que os helpers `insert*` em `fixtures.ts` aceitem tanto a connection-level (`admin()`) como a transactional (`asUser()`).
6. **Padrão de teste para REVOKE explícito:** `audit_log` UPDATE/DELETE têm `revoke ... from authenticated` para além das policies. Isso causa `permission denied` (excepção) em vez de `count = 0`. Os testes usam `expect(...).rejects.toThrow(/permission denied|row-level security/i)` para validar o bloqueio.
7. **`expectRlsBlocks` helper:** abstracção sobre `asUser` que devolve `boolean` em vez de propagar excepção — utilizado nos testes de INSERT cross-household para asserção mais clara.

### Áreas para QA prestar atenção

- **AC7 / tempo CI:** local foi 7.88s mas o primeiro run em CI tem cold pull da imagem `pgvector/pgvector:pg16` (~150MB). Estimativa: 30-45s. Se em CI passar dos 60s, considerar pre-pull com `docker pull` num step antes ou cache da layer via `docker/setup-buildx-action`.
- **Testcontainers em CI ubuntu-latest:** Docker está disponível por default mas Testcontainers requer DOCKER_HOST acessível. Se houver problemas, adicionar step `docker version` no workflow para diagnose.
- **Categorias defaults:** o teste `categories.rls.test.ts` faz `delete from public.categories` no `beforeEach` para limpar tudo (incluindo defaults). Se a Story 1.5 ou outra Story adicionar mais defaults via seed, este teste continua válido (testa-se directamente a policy, não a contagem).
- **payment_events SELECT bloqueado universalmente:** authenticated nunca pode ver — apenas service_role via webhook. Confirmar se este nível de restrição é o desejado para debugging em Supabase Studio (alternativa: permitir owner/admin ver mas continuar a bloquear escrita).
