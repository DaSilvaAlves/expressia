# Story 6.7: Convite e remoção de membros do household + limites por plano

## Status

Done v1.2-ARCH-APPROVED — `@architect *qa-gate 6.7` PASS 9,3/10 HIGH (Aria, 2026-06-01). Pronta para `@devops *push 6.7`.

**Estimate:** M (alinhado ao epic-6 §5; ciclo E2E: 1 migration SQL function + 4 endpoints + 1 rota + UI + 2 suites de teste).

## Story

**As a** owner (ou admin) de um household na Expressia,
**I want** convidar pessoas por email, ver e revogar convites pendentes, e remover membros — e que o convidado possa aceitar o convite por um link e entrar no household,
**so that** a minha família partilha tarefas e finanças no mesmo espaço, respeitando o limite de membros do plano e o isolamento RLS entre households.

## Contexto e âmbito (ler antes das ACs)

Esta story fecha o **ciclo E2E de gestão de membros** do household, complementando a feature Família já em produção (`/conta/household` GET+PATCH — renomear + listar membros, commit `3470f0f`). Decisão de âmbito validada por Eurico (2026-06-01): **ciclo completo numa story** (convidar → aceitar → entrar + remover/revogar), porque sem aceitação não há valor E2E (alinha com a directiva `refocus-core`: o core tem de funcionar ponta-a-ponta, não "Done com mocks").

**Pré-condição de DB verificada no código real (Article IV):** ao contrário do que o handoff de origem fazia recear, **a fundação de DB já está quase toda feita** — esta story **não recria** a tabela nem o RLS:

- **Tabela `public.household_invites` EXISTE** em `packages/db/migrations/0000_initial_schema.sql:212-229` (alinhada ao schema Drizzle `tenancy.ts:115`): `token` unique, `expires_at`, `accepted_at`/`accepted_by_user_id`, unique parcial `household_invites_unique_pending(household_id, email)`, índices.
- **As 4 RLS policies EXISTEM** em `0001_rls_policies.sql:114-148`. Nota importante: ao contrário de `households` (cujo UPDATE a RLS permite a qualquer membro), aqui a RLS **já restringe owner/admin** no INSERT/UPDATE/DELETE (`is_household_owner_or_admin(household_id)`), e o SELECT já permite ao convidado ver o seu próprio convite (`email = auth.users.email`). O helper `is_household_owner_or_admin()` também já existe (`0000:70`).
- **As policies de `household_members` já suportam self-insert via `accept_invite`** (`0001:87-92`: INSERT `with check (is_household_owner_or_admin OR user_id = auth.uid())`).
- **`pnpm check:rls` passa naturalmente** — `household_invites` já tem as 4 policies; não há nada a alterar no gate.
- **A suite RLS `packages/db-test/src/tests/household_invites.rls.test.ts` já existe e está verde** (3 testes: SELECT/INSERT cross-household bloqueado + owner vê os seus), com a fixture `insertHouseholdInvite` em `helpers/fixtures.ts:378`.

**O que NÃO existe (é o trabalho desta story):** a função SQL **`accept_invite()`** (referida nos comentários `0000:209`/`0001:141` e na `architecture §5.3`, mas **nunca implementada** — confirmado: as únicas funções em `0000` são `current_household_id`, `is_household_member`, `is_household_owner_or_admin`, `set_updated_at`); os **endpoints REST** de convite/remoção; a **rota `/aceitar-convite/{token}`**; a **UI** de gestão; e os **testes** de aplicação + de `accept_invite()`.

**Email do convite:** Resend está congelado (GAP-2 / directiva `refocus-core`). Decisão validada por Eurico (2026-06-01): **no MVP a API devolve o link `/aceitar-convite/{token}` na resposta** para o owner partilhar manualmente (copiar/WhatsApp). Sem dependência do Resend. Diverge do epic-6 §5 ("email Resend") — envio automático fica como follow-up quando o Resend ligar.

## Acceptance Criteria

1. **Função SQL `accept_invite()` `SECURITY DEFINER`.** Nova migration (`packages/db/migrations/0020_accept_invite_function.sql`) cria `public.accept_invite(p_token text) returns uuid` `SECURITY DEFINER` (`set search_path = public`) que, numa **transação única com lock** (defesa em profundidade vs UI — R-6.5): (a) localiza o convite por `token` e fá-lo `for update`; (b) valida que existe, não expirou (`expires_at > now()`), não foi aceite (`accepted_at is null`) e que o `email` do convite corresponde ao email do utilizador autenticado; (c) valida que `count(household_members do household) < limite_do_plano` (lê `households.plan`: Pessoal=1, **Família=4**, Pro=10; **Free=1 — [PO-NOTE]: o epic-6/FR27 só lista os 3 planos pagos; `Free=1` é inferência sensata (Free é single-user) — @dev confirma com `architecture §5.3`/§6.4 ou regista como [DEV-DECISION]; não é bloqueante porque o trial entra como `familia`**); (d) valida que `auth.uid()` ainda não é membro (evita violar a PK composta); (e) insere `household_members(household_id, auth.uid(), role do convite)`; (f) marca `accepted_at=now()`, `accepted_by_user_id=auth.uid()`; (g) devolve o `household_id`. Caminhos de erro com `raise exception` tipada (mensagem distinguível: expirado / já aceite / limite atingido / token inválido / email não corresponde / já membro). Idempotente: segunda chamada falha "já aceite". A migration deve incluir **`grant execute on function public.accept_invite(text) to authenticated`** (a função é chamada via `getDb()` com role `authenticated`; sem o GRANT a chamada falha). Usar `set local check_function_bodies = off` se necessário (padrão `apply-migrations.ts`). **[PO-NOTE] email-match (passo b):** exigir que o `email` do convite == email de `auth.uid()` é uma decisão de segurança (o convite é nominal); @dev documenta como [DEV-DECISION] e a mensagem de erro PT-PT deve ser clara ("Este convite foi enviado para outro email."). [Source: tenancy.ts:113 + 0000:209 + 0001:141 (comentários "accept_invite() SECURITY DEFINER"); architecture.md §5.3 (limites de plano); epic-6 §2/§5/§10 FR27 (Pessoal=1/Família=4/Pro=10); R-6.5; 0001:87-92 (household_members INSERT já permite self-insert)]
2. **`POST /api/conta/household/invites`** cria um convite: body `{ email, role? }` validado por Zod; **só `owner`/`admin`** — validar o papel na app e devolver **403 limpo** (a RLS `household_invites_insert_owner_admin` também bloqueia, mas a app não deve depender só do erro de RLS); gera `token` aleatório (≥32 bytes — `gen_random_bytes(32)` na DB ou `crypto.randomBytes(32).toString('hex')`) e `expires_at` (now()+7d); respeita o unique parcial `household_invites_unique_pending` (409 se já há convite pendente para o email no household); **devolve o link `/aceitar-convite/{token}` na resposta** (MVP sem Resend). 401 sem auth · 403 papel insuficiente · 400 Zod · 409 duplicado. [Source: route.ts:52 (ROLES_CAN_EDIT), :235-368 (padrão de erros/audit); 0001:128-133 (RLS insert owner/admin); tenancy.ts:142 (unique pending); decisão Eurico 2026-06-01 (link manual)]
3. **`GET /api/conta/household/invites`** lista os convites **pendentes** do household (`accepted_at is null and expires_at > now()`): `{ id, email, role, expiresAt, createdAt }`. **Não expõe o `token`** na listagem (o token só é devolvido no POST de criação). `getDb()` (a RLS de SELECT já filtra por household). [Source: route.ts (padrão GET); 0001:117-126 (RLS select); db-schema §2]
4. **`DELETE /api/conta/household/invites/{id}`** revoga (elimina) um convite pendente do household; só `owner`/`admin` (403 na app + RLS delete owner/admin); 404 se não existir/não pertencer ao household; 401. [Source: route.ts:52; 0001:143-148 (RLS delete owner/admin)]
5. **`DELETE /api/conta/household/members/{userId}`** remove um membro (elimina a row `household_members`); só `owner`/`admin`; **guard inegociável: nunca remover um membro com `role='owner'`** (422/403); não permitir auto-remoção do único owner. Remove apenas o membership — os dados do household ficam (o `household_id` não muda). 401/403/404. Nota: a RLS de `household_members` DELETE permite owner/admin **ou** auto-saída (`0001:108-110`) — o guard "nunca o owner" é regra de negócio aplicada na app. [Source: 0001:94-110 (RLS household_members update/delete); tenancy.ts:81; handoff (guard owner)]
