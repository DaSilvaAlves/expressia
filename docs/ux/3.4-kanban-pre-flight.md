## 1. Contexto

### 1.1 O que esta story entrega

Página `/tarefas/kanban` com:
- Vista visual board com **3 a 6 colunas configuráveis per household** (schema `kanban_columns` já existe, seed default `'A fazer' / 'Em curso' / 'Concluído'` aplicado em Story 3.1)
- **Drag-and-drop atómico** entre colunas (PATCH `/api/tasks/[id]/move` — endpoint já criado em Story 3.2) e intra-coluna (reorder `kanban_position`)
- Configurar colunas via **Sheet lateral** (adicionar/renomear/eliminar/reordenar)
- **Optimistic UI** com revert + toast erro PT-PT em caso de PATCH failure
- **Acessibilidade keyboard-first** (Space pick, Arrow move, Enter drop, Esc cancel)
- **Mobile UX** com tap-and-hold sensor + scroll horizontal entre colunas

## 5. Drag-and-Drop Spec (@dnd-kit)

### 5.1 Sensors config

```typescript
import {
  DndContext,
  PointerSensor,
  KeyboardSensor,
  TouchSensor,
  useSensor,
  useSensors,
  closestCorners,
} from '@dnd-kit/core';
import { sortableKeyboardCoordinates } from '@dnd-kit/sortable';

const sensors = useSensors(
  useSensor(PointerSensor, {
    activationConstraint: { distance: 8 },          // 8px move antes de iniciar drag (evita cliques acidentais)
  }),
  useSensor(TouchSensor, {
    activationConstraint: {
      delay: 250,                                    // tap-and-hold 250ms (R7 — mobile)
      tolerance: 5,                                  // tolerância de movimento durante delay
    },
  }),
  useSensor(KeyboardSensor, {
    coordinateGetter: sortableKeyboardCoordinates,
  }),
);
```

### 5.2 Collision detection

`closestCorners` (não `closestCenter`) — preferível para vertical sortable porque dá precisão de inserção entre cards. Fonte: @dnd-kit docs `closestCorners` para sortable lists.

### 5.3 Drag lifecycle

```
onDragStart(event)
  → identificar `activeTask`
  → setActiveId(event.active.id)
  → activeTask.opacity = 0.4 (visual fade)
  → renderizar <DragOverlay> com KanbanCardOverlay clone

onDragOver(event)
  → identificar `overId` (column ou card)
  → calcular destino (column.id + posição)
  → aplicar visual drop-target (background + border)

onDragEnd(event)
  → optimistic update local state (move card)
  → PATCH /api/tasks/[id]/move { kanbanColumnId, kanbanPosition }
  → on success: silent (já mostrámos)
  → on error: revert state + toast danger PT-PT

onDragCancel()
  → restaurar visual original sem mexer state
```

### 5.4 Endpoint PATCH atómico (relembrar Story 3.2)

```
PATCH /api/tasks/[id]/move
Body: { kanbanColumnId: uuid, kanbanPosition: integer }
Behavior: UPDATE atómico em transacção; rebalanceia positions sibling se necessário
```

**Falhas possíveis tratadas no UI:**
- `403` (RLS — task não pertence ao household): revert + toast "Não tens permissão para mover esta tarefa."
- `404` (task eliminada por outro membro): revert + toast "Esta tarefa foi eliminada. A lista vai actualizar." + refresh server data
- `409` (conflict — coluna eliminada entretanto): revert + toast "A coluna foi removida. A lista vai actualizar." + refresh
- `500`: revert + toast "Ocorreu um erro a mover a tarefa. Tenta de novo." + log Sentry

### 5.5 Reorder intra-coluna

Mesma lógica: PATCH `/api/tasks/[id]/move` com `kanbanColumnId` igual ao actual + novo `kanbanPosition`. O backend rebalanceia siblings (Story 3.2 implementou; confirmar com `@architect`/Dex).

## 6. Mobile UX Strategy (R7 Mitigation)

### 6.1 Padrão escolhido

| Opção | Decisão |
|-------|---------|
| Tap-and-hold 250ms para iniciar drag (`TouchSensor` delay) | **Adoptado** — padrão iOS/Android nativo; previne false-positives durante scroll |
| Drag handle explícito (`⋮⋮` ícone à esquerda do card) | **Rejeitado** — adiciona ruído visual; tap-and-hold é universal |
| Scroll horizontal entre colunas via swipe | **Adoptado** — `overflow-x: auto` + `scroll-snap-type: x mandatory` natural no mobile |
| Long-press menu (alternative para utilizadores sem drag confortável) | **Adoptado** — long-press no card (≥ 600ms) abre menu "Mover para → coluna X" como fallback acessível |

### 6.2 Detalhe sensor config mobile

```typescript
TouchSensor activationConstraint: {
  delay: 250,
  tolerance: 5,   // 5px de movimento permitido durante delay (evita cancel por tremor)
}
```

### 6.3 Edge cases mobile

| Cenário | Comportamento |
|---------|---------------|
| Drag para fora do viewport (coluna à direita não visível) | Auto-scroll horizontal quando cursor a < 50px da edge (@dnd-kit `autoScroll` config) |
| Drag durante scroll vertical na coluna | `TouchSensor delay` previne ambiguidade — scroll wins se < 250ms |
| Drag iniciado mas utilizador desbloqueia ecrã | `onDragCancel` chamado automaticamente por `pointercancel` event |
| Drop em zona ambígua (entre 2 colunas) | `closestCorners` resolve para coluna mais próxima do cursor |

### 6.4 Visual mobile-specific

- `KanbanCard` tem `touch-action: manipulation` para previne pinch-zoom acidental durante long-press
- Cursor `grab/grabbing` substituído por feedback haptic se disponível (`navigator.vibrate(10)` no `onDragStart`) — **DP-3.4.5**

## 7. Performance Strategy (R8 Mitigation)

### 7.1 Risco

Coluna com >100 tasks renderiza 100+ `KanbanCard` no DOM. Drag operations tornam-se laggy. R8 do Epic plan flagga isto.

### 7.2 Estratégia escolhida — naive-first, virtualize-on-demand

**Decisão default:** começar com render naive (todas tarefas no DOM). Adicionar virtualização **só se threshold for atingido** numa coluna.

**Threshold:** coluna com `> 50` tasks renderiza com `react-virtual` (tanstack/react-virtual) ou `react-window`.

**Racional:**
- MVP Fase 1 famílias 2-6 utilizadores; raro ter coluna com >50 tasks
- Virtualização adiciona complexidade ao drag (cards fora do DOM precisam de detecção especial)
- @dnd-kit suporta virtualization mas requer `<SortableContext strategy={verticalListSortingStrategy}>` config diferente

**Threshold token:** `--kanban-column-min-tasks-virtualize: 50` (configurable; pode subir/descer pós-validação).

### 7.3 Optimizações additivas (sempre activas)

| Optimização | Detalhe |
|-------------|---------|
| Pagination implícita | Quando `>50` cards na coluna, mostrar 50 + footer "Ver mais 12 →" (FR9 inheritance) |
| Drag throttle | `onDragOver` throttled a 60fps via @dnd-kit nativo (não custom) |
| Memo cards | `React.memo(KanbanCard, deepEqual)` com props pegada — evita re-render durante drag de outro card |
| Lazy column rendering | Colunas fora do viewport (scroll horizontal) com `content-visibility: auto` CSS |
| Suspense boundaries | Cada coluna em `<Suspense fallback={<ColumnSkeleton />}>` para streaming |

### 7.4 Performance budget (target metrics)

| Métrica | Target | Validação |
|---------|--------|-----------|
| Initial render (4 colunas × 25 tasks = 100 cards) | < 200ms | React Profiler |
| Drag start latency | < 50ms (entre pointerdown + visual feedback) | Lighthouse / manual timing |
| Drag move FPS | ≥ 50fps em desktop, ≥ 30fps em mobile mid-range | Chrome DevTools Performance |
| Drop confirm UI latency (optimistic) | 0ms (synchronous) | — |
| Drop API roundtrip target | < 300ms p50, < 1000ms p99 | Vercel logs |

## 8. Acessibilidade (WCAG AA + AAA where possible)

### 8.1 Keyboard navigation

| Tecla | Comportamento |
|-------|---------------|
| `Tab` | Navega entre colunas (header → cards → footer add-inline) |
| `Tab` dentro de coluna | Navega entre cards sequencialmente |
| `Space` (em card focado) | **Pick up** card (inicia keyboard drag); aria-pressed=true |
| `Arrow Up/Down` (durante drag) | Move card uma posição cima/baixo dentro da coluna |
| `Arrow Left/Right` (durante drag) | Move card para coluna anterior/próxima (mesma posição relativa) |
| `Enter` ou `Space` (durante drag) | **Drop** card na posição actual |
| `Esc` (durante drag) | **Cancel** drag, restaura posição original |
| `Enter` (em card sem drag) | Abrir `EditTaskModal` |
| `Cmd/Ctrl + Enter` (em add-inline input) | Criar tarefa nessa coluna |

**Implementação:** @dnd-kit `KeyboardSensor` + `sortableKeyboardCoordinates` (já configurado §5.1) fornece tudo isto out-of-box.

### 8.5 Reduced motion

- `prefers-reduced-motion: reduce`: drag overlay sem rotate/scale, transitions a 0ms, fade-only para enter/exit

### 11.1 DP-3.4.7 Resolution — endpoints `/api/kanban-columns` (Aria, 2026-05-17)

**Verdict:** Opção A. Não existem. Story 3.2 ratificou defer via DP4-3.2 = B ACCEPT.

**Estado actual `apps/web/src/app/api/`:** directórios `agent`, `conta`, `inngest`, `me`, `recurrences`, `tags`, `tasks`. Ausentes `kanban-columns` ou variantes. Endpoint `/api/tasks/[id]/move/route.ts` existe (Story 3.2 entregou).

**Schema `kanban_columns` (`packages/db/src/schema/tenancy.ts:152-173`):**
- Colunas: `id`, `household_id` (FK CASCADE), `name`, `sort_order` (int default 0), `color` (default `#6B7280`), `is_done_column` (**`text` not boolean — default `'false'`**), `created_at`, `updated_at`
- Constraints existentes: `householdIdx`, `unique(household_id, sort_order)`
- **Gap schema HIGH:** sem CHECK `count ≤ 6 per household` nem `unique partial is_done_column=true per household`
- **Gap schema MEDIUM:** `is_done_column` é `text` (`'true'`/`'false'`) em vez de `boolean` — inconsistente com NFR best practice

**RLS (`0001_rls_policies.sql:150-168`):** 4 policies confirmadas:
- `kanban_columns_select_member` · `kanban_columns_insert_member` · `kanban_columns_update_member` · `kanban_columns_delete_owner_admin` (**variant: só owner/admin elimina**)

**5 ACs adicionais para Story 3.4 (Aria recommendation):**

```
AC-7: GET /api/kanban-columns → lista ordenada por sort_order ASC, household via RLS authDb.
      Response: { id, name, sort_order, color, is_done_column }. Cache: no-store.

AC-8: POST /api/kanban-columns → Zod { name: 1-40 chars, color?: hex, sort_order?: int }.
      Server valida count actual < 6 (SELECT count(*) WHERE household_id). sort_order = MAX+1 se omitido.
      audit_log entry { entity_type: 'kanban_column', action: 'create' }.

AC-9: PATCH /api/kanban-columns/[id] → partial { name?, color?, is_done_column?, sort_order? }.
      Se is_done_column=true: server desliga todos os outros (UPDATE SET is_done_column='false'
        WHERE household_id AND id != [id]) na mesma transaction.
      Se sort_order muda: rebalanceia siblings (shift pattern semelhante kanban_position em tasks).
      audit_log action='update' com diff.

AC-10: DELETE /api/kanban-columns/[id] → role owner OU admin (resolveHouseholdRole + RLS reforça).
       Se coluna tem tasks: query param ?move_to=<column_id> obrigatório (validar destino ∈ household, ≠ id).
       UPDATE tasks SET kanban_column_id=move_to em transaction com DELETE.
       Sem move_to: 409 Conflict com payload { tasks_count: N }. audit_log action='delete' com snapshot.

AC-11: PATCH /api/kanban-columns/batch (DP-3.4.6 opção A) → { columns: [{id, sort_order, name?, is_done_column?}],
       creates?: [...], deletes?: [...] }. Tudo em db.transaction() — rollback total se qualquer falhar.
       Valida invariants pós-batch: 3 ≤ count ≤ 6, exactly 1 is_done_column=true, names únicos.
       Returns final state. audit_log action='batch_update'.
```

**Sub-tasks adicionais sugeridas a `@sm`:**

- **T-X.1 (coordenação @data-engineer):** decidir migration `0010_kanban_columns_constraints.sql` — CHECK `count ≤ 6` + partial unique `is_done_column=true per household`. Recomendação Aria: **migração DB** (defesa em profundidade NFR5) vs só app-layer
- **T-X.2 (housekeeping):** converter `is_done_column text → boolean` na mesma migration 0010 (gap MEDIUM)

**Impacto estimate:** **L → XL** (~25-30% scope adicional). Aria recomenda **manter 3.4 como XL único** (não spin-off Story 3.4a) por coesão narrativa + evitar 2 ciclos QA gate.

**Risk flags integrados:**

| Risk | Severidade | Mitigação |
|------|------------|-----------|
| NFR5 RLS DELETE owner/admin only | LOW (RLS bloqueia naturalmente) | UI trata 403 com toast PT-PT "Apenas administradores podem eliminar colunas." |
| NFR9 audit POST/PATCH/DELETE | MEDIUM | Confirmar `audit_log.entity_type` aceita `'kanban_column'` (com `@data-engineer`); aplicar `insertAuditLog` nos 4 endpoints write |

### 14.3 Pattern optimistic update + revert

```tsx
async function handleDragEnd(event: DragEndEvent) {
  const { active, over } = event;
  if (!over) return;

  const taskId = active.id as string;
  const targetColumnId = /* extract from over */;
  const targetPosition = /* compute */;

  const previousTasks = tasks;
  // 1. Optimistic update
  setTasks(prev => moveTask(prev, taskId, targetColumnId, targetPosition));

  // 2. PATCH API
  try {
    const res = await fetch(`/api/tasks/${taskId}/move`, {
      method: 'PATCH',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ kanbanColumnId: targetColumnId, kanbanPosition: targetPosition }),
    });
    if (!res.ok) throw new ApiError(res.status, await res.text());
  } catch (err) {
    // 3. Revert + toast
    setTasks(previousTasks);
    toast.error(mapErrorToCopy(err));   // §5.4 mapping
    captureException(err);              // Sentry
  }
}
```
