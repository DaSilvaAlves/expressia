## 4. AC4 — Verificação runtime de fuga de dados (núcleo do security gate)

O @po confirmou a condição estrutural (gate quebrado) mas **não** testou a profundidade da fuga. O @qa fechou essa lacuna por inspecção do caminho de dados real:

**Mapa de guardas `getUser()` ao nível de página (12 rotas `(app)/`):**

- **10/12 têm guarda própria** (`getUser()` + `redirect('/entrar')` *antes* de qualquer `getDb()`): `visao`, `conta/preferencias`, `financas/{este-mes,cartoes,patrimonio,recorrentes,variaveis}`, `tarefas`, `tarefas/{kanban,calendario}`.
- **2/12 sem guarda própria:** `financas` (índice) e `jarvis`.
  - `financas/page.tsx:9-10` — corpo é literalmente `redirect('/financas/este-mes')`. **Zero fetch de dados.** O destino (`este-mes`) tem guarda própria.
  - `jarvis/page.tsx` — já estava coberto pelo middleware **antes** do fix (`/jarvis` ∈ array desde a 2.7).

**Cadeia de um GET anónimo a `/financas/este-mes` ANTES do fix:**
1. Middleware: sem redirect (o bug).
2. `este-mes/page.tsx:59-63`: `getUser()` → `null` → `redirect('/entrar')` — **lança antes** do `getDb()` (linha 93).
3. Resultado: **nenhuma query executada, nenhum dado renderizado.**

**Conclusão AC4:** a profundidade real da fuga era **NIL**. O bug era uma lacuna genuína de auth gate (NFR8 — utilizadores anónimos não eram redireccionados ao nível do gate universal), mas a defesa-em-profundidade ao nível da página já impedia exposição de dados em todas as rotas portadoras de dados. **Depois do fix:** o middleware devolve 307 → `/entrar` antes mesmo de o RSC renderizar — defesa-em-profundidade restaurada no gate universal.

**Superfícies fora do middleware:** o `matcher` exclui `/api/billing` e `/api/cron` (`middleware.ts:86` — webhooks externos sem JWT, sem dados de utilizador). Restantes `/api/*` passam o middleware mas não estão em `APP_PATH_PREFIXES` (não são rotas UI; as route handlers usam `requireAuth` → 401, ver `lib/api-helpers/auth.ts:63-77`). Sem fuga.
