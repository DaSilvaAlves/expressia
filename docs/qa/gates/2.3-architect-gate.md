**Pergunta arquitectural:** O facto do package `@meu-jarvis/tools` não conseguir importar `agentReverseOps` (Drizzle table) de `@meu-jarvis/db` é uma violação de Article IV (No Invention) ou má arquitectura?

| Source | Localização | Confirma |
|--------|-------------|----------|
| Schema real Drizzle | `packages/db/src/schema/agent.ts:188-215` | Tabela `agent_reverse_ops` com `agent_run_id`, `household_id`, `reverse_op`, `expires_at` |
| SQL na implementação | `packages/tools/src/atomic.ts:244-250` | `insert into agent_reverse_ops (agent_run_id, household_id, reverse_op, expires_at) values (..., now() + interval '30 seconds') returning id` |
| Sanity test 1 | `packages/tools/src/__tests__/atomic.test.ts:289-300` | Captura `sqlText` do template e valida regex `/now\(\)\s*\+\s*interval\s*'30\s*seconds'/i` |
| Sanity test 2 | `packages/tools/src/__tests__/atomic.test.ts:303-317` | Valida que SQL inclui referências às colunas `agent_run_id`, `household_id`, `reverse_op`, `returning id` |
| Story dependência | Story 2.3 v1.2 line 47 (Dependências) | "DEPENDE de Story 1.3 Done (schema `agent_reverse_ops` aplicado: migration `0005`, coluna `expires_at DEFAULT now() + interval '30 seconds'`)" |

**5 fontes coerentes.** A divergência entre o SQL e a tabela real seria detectada em runtime production (PostgresError tipo `column "x" does not exist` ou `relation "agent_reverse_ops" does not exist`). O sanity test em mock garante que o SQL string contém os elementos correctos antes de chegar a produção.

Esta decisão é **isomorfa a D1 da Story 2.2** (LLM_MODEL_VALUES hardcoded + sanity test) — pattern arquitectural consistente do monorepo: quando TS paths não-propagam-cross-project, preferir documentação + test sobre import directo.

### D2 — Composite per-level recursivo — análise definitiva

**Pergunta arquitectural:** Por que escolher per-level em vez de top-level, dado que a story Pax v1.1 dizia "qualquer escolha é defensável"?

**Resposta:** **Per-level é a escolha defensiva correcta.**

Top-level only permitiria `10^depth` ops em aninhamento, gerando um vector de attack via payload jsonb:

```
{ kind: 'composite', ops: [10× { kind: 'composite', ops: [10× { kind: 'composite', ops: [10× delete_row] }] }] }
// = 1000 ops num único reverse_op — bytea aceitar mas o cleanup futuro (Story 2.7+) pagaria o preço
```

Per-level mantém o payload bounded em `O(10^depth)` mas **a profundidade de aninhamento natural do uso real esperado é ≤2 níveis** (ex: `criar_parcelada` cria parent transaction + 12 child installments → 1 composite com ~13 ops top-level, sem aninhamento adicional). 

A implementação via `z.lazy(() => discriminatedUnion(...))` (`contracts.ts:192-201`) é elegante: o `.max(COMPOSITE_REVERSE_OP_MAX_OPS)` aplica-se em cada nível recursivamente sem precisar de checks adicionais. Zero overhead de implementação para máxima defesa.

**Cobertura completa:** 4 testes em `contracts.test.ts`:
- linha 80-88: 10 ops OK (limite inclusivo)
- linha 90-98: 11 ops fail (limite exclusivo)
- linha 100-113: nested composite com 11 ops fail (per-level enforced)
- linha 60-76: nested 1-nível válido passa (round-trip preserva structure)

**Endereça PO suggestion #2 conclusivamente** — não é apenas "documentação", é runtime enforcement.

### D4 — `InternalAtomicAbort` marker class privada — semântica correcta

Drizzle's `transaction(fn)` faz rollback automático sempre que `fn` lança. O desafio é distinguir **2 cenários do callback fn**:

| Cenário | Causa | Resultado para caller |
|---------|-------|----------------------|
| (a) | Tool falhou (input inválido, execute lança, output Zod inválido, reverse lança) | `AtomicFailure { success: false, failedToolName, error: ToolError, rolledBack: true }` |
| (b) | Transacção falhou (deadlock, connection drop, exception Postgres) | Throw `ToolTransactionError { retryable: true }` |

O `InternalAtomicAbort` (atomic.ts:335-345) resolve elegantemente:
- Quando tool falha, `executeAtomic` faz `throw new InternalAtomicAbort(toolName, toolError)` (linha 296), que envolve o `ToolError` original.
- Drizzle vê a excepção e faz rollback (cenário esperado).
- O catch fora da transacção (linha 310-323) faz `instanceof InternalAtomicAbort` check:
  - Match → retorna `AtomicFailure` (cenário a)
  - Outro tipo → wraps em `ToolTransactionError` retryable (cenário b)

A class é privada ao módulo (não exportada no barrel `index.ts`). Consumers só vêem `AtomicFailure` ou `ToolTransactionError`. **Type-safe + stack-trace preserved + semanticamente claro.**

Cobertura: 4 testes em `atomic.test.ts:194-285` cobrem todos os cenários de rollback (tool 2 falha, input inválido, schema output bad, reverse() lança), e o cenário (b) é coberto pela path `err instanceof ToolError ? err : new ToolTransactionError(err)` (atomic.ts:323) — não é exercitado directamente nos testes (porque o mock `ctx.db.transaction` não simula deadlock natural), mas a lógica é trivialmente verificável por inspecção.

**Trade-off:** 1 class extra (em troca de type safety completo). Aceitável — uma sentinel value perderia o stack trace.

### D5 — Tracing 2-níveis whitelist 10 keys — análise

**Mecânica:** `executeAtomic` cria 2 níveis de spans:

1. **`agent.tool.atomic`** (1 por chamada de `executeAtomic`, criado primeiro em `withAtomicSpan`):
   - `tool.atomic.tool_count` (number) — N tools no batch
   - `tool.atomic.run_id` (string) — UUID de `agent_runs.id` para correlação
   - `tool.atomic.success` (boolean) — finalizado em `annotateAtomicMetrics`
   - `tool.atomic.rolled_back` (boolean) — true se houve falha controlada

2. **`agent.tool.call`** (N spans, 1 por tool individual, criado em `withToolSpan`):
   - `tool.name` (string)
   - `tool.domain` (string)
   - `tool.duration_ms` (number)
   - `tool.success` (boolean)
   - `tool.household_hash` (string sha256 via `hashForCorrelation`)
   - `tool.trace_id` (string)

Total: **10 keys whitelist**. Validado em runtime pelo teste `tracing.test.ts:198-206`:

```ts
expect(TOOL_SPAN_ATTRIBUTE_KEYS.length).toBe(10);
const flat = (TOOL_SPAN_ATTRIBUTE_KEYS as ReadonlyArray<string>).join(' ').toLowerCase();
expect(flat).not.toContain('input');
expect(flat).not.toContain('output');
expect(flat).not.toContain('snapshot');
expect(flat).not.toContain('prompt');
expect(flat).not.toContain('message');
```

**Padrão consistente com Story 2.2** (`PROVIDER_SPAN_ATTRIBUTE_KEYS` também 10 keys). Os 2 níveis permitem queries Grafana distintas:
- "Latência média da tool `criar_tarefa` no domain `tasks`" → query sobre `agent.tool.call`
- "Rollback rate de `executeAtomic` por hora" → query sobre `agent.tool.atomic`

`tool.household_hash` usa `hashForCorrelation` (verificado pelo teste `tracing.test.ts:127-145`) — UUID raw NUNCA emitido. Defesa NFR12 sólida.

### D6 — `ToolRegistry.has()` + `clear()` @internal

`has(name): boolean` adicional fora do AC: **decisão pragmática correcta**. Story 2.5 Planner pode querer verificar tool existence antes de fazer `get()` (que lança ToolNotFoundError), evitando try/catch wrap. Custo zero (Map.has é O(1)), ganho útil. JSDoc explica o uso.
