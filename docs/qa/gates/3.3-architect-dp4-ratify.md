## Rationale (5 razões em ordem decrescente de peso)

### 1. Pattern continuity — D-3.2.1 explicitamente antecipou este uso

O meu próprio gate `docs/qa/gates/3.2-architect-gate.md` linhas 117-133 ratifica D-3.2.1 como "HIGH-VALUE engineering judgment" e diz literalmente: *"Pattern reusable Stories 3.3-3.8 + future epics que adicionem routes"*. `listTasksHelper` é exactamente o tipo de reuse antecipado — não um pattern novo. Rejeitar agora seria contradizer o rationale do meu próprio gate Story 3.2.

### 2. Extraction surface é arquitecturalmente limpa

Analisei `route.ts:59-157`:

| Linhas | Conteúdo | Destino |
|--------|----------|---------|
| 65-91 | auth (`requireAuth`) + Zod parse `TaskFiltersSchema` + cursor decode | **Permanece em wrapper** (HTTP-bound) |
| 93-145 | build dynamic WHERE + SELECT + cursor encode + shape `{ tasks, next_cursor }` | **Move para helper puro** |
| 146-154 | catch + observability + `apiError` | **Permanece em wrapper** (Response-bound) |

A separação é natural: o helper recebe inputs já validados (`{ filters: TaskFiltersInput, householdId, userId, db: DbShim }`) e devolve um objecto serializável. Sem dependências em `NextRequest`/`NextResponse`/`Span`. Testable e reusable.

### 3. Precedente exacto da forma em `audit.ts`

`apps/web/src/lib/api-helpers/audit.ts:42` já implementa o pattern proposto: `insertAuditLog(params: { db: DbShim, ... })` aceita `db` como parâmetro injectado. Aplicar a mesma forma a `listTasksHelper(params: { db, filters, householdId, userId })` é literalmente copy-pattern, não novo design.

### 4. Cobertura de regression — 5 tests do SQL surface garantem

O handoff PO refere "12 GET regression tests"; contagem real em `apps/web/src/app/api/tasks/__tests__/list.test.ts` é **9 GET tests**. **Drift cosmético** (provável conflate POST + GET tests). Não é blocker porque dos 9:

- 4 tests validam o wrapper (401 auth, 404 household, 400 validation × 2) — não tocam helper
- **5 tests exercitam a SQL surface que migra para o helper:** happy path 200 (linhas 93-107), cursor next 200 (109-127), composite filters 200 (129-139), DB error 500 (141-153), PT-PT CON3 (155-160)

Os 5 tests críticos cobrem precisamente o código que move. T2.5 adiciona ~4 sort tests novos como cobertura adicional. Regression risk **LOW** mantém-se.

> NIT-non-blocking: contagem real é 9 GET tests, não 12 — handoff future deve corrigir. Não bloqueia esta decisão.

### 5. Anti-tech-debt multi-story (Epic 3)

Stories 3.4 (Kanban), 3.5 (calendário), 3.6 (tags) consomem listings de tarefas com projecções de filter diferentes. Sem `listTasksHelper`:
- Cada vista RSC duplica WHERE-building + SELECT + cursor encoding (3× drift surface), OU
- RSC invoca recursivamente o route handler internamente (loopback HTTP latency + bundle auth/Zod que RSC tem de forma diferente)

Extracted helper é o único path que escala cleanly. Rejeitar agora cria tech debt explícito documentado para Epic 4+ housekeeping.

## Guidance Notes para @dev

Recomendações não-bloqueantes que melhoram qualidade da implementação:

### G1. Helper aceita `db` como param (NÃO chama `getDb()` internamente)

**Why:** Testability (mocks de `db-shim` em RSC tests sem ter de stubbar internamente) + futuro reuse com `getServiceDb()` em Inngest jobs (Story 3.7 generate-recurring-tasks pode reusar a mesma SQL surface com role `service_role`).

**How:**
```typescript
// list-tasks.ts
export async function listTasksHelper(params: {
  readonly filters: TaskFiltersInput;
  readonly householdId: string;
  readonly userId: string;
  readonly db: DbShim;
}): Promise<{ tasks: TaskRow[]; next_cursor: string | null }> {
  // ... pure SQL build + execute + cursor encode
}
```

Route handler injecta `getDb()`; RSC `tarefas/page.tsx` injecta `getDb()`; Inngest jobs futuros podem injectar `getServiceDb()`.

### G2. Mover `TaskRow` interface para o helper file

`TaskRow` (route.ts:38-57) é o output type que ambos handler e helper precisam. Move para `list-tasks.ts` e re-export para evitar duplicate type definitions.

### G3. Preservar `1=1` placeholder pattern em WHERE building

O pattern `const conditions = [sql\`1=1\`]; ... conditions.reduce(...)` (route.ts:98) é defensive (sempre devolve um WHERE válido mesmo sem filters). Manter no helper sem optimizar — passou o gate Story 3.2 e é a forma idiomática Drizzle aqui.

### G4. `householdId` é redundant param mas mantém para clarity

O JWT já filtra `household_id` via RLS (`getDb()` authenticated role); o param `householdId` não é injectado no SQL no helper. Mantém na signature mesmo assim para:
- Audit trail consistency (helper pode logar `user_hash`/`household_id` para correlation futura)
- Eventual non-RLS path (se Story Inngest futura usar `getServiceDb()` precisa de filter explícito)

### G5. `cursorPayload` decode permanece no wrapper

`decodeCursor(filters.cursor)` (route.ts:84-91) devolve 400 se cursor malformado. **Isto é HTTP-concern, fica no wrapper.** O helper recebe `cursorPayload` já decoded (ou null). Signature ajustada:

```typescript
listTasksHelper(params: {
  filters: TaskFiltersInput;
  cursorPayload: { last_due_date: string | null; last_id: string } | null;
  householdId: string;
  userId: string;
  db: DbShim;
})
```

RSC `tarefas/page.tsx` faz o mesmo: `searchParams.cursor` → `decodeCursor` → render `<EmptyState variant="error">` se inválido → caso contrário invoke helper.

### G6. Sort param (T2.3-T2.4 — DP5 separate ratify)

Quando adicionares `filters.sort` ao schema, a `ORDER BY` clause vira branched no helper:

```typescript
let orderBy;
switch (filters.sort) {
  case 'due_date_asc': orderBy = sql`due_date asc nulls last, id asc`; break;
  case 'created_at_desc': orderBy = sql`created_at desc, id desc`; break;
  case 'priority_desc': orderBy = sql`case priority when 'high' then 1 when 'medium' then 2 else 3 end, due_date asc nulls last, id asc`; break;
  case 'title_asc': orderBy = sql`title asc, id asc`; break;
}
```

**Cursor pagination optimal só para `due_date_asc`** (default). Documentar em comment + na DoD line 874 (já feito v1.1).
