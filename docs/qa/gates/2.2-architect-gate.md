## Análise específica: as 10 decisões D1-D10

### D1 — `LLM_MODEL_VALUES` hardcoded — análise definitiva

**Pergunta arquitectural:** A duplicação dos 3 nomes de model é uma violação de Article IV (No Invention)?

**Resposta:** **Não. É replicação documentada com sanity test, não invenção.**

| Source | Localização | 3 valores |
|--------|-------------|-----------|
| Postgres enum (truth) | `packages/db/src/schema/agent.ts:57-60` | `'gpt-4o-mini', 'claude-sonnet-4-5', 'claude-opus-4-7'` |
| Agent contracts | `packages/agent/src/contracts.ts:27-31` | `'gpt-4o-mini', 'claude-sonnet-4-5', 'claude-opus-4-7'` |
| Sanity test | `packages/agent/src/__tests__/contracts.test.ts:13-21` | Valida `length === 3` + 3 toContain |
| Story 2.1 dependência | Story 2.2 v1.2 line 73 (Contexto de entrada) | "llm_model: gpt-4o-mini, claude-sonnet-4-5, claude-opus-4-7" |

**4 fontes coerentes. Zero invenção.** A duplicação resulta de uma limitação técnica real (TS paths não-propagam-cross-project entre packages do mesmo monorepo) — pattern consistente com `@meu-jarvis/web`, `@meu-jarvis/auth` e `@meu-jarvis/db-test` que também não importam directamente de `@meu-jarvis/db`. A solução com sanity test é o equivalente determinístico a um import directo: se o enum mudar e o tuple não, o test parte. Article IV preservado.

### D2-D3 — TS strict + interfaces customizadas

`override` em `NetworkError.cause` é tecnicamente correcto (TS4114 evitado). Interfaces `AnthropicClientLike` / `OpenAIClientLike` são uma boa decisão de design — desacopla de breaking changes SDKs e permite `clientOverride` tipado em tests sem `as unknown as Anthropic`. Pattern ergonómico para o test estilo "explicit injection" usado em `anthropic.test.ts` (vs `vi.mock` hoisted style das Dev Notes — ambos válidos, o primeiro é mais legível para gates arquitecturais).

### D4 — `sanitizeHint` regex composto

Esta é uma decisão de qualidade que reflecte conhecimento de domínio PT-PT. NIFs portugueses (9 dígitos) são literais que o regex `[A-Za-z0-9_-]{10,}` original deixaria escapar (9 < 10). O segundo pass `\d{9,}` é específico para identificadores numéricos longos: NIF (PT, 9), CPF (BR, 11), NIB pré-IBAN (PT, 21). Test `errors.test.ts:61-75` valida com mock realístico que combina ambos os tipos de leak. **Decisão sólida e bem-testada.**

### D5 — Cooldown progressivo

A escolha 30s→60s é justificada como "defensive engineering" no comment SQL (`circuit-breaker.ts:13-15`). Em provider outages prolongados (>5min), permitir half_open probe a cada 30s gera traffic sustained que pode prolongar o outage do lado do provider (thundering herd quando recover). Duplicar o cooldown após half_open fail é o pattern padrão de exponential backoff aplicado a circuit breakers. **Decisão idiomática.**

Trade-off: em outages curtos (1-2min), o caller pode ver `CircuitOpenError` por mais tempo do que estritamente necessário. Aceitável porque (1) `CircuitOpenError.userMessage` em PT-PT é informativo, (2) Story 2.9 (Upstash distributed) vai introduzir métricas globais que permitem decisões mais finas.

### D6 — Cost calculation devolve `costUsd` + `costEur`

Não-bloqueante. AC pedia só `costEur` mas a interface `CostBreakdown` (`pricing.ts:70-73`) expõe ambos. Custo zero, ganho potencial: Story 2.9 (cost router) vai auditar conversão FX em tempo real e ter o intermediário sem recomputar é cómodo. Não há leak — ambos os campos são públicos por design. **Aceito como antecipação útil.**

### D7 — Anthropic prompt cache: `cache_control` no último tool

Esta decisão tem nuance que vale a pena destacar para futuros maintainers:

**Mecânica Anthropic:** o array `tools: [...]` numa request a `messages.create` pode receber `cache_control: { type: 'ephemeral' }` em **qualquer** elemento. Quando aplicado ao **último** tool, o marker propaga **para trás** cobrindo system + tool defs como uma unidade coesa de cache. Aplicar a um tool intermédio criaria 2 unidades de cache separadas (menos eficiente).

**Implementação (`providers/anthropic.ts:183-199`):**
```ts
const last = input.tools.length - 1;
return input.tools.map((t, idx) => {
  const base = { name: t.name, description: t.description, input_schema: t.input_schema };
  if (input.cacheControl === 'ephemeral' && idx === last) {
    base.cache_control = { type: 'ephemeral' };
  }
  return base;
});
```

**Cobertura test:** `anthropic.test.ts:78-95` valida `systemBlocks[0]?.cache_control === { type: 'ephemeral' }` quando `cacheControl: 'ephemeral'` no input. Não há test específico que valide o cache marker no último tool (apenas no system block) — não bloqueante porque a lógica `buildToolsParam` é trivialmente verificável por inspecção, mas registado como [TEST-001-NB] abaixo para Story 2.3 (Tool Registry) que vai exercitar este caminho com tools reais.

### D8 — OpenAI tool_calls mal-formados → `input: {}`

Decisão de robustez explícita. Permite que Story 2.5 Planner+Executor implemente lookup do schema da tool (de um registry) e reje calls com input vazio sem precisar de `try/catch JSON.parse` em cada call site. Test `openai.test.ts:127-145` valida directamente. **Pattern correcto para chains de tool use onde retry-then-regenerate > propagate-error.**

### D9 — Factory cache não considera `AGENT_FALLBACK_OPENAI_ENABLED`

Esta é a decisão mais delicada das 10. **Arquitecturalmente aceito para MVP, mas com nota explícita para Story 2.9.**

**Comportamento:** `getProvider()` faz `cacheKey = providerId::model` e armazena instâncias em `Map`. Se `AGENT_FALLBACK_OPENAI_ENABLED` mudar mid-session, o cache não invalida. Em runtime real (Vercel functions), env vars são fixas durante o lifetime do process — não há mutação observável. O cache é correcto.

**Risco residual:** local dev / tests onde alguém faz `process.env.AGENT_FALLBACK_OPENAI_ENABLED = 'true'` mid-test sem chamar `resetProviderCache()`. Test `factory.test.ts:8-22` cobre isto correctamente com `beforeEach`/`afterEach` que chamam `resetProviderCache()` + `CircuitBreaker.resetAll()`.

**Story 2.9 substituição:** o handoff explicitamente nota que Story 2.9 introduz `RouteDecision` Upstash-backed que substitui completamente este factory. Não é technical debt — é uma escolha de scope deliberada. **Aprovado como interim solution.**

### D10 — `agent.fallback_used` sempre `false` no MVP

`tracing.ts:44-54` declara `fallbackUsed` no interface `ProviderCallMetrics`, e `annotateProviderMetrics` aplica como span attribute. Em `anthropic.ts:118` e `openai.ts:113`, é hardcoded `fallbackUsed: false` em ambos os providers — porque na shape actual nenhum provider sabe se é o "primary" ou "fallback" (essa decisão é do factory).

**Trade-off:** o span attribute existe e é whitelisted, mas não carrega informação útil até Story 2.9 introduzir routing centralizado que injecta o flag no input. Para o MVP, qualquer downstream consumer (Grafana, Sentry) verá `agent.fallback_used: false` em 100% dos casos — equivalente a não emitir o attribute.

**Não-bloqueante** porque (a) a infraestrutura existe e está testada (`tracing.test.ts:101-108` valida `fallbackUsed: true` é aplicado quando passado), (b) Story 2.9 vai actualizar o caller para passar `true` no fallback path. Aceito como hook hierárquico para a história próxima.

| **2 — Span attribute whitelist** | `tracing.ts:28-39` (10 keys) + `annotateProviderMetrics` (linha 95-104) | Só keys em `PROVIDER_SPAN_ATTRIBUTE_KEYS` podem ser emitidas. **Sobreposição minimal e necessária:** as 3 camadas operam em vectores diferentes (payload struct vs span KV vs error message string). Remoção de qualquer uma cria buraco específico:

- Sem camada 2 → developer pode acidentalmente fazer `span.setAttribute('agent.prompt', input.system)` e PII flui para Grafana.
- Sem camada 3 → SDK error message com NIF/token aparece em error events Sentry.

**Ortogonalidade verificada.** PII guardrail completo.

**[TEST-001-NB] Sem test específico para `cache_control` no último tool (D7)**
- **File:** `packages/agent/src/__tests__/providers/anthropic.test.ts`
- **Estado actual:** Test "aplica cache_control: ephemeral ao system prompt" (linha 78-95) valida `systemBlocks[0]?.cache_control` mas não inspecciona o array `tools` resultante.

- **Não corrigido neste gate:** custo > benefício para algo cosmético. A lógica está cobrindo o caso real (system block) e o pattern mecânico do array é legível.
- **Recomendação:** Story 2.3 (Tool Registry) deve adicionar 1 test que inspecciona `params.tools[lastIdx]?.cache_control` quando `cacheControl: 'ephemeral'` + tools array com ≥2 elementos. Não criar story dedicada.

**[ARCH-001-NB] `ProviderCompleteOutputSchema.parse` corre dentro do retry loop**
- **File:** `packages/agent/src/providers/anthropic.ts:97` + `providers/openai.ts:93`
- **Comportamento actual:** Em `callOnce`, depois de `mapAnthropicResponse` retornar um objecto, `ProviderCompleteOutputSchema.parse(mapped)` valida runtime. Se a SDK retornar shape inesperada (ex: tokens negativos), `ZodError` é lançado **antes** do `mapAnthropicError` ser chamado — logo o `ZodError` propaga sem ser convertido em `ProviderError`, e `withRetry` (que só retenta `ProviderError.retryable === true`) propaga imediatamente.
- **Avaliação:** Comportamento aceite — se a SDK retorna shape inválido, é um bug do provider e retry não vai resolver. Mas o caller superior (Story 2.5 Planner+Executor) vai precisar de `try/catch` para `ZodError` específico, em vez de apenas `instanceof ProviderError`. Não está documentado.
- **Não corrigido neste gate:** mudar isto requer envolver o parse em try/catch que mapeia ZodError → BadRequestError ou similar. Não é trivial e tem trade-offs (perde-se detalhe da validação Zod). Aceito como design current.
- **Recomendação:** Adicionar JSDoc em `complete` documentando que `ZodError` também pode ser thrown além de `ProviderError`. Story 2.5 deve handle ambos. **Não bloqueante.**

**[DOC-001-NB] `factory.test.ts` não testa cache stale após mudança de `AGENT_FALLBACK_OPENAI_ENABLED` (D9)**
- **File:** `packages/agent/src/__tests__/providers/factory.test.ts`
- **Estado actual:** Test "isFallbackOpenAIEnabled reflete env var" (linha 93-100) valida que a função lê env correctamente, mas não há test que demonstre o cache stale: `getProvider()` cacheia AnthropicProvider → flag muda → `getProvider()` continua a retornar o cached.

- **Não corrigido neste gate:** documentar comportamento conhecido em test seria valioso para futuros maintainers. Mas adicionar o test pode ser interpretado como "endorsement" do comportamento — ambíguo.
- **Recomendação:** Story 2.9 (Cost Router) deve incluir explicit test que demonstra a invalidação de cache quando flag muda — a partir desse momento, é um requisito real, não trade-off MVP. **Não bloqueante.**
