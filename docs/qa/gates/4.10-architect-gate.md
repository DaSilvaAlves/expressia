- `packages/tools/src/finance/create-finance-variable.ts` confere: inputSchema com `description: z.string().min(1).max(500)` (F1 NOT NULL ✓), `paymentMethod` opcional inferido (F1 ✓), `accountId XOR cardId` via `.refine()` (CHECK `transactions_account_or_card` ✓), `kind: enum(['expense','income'])` (DP-4.10.C — sem `transfer` ✓), `categoryId` opcional (DP-4.10.D ✓).
- `execute()` resolve `categoryId` via `resolveDefaultCategory({db: ctx.db, kind})` (F6 ✓), `paymentMethod` via fallback `cardId → 'card' / accountId → 'transfer'` (IMPL-4.10.4 ✓), INSERT em `transactions` com `household_id=ctx.householdId, created_by_user_id=ctx.userId, is_projected=false, agent_run_id=ctx.runId` (RLS via ctx.db ✓).
- `reverse(): {kind: 'delete_row', table: 'transactions', id: output.transactionId}` ✓.
- 31 testes GREEN.

### AC2 — `create_finance_recurrence` ✅ PASS
- `create-finance-recurrence.ts` confere: `description NOT NULL` (F2 ✓), `startsOn` no Zod (não `startDate`) (F2 ✓), sem `dayOfMonth` (F2 ✓), `frequency: enum(['monthly','weekly','yearly'])` (subset MVP documentado ✓).
- 20 testes GREEN.

### AC3 — `create_card` ✅ PASS
- `create-card.ts` confere: `accountId: z.string().uuid()` NOT NULL (F3 ✓), `cardType: enum(['credit','debit'])` EN literal (F3 ✓), `creditLimitCents` (não `limitCents`) (F3 ✓), `.refine()` força `closingDay + dueDay + creditLimitCents` quando `cardType='credit'` (CHECK `cards_credit_needs_limit` ✓), sem `brand` ou `bankName` (não existem em schema cards) (F3 ✓).
- 20 testes GREEN.

### AC4 — `create_installment` ✅ PASS (architectural deep-dive)
- `create-installment.ts` confere: `firstInstallmentOn` NOT NULL (F4 ✓), cálculo `transaction_date = addMonthsSafe(firstInstallmentOn, i-1)` (linha 158 ✓), helper puro `computeInstallmentSplit` (D-4.10.7 ✓), execute atómico via Postgres tx provida por `executeAtomic` (DP8=A ✓).
- **Composite reverse_op aninhado (D-4.10.4) — verificação empírica detalhada:**
  - Código (linhas 224-246): `chunkArray(output.transactionIds, COMPOSITE_REVERSE_OP_MAX_OPS=10)` produz array de sub-composites; estrutura final é `{kind: 'composite', ops: [...subComposites, {delete_row installments}]}` — ordem CRÍTICA preservada.
  - Test assertion (test linha 322-330): para N=12 → `op.ops.length === 3`, `op.ops[2] === {kind:'delete_row', table:'installments', id:...}` LAST.
  - Test assertion (test linha 355-360): para N=60 → `op.ops.length === 7` (6 sub-composites de 10 tx + 1 delete installments LAST).
  - Test assertion (test linha 423-428): "Ordem: sub-composites de transactions PRIMEIRO, installments POR ÚLTIMO" — `lastOp.table === 'installments'`.
  - **FIFO no /undo route confirmado** (linha 280-289 de `apps/web/src/app/api/agent/prompt/[runId]/undo/route.ts`): `for (const sub of op.ops) { await applyReverseOp(sub, db); }` — linear, em ordem do payload. Logo transactions são apagadas ANTES do installment row, preservando CHECK `transactions_installment_index_coherent`.
- 20 testes GREEN incluindo invariante de cálculo (R-4.1) e N=60 composite válido.

### AC5 — `query_finance_summary` ✅ PASS
