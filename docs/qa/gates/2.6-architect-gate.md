## Sumário executivo

A Story 2.6 entrega exactamente o **consumidor canónico** prometido pela arquitectura §4.1 — um endpoint Next.js Route Handler `POST /api/agent/prompt` que **orquestra os 3 estágios do pipeline AI multi-intent** consumindo as foundations entregues pelas 5 stories Done predecessoras (2.1 schema, 2.2 agent provider abstraction, 2.3 tools+executeAtomic, 2.4 classifier, 2.5 planner-executor). É a story que **fecha o caminho crítico do Epic 2 a 6/11** transformando a pipeline de packages mockable-only num endpoint HTTP real, autenticado, multi-tenant via RLS, com idempotency, rate limiting, audit log, PII redaction defense-in-depth, OTel + Sentry, preview-then-confirm (FR4) e undo de 30 segundos (FR6). A disciplina "Foundation, not behaviour" das stories 2.2-2.5 quebra-se aqui — e bem — porque é exactamente este o ponto da pipeline onde a behaviour real é exposta ao utilizador. Mas a coerência arquitectural é preservada: Classifier continua mockável (`OpenAIClientLike` 2.4), Planner continua mockável (`AnthropicClientLike` 2.5), Executor delega tudo a `executeAtomic` 2.3 — zero re-implementação de transaction wrapper, zero re-implementação de RLS enforcement, zero re-implementação de `agent_reverse_ops` persistence. EB1+EB2+EB3 **não bloquearam** porque `pnpm test` corre 64 testes novos mockable-only sem `ANTHROPIC_API_KEY`/`OPENAI_API_KEY`/Upstash, entregando 480 testes totais (target ≥456 — 5% acima do mínimo).

A decisão arquitectural mais sensível desta story foi a **D27 — refactor cross-package `@/*` → `./*` em packages Done `@meu-jarvis/db` (8 ficheiros) + `@meu-jarvis/planner-executor` (3 ficheiros)**. Esta decisão é a **completion natural da Directive D16** que estabeleci no gate 2.5: "*quando criar um package novo que importa de outro package interno do monorepo, evitar `@/*` em ficheiros source files cross-package*". A 2.5 aplicou D16 em `packages/tools` + `packages/classifier` (que era o scope necessário para 2.5 desbloquear typecheck do `planner-executor`). A 2.6 é o **primeiro consumer real cross-package** dos packages `@meu-jarvis/db` e `@meu-jarvis/planner-executor` (`apps/web` toca-os pela primeira vez como consumer real, não apenas via dev/test isolado), e por isso herda o mesmo problema de resolução: `tsc --noEmit` em `apps/web` falha com `Cannot find module '@/schema'` quando segue um import de `packages/db/src/index.ts` que faz `export * from '@/schema'`. A decisão **D25 — `redactPii` opção (b) implementação local em `apps/web/src/lib/agent/redaction.ts`** é arquitecturalmente correcta e **fecha o gap NIT-002-NB que registei no gate 2.5**. A heurística `sanitizeHint` da 2.2 cobre NIF (9 dígitos) e IBAN PT (PT50... 25 chars alphanum) mas NÃO cobre email completo (`@` quebra alphanum sequence) nem telefone PT (`+351912345678` curto). O @dev avaliou as 3 opções que deixei em aberto no PO_FIX_INLINE da v1.1: (a) usar `sanitizeHint` exportado (gap permanece), (b) implementação local cobrindo o gap, (c) D9-redactPii equivalente (re-export `redactProviderPayload` do `@meu-jarvis/agent`). Escolheu **(b)** com justificação correcta: `redactProviderPayload` é privado intencionalmente per `packages/agent/src/index.ts:12`, e expô-lo via re-export quebraria o encapsulamento por uma necessidade endpoint-específica. A implementação local (175 linhas) cobre **email + telefone PT 9 dígitos + NIF + IBAN PT + cartão crédito 13-19 dígitos** — coberto por **21 testes unitários** em `redaction.test.ts` (3 hashPrompt + 10 redactPiiText + 1 redactEndpointInput + 5 redactEndpointOutput + 3 sentrySafeContext) + 1 assertion E2E em `route.test.ts` que verifica que email aparece redacted no `results.summary` antes da serialização da response. **Camada 4 endpoint-level** complementa as 3 camadas existentes (errors 2.2, OTel whitelist 2.5, hashForCorrelation 1.7) — defense-in-depth NFR12 totalmente coberta com 4 camadas independentes.

A decisão **D26 — skip CodeRabbit pre-commit** é consistente com o precedente das 4 stories anteriores Done (D12 da 2.4, D14 da 2.5, D22 da 2.6 @sm) e arquitecturalmente defensável: este gate manual examinou directamente os 14 ficheiros novos + 8 modificados + 3 docs, re-validou os 5 quality gates, e as decisões arquitecturais críticas desta story (path normalization D21, redaction strategy D25, cross-package refactor D27, undo service_role exception, ALLOWED_REVERSE_TABLES whitelist) estão **fora do escopo automated do CodeRabbit** que brilha em input validation e syntactic patterns. **Aceito o skip** com nota: para Story 2.10 (benchmark E2E com credentials reais Anthropic/OpenAI — toca env handling, secrets, credenciais reais) recomendo CodeRabbit pre-commit por se aproximar mais do focus-area onde brilha (auth, secrets, env handling).

A decisão **D21 — path nested `/api/agent/prompt/[runId]/{confirm|undo}`** vinda da v1.1 da story (com flag CONCERNS-ACCEPTABLE da @po e DEV-DECISION pelo @dev mantendo nested) é **ratificada como ARCHITECT-PASS com Doc Fix inline aplicada**. A architecture.md original tinha drift histórico de duas formas: (1) §1.3 referia `/api/agent/undo/{token}` (token genérico) e (2) §4.5 referia `/api/agent/undo/{runId}` (run_id explícito mas path flat). A escolha do @sm/draft v1.0 + DEV-DECISION D21 do @dev de adoptar `[runId]` nested sob `/api/agent/prompt/` é **arquitecturalmente superior** por 4 razões: (1) **Next.js App Router file-based routing idiomático** — co-localização de `[runId]/confirm/route.ts` + `[runId]/undo/route.ts` sob o feature directory permite middleware partilhado (auth, RLS context, OTel span propagation) sem duplicação; (2) **HATEOAS-style discoverability** — o endpoint principal retorna `confirmation_url` + `undo_url` com paths absolutos que o cliente segue directamente, eliminando ambiguidade entre "token" (genérico) e "run_id" (explícito); (3) **co-localização semântica** — `confirm` e `undo` são operações sobre uma run específica que pertencem semanticamente sob a feature `agent/prompt/`; (4) **codebase é fonte de verdade da architecture** — quando há drift entre architecture.md e implementação correctamente justificada, a implementação ganha e a doc é actualizada (princípio standard de specs vivas). **Apliquei DOC-FIX-001 inline neste gate** em `docs/architecture.md`:

| Secção | Antes (drift histórico) | Depois (DOC-FIX-001) |
|--------|--------------------------|----------------------|
| **§1.3 Fluxo Undo (30s)** | `POST /api/agent/undo/{token}` → lê `agent_runs.reverse_ops` → marca `reverted_at` | `POST /api/agent/prompt/{run_id}/undo` → lê `agent_reverse_ops` (com `executed_at IS NULL`) → executa transacção inversa → marca `executed_at` (row-level) + `agent_runs.reverted_at` (run-level) |
| **§4.5 Endpoint** | `POST /api/agent/undo/{runId}` flat path único | Bloco de 2 endpoints nested REST sob `/api/agent/prompt/[runId]/`: `/undo` (lê `agent_reverse_ops` + `executed_at IS NULL`) + `/confirm` (5min TTL D20 + relê `intents_detected` JSONB) |
| **§4.x Source tree apps/web/** | `agent/{prompt,undo}/route.ts` (flat) | `agent/prompt/route.ts` + `agent/prompt/[runId]/confirm/route.ts` + `agent/prompt/[runId]/undo/route.ts` (nested) |

A DOC-FIX-001 também explicita pela primeira vez na architecture.md a **distinção semântica obrigatória entre `executed_at` (row-level em `agent_reverse_ops` — marca que o reverse op foi aplicado) vs `reverted_at` (run-level em `agent_runs` — marca que a run foi revertida)**. Esta clarificação fecha definitivamente o `DOC-001-NB` que registei no gate 2.3 (architecture §4.5 estava stale com `reverted_at` confundido entre as duas tabelas) — Story 2.3 introduziu o nome correcto da coluna `executed_at` em `agent_reverse_ops` e desde então a doc estava em débito.

A decisão crítica de **`getServiceDb()` no /undo path** é **arquitecturalmente justificada e bem isolada**. O trigger `trg_agent_runs_immutability` (Story 2.1) bloqueia mutação de `status='success'` para role `authenticated` — necessário para preservar a invariante de imutabilidade do audit log NFR9. Mas o undo legítimo precisa transitar `success → reverted` + escrever `reverted_at`. A excepção do `getServiceDb()` é **única, justificada como análoga a GDPR purge**, e bem isolada: (1) o `run_id` é validado via `household_id` ANTES da chamada `getServiceDb()` (cross-household bloqueado pela query inicial com `getDb()` RLS); (2) a janela de 30s + `executed_at IS NULL` é validada antes de executar; (3) a `ALLOWED_REVERSE_TABLES` whitelist em `applyReverseOp` defende SQL injection no campo `op.table` — só permite `tasks`, `transactions`, `recurrences`, `card_accounts`, `card_transactions`, `installments` (verificável em `[runId]/undo/route.ts`). Esta é uma **constraint de hardening que merece atenção em stories futuras**: à medida que novas tools com novas tabelas são adicionadas (Stories 2.7/2.8/2.9), a whitelist tem de ser extendida explicitamente — registado como `NIT-005-NB` para Story 2.9.

A decisão **D18 — Postgres counter para rate limit MVP** (vinda da story v1.0 ratificada pela @po na v1.1) é arquitecturalmente correcta para o estado actual do projecto onde EB3 (Upstash Redis EU) está PENDING. A implementação em `apps/web/src/lib/agent/rate-limiter.ts` (110 linhas) usa UPSERT atómico com `date_trunc('minute', now())` + counter em `agent_rate_limit_counters` (PK `household_id` + `window_start`). A nova tabela traz o `check:rls` de **26 tabelas / 104 policies** (gate 2.5) para **27 tabelas / 108 policies** com 4 RLS policies idempotentes adicionadas em `0001_rls_policies.sql` via DO block condicional (`if exists`). Esta é a **primeira tabela do projecto que escapa ao pattern DDL inline do migration 0001** — o DO block é necessário porque `agent_rate_limit_counters` só é criada em `0006`, não em `0000`, e o regex do `check:rls` gate inspecciona literais `create policy "..." on public.agent_rate_limit_counters for <cmd>` que **funcionam dentro de `EXECUTE`**. Verifiquei independentemente que: (a) `pnpm check:rls` exit 0 com 27/108; (b) a abordagem é compatível com fresh install (DO block só executa se a tabela existir, evitando falha em ambientes pre-0006); (c) o pattern é replicável para futuras tabelas que sejam adicionadas em migrations posteriores ao 0001. A migração para Upstash + sliding window real (Architecture §7.2) fica deferida para Story 2.9 (EB3 unblock + Cost Router + Cache + Quotas) — registado como `NIT-002-NB`.

A decisão de **quota tier-derived hardcoded** (NFR20 implementação MVP) — `free=100/mês, pessoal=500/mês, familia=2000/mês, pro=10_000/mês` — vem do PO_FIX_INLINE da v1.1 que detectou que a coluna `quota_limit` referida na story original v1.0 NÃO existe em `agent_quotas` (real schema tem `prompts_used` + `cost_eur_used` + `month` apenas — `agent.ts:230-252`). O @dev implementou o tier-derived em `rate-limiter.ts` lendo `households.plan` e mapeando para o limite via constante. A escolha dos valores literais (100/500/2000/10000) **reflecte literalmente Architecture §4.6** (Cost Router + Quotas) e o pricing PT-PT do PRD (Família €8,88). A migração para valores configuráveis (env vars ou DB row) fica deferida para Story 2.9.

A escolha entre re-validar directamente vs delegar é uma optimização de eficiência do gate — delego nesta story porque os números entregues pelo @dev são consistentes com a expansão esperada (53 → 53+11 db = 64 tooling base + 64 web — match exacto com 480 = 416+64) e não há sinal de divergência.

**Causa raiz da decisão (documentada em Debug Incidents 1 + 5 do Dev Agent Record):**

TypeScript em monorepo source-only resolve `@/*` no scope do **package que está a fazer typecheck**, não no scope **do package origem do import**.
