## Sumário executivo

A Story 2.7 é o **primeiro material UI** entregue ao utilizador final no Epic 2 — transforma o pipeline canónico Story 2.6 (`POST /api/agent/prompt`) numa experiência observável: o user abre `/jarvis`, escreve um prompt PT-PT como *"amanhã reunião 15h e paguei 78,70 no supermercado"*, e a resposta do cérebro AI multi-intent torna-se visível como um cartão `PreviewCard` (quando `confidence < 0.70` OU `always_preview=true`) ou um `ResultMessage` (quando executado directamente). É a story que **fecha o gap funcional do Epic 2** identificado por Eurico em sessão 2026-05-09 — antes de 2.7, o cérebro AI vivia apenas em endpoints e testes; depois de 2.7, vive na UI real. Adicionalmente, materializa **FR4 user-pref override** (`always_preview`) que tinha cobertura literal apenas no DP2 do EPIC mas zero materialização no codebase — fecha-se com migration 0007 + RLS append em 0001 + endpoint `/api/conta/preferencias` GET/PATCH (lazy-init UPSERT idempotente D32) + UI toggle em `/conta/preferencias`.

A decisão arquitectural mais sensível desta story foi **D28 — scope-shift M → L cross-domain**. O EPIC plan original tratava 2.7 como story backend M (schema + endpoint logic apenas), com a UI deferida implicitamente para Epic 5. O @sm/Orion e Eurico decidiram em sessão (com 4-option AskUserQuestion) expandir para L incluindo `/jarvis` chat UI + `/conta/preferencias` toggle — porque sem UI o cérebro AI não é demonstrável fora de testes, e a UI principal não estava endereçada por nenhuma das 11 stories do Epic 2. **Ratifico D28 como PASS** com a observação policy importante que segue: **stories cross-domain backend+frontend numa única L são aceitáveis quando (a) a fronteira é técnicamente atómica (mesma migration suporta backend + frontend), (b) o scope de UI é minimal e não requer design system extraction, (c) os testes cobrem ambas as camadas no mesmo CI run sem novo setup de infra**. As 3 condições verificam-se em 2.7. Para **Story 2.8 (Undo UI)** + futuras Epic 5 design system stories, a policy é diferente — ver `Lessons for Story 2.8+` abaixo.

A decisão **D31 — UI components inline `_components/`** é arquitecturalmente correcta para o estado actual (~3 componentes únicos consumidos por 2 páginas), mas **define um precedente que precisa de threshold explícito**. Quando Story 2.8 implementar Undo UI, vai precisar de **ResultMessage** (já criado em 2.7) e provavelmente de um novo **UndoConfirmCard** semelhante ao **PreviewCard**. Há 3 caminhos possíveis: (1) duplicar inline em `app/(app)/jarvis/_components/` (KISS) — viável mas começa a ferir DRY; (2) extrair para `apps/web/src/components/` e refactor 2.7 — ~30min mecânico; (3) criar um package interno `@meu-jarvis/ui` (heavyweight, prematuro). **Recomendo policy explícita:** *trigger de extracção é 3+ componentes shared OR 1+ componente shared cross-feature directory (ex: `/jarvis` + `/conta` partilham `PrefsToggle`-style)*. Se Story 2.8 triggar a condição, fazer a extracção como sub-task M dentro da própria 2.8 (não criar story dedicada — overhead SDC desnecessário). Se Epic 5 explicitamente engaja `@ux-design-expert` para design system, então é o ponto natural de extracção formal.

A decisão **D29 — `user_prefs` schema 1:1 user (não composite PK user×household)** é arquitecturalmente correcta. A preferência cognitiva `always_preview` (UX choice — "quero ver tudo antes de gravar") é **natural a um user**, não a um membership. Multi-household users (edge case Família+Pessoal) partilham a mesma escolha mental. O composite PK introduziria complexidade sem benefício real para o MVP — `user_prefs` queries sempre filtram por `user_id` directo (PK lookup O(1)). Se o edge case multi-household-com-prefs-diferentes vier a materializar-se via feedback real, a evolução é mecânica: adicionar `(user_id, household_id)` composite na nova migration + backfill com `household_id = first_household_of(user)`. Aceito como PASS com nota DP futura registada no story Dev Notes.

A decisão **D30 — `run_id`-based confirm (sem HMAC adicional)** é a continuação natural de Story 2.6 D21+DOC-FIX-001. O EPIC plan original (escrito antes de 2.6) sugeria HMAC `previewToken` — proposta superseded pela nested REST + RLS ownership + status check + TTL. O endpoint `POST /api/agent/prompt/[runId]/confirm` é canónico desde 2.6 e Story 2.7 consome-o directamente sem introduzir token adicional. **Defense-in-depth equivalente sem secret novo a gerir.** PASS pure.

A decisão **D32 — lazy-init UPSERT em GET `/api/conta/preferencias`** é uma escolha pragmática excelente. Alternativas consideradas: (a) seed migration sobre `auth.users` existentes (slow, race conditions com novos signups), (b) trigger `on_auth_user_created` (precedente Story 1.3 para `households` — funciona mas adiciona moving piece), (c) lazy-init no endpoint (escolhido). A escolha (c) é **idempotente por construção** (`INSERT … ON CONFLICT (user_id) DO NOTHING`), zero overhead na maioria dos requests (1 INSERT no primeiro acesso por user, depois SELECT puro), e auto-contida em `apps/web/src/app/api/conta/preferencias/route.ts`. **PATCH faz UPSERT também** com `ON CONFLICT (user_id) DO UPDATE SET always_preview = EXCLUDED.always_preview, updated_at = now()` — cobre o caso edge "user nunca abriu /conta/preferencias mas activou via API directa". PASS pure.

A decisão crítica **D34 — testes UI Vitest+Testing Library+jsdom** (sem Playwright) é arquitecturalmente correcta para 2.7. `vitest.config.ts` já tinha `environment: 'jsdom'` desde Story 1.5; `@testing-library/react ^16.0.1` + `@testing-library/jest-dom` já em `apps/web/package.json`. Adicionar Playwright introduziria dependência nova + CI infra (browsers headless) + flakiness em runners GitHub Actions — overhead injustificado para o scope desta story. Os 38 testes UI mockable-only com `vi.mock('@/lib/agent/db-shim')` + `global.fetch = vi.fn()` cobrem cenários determinísticos (render initial, submit happy paths preview/executed, error states 401/429/5xx, countdown timer, badges colour-coded, optimistic UI revert on error). **Story 2.10 (E2E benchmark com credentials reais) é o ponto natural de adoptar Playwright** — já estava previsto para benchmark E2E. PASS pure.

A decisão crítica **D36 — audit log INSERT removido em PATCH `/api/conta/preferencias`** merece análise dedicada. O Dev encontrou em runtime que `audit_action` enum em `packages/db/src/schema/audit.ts:30-50` não inclui valor `user_prefs.updated`. Adicionar exigia migration nova `0008_audit_action_user_prefs` (extensão do enum Postgres + drizzle schema regen). **Decisão correcta no momento:** scope creep injustificado em 2.7 (a) o INSERT real é facilmente diferível porque NFR16 é progressivo (audit log é defesa contra ataques + GDPR access requests, não dependência funcional do toggle), (b) a substituição via Pino structured logger (`{action: 'user_prefs.updated', user_hash, always_preview}`) cumpre a propriedade observável NFR16 de "we know it changed" — vai para Sentry breadcrumb (Story 1.7 instrumentation) onde fica acessível via SIEM. **Mas isto introduz dívida técnica explícita que o gate ratifica como `NIT-001-NB` a fechar em Story 2.8 (já requer enum extension para `agent_run_reverted` ou similar) ou story follow-up curto M dedicado.** A escolha entre piggyback-2.8 vs follow-up-dedicado depende de se Story 2.8 também precisa de enum extension — se sim, é eficiente combinar; se não, follow-up M independente é mais limpo. **Architect recomendação: piggyback Story 2.8 se Undo precisar de novos enum values; senão follow-up M independente em sprint pós-2.8.**

A decisão **D37 — `0001_rls_policies.sql` re-apply em prod** é a decisão operacional mais sensível desta gate. O `pnpm db:migrate` runner em `packages/db/src/scripts/apply-migrations.ts` regista migrations por filename em `__schema_migrations`. Como `0001_rls_policies.sql` já estava registado em prod desde Story 1.3 bootstrap, **o DO block novo de `user_prefs` policies NÃO foi aplicado em DBs existentes via este migration run**. Para CI containers efémeros + dev fresh: zero impacto (todas as migrations correm em ordem do zero). Para prod: necessário ou (a) re-execute manual de 0001 (operacionalmente arriscado se outras tabelas tiverem drifted), ou (b) migration follow-up `0008_user_prefs_rls.sql` defensiva idempotente (mesmo DO block, CREATE-IF-NOT-EXISTS pattern). **Decido (b) como caminho preferido** — é defensável por: (1) idempotência por construção (DO block + `if exists` + `drop policy if exists` + `execute create policy`), (2) auditável no migration history (vs manual operação fora de CI), (3) zero risco de drift em outras tabelas (apenas toca `user_prefs`), (4) replicável em qualquer ambiente (staging, prod-EU, prod-rollback). **NIT-002-NB:** criar `packages/db/migrations/0008_user_prefs_rls.sql` antes de Story 2.7 chegar a prod EU. Pode ser feito como sub-task de @devops no push (não bloqueia commit) ou story follow-up curto S.

A decisão **D35 — RLS coverage final 28/120** é cosmética. Verifiquei independentemente neste gate que `pnpm check:rls` reporta `Tabelas com household_id encontradas: 28` + `Policies existentes em 0001_rls_policies.sql: 120` + `✅ Todas as tabelas multi-tenant têm coverage completa (SELECT/INSERT/UPDATE/DELETE)` + `✓ user_prefs (prefs.ts)` na lista de tabelas verificadas. O número 120 (vs 112 esperado pelo handoff PO) reflecte heurística do parser regex em `scripts/check-rls-coverage.ts` que conta linhas `EXECUTE` strings dentro do DO block como ocorrências separadas (DO block para `user_prefs` tem 8 linhas EXECUTE: 2 ALTER + 4 DROP + 4 EXECUTE CREATE POLICY, vs as 4 linhas literais `create policy` que o parser conta para tabelas DDL-inline). **Gate substantive PASS** porque todas as 28 tabelas multi-tenant têm coverage SELECT/INSERT/UPDATE/DELETE — a função do gate NFR5 é detectar ausência de coverage, não contar policies precisamente. **NIT-003-NB:** patch opcional ao parser para normalizar counting dentro de DO blocks (low-priority polish — story follow-up Epic 2.x housekeeping).

A decisão **D33 — skip CodeRabbit pre-commit** é consistente com 5 stories Done predecessoras (D12 da 2.4, D14 da 2.5, D22+D26 da 2.6, D33 da 2.7). Architect gate exhaustivo (~24 ficheiros: 16 novos + 8 modificados + DOC-FIX-002 architecture+db-schema) substitui review automatizado. Recomendação para Story 2.10 (E2E benchmark com credentials reais) mantém-se: executar CodeRabbit pre-commit por se aproximar mais do focus-area onde brilha (env handling, secrets, credenciais reais). PASS.

### D28 ratification + future policy — Scope-shift M → L cross-domain

**Pergunta arquitectural:** A escolha de expandir Story 2.7 de M (backend toggle) para L (backend + UI cross-domain numa única story) é arquitecturalmente correcta? E qual deve ser a policy para futuras stories cross-domain?

**Análise do trade-off:**

| Critério | Single L story (escolhida 2.7) | Split 2.7-backend (M) + 2.7-UI (M) |
|----------|----------------------------------|--------------------------------------|
| **Atomicidade técnica** | ✓ migration 0007 + endpoint + UI no mesmo PR — fácil de rever | ✗ split em 2 PRs = handoff overhead |
| **Overhead SDC** | 1× ciclo @sm→@po→@dev→@architect→@devops | 2× ciclos = ~2× tempo wall-clock |
| **Risco de scope creep** | ~ presente mas mitigável (sub-tasks claros + rules-from-precedent) | ✗ split não elimina, só distribui |
| **Test coverage cross-domain** | ✓ mesma CI run cobre backend + frontend | ✗ split duplica setup CI |
| **Design system requirements** | UI minimal (3 componentes únicos) — sem extracção necessária | — |
| **Risco se UI requer design system extraction** | scope explode XL — contention point | menor risco se UI é story dedicada |

**Verdict arquitectural:** **Aceitável para 2.7 porque as 3 condições verificam-se:**
1. **Fronteira atomicamente técnica:** migration 0007 SUPORTA backend (RLS) + frontend (toggle UI). Schema → endpoint → UI é cadeia natural sem ramificações.

3. **Testes mesma CI run:** Vitest+jsdom já configurado desde Story 1.5. Zero novo setup de infra (sem Playwright — D34).

**Future policy explícita:**

| Cenário Story 2.8+ | Recomendação |
|--------------------|--------------|
| Story L cross-domain mas UI minimal e sem design system extraction | **Single story L** (precedent 2.7) |
| Story L cross-domain mas UI requer 3+ shared components OR cross-feature directory sharing | **Split em 2 stories M+M** OR **Single L com sub-task explícita de extracção dentro da story** |
| Story L cross-domain + Epic 5 design system trigger condition activado | **Single story M (backend) + Epic 5 story (UI)** — separação por equipa especializada (`@dev` backend + `@ux-design-expert` UI) |
| Story L com novo subsystem (ex: novo package interno) | **Sempre split** — packages novos requerem story dedicada |

### D37 production deploy approach — Migration 0008 defensiva vs manual re-execute

**Pergunta arquitectural:** Como aplicar o DO block de `user_prefs` policies em prod EU dado que `__schema_migrations` registou `0001_rls_policies.sql` por filename desde Story 1.3?

**Análise das opções:**

| Opção | Pros | Contras |
|-------|------|---------|
| **(a) Manual re-execute de 0001 em prod** | direct, sem migration nova | risco operacional (drift em outras tabelas pode estourar), não auditável no migration history, requer DBA intervention manual |
| **(b) Migration `0008_user_prefs_rls.sql` defensiva (idempotente)** | auditável no migration history, idempotência por construção (DO block + `if exists` + `drop if exists` + `execute create policy`), zero risco drift outras tabelas, replicável em qualquer ambiente | scope creep mínimo (1 file novo, ~50 linhas — copy do DO block existente em 0001) |
| **(c) Refactor `apply-migrations.ts` para registar checksum em vez de filename** | resolve fundamentalmente o problema | scope creep enorme (refactor runner core + risco de re-aplicar TODAS migrations em prod), fora do scope 2.7 |

**Decisão arquitectural:** **Opção (b) — migration 0008 defensiva preferida.**

**Justificação:**
1. **Idempotência por construção:** DO block + `if exists (select 1 from pg_tables where tablename = 'user_prefs')` + `drop policy if exists` + `execute create policy` — pode correr múltiplas vezes sem erro.
2. **Auditabilidade:** entry em `__schema_migrations` linkable ao Story 2.7.
3. **Replicabilidade:** mesmo migration em CI/dev/staging/prod-EU — zero ambient drift.
4. **Pode ser feito como sub-task @devops no push** sem bloquear o commit do gate (DOC-FIX-002 + housekeeping). @devops cria `0008_user_prefs_rls.sql` antes de `pnpm db:migrate` em prod.

**NIT-002-NB:** criar `packages/db/migrations/0008_user_prefs_rls.sql` antes de Story 2.7 chegar a prod EU. Conteúdo: copy do DO block `$rls_user_prefs$` de `0001_rls_policies.sql:709-754`. Sub-task @devops no push OR story follow-up curto S.

| Story L com novo subsystem (ex: novo package interno) | **Sempre split** |
