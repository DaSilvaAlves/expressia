A Story 2.5 entrega exactamente o **Estágio 2+3 da pipeline AI multi-intent** prometido pela arquitectura §4.1 + §4.3: **um package source-only `@meu-jarvis/planner-executor` que expõe duas classes injectáveis (`Planner`, `Executor`) com fronteiras tipadas estáveis para a Story 2.6 (endpoint `/api/agent/prompt`) e Story 2.10 (benchmark)**. O Executor é um **thin wrapper** sobre `executeAtomic` da 2.3 — sem recriar transaction wrapper, sem recriar RLS enforcement, sem recriar `agent_reverse_ops` persistence. A disciplina "Foundation, not behaviour" estabelecida pelas Stories 2.2/2.3/2.4 é aqui rigorosamente respeitada: zero wiring com endpoint produção (fica para 2.6), zero benchmark com prompts reais (fica para 2.10), zero modificação a `agent_runs` schema (fica para 2.6). A decisão arquitectural mais sensível desta story (D16 — refactor mecânico `@/*` → `./` em **packages Done** `@meu-jarvis/tools` e `@meu-jarvis/classifier`) está **bem fundamentada e zero-comportamental**, mas merece análise dedicada porque **toca código produção em scope expandido** vs as decisões análogas das stories anteriores (D9-2.2/2.4 modificavam apenas barrels via re-exports). O diff (`git diff acb949f^ acb949f -- packages/tools/src/atomic.ts`) revela exclusivamente mudanças sintácticas no scope `import` (substituição literal de prefix path alias `@/` por `./` em 9 imports tools + 15 imports classifier — 6 ficheiros total). **Nenhum identifier renomeado, nenhuma assinatura alterada, nenhuma semântica de execução modificada**. A causa raiz é fundamentada e foi descoberta pelo @dev em iteração 1 do typecheck (documentada em Debug Log References): TypeScript em monorepo source-only resolve `@/*` no scope do package que está a fazer typecheck — quando `@meu-jarvis/planner-executor` segue um import até `packages/tools/src/atomic.ts` e este faz `import { ... } from '@/contracts'`, o TS tenta resolver `@/contracts` no scope **do planner-executor** (que aponta para `packages/planner-executor/src/contracts.ts` — inexistente) e não no scope **do tools** (onde resolveria correctamente para `packages/tools/src/contracts.ts`). Sem `composite: true` + project references no `tsconfig` (refactoring extensivo de toda a build chain Turbo), **a única solução robusta é evitar `@/*` cross-internal em source files cross-package**. O pattern `packages/agent/src/*` já usa relativos `./*` desde Story 2.2 — o D16 estende esta convenção a `tools` e `classifier`, garantindo consistência cross-package. Tests internos (que executam isoladamente) mantêm `@/*` porque resolvem local — verificável em `packages/tools/src/__tests__/*.ts` e `packages/classifier/src/__tests__/*.ts` que continuam a usar `@/...`.

Justificação: (1) **a refactoring é mecânica e auditável** — 1 commit (`acb949f`) contém todas as mudanças de imports + adição do package novo; reverter implicaria fragmentar a história desta story em 2 entregas, complicando audit trail; (2) **o pattern já existia em `packages/agent/src/*` desde Story 2.2** — D16 não inventa um novo padrão, apenas estende um existente para coerência; (3) **a alternativa "open story dedicada" criaria débito técnico** — story 2.5 dependeria de uma story refactoring para conseguir typecheck, bloqueando o caminho crítico Epic 2 sem benefício prático; (4) **372 linhas de código produção tocadas (`git diff acb949f^ acb949f --stat`) cabem confortavelmente no audit deste gate** — não há volume suficiente para justificar fragmentar. O custo do D16 é trivial (find/replace mecânico) e o benefício é alto (desbloqueia Story 2.5 + qualquer story futura que importe de `tools`/`classifier`).

A decisão D9-anthropic-applied (Task 2 — re-export `AnthropicClientLike` em `packages/agent/src/index.ts:20`, 1 linha) é **idêntica em forma e função** ao D9 da Story 2.4 (`OpenAIClientLike` re-export). É o pattern correcto de evolução incremental do `@meu-jarvis/agent` à medida que mais consumers (Stories 2.4/2.5/2.6) precisam de constructs internos legítimos.

A decisão D5 (`toolCalls.max(10)` anti-hallucination guardrail) é **defensável como expansão linear D8 da 2.4** (max 5 intents × 2-3 tools/intent = 10-15) e tem **fonte explícita correcta** após o PO_FIX_INLINE da v1.1: o cap de 5 intents vem da Story 2.4 D8 (já em produção, gate APPROVED 9.4/10), e o cap de 10 desta story deriva por expansão. Avaliei 3 alternativas:
- **Manter max(10)** (recomendado): equilíbrio entre cobrir cenários multi-intent realistas (`criar_parcelada` legítima pode expandir a 3 tool calls) e limite duro contra hallucination (LLM gerar arrays de 20+ tool calls).
- **max(5)**: alinharia exactamente com 2.4 D8 mas rejeitaria cenários `criar_parcelada` legítimos (ex: 1 intent → 3 tool calls).
- **max(15)**: máximo teórico aceitando 5 intents × 3 tools cada — mais permissivo, sem evidência empírica de necessidade.

**Verdict arquitectural:** PASS para max(10) + nota para Story 2.11 telemetry validation. Test `schemas.test.ts:9` valida que 11 tool calls falham — evidência directa do guardrail. Se telemetria 2.11 mostrar p99 < 6 em produção, max(10) é over-cautious mas não-prejudicial. Se mostrar saturação ao teto, escalar para max(15) com bumping de `PLANNER_SYSTEM_PROMPT_VERSION` (snapshot test apanha — embora ver NIT-001 abaixo sobre esta protecção).

A decisão D6 (`TOOL_TO_INTENT_MAP` declarativo estático) é **arquitecturalmente correcta**: determinismo + testabilidade + zero custo LLM extra + fallback `'unknown'` para tools fora do MAP. Cobertura: `contract.test.ts` valida que cada uma das 8 intents do `IntentSchema` (Story 2.4 AC2) tem ≥1 tool name mapeado. A alternativa rejeitada (LLM anota intent no schema da tool) duplicaria informação já estática e introduziria custo de tokens output extra sem benefício real. Para tools que possam ser introduzidas em stories futuras sem mapping, a fallback `'unknown'` produz degradação graceful — **não bloqueia** o Executor (que não usa `intent` para validação semântica).

A decisão D8 (defense-in-depth tool name validation no Executor mesmo já validada no Planner) é **canónica para código transaccional**. Custo: 1 lookup Map O(1) por tool call × max 10 = trivial. Benefício: (1) early-fail antes de transacção poupa Postgres round-trip; (2) o Executor pode receber `PlanResult` de outras origens em stories futuras (preview-then-confirm 2.7, undo redo 2.8) onde a validação Planner não foi exercitada; (3) defense-in-depth é padrão obrigatório em código que abre transacções. Verificável em `executor.test.ts:91-130` que cobrem 2 cenários (tool inválido SEM abrir tx + mistura válido/inválido SEM abrir tx). Os logs Pino capturados durante `pnpm test` confirmam o comportamento fail-rapid: `{"level":40,"toolName":"tool_inexistente","msg":"Executor defense-in-depth: tool name não encontrada no registry"}`.

A decisão D11 (`cacheControl: 'ephemeral'` default não opt-in) é **literal Architecture §4.3** (linha 367): *"Anthropic prompt caching: o system prompt + tool definitions JSON (~3-5k tokens) são marcados com `cache_control: ephemeral`. Cache hit reduz input cost em ~90% após o primeiro request por cada 5min de inactividade."*. Em regime estacionário, o custo médio do Planner é dominado por output tokens — fazer cache opt-in seria deixar dinheiro na mesa em produção. O override `cacheControl: null` continua disponível para testes/debug, e o teste `planner.test.ts:258-268` exercita explicitamente este override. Vale notar: durante implementação, o @dev detectou e fixed um **bug de design subtle no constructor**: `this.cacheControl = opts.cacheControl ?? 'ephemeral'` falharia silenciosamente porque `null ?? 'ephemeral' === 'ephemeral'` (null é nullish para `??`), tornando o override impossível. Fix: comparação explícita `opts.cacheControl === undefined ? 'ephemeral' : opts.cacheControl` (preserva `null` explícito). Este é um bom exemplo de **TDD revelar um defeito antes de chegar a produção** — o teste `cacheControl=null desliga cache` partiu na primeira execução, levou ao debug, e ficou como regression test permanente. Atribuição: caso clássico onde tests permitiram apanhar o que a leitura visual do código não apanhou.

A decisão D13 (Executor delega `ToolError` da 2.3 — não cria hierarquia paralela) é **arquitecturalmente correcta** e consistente com o princípio "thin wrapper" estabelecido na story. A única excepção criada (`ExecutorValidationError` para input parsing antes de chegar a `executeAtomic`) é semanticamente análoga a `PlannerValidationError` e fica agrupada no mesmo `errors.ts` por coesão de package. Reinventar `ExecutorExecutionError` / `ExecutorTransactionError` etc. duplicaria a hierarquia da 2.3 e quebraria error mapping no endpoint da Story 2.6 (que vai mapear `ToolError` → HTTP response com `userMessage` PT-PT — ver Story 2.6 plan).

A decisão D14 (skip CodeRabbit pre-commit) é **arquitecturalmente defensável** dado que: (1) Stories 2.2/2.3/2.4 estabeleceram o precedente; (2) este gate manual examinou directamente os 21 ficheiros novos + 8 modificados; (3) os 5 quality gates obrigatórios foram **re-verificados independentemente neste gate**; (4) as decisões arquitecturais críticas — tool calling integration, atomicity delegation, defense-in-depth, cache strategy, error taxonomy — estão tipicamente FORA do escopo automated do CodeRabbit. **Aceito o skip** com nota: para Story 2.6 (endpoint `/api/agent/prompt` — toca handlers HTTP, autenticação JWT, response formatting, undo_token HMAC) recomendo CodeRabbit pre-commit por se aproximar mais dos focus-areas onde brilha (input validation, auth, secrets handling).

### D16 — Refactor `@/*` → `./` em packages **Done** `tools` (4 ficheiros) + `classifier` (2 ficheiros)

**Pergunta arquitectural:** Modificar **6 ficheiros source files de packages Done** (Story 2.3 e 2.4) para refactoring de imports é aceitável dentro do scope de uma story de feature?

**Causa raiz da decisão (documentada em Debug Log References da story v1.2):**

TypeScript em monorepo source-only resolve `@/*` no scope do **package que está a fazer typecheck**, não no scope **do package origem do import**. Quando `@meu-jarvis/planner-executor` typecheck segue um import até `packages/tools/src/atomic.ts` e este ficheiro faz `import { ... } from '@/contracts'`, o TypeScript tenta resolver `@/contracts` no scope do `planner-executor` (que aponta para `packages/planner-executor/src/contracts.ts` — ficheiro **inexistente**) em vez de no scope do `tools` (onde resolveria correctamente para `packages/tools/src/contracts.ts`). Sem `composite: true` + project references no `tsconfig.json` (refactoring extensivo de toda a build chain Turbo + scripts `tsc --build` em vez de `tsc --noEmit` + coordenação com turborepo task dependencies), a única solução robusta é evitar `@/*` cross-internal em source files que possam ser seguidos por outro package.

**Pattern já existia:** `packages/agent/src/*` usa relativos `./*` desde Story 2.2 — D16 estende este padrão para coerência cross-package. Uma vez que o monorepo cresça (Stories 2.6/2.7/2.8 vão importar de `tools`, `classifier`, `planner-executor`), aplicar a convenção de relativos em source files cross-package é o caminho técnico canónico — alternativa significaria configurar project references mais tarde com mais código a refactorizar.

**Tests internos preservam `@/*`:** verificável em `packages/tools/src/__tests__/*.ts` e `packages/classifier/src/__tests__/*.ts` — testes resolvem `@/*` localmente porque o `vitest.config.ts` de cada package configura o alias para o seu próprio `src/`. **Esta assimetria source/test é intencional e aceitável** — testes não são seguidos por outros packages, então `@/*` é safe interno.

**Alternativa avaliada e rejeitada (reverter D16 + abrir story dedicada):**

| Critério | Aplicar inline (recomendado) | Abrir story dedicada |
|----------|------------------------------|---------------------|
| **Audit trail** | 1 commit unificado feat planner-executor + refactor | 2 stories fragmentadas, dependência entre elas |
| **Caminho crítico Epic 2** | Story 2.5 desbloqueada imediatamente | Story 2.5 bloqueada até story refactoring fechar |
| **Risco** | Zero (testes confirmam zero-comportamental) | Risco de drift entre as 2 entregas |
| **Custo @architect gate review** | Análise dedicada nesta secção (~10 min) | 2 architect gates separados |
| **Pattern já existia** | Sim, em `packages/agent/src/*` desde 2.2 | — |

**Verdict:** **PASS**. Aceitar D16 inline. A refactoring é defensável arquitecturalmente, os 363 testes existentes confirmam zero-comportamentalidade, e a alternativa (story dedicada) cria mais débito técnico do que resolve.

**Directive para futuras stories:** quando criar um package novo que importa de outro package interno do monorepo, **evitar `@/*` em ficheiros source files cross-package**. Tests podem manter `@/*` (resolução local). Este pattern fica documentado como convenção implícita do monorepo até o `composite: true` + project references serem implementados (provável Story de tooling se o monorepo escalar a 15+ packages).
