Não há try/catch silencioso. |
| **D4** | `audit_log` scrita NFR9 | PASS | Trigger escreve `audit_log` entry completo (`packages/db/migrations/0003_auth_user_trigger.sql:100-120`) com `household_id`, `user_id`, `action='household_created'`, `entity_table='households'`, `entity_id`, `before_state=null`, `after_state` jsonb com snapshot completo (household_id, household_name, owner_user_id, plan, trial_ends_at, currency, locale, timezone, created_via). Validado por `handle-new-user.trigger.test.ts:122-152` que verifica todos os 8 campos do snapshot via `toMatchObject`. |
| **D8** | Email confirmation Dashboard ON vs código diz OFF | CONCERN (não-bloqueante) | Discrepância documental confirmada. `apps/web/src/app/(auth)/actions.ts:69-71,97-98` afirma `"D8 (decidido): email confirmation OFF para MVP — o utilizador entra directamente após signup"` e faz `redirect('/visao')` imediatamente após `supabase.auth.signUp()`. Mas o Supabase Dashboard tem Email Confirmations ENABLED (provado pelo facto do Eurico ter recebido email de confirmação durante o smoke test). **Impacto real:** o `redirect('/visao')` post-signUp atinge a rota protegida sem sessão válida (a sessão só existe após confirmação + login), pelo que o middleware faz redirect para `/entrar?next=/visao`. Comportamento aceitável mas inelegante: utilizador vê uma piscadela curta antes da página `/entrar`. Não há erro funcional. **Recomendação:** alinhar como follow-up Epic 6 (custom email PT-PT via Resend integrar), idealmente com (a) actualizar comentário em `actions.ts:69-71` para reflectir realidade Dashboard, ou (b) trocar `redirect('/visao')` por `redirect('/entrar?registo=ok')` com mensagem "Verifica o teu email para confirmares a conta". |
| **smoke_test_manual_unico** | Smoke test apenas 1 utilizador real | CONCERN (não-bloqueante) | Smoke test manual cobriu happy path (1 user real, email Gmail). Não foram testados os seguintes cenários: signup com email já registado, password fraca a passar a validação client-side mas falhar no Supabase, recovery flow completo (`resetPasswordAction` envia email mas a UX de "definir nova palavra-passe" é Epic 6), logout idempotência (logoutAction → middleware → redirect chain), sessão expirada com refresh token revogado, entrada via `next` query param após middleware redirect. **Recomendação:** criar story Epic 6 para suite Playwright cobrindo estes 6 cenários. Para Story 1.5 esta cobertura é aceitável dado que o objectivo da story é integração funcional, não UX completa (a story foi explícita em Task 5 "Esta story cria apenas scaffolding funcional — UX completa é Epic 6"). |
