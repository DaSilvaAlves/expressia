## 1. DPs HIGH — ratify explícito

### DP-3.4.4 — Threshold virtualização (50 vs 100 vs sempre)

**Recomendação Uma:** A — 50 tasks/coluna (naive-first sem virtualização + footer "Ver mais N →" implícito acima do threshold; token `--kanban-column-min-tasks-virtualize: 50` configurável).

**Verdict:** **APPROVED HIGH confidence** — Opção A é a escolha tecnicamente correcta para MVP Fase 1.

**Razões:**

1. **Realidade demográfica MVP Fase 1.** Famílias 2-6 utilizadores com horizonte tarefas 90 dias (PRD CON5). Mesmo cenário pessimista (3 utilizadores × ~10 tasks/semana × 4 semanas = 120 tasks total spread across 3-6 colunas) raramente excede 40 tasks numa única coluna. Threshold 50 dá folga ~25%.

2. **Virtualização adiciona complexidade real ao drag-and-drop.** `@dnd-kit` suporta virtualization via `SortableContext strategy={verticalListSortingStrategy}` mas requer registo manual de items off-DOM. Cards fora do viewport não respondem a `over` events sem custom collision logic. Trade-off complexity/benefit não justifica para MVP.

3. **`React.memo(KanbanCard, deepEqual)` + `content-visibility: auto` cobrem 80% do ganho** (pre-flight §7.3). React não re-renderiza cards estáveis durante drag; browser saltar paint/layout em colunas off-viewport. Estes são free-lunch — virtualização vem só se React.memo + content-visibility falharem.

4. **Token configurável dá kill-switch barato.** Se pós-launch coluna real ultrapassa 50 + perf degrada, basta subir token para 200 (ou ligar virtualização) sem refactor. Decisão reversível.

5. **Paginação implícita "Ver mais N →" preserva semântica.** Acima de 50 cards, footer link substitui virtualização — utilizadores ainda conseguem aceder a todas as tarefas, performance permanece linear no que mostra. Alinha FR9 (vista Kanban) + UX pattern Trello free-tier (limite implícito 100 cards/coluna).

**Guidance notes (G1):**

| ID | Note | Why |
|----|------|-----|
| **G1.1** | `React.memo(KanbanCard, deepEqual)` deve usar comparador **shallow não-deep** para evitar custo cumulativo. Props devem ser primitivas/refs estáveis (taskId, columnId, position, dueDate ISO string, priority enum, isOverdue boolean). | Deep equal em 50+ cards comparando objectos complexos pode comer ganho de memo. Sticking com primitivas em props faz `React.memo` default shallow ser suficiente. |
| **G1.2** | Token `--kanban-column-min-tasks-virtualize: 50` deve estar declarado em `apps/web/src/app/globals.css` (ou módulo CSS Kanban dedicado) **com comentário "configurável — subir se famílias reais excederem"**. | Auto-documenta o kill-switch para futuro maintainer. Token sem comentário é magia. |
| **G1.3** | Footer "Ver mais N →" deve fazer **scroll vertical interno à coluna** (não modal nem navegação) — preserva UX board-as-canvas. Implementação: clica → `column.scrollIntoView({ block: 'end', behavior: 'smooth' })` + revela cards adicionais incrementalmente (chunk size 25). | Modal/navegação quebra fluxo drag-and-drop. Scroll incremental mantém contexto visual. |

**Risk note LOW:** Se telemetria pós-launch mostrar coluna > 50 cards em > 5% households, considerar virtualização real (`@tanstack/react-virtual` recomendado vs `react-window` por compatibilidade @dnd-kit). Não bloqueia 3.4.

### DP-3.4.6 — Batch save column config (single PATCH batch vs múltiplos PATCHs)

**Recomendação Uma:** A — Single PATCH `/api/kanban-columns/batch` com `db.transaction()` rollback-all-on-failure.

**Verdict:** **APPROVED HIGH confidence** — Opção A é a escolha tecnicamente correcta + alinha pattern atómico Story 3.2 `/api/tasks/[id]/move`.

**Razões:**

1. **Invariants pós-batch são multi-row.** `3 ≤ count ≤ 6` + `exactly 1 is_done_column=true` + `names únicos per household` exigem leitura de TODAS as colunas do household antes de COMMIT. Sem transaction, race condition entre browser-side parallel PATCHs viola estes invariants (ex: 2 utilizadores admin movem `is_done_column=true` simultaneamente → 0 ou 2 done columns).

2. **Pattern reusable Story 3.2 já APPROVED 9.3/10.** D-3.2.1 (DRY helpers `requireAuth/resolveHouseholdId/resolveHouseholdRole/insertAuditLog`) + AC4 PATCH `/api/tasks/[id]/move` corre em `db.transaction()` para preservar `unique(household_id, kanban_position)`. Story 3.4 batch endpoint é caso extremo do mesmo padrão — múltiplas linhas vs 1 linha.

3. **`unique(household_id, sort_order)` força transaction obrigatória.** Já confirmado em `tenancy.ts:171` (`uniqueOrder: unique('kanban_columns_unique_order').on(t.householdId, t.sortOrder)`). Reorder de 6 colunas (1,2,3 → 3,2,1) sem transaction tem janela onde 2 colunas têm mesmo `sort_order` → constraint violation. Migração 0011 partial unique `is_done_column=true` adiciona segundo gate constraint que igualmente requer transaction.

4. **Atomicidade UX equivalence.** Utilizador clica "Guardar" e expecta: ou todas as alterações ficam persistidas ou nenhuma (com mensagem clara). Múltiplos PATCHs paralelos podem deixar estado parcial (ex: 2 colunas renomeadas, 1 falhou) que confunde mental model do utilizador. Batch + rollback total = estado consistente sempre.

5. **Audit log unico vs N entries.** `kanban_column.batch_updated` (1 entry com `after_state.changes_summary`) é mais útil para auditoria/debugging do que N entries fragmentadas. NFR9 atende — o que importa é traceability + reproducibility, não granularidade artificial.

**Guidance notes (G2):**

| ID | Note | Why |
|----|------|-----|
| **G2.1** | **Input schema obrigatório (Zod `.strict()`):** `{ columns: [{ id: uuid, sort_order: int, name?: string, is_done_column?: boolean }], creates?: [{ name: string, color?: hex, sort_order?: int }], deletes?: [{ id: uuid, move_to?: uuid }] }`. Validar: (a) `columns[].id` ∈ household, (b) `deletes[].move_to` ∈ household e ≠ `deletes[].id`, (c) `creates[].name` ≠ existing names. | Schema explícito previne payload malicioso (inject coluna doutro household via UUID). `.strict()` rejeita campos extra (defesa em profundidade ataque payload bloat). |
| **G2.2** | **Ordem de operações dentro de `db.transaction()`:** **(1) DELETEs primeiro** (com UPDATE tasks `kanban_column_id = move_to` antes de DELETE), **(2) CREATEs depois**, **(3) UPDATEs por último** (incluindo `sort_order` rebalance). Razão: deletes podem libertar `sort_order` slots; creates podem ocupar new slots; updates finais reposicionam tudo. | Evita constraint violations transitórias (ex: UPDATE 2 colunas para `sort_order=1` mid-transaction). Postgres faz unique check no COMMIT em transactions, mas separação por fase reduz superfície de bug. |
| **G2.3** | **Invariants validation pós-batch (server-side, pré-COMMIT):** SELECT count(*) + is_done_column count + names array → reject 422 UNPROCESSABLE_ENTITY com payload `{ violations: ['count_out_of_range', ...] }` se qualquer falhar. Client recebe lista exacta de problemas para inline display. | App-layer enforcement em cima de DB constraints (migration 0011 cobre count ≤ 6 + partial unique done). Defesa em profundidade NFR5. Mensagens PT-PT no `mapErrorToCopy()`. |
| **G2.4** | **Response shape:** `{ columns: KanbanColumn[] }` — estado final completo do household pós-batch (não diff). Client substitui state inteiro sem race com optimistic update. `audit_log` entry com `after_state.changes_summary = { creates_count, updates_count, deletes_count, renames: [{id, from, to}] }`. | Optimistic update na Sheet pode divergir do estado server (sort_order rebalance). Response com estado completo permite client reconciliar deterministicamente. Summary em audit log dá traceability sem flooding `after_state` com snapshot completo. |

**Risk note MEDIUM:** Endpoint batch é novo — não tem precedent em packages/web. Atenção especial em testes T3.4 e T11.6: cenário "batch rollback em falha parcial" deve usar mock/fake DB para validar rollback é total (não pode confiar só em integration test live com Postgres).

## 2. DPs MEDIUM — APPROVE silent

### DP-3.4.1 — Drag handle (todo o card vs ícone `⋮⋮` explícito)

**Recomendação Uma:** A — Todo o card draggable. **APPROVED silent.**

Trello/Notion/Linear convergiram para card-as-handle. `activationConstraint: { distance: 8 }` (pre-flight §5.1) distingue click de drag com tolerância razoável. Drag handle explícito adiciona ruído visual em densidade Kanban — não justifica.

### DP-3.4.2 — Click no body (EditTaskModal vs inline edit)

**Recomendação Uma:** A — EditTaskModal. **APPROVED silent.**

Conflito comportamental drag + inline edit (mesmo activator pointer). Modal dá superfície maior para edit completo (notes, due_date picker, tag picker, assignee, priority). Alinha pattern Story 3.3 `EditTaskModal` re-usado (DP-3.4 §2.3 pre-flight) — zero código novo.

### DP-3.4.3 — Filter bar (sem vs subset vs completo)

**Recomendação Uma:** B — Subset (search + tag). **APPROVED silent.**

KISS: Kanban visual board é instrumento de **organização**, não de **filtro pesado**. Search + tag cobrem 80% dos casos UX (Trello free-tier idêntico). Filtros pesados (date range, project, assignee) ficam para Lista (Story 3.3 já implementou). Story 3.6 substituirá tag dropdown placeholder por TagPicker real.

### DP-3.4.5 — Haptic feedback mobile (`navigator.vibrate(10)`)

**Recomendação Uma:** A — Sim, com feature-detect. **APPROVED silent.**

Custo: 1 linha. Benefício UX mobile: alto (feedback tátil reforça drag-init). Feature-detect `if (typeof navigator !== 'undefined' && 'vibrate' in navigator) navigator.vibrate(10)` evita TypeError em browsers sem suporte (Safari iOS antigo). Sem implicações privacy/perms.

## 3. DPs LOW — APPROVE silent

### DP-3.4.8 — Mobile coluna (full-width snap forte vs 1.5 vs 2 colunas)

**Recomendação Uma:** A — 1 coluna full-width com snap forte. **APPROVED silent.**

Trello/Asana mobile convergiram para este padrão. 1.5 colunas (peek) gera tap targets ambíguos. 2 colunas a < 768px torna cards minúsculos. Snap forte com `scroll-snap-type: x mandatory` + `scroll-snap-align: start` é UX mobile standard. Alinha R7 mitigation (pre-flight §6).

### DP-3.4.9 — Concluído animation (normal vs check icon + fade strike-through 800ms)

**Recomendação Uma:** B — Check icon + fade strike-through 800ms. **APPROVED silent.**

Micro-interaction delight ratifica completion. Consistente com Lista §5.6 (Story 3.3 TaskCheckbox completou pattern). 800ms é threshold inferior para felt-time (< 500ms = imperceptível, > 1500ms = chato). `prefers-reduced-motion: reduce` deve desactivar animation (fade only sem strike-through delay). Cost: 1 componente animation isolado.
