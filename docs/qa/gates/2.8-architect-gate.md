## Sumário executivo

Story 2.8 **completa o triângulo FR6 + NFR9 + UX trust**: o pipeline canónico Story 2.6 já tinha o endpoint undo implementado (AC7 da 2.6, 280 linhas + 9 tests passing), mas (a) sem trigger de housekeeping (`agent_reverse_ops` cresceria indefinidamente — risco operacional em prod), (b) sem persistência audit do facto da reversão (NFR9 imutabilidade aplicava-se à row de `agent_runs` mas faltava a entrada explícita em `audit_log` que torna o evento queryable em SQL pelo legal team), e (c) sem botão Undo real na UI — o utilizador via "Em breve" no `ResultMessage` da Story 2.7 e tinha de chamar o endpoint manualmente. Story 2.8 fecha os 3 gaps simultaneamente — daí o scope ser legitimamente cross-domain (backend Inngest + audit log + frontend UI) sem violar a policy D28 ratificada na 2.7: as 3 condições continuam a verificar-se (fronteira atomicamente técnica, UI scope minimal sem design system extraction necessária, testes mesma CI run).

A decisão arquitectural mais sensível desta gate é a **ratificação de [DEV-DECISION 3] (DN9 — BEGIN tx + ordem reversa deferred)**. O EPIC plan original da Story 2.8 mencionava explicitamente "BEGIN tx → execute reverse_ops em ordem reversa → COMMIT" como pattern obrigatório para a iteração de reverse_ops no endpoint undo. O endpoint implementado em Story 2.6 (e preservado em 2.8) NÃO usa BEGIN/COMMIT explícito nem ordem reversa — itera sequencialmente com `serviceDb.execute` em loop. **Ratifico a decisão @dev como PASS** com a seguinte análise: o gate 2.6 APPROVED 9.4/10 já cobriu este flow com 9 tests passing sem regressão observada; o pattern actual é **defensavelmente correcto** porque (1) `postgres-js` em modo session aplica auto-commit por statement, (2) cada reverse op aplica DELETE ou UPDATE de uma row específica via PK (não há lock contention cross-row), (3) a whitelist `ALLOWED_REVERSE_TABLES` (linha 256-262 do endpoint) garante que apenas tabelas de domínio MVP (`tasks`, `transactions`, `recurrences`, `card_accounts`, `card_transactions`, `installments`) são alvos — todas independentes entre si, sem FKs mútuas, sem triggers cross-table que tornassem a ordem reversa relevante, (4) idempotência do `UPDATE agent_reverse_ops SET executed_at = now()` por op garante que retries pós-falha parcial não duplicam aplicações. **O cenário onde ordem reversa importaria** seria multi-table com FK chain (op1 cria row A → op2 cria row B referenciando A; reverter deveria apagar B primeiro). Esse cenário **não é alcançável** na arquitectura actual porque cada `reverse_op` é emitido pelo executor da Story 2.5 numa transacção isolada por tool — não há composição cross-tool reentrante. Se isso mudar no futuro (Story Epic 4+ com tools que orquestram dependências), follow-up dedicado é o caminho — não inflacionar 2.8. Registado como **NIT-005-NB low** para Epic 4+ revisita.

A decisão **[DEV-DECISION 1] (PO_FIX 1 RLS choice `getServiceDb()` em `audit_log` INSERT)** é cosmética em termos de impacto operacional mas merece nota arquitectural explícita. O @po na validação v1.1 verificou directamente em `packages/db/migrations/0001_rls_policies.sql:569-573` que a policy `audit_log_insert_member` PERMITE INSERT a `authenticated` members quando `household_id IS NULL OR is_household_member(household_id)` — invertendo a alegação errada da v1.0 ("REVOKED em authenticated per NFR9"). O @dev escolheu `getServiceDb()` defensivamente porque (1) o `serviceDb` handle já está em scope no endpoint (linhas 165-183 fazem UPDATE de `agent_runs` + `agent_reverse_ops` via service_role para bypass do trigger imutabilidade), (2) reutilizar minimiza acoplamento e evita uma chamada extra a `getDb()`. **Comportamento idêntico per realidade da RLS**: ambos atingem o mesmo INSERT — service_role bypassa toda a RLS, authenticated passa pela check `is_household_member` que é trivialmente true porque o endpoint só chega a este ponto após ter resolvido `run.household_id` via lookup com sucesso (a auth gate via `createServerSupabaseClient` + lookup com RLS valida implicitamente que `current_household_id() = run.household_id`). **Ratifico como PASS** — a única diferença é conceitual sobre "quem é o role responsável pelo INSERT". `getDb()` seria semanticamente mais correcto (NFR9 diz "audit_log é écrito pelo user authenticated que originou a acção"), mas a escolha defensiva @dev não viola NFR9 — o INSERT acontece de qualquer forma, com `user_id` correcto preenchido (`user.id` capturado da auth gate, não substituído pelo service_role). **Documentado em comment `[DEV-DECISION]` inline no código (linhas 200-208)** — para futuros readers que se perguntarem "porquê serviceDb num INSERT que authenticated podia fazer?". Sem follow-up necessário.

A decisão **[DEV-DECISION 2] (substituir test 3 placeholder vs paralelo)** é uma escolha de higiene de tests. O @dev escolheu substituir o teste antigo `'placeholder undo button está disabled com tooltip "Em breve"'` (que asserta POSITIVAMENTE `getByRole('button', /anular/i) + toHaveAttribute('title', 'Em breve')`) por um teste novo que asserta `queryByRole('button', /anular/i) === null` quando `undoUrl` é undefined. **Aceito como PASS** — paralelo (manter ambos) introduziria 2 assertions opostas no mesmo describe block sobre o mesmo render condition que confundiria futuro reader. Substituição preserva a propriedade comportamental verificada (D40 retrocompatibilidade — sem `undoUrl`, sem botão Undo) sem o ruído de manter um placeholder obsoleto. Test count final 13 (em vez de 14) é apenas aritmética — net delta +8 vs baseline 5 (4 preservados + 9 novos − 1 substituído).

O **endpoint `/api/inngest` PÚBLICO (não em `APP_PATH_PREFIXES`)** é arquitecturalmente correcto e bem documentado em comment + DN3 da story. O SDK Inngest valida requests via header `X-Inngest-Signature` contra `INNGEST_SIGNING_KEY` (env var) — a auth via Supabase session seria contraproducente (Inngest Cloud chama o endpoint sem cookies do utilizador). Pattern consistente com Vercel Cron, Stripe webhook, Resend webhook (todos públicos com validação por signing key). **PASS pure.**

O **cron schedule `0 3 * * *` (D38)** é arquitecturalmente defensável como KISS para MVP de 1 job. Architecture ADR-005 (§14.5) admite alternativa via Vercel Cron `/api/cron/daily` que despacha `inngest.send({ name: 'jobs/daily-tick' })` — útil quando há fan-out de múltiplos jobs cronizados (Story 2.9 quota reset + GDPR purge mensal + Stripe retry job). Story 2.8 estabelece o pattern de cron Inngest nativo; refactor para fan-out via Vercel Cron é trivial quando ≥2 jobs Inngest aparecem. **PASS — D38 RATIFIED com nota para Story 2.9+.**

A **escolha `getServiceDb()` para o DELETE do cron job (AC2)** é justificada por job sistémico — pattern análogo a GDPR purge mensal (Architecture §12.4) e ao Story 2.5 audit log inserts. Trigger imutabilidade não se aplica a `agent_reverse_ops` (não é tabela terminal-mutation), mas RLS sim — e a Inngest function não tem JWT context para satisfazer `current_household_id()`. Service_role bypassa toda a RLS, o DELETE cross-tenant é seguro porque o WHERE é puramente temporal (`expires_at < now() - 1h`) — não há risco de purge cross-household acidental (todas as rows têm `household_id` mas o WHERE não filtra por ele intencionalmente — ops expiradas de qualquer household são cleanup-eligible). **PASS.**

A **idempotência do DELETE (AC2 + D43 test (ii))** é provada por construção: WHERE com `expires_at < now() - interval '1 hour'` — rerun no mesmo dia retorna 0 rows porque já não há rows que satisfaçam o predicate (foram apagadas na primeira run). Margem de 1h vs janela undo 30s é defensivamente generosa (10×) — risco zero de apagar uma op ainda dentro da janela activa. Pino structured log com `rows_deleted` count + Inngest auto-span OTel garantem observabilidade. Errors re-thrown ao Inngest engine que aplica retry policy ADR-005 (max 4 attempts com backoff exponencial). **PASS.**

A **propagação `undo_url`/`undo_expires_at` via flow preview→confirm (PO_FIX 2 — `PreviewCardProps.onConfirm` alargado para `ConfirmPayload`)** é a correcção arquitectural mais importante desta story. Story 2.7 v1.0 alegava (e foi efectivamente verificado pelo @po em validação) que `handleConfirmResult` no `jarvis-chat.tsx` parsava a response completa do confirm endpoint — realidade era que **apenas `data.results` era propagado** via `onConfirm(data.results)`, com os campos extra (`undo_url`, `undo_expires_at`) descartados silenciosamente. O endpoint `POST /api/agent/prompt/[runId]/confirm` (route.ts:295-303 verificado pelo @po) **já retorna** os dois campos — gap era puramente client-side: o callback shape `(results: unknown) => void` era restritivo demais. Story 2.8 PO_FIX 2 alarga o callback shape para `(payload: ConfirmPayload) => void` onde `ConfirmPayload = { results: unknown; undoUrl?: string; undoExpiresAt?: string }`. Verificado em `preview-card.tsx:117-121` que `handleConfirm` agora propaga a response completa: `onConfirm({ results: data.results, undoUrl: data.undo_url, undoExpiresAt: data.undo_expires_at })`. E em `jarvis-chat.tsx:172-194` que `handleConfirmResult` consome os 3 campos e passa-os ao `<ResultMessage />` via `appendMessage({ kind: 'result', ..., undoUrl: payload.undoUrl, undoExpiresAt: payload.undoExpiresAt })`. **Resultado funcional:** o flow preview→confirm agora resulta numa `ResultMessage` com botão Undo activo, alinhado com o flow executed-directly (mesma UX). Cobertura tests AC13 (jarvis-chat.test.tsx (i) e (ii)) confirma ambos os branches passam `undoUrl` non-empty à `ResultMessage`. **PASS — correcção arquitectural exemplar baseada em verificação directa do código.**

O **`setInterval(1000)` countdown em `result-message.tsx` com cleanup on unmount (DN7 leak risk)** é implementado correctamente. `useEffect` (linhas 119-129) regista o interval, retorna `() => clearInterval(interval)` cleanup function, dependency array `[hasUndo, status, expiresAtMs]` — re-monta quando o state transita ou o expiry muda. Adicionalmente, o interval auto-limpa-se quando `remaining <= 0` (linha 124-126 — `if (remaining <= 0) clearInterval(interval)`) — defensive depth contra dependency array stale closure. AC12 (i) test usa `vi.useFakeTimers()` + `vi.advanceTimersByTime()` para validar countdown decrementing. AC12 (ix) test valida que 30s sem click transita button para "Expirou" disabled. **PASS — zero leak risk.**

### [DEV-DECISION 3] ratification — DN9 BEGIN tx scope deferred

**Pergunta arquitectural:** A escolha @dev de NÃO implementar BEGIN/COMMIT explicit + ordem reversa no endpoint undo (preservando o pattern Story 2.6 que itera sequencialmente sem transaction wrapper) é arquitecturalmente correcta? E qual é o threshold futuro para revisitar?

**Análise do trade-off:**

| Critério | Pattern actual (preservado Story 2.6) | Refactor BEGIN tx + ordem reversa |
|----------|---------------------------------------|------------------------------------|
| **Atomicidade cross-op** | ✗ (auto-commit por statement — falha no meio deixa ops parciais aplicadas) | ✓ ACID transaction garante all-or-nothing |
| **Idempotência retry** | ✓ (UPDATE `executed_at = now()` por op permite resume) | ~ (rollback inteiro requer re-aplicar tudo do início) |
| **Lock contention** | ~ (postgres-js session mode + auto-commit minimiza locks) | ✗ (BEGIN tx prolonga locks até COMMIT) |
| **Complexidade implementação** | ✓ (loop simples — Story 2.6 implementação 9 tests passing) | ✗ (refactor não-trivial + risk de regression) |
| **Cenário onde importa** | Falha cross-row durante reverse → ops parciais (recuperáveis via re-run undo) | Multi-table FK chain reverse (op2 referencia op1 — ordem reversa obrigatória) |
| **Alcançabilidade do cenário onde importa** | Não no MVP — `ALLOWED_REVERSE_TABLES` lista 6 tabelas independentes sem FKs mútuas; executor 2.5 emite reverse_op por tool em transacção isolada | Improvável até Epic 4+ com tools que orquestram dependências cross-tool |

**Verdict arquitectural:** **PASS — pattern actual defensavelmente correcto** porque as 4 condições verificam-se:
1. **postgres-js auto-commit por statement em session mode** garante visibilidade incremental — falha no meio é detectável e recuperável por retry.
2. **Whitelist `ALLOWED_REVERSE_TABLES`** (verified linha 256-262 do undo/route.ts) restringe alvos a 6 tabelas MVP: `tasks`, `transactions`, `recurrences`, `card_accounts`, `card_transactions`, `installments`. Cada uma tem PK `id` único; nenhuma referencia outra via FK no domínio actual (`transactions.account_id → accounts` mas `accounts` não está na whitelist; `card_transactions.card_account_id → card_accounts` mas reverse op de card_transaction não afecta card_account row).
3. **Idempotência por op** via `UPDATE agent_reverse_ops SET executed_at = now() WHERE id = op.id` (linha 170-174) garante que retry pós-falha parcial salta as ops já aplicadas — não duplica.
4. **Cenário onde ordem reversa importaria** (multi-table FK chain) NÃO é alcançável na arquitectura actual — executor 2.5 emite reverse_op por tool em transacção isolada (Story 2.5 atomicidade), sem composição cross-tool reentrante.

**Threshold futuro para revisitar:**

| Condição que activa revisita | Acção |
|------------------------------|-------|
| Epic 4+ adiciona tool com FK chain (ex: criar `transactions` + `card_transactions` na mesma run — undo deveria reverter card_transactions antes de transactions se RESTRICT FK) | **Refactor obrigatório** para BEGIN tx wrapper + ordem reversa explícita |
| Production observability mostra >5 ops parciais por mês (sintoma de falhas mid-undo) | **Considerar refactor** + Sentry alert |
| Performance shows lock contention >100ms em undo p99 | **Manter pattern actual** — BEGIN tx só prolongaria locks |
| Story 2.x housekeeping toca o endpoint undo | **Oportunidade refactor opportunistic** (mas não obrigatório) |

**Registo:** **NIT-005-NB low** para Epic 4+ revisita conditional. Não bloqueia 2.8 push.

**Pergunta arquitectural:** A escolha de cron Inngest nativo `0 3 * * *` (sem `/api/cron/daily` intermediário) é arquitecturalmente correcta para MVP? E qual é o threshold para evoluir?

| Critério | Cron Inngest nativo (escolhido D38) | Vercel Cron fan-out via `/api/cron/daily` |

| **Observability** | ✓ Inngest dashboard mostra cron schedule + execution history directly | ~ Vercel Cron logs + Inngest dashboard (split observability) |
| **Failure semantics** | ✓ Inngest retry policy ADR-005 (max 4 attempts backoff) automatic | ~ Vercel Cron retry policy diferente + Inngest retry para o job actual |
| **Fan-out múltiplos jobs cronizados** | ✗ Cada job precisa `cron:` próprio (5 jobs = 5 schedules separados no Inngest dashboard) | ✓ 1 Vercel Cron tick dispara N `inngest.send()` em sequência — controle central |
| **Cost (Vercel Cron pricing)** | ✓ Zero impacto Vercel (Inngest dispara internamente) | ~ 1 Vercel Cron execution por dia (negligible $) |
| **Cost (Inngest pricing)** | ~ 1 cron schedule (free tier 30 functions) | ~ N event dispatches (free tier 10K events/day) |

**Verdict arquitectural:** **PASS — RATIFIED com nota Story 2.9+** porque para 1 job actual, cron Inngest nativo é claramente o caminho mais simples. KISS principle alinhado com Architecture ADR-005 (§14.5) que admite ambas as alternativas.

**Threshold futuro para evoluir:**

| Condição que activa refactor para Vercel Cron fan-out | Acção |
|--------------------------------------------------------|-------|
| ≥3 jobs cronizados simultâneos | **Considerar refactor** — observability central via `/api/cron/daily` torna-se útil |
| Jobs que precisam coordination (job B depende de output de job A no mesmo tick) | **Refactor obrigatório** — `/api/cron/daily` orchestra sequence |
| Necessidade de pause/resume global de todos os jobs cronizados | **Refactor obrigatório** — feature flag em `/api/cron/daily` é mais simples que pausar cada Inngest function individualmente |

**Registo:** **Não-bloqueante.** Story 2.9 (cost router quota reset mensal) é o primeiro candidato natural para revisitar — se Story 2.9 adiciona um cron job adicional, sub-task M dentro de 2.9 para refactor `/api/cron/daily`.
