## Sumário executivo

A Story 2.1 entrega exactamente aquilo a que se propõe: **valida o schema agent existente (Story 1.3), formaliza o enforcement de imutabilidade NFR9 via trigger, e adiciona o safety net DEFAULT para o TTL FR6 — sem inventar tabelas, colunas ou padrões.** A migration 0005 é a coisa mais pequena que pode possivelmente funcionar para satisfazer NFR9 + FR6, e isso é uma virtude.

A decisão arquitectural mais escrutinável (D1 — bloquear apenas `authenticated`+`anon` no trigger em vez de "permitir apenas `service_role`") está **correctamente desenhada** e bem-fundamentada:

1. **`current_user` é o role efectivo após `SET ROLE`** — em Supabase, queries de utilizador autenticado correm como `authenticated` (PostgREST faz `SET ROLE` per-request); jobs Inngest correm como `service_role`; migrations correm como `postgres` superuser. O trigger `current_user IN ('authenticated', 'anon')` resolve correctamente em todos os 3 caminhos.
2. **`security invoker` está explicitado** na função (linha 53 da migration) — sem isto, `current_user` resolveria para o owner da função (postgres), bypassando todos os checks. A escolha é deliberada e correcta.
3. **A asimetria "block list" vs "allow list" é defensável** — bloquear `authenticated`+`anon` permite que o migration runner (`apply-migrations.ts`) conecte como `postgres` e aplique data corrections / retention sem ginástica de `set local role`. Aprisionar tudo no `service_role` partia o próprio sistema de migrações e DBA ops.
4. **Defense-in-depth real** — o trigger é layer 2; a policy `agent_runs_update_self` (layer 1) continua a bloquear cross-tenant + user_id mismatch. Nenhuma se sobrepõe ou tornaria a outra redundante.

A naming reality (D5) está coerente em toda a stack: `agent_runs`, `intent_classifications`, `agent_reverse_ops`, `agent_quotas` aparecem identicamente em (a) schema Drizzle, (b) migration 0005, (c) story v1.2, (d) `docs/db-schema.md §4.4`. Article IV (No Invention) preservado.

A correcção do bootstrap dos Testcontainers (D4 — grants explícitos para `service_role`) é **alinhamento legítimo com produção**, não invenção: em Supabase, `service_role` recebe `grant select/insert/update/delete on tables` por default; o test harness só tinha o flag `bypassrls`, daí o `permission denied for table agent_runs` que apareceu ao testar o bypass do trigger. Sem este fix, o teste arquitectural de "service_role bypassa trigger" falhava por motivo errado (grants ausentes), mascarando o que se quer realmente validar.

**Zero issues bloqueantes. 1 nit minor não-bloqueante** registado em CONCERNS (semântica do `agent_quotas_update_blocked` — a story tem invariante "agent_quotas é write-only via service_role", mas tecnicamente é "update-only via service_role", coerente com o INSERT inicial que é feito pelo trigger `handle_new_user` da Story 1.5; doc §4.4 já está correcta — só nota arquitectural para futuro).

## Decisão sobre D1 (trigger bypass) — análise definitiva

### Pergunta arquitectural

O `current_user` resolve correctamente em Supabase deployment, ou pode ser `authenticator` (pgbouncer pool user) em vez de `authenticated` (role efectivo após `SET ROLE`)?

### Resposta

**`current_user` resolve para o role efectivo após `SET ROLE`. A escolha é correcta.**

| Cenário | `session_user` | `current_user` | Trigger fires? |
|---------|---------------|----------------|----------------|
| Web app (PostgREST → Supabase Pooler → query) | `authenticator` | `authenticated` (após `SET ROLE`) | **YES** — bloqueia |
| Inngest job (Stripe webhook, GDPR purge) | `service_role` | `service_role` | NO — passa |
| Migration runner (`apply-migrations.ts`) | `postgres` | `postgres` | NO — passa |
| DBA manual (`psql` via tunnel) | `postgres` | `postgres` | NO — passa |
| Auth Hook (`supabase_auth_admin`) | `supabase_auth_admin` | `supabase_auth_admin` | NO — passa |

A literatura Postgres é clara: `current_user` reflecte **o role efectivo da execução actual**, que em SQL inclui o efeito de `SET ROLE` e `SECURITY DEFINER` functions; `session_user` é o role original do login (não muda com `SET ROLE`). A função `prevent_update_terminal_agent_runs()` é declarada `security invoker`, logo `current_user` reflecte o caller — não o owner da função.

### Verificado no test harness

`packages/db-test/src/rls-harness.ts:320` confirma que `asUser()` faz `set local role authenticated` antes de correr a query — exactamente o flow Supabase. O teste "UPDATE em agent_run com status='success' bloqueado para authenticated" (`agent_runs.rls.test.ts:88-121`) valida que o trigger dispara dentro deste contexto. O teste "UPDATE PERMITIDO via service_role" (linha 171-198) usa `tx.unsafe('set local role service_role')` para simular o flow Inngest — bypass funciona.

### Possível alternativa idiomática Supabase — `pg_has_role`

Uma alternativa seria `pg_has_role(current_user, 'service_role', 'MEMBER')` — mais flexível para herança de roles. **Não foi necessária** porque:

- O design do schema não usa herança de roles (não há `granted` membership entre `authenticated` e `service_role`).
- Postgres `bypassrls` flag é per-role, não herdado — `service_role` tem-no, `authenticated` não.
- A igualdade simples `current_user IN (...)` é semanticamente equivalente E mais fácil de auditar.

A escolha @dev é defensável.

### Escalation paths possíveis

| Caminho | Possível? | Mitigação |
|---------|-----------|-----------|
| `authenticated` → `SET ROLE service_role` mid-query | NO | `service_role` tem `nologin` mas isso não é o bloqueio — o role `authenticated` simplesmente não tem permissão para executar `SET ROLE service_role` em Supabase (o grant não existe). |
| `authenticated` → `SECURITY DEFINER` function que faz UPDATE | Possível em teoria | Não há tal função no schema actual. Adicionar uma seria uma decisão arquitectural nova (e mau cheiro — a única razão para ter `SECURITY DEFINER` neste schema seria contornar este trigger). |
| `authenticated` → CTE com `current_setting('role')` manipulado | NO | `set_config('role', ...)` requer privilégio que `authenticated` não tem em Supabase. |

**Veredicto:** D1 é arquitecturalmente sólido. Sem hole de segurança detectado.

## Decisão sobre D4 (bootstrap grants) — alinhamento com produção

### Pergunta arquitectural

A expansão do bootstrap de testes (`grant ... to service_role`) introduz divergência entre test harness e production runtime?

### Resposta

**Não. É alinhamento, não divergência.**

Em Supabase produção, o role `service_role` é configurado pela própria infrastructure Supabase com:

1. `bypassrls` flag (presente já no bootstrap original).
2. `grant select/insert/update/delete on all tables in schema public` (default privileges).
3. `grant usage on schema public, sequences`.
4. `grant execute on functions`.

O bootstrap original (`bootstrap.ts:42`) só tinha `bypassrls`. Os tests da Story 1.4 funcionavam apesar disso porque nenhum teste tentava INSERT/UPDATE como `service_role` directamente — só usavam `admin()` (postgres superuser, que tem TUDO). A Story 2.1 introduziu o primeiro teste que faz `set local role service_role` para simular o caller Inngest — daí o `permission denied` que o @dev detectou.

A correcção (adicionar os grants no bootstrap) **espelha a configuração default Supabase** que já existe em produção. Não há divergência — pelo contrário, o bootstrap original é que estava em divergência (incompleto).

**Veredicto:** D4 está correcto. Test harness agora é fiel à produção.
