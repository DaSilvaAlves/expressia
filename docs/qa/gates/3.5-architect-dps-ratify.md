## 1. DPs HIGH — ratify explícito

### DP-3.5.1 — Calendar library (Custom CSS grid + @dnd-kit vs react-big-calendar vs FullCalendar vs @schedule-x)

**Recomendação @sm:** A — Custom 7-col CSS grid + @dnd-kit (zero new dep, UX consistente com Kanban Story 3.4, reuse pattern).
**Recomendação @pm Morgan:** A (registada `docs/epics/EPIC-3-EXECUTION.yaml:131-140` DP2 NEW + critical_note sobre `react-day-picker` ser date-picker NÃO week-view).

**Verdict:** **APPROVED A — HIGH confidence técnica + Eurico-pending business confirmation** — Recomendação técnica unânime entre @pm/@sm/@architect. Decisão final pertence ao Eurico (silent accept ou override).

**Razões técnicas (5 layers — pattern Story 3.4 DP-3.4.4):**

1. **Stack canónica já existe em `apps/web/package.json` (cross-confirmed linhas 18-37).** `@dnd-kit/core ^6.3.1` + `@dnd-kit/sortable ^10.0.0` + `@dnd-kit/utilities ^3.2.2` + `date-fns ^4` + `date-fns-tz ^3` instalados em `dependencies` (não devDependencies). Opção A consome zero deps novas. Opção B (`react-big-calendar`) adiciona ~80kb minified + peer-deps `moment`/`luxon` (date-fns NÃO suportado nativamente — requer wrapper `dateFnsLocalizer`). Opção C (`fullcalendar`) tem license **commercial** para uso comercial (`expressia.pt` é SaaS comercial pago €8,88/mês, fora do MIT scope) — **REJECT absoluto, não negociável**. Opção D (`@schedule-x/react`) é ~60kb mas API beta + comunidade pequena (~2k weekly downloads vs 350k react-big-calendar) — risco maturidade.

2. **UX consistency com Story 3.4 Kanban (pattern reuse).** `@dnd-kit` lifecycle (DndContext + DragOverlay + sensors PointerSensor/TouchSensor/KeyboardSensor + 6 announcements PT-PT WCAG AA) é code battle-tested 393/393 tests `apps/web` em `KanbanBoardClient.tsx` — pre-flight Uma + Final Gate Aria PASS 9.3/10. Story 3.5 escolher react-big-calendar fragmenta o pattern entre vistas Kanban (dnd-kit) + Calendário (react-big-calendar own drag-and-drop API `react-dnd` peer-dep ~30kb additional) — bug surface 2x, mental load contributor 2x, test setup 2x.

3. **CSS Grid 7-col é trivial em 2026.** `grid-template-columns: var(--calendar-day-min-width) repeat(7, 1fr)` + `grid-template-rows: auto 1fr` é 2 linhas CSS. Story 3.5 AC2 documenta tokens novos (`--calendar-day-min-width: 140px`, `--calendar-day-gap: 8px`, `--calendar-day-header-height: 56px`, etc) — pattern Kanban Story 3.4 globals.css escalado para 6 tokens novos. `react-big-calendar` força adopção do seu próprio CSS framework + override class names — anti-pattern KISS.

4. **`front-end-spec.md:1500-1520` confirma stack canónica.** Documento `Stack UI` lista `date-fns` + `@dnd-kit` como stack principal Epic 3 vistas tarefas. Linha 1511 menciona `react-day-picker` mas **Morgan já registou em EPIC-3-EXECUTION.yaml:138-140 `critical_note` que essa lib é date-picker (mês/ano), NÃO week-view-with-drag** — DP2 é NEW NÃO duplicate spec.

5. **Schema reality favorece custom (cross-confirmed `packages/db/src/schema/tasks.ts:50-117`).** `tasks.due_date` é PG `date` type calendar-only (sem hora, sem fuso). String YYYY-MM-DD é o formato canónico DB. `react-big-calendar` espera `Date` objects com `start`/`end` timestamps — wrapper de conversão `date → Date midnight` adiciona surface DST bug que Story 3.5 quer evitar (R3 risk). Custom CSS grid trabalha directamente com `task.due_date === dayIso` string comparison (PO_FIX 2 v1.1).

**Trade-off honesto vs B (react-big-calendar):**

| Critério | A — Custom + @dnd-kit | B — react-big-calendar |
|----------|----------------------|------------------------|
| Bundle size delta | 0kb | +~80kb |
| Dev time | ~20h (12 tasks story 3.5) | ~14h (lib handles grid/nav) − ~6h adaptation (wrappers, theme override, drag integration) = ~8h |
| UX consistency Kanban | 100% (same lib) | 60% (different drag UX) |
| Drag-and-drop quality | @dnd-kit (best-in-class 2026) | react-dnd (older, less polished mobile) |
| Mobile experience | TouchSensor delay 250ms (battle-tested 3.4) | requires `react-big-calendar/addons/dragAndDrop` + custom touch |
| Tests | Vitest + RTL same pattern Kanban | requires separate test harness (lib mocks) |
| Future flexibility (e.g., custom UX requirements) | Total | Limited by lib API |
| Maintenance burden | Custom code | Lib upgrades, breaking changes |
| License | MIT (zero risk) | MIT |

**Resultado:** A vence em 8/9 critérios; B só vence em "dev time inicial" (estimativa optimista). Considerando R8 (perf) + R3 (DST) + reuso pattern 3.4 + bundle size matter para mobile users, **A é a escolha estrategicamente correcta**.

## 2. DPs MEDIUM/LOW — CONFIRM / APPROVE silent

### DP-3.5.5 — Drag → due_date format (CONFIRM RESOLVED by @po)

**Recomendação @po Pax (v1.1):** A trivial — string `YYYY-MM-DD` calendar-only. B+C REJECT (não aplicáveis a PG `date` type).

**Verdict:** **CONFIRM RESOLVED — HIGH confidence técnica.** Pax cross-confirm de schema (`packages/db/src/schema/tasks.ts:77` `dueDate: date('due_date')` + linha 79 `dueTime: text('due_time')`) é tecnicamente impecável. Aria valida:

- PG `date` type armazena calendar date sem hora nem fuso. Sem ambiguidade UTC/Lisbon. Sem DST storage edge cases.
- Conversão drag `dayIso` → DB é literal `{ due_date: 'YYYY-MM-DD' }` PATCH body.
- Hora opcional vive em campo separado `due_time text` HH:MM — Story 3.5 não toca este campo (drag preserva `due_time` se existir, apenas altera `due_date`).
- `date-fns-tz` continua relevante para `completed_at`/`created_at`/`updated_at` que SÃO `timestamp withTimezone` (cross-confirmed `tasks.ts:94-96`) — mas NÃO para `due_date`.
- DP-3.5.5 NÃO requer ratify adicional. Reduz 6 DPs nominais → **5 efectivos activos** para @architect.

**Sem guidance G adicional** — PO Validation Block v1.1 já documenta correctamente. @dev consome AC5(c) + AC8(f) + DP-3.5.5 RESOLVED conforme story v1.1.
