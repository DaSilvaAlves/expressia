### [DEV_DECISION D64] `traces_spanmetrics` strategy → **RATIFIED PARTIAL with REFINEMENT (NIT-003-NB)**

**Análise técnica @architect:**

Grafana Cloud Tempo `metrics-generator` (span-metrics processor) emite por defeito:
- `traces_spanmetrics_calls_total` (counter por span_name + dimensions)
- `traces_spanmetrics_latency_*` ou `_duration_*` (histogram)
- `traces_spanmetrics_size_bytes_*`

**NÃO emite** automaticamente `traces_spanmetrics_sum_<attr>` para attrs numéricos arbitrários.

**Implicações por painel:**

| Painel | Strategy | Works with default Grafana span-metrics? | Action |
|--------|----------|------------------------------------------|--------|
| 1 (latência) | OTel HTTP histogram | ✅ via `@vercel/otel` | Approved |
| 2 (error rate) | HTTP metrics + `intent_class` dimension | ✅ via OTel + span-metrics dimensions | Approved |
| 3 (custo planner) | `traces_spanmetrics_sum_planner_cost_eur` | ❌ **não funciona** | **NIT-003-NB** |
| 4 (accuracy) | Custom OTel meter Gauge (Story 2.10) | ✅ via custom emission | Approved |
| 5 (cache hit) | `traces_spanmetrics_calls_total` + boolean dimension | ✅ via dimensions config | Approved |
| 6 (RLS) | LogQL Loki | ✅ via logger emit (placeholder) | Approved (NIT-002-NB existente) |

**Veredito D64:** **RATIFIED PARTIAL** — strategy correct for 4/6 painéis. Painel 3 + Alarme 2 requerem **NIT-003-NB follow-up** (Epic 2.x): adicionar custom OTel meter emission em `packages/planner-executor/src/planner.ts` — `meter.createCounter('agent.planner.cost_eur').add(metrics.costEur, { household_hash })`. Alternativa secundária: OTel collector pipeline com `transform` processor (mais infra-overhead, descartada).

**Empty state esperado pre-NIT-003-NB fix:** Painel 3 + Alarme 2 renderão "No data". Documentado em runbook §7.3.

**Padrão arquitectónico oficial pós-gate:** Métricas derivadas de spans seguem hierarchy:
1. **Standard span metrics** (calls/duration/dimensions/boolean) → use `traces_spanmetrics_*` queries
2. **Numeric attribute sums** (cost, tokens) → use custom OTel meter emission (`@opentelemetry/api-metrics`)
3. **Custom domain metrics** (accuracy) → use custom OTel meter Gauge (precedente Story 2.10)

## Lessons for Story 2.12+ (Epic 3+ Future)

### L1 — Dashboard query strategy hierarchy

Métricas derivadas de OTel spans seguem hierarquia oficial (post-D64 ratification):

1. **Calls + Duration + Boolean dimensions** → `traces_spanmetrics_calls_total` + `traces_spanmetrics_duration_*` (default Grafana span-metrics generator)
2. **Numeric attribute sums (cost, tokens, sizes)** → Custom OTel meter emission no código (`@opentelemetry/api-metrics`)
3. **Custom domain metrics (accuracy, business KPIs)** → Custom OTel meter Gauge (precedente Story 2.10 + 2.11 painel 4)

### L2 — Dashboard scaffold pattern

Story 1.7 + Story 2.11 estabeleceram pattern oficial: JSON scaffold versionado em `docs/dashboards/` + queries documentadas em runbook + UI activation deferred pós-deploy via handoff post-deploy. Pattern aceitável e reutilizável Epic 3+.

### L3 — PO_FIX_INLINE process value

Story 2.11 v1.0 → v1.1 caught 6 hallucinations vs codebase real via @po cross-confirm rigoroso. Sem este step, @dev teria implementado queries impossíveis de funcionar. **Process value: ~45min @po validation salvou downstream debug cycles.** Recommend continuing rigorous cross-confirm process.

### L4 — Architect catch beyond @po reach

@architect identificou D64 refinement (`traces_spanmetrics_sum_*` não funciona para arbitrary numeric attrs) — este é conhecimento técnico Grafana Tempo que escapou ao @po cross-confirm. Architect gate adiciona valor único pela perspectiva técnica deep-stack.

### L5 — Mockable-friendly pattern continuation

Padrão Stories 2.2-2.11 (10 stories mockable-only ou mockable-friendly) demonstra que Epic 2 fechou efectivamente sem bloquear em decisão humana (DPA UE Anthropic). Pattern continuará Epic 3+ até EB1 resolved.
