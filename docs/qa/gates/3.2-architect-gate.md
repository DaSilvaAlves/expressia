### AF7 — Audit_log best-effort gap risk NFR9 ⚠️

⚠️ **APPROVED com NIT-001-3.2 deferred.**

**Analysis:** `insertAuditLog` em `audit.ts:32-55` envolve INSERT em try/catch — falhas (DB connection down, FK violation, enum mismatch) são `log.warn` mas mutation principal continua succeed. Pattern em route handlers:

```typescript
try {
  await insertAuditLog({...});
} catch (auditErr) {
  log.warn({ err: auditErr }, 'audit_log INSERT falhou (best-effort)');
}
```

**NFR9 Architectural concerno:**
- **Pro best-effort:** UX não affected por audit gap; mutation succeeds (user feedback preservado).
- **Con best-effort:** Audit row pode missing → NFR9 imutabilidade preservada para rows EXISTENTES, mas observability gap potencial. Em compliance audit, "no audit row" pode ser interpretado como "no mutation occurred" — incorrect.

**Alternative fail-fast pattern:**
```typescript
// Fail-fast: rollback mutation se audit falha
await db.transaction(async (tx) => {
  await tx.execute(sql`INSERT INTO tasks ...`);
  await tx.execute(sql`INSERT INTO audit_log ...`); // throws → ROLLBACK
});
```

**Trade-off:** Fail-fast garantia audit completeness mas degrade UX (mutation falha por audit error não user-facing).

**Architect recommendation:** Current best-effort is pragmatic for MVP. **NIT-001-3.2 deferred** — Epic 4 housekeeping batch consider:
1. Add structured metric `agent.audit_log.insert_failures` counter para observability.
2. Alarme Grafana se rate > 1% durante 5min.
3. Eventually migrate para fail-fast (transactional) quando volume + SLO justifica.

Non-blocking gate. NFR9 imutabilidade for stored rows preservada (RLS revoga UPDATE/DELETE em audit_log per Story 2.8 migration).
