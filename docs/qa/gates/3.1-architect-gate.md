## Architect Focus Areas — AF1-AF6 Analysis

### AF1 — Migration `0009_kanban_seed.sql` correctness

**Q1: SQL function `seed_household_kanban_defaults` correto? `ON CONFLICT (household_id, sort_order)` é a única unique constraint disponível?**

✅ **APPROVED.** Verified em `tenancy.ts:171`:
```typescript
uniqueOrder: unique('kanban_columns_unique_order').on(t.householdId, t.sortOrder),
```
Esta é a **única** unique constraint multi-column na tabela. Outros constraints existentes:
- `id` PK (uuid) — automaticamente unique mas não composite
- Index `kanban_columns_household_idx` em `(household_id)` apenas — NÃO unique

`ON CONFLICT (household_id, sort_order) DO NOTHING` é a escolha correcta para idempotência sem race conditions. Alternativas rejeitadas:
- `ON CONFLICT (household_id, name)` — não há unique constraint nessa combinação (válido para customizações user que podem ter mesma name em sort_orders diferentes).
- `IF NOT EXISTS` em SELECT pré-check — race-prone (TOCTOU window entre check e insert).

**Q2: Trigger wrapper pattern `tr_seed_kanban_for_new_household` é overkill vs inline procedure call?**

✅ **APPROVED — best practice Postgres.** Razões:
- **Signature mismatch:** Trigger functions em Postgres devem retornar `trigger` (não `void`). Business function `seed_household_kanban_defaults` retorna `void` para ser chamável directamente (backfill DO block + future API calls). Wrapper resolve gap signature.
- **Separation of concerns:** Business function é callable directamente (testes unit, debug ad-hoc, future signup hook); trigger function tem contrato Postgres específico (`new`/`old` records, `returns trigger`).
- **Pattern matching:** `0003_auth_user_trigger.sql` usa mesmo pattern (trigger `on_auth_user_created` → wrapper `handle_new_user()` retorna trigger → chama business logic).
- **Future-proof:** Se accept_invite() workflow precisar chamar `seed_household_kanban_defaults` directamente (sem inserir household), business function está disponível sem refactor.

**Q3: Backfill DO block é seguro em prod (transação automática + idempotente)?**

✅ **APPROVED — atómico per migration.** Razões:
- `apply-migrations.ts:48-61` aplica cada ficheiro `.sql` numa transação dedicada — DO block + function creates + trigger creates corre tudo numa transação atómica. Rollback automático em qualquer erro.
- Idempotência via `ON CONFLICT DO NOTHING` na função alvo — re-run em households com colunas pre-existing absorve duplicates sem erro.
- `RAISE NOTICE` no fim do DO block produz output observável em CI logs — `total_processed: N households` é debugável.
- DO block usa `record` cursor com `FOR ... LOOP` — eficiente para dataset pequeno (1 household actual) mas escalável (loop sequencial, sem JOIN costs).

**Performance check:** Para ~1k households (futuro escala), backfill seria ~1k function calls × ~200µs = ~200ms total. Aceitável para migration runtime. Para 100k+ households, alternativa seria batch INSERT com `INSERT ... SELECT` — premature optimization para MVP.

**Q4: PT-PT names byte-for-byte UTF-8 — risco encoding em CI Linux vs dev Windows?**

✅ **APPROVED — UTF-8 sem BOM verified.** Razões:
- `file packages/db/migrations/0009_kanban_seed.sql` retornou "Unicode text, UTF-8 text" — encoding correcto.
- Sem BOM detectado (BOM causaria "UTF-8 Unicode (with BOM) text" output) — alguns parsers SQL choke em BOM no início do ficheiro.
- `grep` byte-pattern matching retornou os 3 nomes literais nas linhas 13, 22, 63, 67-69, 75 — caracteres `ç` (U+00E7, 0xC3 0xA7 em UTF-8) e `í` (U+00ED, 0xC3 0xAD em UTF-8) preservados.

**Q4: Variants `delete_owner_admin` (tags + kanban_columns) — comentários explicam PO_FIX F1 satisfaction, mas testes assumem RLS USING filter ordering (RLS antes role check). Confirmar ordering correto?**

✅ **APPROVED — Postgres RLS USING semantics verified.** Postgres RLS policy execution order:

1. **USING clause** (filter rows visible para SELECT/UPDATE/DELETE) — applied PRIMEIRO
2. **WITH CHECK clause** (validate new rows for INSERT/UPDATE) — applied DEPOIS no INSERT/UPDATE results
3. **Role permissions** (GRANT/REVOKE) — checked separately mas tipicamente SET ROLE authenticated permite todas operações DML, RLS é o filter real

Para `tags_delete_owner_admin`:
```sql
create policy "tags_delete_owner_admin"
  on public.tags for delete
  to authenticated using (public.is_household_owner_or_admin(household_id));
```

Quando userB (do householdB) executa `DELETE FROM tags WHERE id = <tagOfHouseholdA>`:
1. Postgres avalia USING clause: `is_household_owner_or_admin(householdA)` para userB → retorna FALSE (userB não é owner/admin de householdA, é de householdB).
2. Row é filtrada do DELETE working set → 0 rows affected.
3. Não há erro permission denied porque o role `authenticated` tem DELETE grant na tabela; é o RLS que filtra silentemente.

Verdict: ordering correcto, comment inline reflecte semantics reais do Postgres.

**Q5: result.count semantics em postgres.js para UPDATE/DELETE 0 rows — confirma não throw?**

✅ **APPROVED — postgres.js library spec.** Library `postgres-js` (used by `rls-harness.ts:103`) retorna result objects com `.count` property para UPDATE/DELETE. Behaviour:
- `UPDATE ... WHERE ...` que matches 0 rows → returns `[]` array com `.count = 0`. **No throw.**
- `INSERT INTO ... WITH CHECK violation` (RLS rejeita) → throws com message `"new row violates row-level security policy for table"`. **Throws.**
- `DELETE FROM ... WHERE ...` que matches 0 rows → returns `[]` array com `.count = 0`. **No throw.**

Por isso:
- Testes INSERT cross-household usam `expectRlsBlocks` helper (catches throw).
- Testes UPDATE/DELETE cross-household usam `result.count === 0` assertion (silent filter via RLS USING).

Pattern correcto para ambas semantics.
