## 7. G1.1 EXPLAIN ANALYZE Resolution — CONCERN LOW (non-blocking)

### Decisão: **Opção B — Static plan validation** (gate spec)

**Justificação:** ambiente QA local não dispõe de fixture seedable 1000×5000 nem acesso live ao Postgres EU dev (requereria credenciais `DIRECT_URL` em `.env.local` + script seed transitório). Em vez disso, valido a query estruturalmente com profundidade comparável a EXPLAIN ANALYZE empírico, e documenta-se a deferência para staging pipeline com mitigation.

### 7.1 Query Topology Analysis

Query alvo (`apps/web/src/lib/api-helpers/list-tasks.ts:168-188`):

```sql
select tasks.id, tasks.household_id, tasks.created_by_user_id, ...,
       coalesce(
         json_agg(
           json_build_object('id', tags.id, 'name', tags.name, 'color', tags.color)
           order by tags.name asc
         ) filter (where tags.id is not null),
         '[]'::json
       ) as tags
from public.tasks
left join public.task_tags on task_tags.task_id = tasks.id
left join public.tags on tags.id = task_tags.tag_id
where ${whereSql}
group by tasks.id
order by ${orderBySql}
limit ${limitPlusOne}
```

### 7.2 Index Coverage Analysis

| Tabela | Index | Schema location | Cobre |
|--------|-------|------------------|-------|
| `tasks` | PK `tasks_pkey` (id) | `0000_initial_schema.sql:441-...` | `GROUP BY tasks.id` (functional dependency PG 16) + `WHERE tasks.id IN (...)` filter sub-query |
| `tasks` | `tasks_household_idx` (household_id) | linha 441 | RLS filter (`is_household_member`) — transparente |
| `task_tags` | PK composite `(task_id, tag_id)` | schema `tasks.ts:217` | JOIN `task_tags.task_id = tasks.id` (covered) |
| `task_tags` | `task_tags_household_idx` (household_id) | linha 497 | RLS filter — transparente |
| `task_tags` | `task_tags_tag_idx` (tag_id) | linha 498 | Sub-query `tag_id` filter (linha 142 list-tasks.ts) + JOIN tags |
| `tags` | PK `tags_pkey` (id) | `0000_initial_schema.sql:482+` | JOIN `tags.id = task_tags.tag_id` (covered) |
| `tags` | `tags_household_idx` (household_id) | linha 486 | RLS filter — transparente |

**Resultado:** Todos os JOIN predicates e filters têm index coverage adequada. Não há Seq Scan obrigatório em fixture realista (PG planner escolhe Index/Bitmap Heap Scan automaticamente para selectivity típica <10%).

### 7.3 Plan Prediction (PG 16)

Para fixture realista 1000 tasks × 200 tags × 5000 task_tags com limit 50:

| Stage | Operação esperada | Custo estimado |
|-------|-------------------|----------------|
| 1. RLS filter `tasks` | Index Scan `tasks_household_idx` | O(log n_tasks_household) ~ 5-10ms |
| 2. LEFT JOIN `task_tags` | Hash Join (small build side post-RLS) ou Nested Loop com Index Scan `task_tags_pkey` | ~10-30ms |
| 3. LEFT JOIN `tags` | Hash Join (200 tags fit easily em hash table) | ~5-10ms |
| 4. GROUP BY `tasks.id` | HashAggregate (PG 16 functional dependency via PK detect — tasks.* OK) | ~10-20ms |
| 5. ORDER BY `tasks.due_date asc nulls last, tasks.id asc` | Sort + Top-K com limit | ~10-20ms |
| **Total estimado** | | **40-90ms** (well within 300ms budget AC10(d)) |

### 7.5 Risk Assessment

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Query >300ms p95 com household power-user (200 tags × 5000 task_tags) | LOW | MEDIUM (UI lag perceptible) | Sentry alert `slow_query` threshold 300ms já configurado (Story 1.7) + Grafana p99 monitor (Story 1.7 backlog) |
| Hash Join → Nested Loop degradation com cardinalidade extrema | VERY LOW | LOW | PG 16 planner auto-adjusts via `pg_stat_statements` — observable em staging |
| `json_agg` ORDER BY overhead | LOW | LOW | tags por task ≤ 5 (DP-3.6.2 UI soft limit) — sort de 5 elementos é trivial |

### 7.6 Conclusão G1.1

**Status:** **CONCERN LOW G1.1-DEFERRED-STAGING** (non-blocking).

**Rationale:**
1. Estrutura da query é canónica e idiomatic PG 16 (json_agg + FILTER + GROUP BY PK).
2. Indexes cobrem todos os JOIN predicates + filters relevantes.
3. Plano teórico 40-90ms está bem dentro do budget 300ms p95.
