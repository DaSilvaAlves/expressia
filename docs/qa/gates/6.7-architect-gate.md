## Segurança — auditoria da `accept_invite()` (migration 0020)

A função é o artefacto crítico (SECURITY DEFINER, insere membership cross-household). Auditada a fundo:

- ✓ **`set search_path = public`** presente — previne search_path hijacking (obrigatório em SECURITY DEFINER).
- ✓ **Lock `for update`** no convite (linha 68) **e** no household (linha 108) — R-6.5: serializa aceitações concorrentes no mesmo household → `count(members) < limite` é fiável (defesa em profundidade vs UI).
- ✓ **email-match case-insensitive** (`lower(invite.email) <> lower(user_email)`) — defesa contra token roubado: mesmo com o token, só o destinatário nominal aceita.
- ✓ **Validação não-membro** antes do insert — evita violar PK composta.
- ✓ **`grant execute ... to authenticated`** presente (PO-FIX) — sem ele a chamada via `getDb()` falharia.
- ✓ **Zero SQL dinâmico** — `p_token` parametrizado; sem `execute`. Sem risco de injection. Token 32 bytes (`crypto.randomBytes`) não-enumerable.
- ✓ **Erros tipados** em todos os caminhos → `mapAcceptInviteError` mapeia para PT-PT + status HTTP.

## CONCERN (não-bloqueante)

- **C-6.7.1 [MEDIUM]** — Smoke E2E no browser (convidar → copiar link → logout → login conta convidada → aceitar) **não exercido manualmente**. Cobertura é unit/integração (web) + db-test contra Postgres real (`accept_invite` provado em 7 cenários: sucesso, limite Família=4, email≠, expirado, dupla aceitação, já-membro, token inválido). A propagação do JWT em `getDb()` para `auth.uid()` segue o mesmo mecanismo de `is_household_member()` (provado em produção desde a Story 1.5). **Risco residual baixo.** Análogo a C-6.1.1 (aceite). Registado como follow-up **INVITE-E2E-SMOKE** (pós-push, Eurico/@qa).

## OBS (LOW, informativas)

- **OBS-6.7.1** — Ordem de validação na `accept_invite()`: `email_mismatch` é verificado **depois** de `already_accepted`/`expired`. Um portador de token (secreto) para outro email recebe o estado do convite antes do mismatch — **information disclosure teórico** (requer o token de 32 bytes, não-enumerable; o email-match continua a impedir a aceitação). Reordenar email-match para antes do estado é opcional (endurecimento). Não-bloqueante.
- **OBS-6.7.2** — `accept_invite()` insere `v_invite.role` directamente; um convite com `role='owner'` criaria 2º owner. **Impossível via endpoint** (`InviteCreateSchema` restringe a `INVITABLE_ROLES=['admin','member']`). Só atingível por SQL directo (que já requer privilégios). Não-issue prático.
