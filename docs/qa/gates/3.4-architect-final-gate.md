### D-3.4.2 — `captureException` em KanbanBoardClient via `@sentry/nextjs` directo (não `@meu-jarvis/observability`)

**Verdict:** ✅ **ACCEPT** silent — pattern precedent confirmado.

**Razões:**
1. **Precedent existe e foi verificado:** `apps/web/src/app/(app)/jarvis/_components/jarvis-chat.tsx:6` importa `import { captureException } from '@sentry/nextjs'` — mesmo pattern aplicado em Story 2.X já em produção. Consistência alta.
2. **Razão técnica válida:** `@meu-jarvis/observability/logger.ts` importa `node:crypto` que parte webpack client bundle com `UnhandledSchemeError`. Documentado em comentário do código `KanbanBoardClient.tsx:21-25`.
3. **Zero runtime risk:** ambos os imports inicializam o mesmo Sentry browser SDK runtime. É apenas decisão de entry point.
4. **Test compatibility:** `KanbanBoardClient.test.tsx:20` mocka `@sentry/nextjs` — mesmo pattern que `jarvis-chat.test.tsx:19`. Conformidade total com convention existente.

**Follow-up (CONCERN-2 LOW non-blocking):** **Spike proposto para Story Epic 4+ ou backlog `@meu-jarvis/observability`** — separar browser/node entry points no package (`@meu-jarvis/observability/browser` vs `@meu-jarvis/observability/node`) para permitir `@meu-jarvis/observability` ser consumível em client islands sem fricção webpack. Não bloqueia 3.4. Pattern actual é o ideal disponível.
