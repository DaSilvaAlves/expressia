## Sumário Executivo

A Story 2.13 fecha o **GAP-6** — a ponte conversacional Finanças ↔ Cérebro que nunca tinha sido construída. Antes desta story, todo o subsistema Finanças via chat estava bloqueado: o utilizador novo tinha 0 contas, o `.refine()` das tools rejeitava antes de `execute()`, e o planner não conhecia as contas reais (placeholder literal `<uuid>` no system prompt). A implementação resolve as três peças da causa raiz na ordem certa (B+A do ADR-002): conta "Dinheiro" default no onboarding + backfill (T1, @data-engineer), refine relaxado + `resolveDefaultAccount` (T2), context injection no payload do planner (T3), bump prompt v2→v3 (T4), `create_card` com conta default (T5).

**Resultado da verificação independente:** zero hallucinations, zero capability loss, 5/5 quality gates re-corridos GREEN pelo gate (não confiei nos números do @dev — corri tudo), todas as 9 ACs verificadas contra código real, as 4 DEV-DECISIONS ratificadas com fundamentação, fronteira PII (NFR12) e RLS (NFR5) verificadas por auditoria negativa (grep + whitelist + redaction layer + test assertions), Constitution Art. IV (No Invention) confirmado.

**E2E (T6) provado manualmente nesta sessão** (Eurico + Orion, localhost): "paguei 18,70 no pingo doce" → criou despesa "Pingo Doce −€18,70 · Outros gastos · Dinheiro" associada à conta default, com undo. Screenshot confirmado. AC9 satisfeito.

**Concerns LOW non-blocking documentados** (não bloqueiam push): NIT-AR-2.13.1 (paridade tipo `AccountType` local — D-2.13.1), NIT-AR-2.13.2 (2 queries no `buildAccountContext` — NFR1 marginal).

## Ratificação das 4 DEV-DECISIONS

### D-2.13.1 — `AccountType` declarado localmente no helper — **RATIFICADA**

O `@meu-jarvis/db` não exporta um tipo `AccountType` nomeado (só o `accountTypeEnum` runtime + `Account`/`NewAccount` via `$inferSelect`). Importar o `accountTypeEnum` para derivar o tipo acoplaria `@meu-jarvis/tools` ao runtime do schema DB — exactamente a fronteira que `contracts.ts` documenta evitar. A declaração local do union de 6 valores (`resolve-default-account.ts:47-53`) espelha 1:1 o enum (finance.ts:39-46), mantém zero coupling cross-package, e é a mesma filosofia do `z.string()` no planner (ADR-002 §9.4). **Fundamentação:** preserva a fronteira de package; a alternativa (importar o enum) violaria-a. **Concern LOW (NIT-AR-2.13.1):** se o enum DB mudar, o union local dessincroniza — mitigável por um futuro teste de paridade (`finance.ts` enum vs union local). Fora de scope; registado como follow-up FUP-2.13.D.

### D-2.13.2 — Precedência 1+2 numa única query SQL — **RATIFICADA**

`ORDER BY (account_type = 'dinheiro') DESC, created_at ASC LIMIT 1` (resolve-default-account.ts:86-92) é semanticamente equivalente à precedência de 3 níveis da story: o booleano `true` (conta dinheiro) ordena primeiro com `DESC`, empate resolvido por `created_at ASC` (mais antiga), e o nível 3 (erro) é `rows[0]` ausente. **Fundamentação:** 1 round-trip DB em vez de 2 (NFR1), determinístico, correcto na semântica de ordenação booleana do Postgres (`false < true`, logo `DESC` põe `true` no topo). Pós-backfill 0018, o nível 1 resolve sempre para households reais. Verificado pelo test `resolve-default-account.test.ts:62-70` que assere a presença literal de `account_type = 'dinheiro'` e `archived_at is null` no SQL. Decisão correcta — preferível às duas queries.

### D-2.13.3 — `<uuid>` substituído também no exemplo 3 (parcelada) — **RATIFICADA**

T4.1 citava só os exemplos 6/7/8, mas o Exemplo 3 (`create_installment`, prompt linha 94) também continha `cardId: '<uuid>'` — placeholder igualmente impossível de resolver. A substituição por `<id do cartão "Activobank" no accountContext>` + nota de que a parcelada exige cartão real (FUP-2.13.A) é coerente: (1) o test negativo `not.toContain('<uuid>')` em `prompts.test.ts` cobre o prompt inteiro, logo exigia limpeza total; (2) não altera scope — `criar_parcelada` continua fora de scope funcional (depende de cartão real). **Fundamentação:** alargamento mínimo e coerente; deixar 1 `<uuid>` no prompt seria inconsistência documental. Coerente com Art. IV (não inventa comportamento, só limpa placeholders).

### D-2.13.4 — `paymentMethod='transfer'` para `accountId` explícito de conta dinheiro — **RATIFICADA**

Quando o utilizador passa um `accountId` explícito que por acaso é de uma conta `dinheiro`, o `paymentMethod` resolve para `'transfer'` (não `'cash'`), porque `resolvedAccountType` só é populado quando `resolveDefaultAccount` corre (create-finance-variable.ts:142-164). **Fundamentação:** (1) preserva o comportamento pré-existente documentado da 4.10 (`accountId presente → 'transfer'`); (2) DP-2.13.A foca o caso da conta **default resolvida** (família-first comum), não a escolha explícita — nesse caso o utilizador pode passar `paymentMethod` se quiser; (3) um lookup extra do tipo por cada `accountId` explícito viola o budget NFR1 sem ganho prático (raro o utilizador nomear explicitamente a conta "Dinheiro" via chat). Os 2 findings MINOR do CodeRabbit sobre isto são aceites como comportamento intencional; o 3.º (dangling-else no `captureSqlText` de teste) foi corrigido. **Trade-off aceite e correcto** — coerente com DP-2.13.A (que decorre do tipo da conta REALMENTE resolvida, não do input).
