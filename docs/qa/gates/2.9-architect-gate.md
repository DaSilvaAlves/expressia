A decisão **[DEV-DECISION D56] (cost router fallback graceful para executor em vez de 500)** é arquitecturalmente correcta e merece análise explícita. Cost router é, por design, optimização de custo — não primary path. Se `executeDirectQuery` falhar (deadlock momentâneo, falha de RLS context, malformed query template), retornar 500 imediatamente seria fragility-by-design — o utilizador veria 500 numa pergunta `consultar_dados` simples quando o sistema sem cost router responderia normalmente via Planner+Executor. **Ratifico D56 como PASS.** Trade-off é o custo de uma chamada Sonnet adicional num caso de falha rara (`directResult = null` → cai no path `if (classification.needs_confirmation || alwaysPreview)` ou no path executed via Planner) — aceitável. Pino warn `'Cost router direct query falhou — degradando para executor'` (linha 383 verificada) garante observabilidade. Padrão consistente com cache lookup falha (também não-fatal — `cacheClient.get` em modo degradado retorna `null` sem throw). Verificado em `route.ts:378-385` que o try/catch envolve `executeDirectQuery` e degrada para `directResult = null`, que faz o fluxo cair fora do bloco `if (directResult) {...}` e continuar para o classifier branch normal.

A decisão **[DEV-DECISION D57] (`selectTemplate` keyword fix `'transac'` → `'transa'` para PT-PT 'ç')** é informativa e ratificada como PASS sem análise extensa. Bug subtle de encoding apanhado pelos próprios tests de @dev em T7 — `'transac'` (length 7) não faz substring match em `'transações'` porque `'transa'` + `'ç'` (U+00E7) ≠ `'transa'` + `'c'` (U+0063). Fix: keyword reduzida a `'transa'` (6 chars cobre todas as variantes PT-PT `'transação'`/`'transações'` + PT-BR `'transac'`/`'transacc'`) + adicionada keyword `'finanç'` (cobre `'finanças'`) e `'financ'` (cobre `'financeiro'`). Implementação verificada em `cost-router.ts:82-90` — keyword matching ordering correcto (saldo primeiro, transac/finanç segundo, default count_tasks). Sem impacto contractual — fix interno encapsulado em `selectTemplate`. Caracteres PT-PT com diacritics requerem caution em substring matching: nota arquitectural geral para o codebase é que **substring matching JavaScript nativo não faz ASCII fold** — abordagens futuras (Epic 3/4 catálogo expandido) devem considerar normalisação via `string.normalize('NFD').replace(/\p{Diacritic}/gu, '')` ou similar quando o número de keywords aumentar. Registado como **NIT-006-NB low** para Epic 3+ quando o catálogo expandir.

A decisão **[AUTO-DECISION D53] (cache key `householdPlan` vs `householdId`)** merece nota arquitectural explícita porque corrige uma hallucination do spawn prompt do @sm. Architecture §4.6 linha 433 literal: `sha256(normalize(prompt) + household_plan)`. Usar `household_id` criaria cache per-household (granularidade demasiado fina → hit rate baixo, especialmente em prod cedo com poucos utilizadores por household). Usar `household_plan` partilha cache entre users do mesmo tier (`free`/`pessoal`/`familia`/`pro`) — maximiza hit rate sem comprometer PII (ClassificationResult contém apenas intents detectadas + raw_span; intents são metadados sobre a forma do prompt, não sobre o conteúdo). **Segurança verificada:** o `cacheKey` é hash SHA-256 do prompt normalizado — input para o hash inclui o prompt original (não cleartext no cache, mas hash não é reversível). O **value cached** é o ClassificationResult (intents + confidences), não o prompt original. DN4 (NUNCA logar prompt cleartext) é respeitado — apenas o `cacheKey` (hash) é logado, e o teste `cache.test.ts (vi)` valida que logs Pino NÃO contêm o prompt string original. **PASS — correcção arquitectural exemplar baseada em verificação directa de Architecture §4.6.**

## Análise D55 — pattern para `service_role_only` writes

A decisão @dev de **remover o parâmetro `db: Database` da signature de `incrementQuota`** é arquitecturalmente correcta como **pattern obrigatório** para todas as operações de escrita em tabelas com `service_role_only` RLS policies. Análise:

### Tabelas service_role_only no codebase actual

Verificação directa contra `packages/db/migrations/0001_rls_policies.sql`:

| Tabela | INSERT policy | UPDATE policy | Caller pattern correcto |
|--------|---------------|---------------|--------------------------|
| `agent_quotas` | `agent_quotas_insert_blocked` `with check (false)` to authenticated | `agent_quotas_update_blocked` `using (false) with check (false)` to authenticated | **`getServiceDb()` interno na função wrapper** (D55) |
| `agent_runs` (UPDATE imutabilidade) | (allowed por owner) | (allowed só status transitions) — trigger imutabilidade em prod via `markRunReverted` | `getServiceDb()` interno (Story 2.8 DN3) — pattern análogo |
| `agent_reverse_ops` | allowed via reverse_op INSERT do executor | (none allowed pós-INSERT) | `getServiceDb()` interno em endpoint undo + Inngest cleanup |
| `audit_log` | `audit_log_insert_member` (authenticated permitido com `is_household_member` check) | — | Authenticated INSERT funcional (Story 2.8 [DEV-DECISION 1] análise: serviceDb defensivo é aceite mas não estritamente necessário) |

### Pattern ratificado

**Para funções wrapper que escrevem em tabelas `service_role_only`:** a função MUST:

1. Aceitar apenas argumentos de domínio (e.g., `householdId`, `runId`) — NUNCA `db: Database` argument.
2. Obter o cliente Drizzle internamente via `getServiceDb()` — não confiar no caller para passar o cliente correcto.
3. Comentário JSDoc explícito documentando a razão (referenciando a migration RLS específica).

**Por quê:** se a função aceitar `db: Database`, qualquer caller futuro pode chamar com `getDb()` (authenticated) e regressar o bug D50. A signature `(householdId: string)` **torna impossível** o erro: o caller não pode escolher o tipo de cliente, a função decide internamente. Análogo a `markRunReverted` da Story 2.8 DN3 — mesmo pattern, mesma motivação (defence-in-depth contra regressão).

**Reverse pattern (NÃO aplicar):** para funções que escrevem em tabelas com policies authenticated permissivas (`agent_runs` UPDATE owner, `audit_log` INSERT member, todas as tabelas de domínio MVP), continuar a passar `db: Database` é correcto — caller decide o contexto. O pattern D55 é específico para `service_role_only` writes.

**Story 2.6/2.8 inconsistências:** verificadas — Story 2.6 `updateAfterExecutor` e Story 2.7 `updatePreviewState` aceitam `db: Database` argument, ambas escrevem em `agent_runs` UPDATE (policies authenticated permitidas para owner). **Não-regressão** — pattern D55 não se aplica retroactivamente porque essas funções escrevem em tabelas com policies authenticated. **Forward-looking:** Story 3+ ou futura função que escreva em `agent_quotas`, `agent_reverse_ops` (post-INSERT UPDATE/DELETE não-Inngest) ou similar deve seguir D55. **NIT-007-NB low** sugere revisitar `incrementQuota` siblings em Story 3+ para garantir consistência.
