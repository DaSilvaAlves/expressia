## Sumário executivo

A implementação da Story 1.7 é **arquitecturalmente sólida e reflete decisões de stack defensáveis com evidência externa**. Os 3 fundamentos críticos da observabilidade Expressia estão presentes e bem desenhados:

1. **OTel via `@vercel/otel`** auto-instrumenta Next.js 15 sem boilerplate, lê `OTEL_EXPORTER_OTLP_ENDPOINT` + `OTEL_EXPORTER_OTLP_HEADERS` directamente, e cobre cold-start serverless (que é o cenário real em Vercel `fra1`). Decisão alinhada com Architecture §9.1 e ADR-004.
2. **Sentry EU em 3 runtimes** (server/client/edge) com data residency Frankfurt confirmada via DSN `.de.sentry.io`, sourcemap upload condicional, PII scrub em `beforeSend` em todos os runtimes, e `replayIntegration` com `maskAllText: true` no client.
3. **PII redaction em 3 camadas defesa-em-profundidade** — Pino paths exportados como constante única `PII_REDACT_PATHS`, reusada literalmente pelo Sentry wrapper (`scrubExtra`) e espelhada pelos `beforeSend` hooks. Não há divergência entre as 3 camadas — a fonte é uma só.

**A decisão autónoma do @dev mais arriscada — remoção do `experimental.instrumentationHook`** — está **VALIDADA por evidência externa autoritativa**:

- Next.js 15 release notes anunciam que "the instrumentation feature is now stable and the experimental.instrumentationHook config option can be removed"
- Documentação oficial Next.js 15 confirma: "Starting with Next.js 15, instrumentation.ts is detected automatically. If your config still contains experimental: { instrumentationHook: true }, remove it — it is ignored and clutters your config."
- Sentry [issue #13551](https://github.com/getsentry/sentry-javascript/issues/13551) tem o título exacto: "[Next.js] Don't set `experimental.instrumentationHook` for Next.js >= 15"

O handoff @po estava baseado em guidance Next 13/14 — o @dev fez a chamada correcta ao remover o flag (que aliás causaria `TS2353` no typecheck). **Spans VÃO chegar ao Grafana sem o flag.** Não há risco de produção neste ponto.

A decisão de **manter dual-emission** (Sentry-SDK para errors + `@vercel/otel` para traces/métricas/logs) em vez do endpoint OTLP nativo do Sentry está correctamente diferida para Epic 2 — evita lock-in vendor agora e permite re-avaliar com dados reais sobre duplicação de span volume.

**Tasks 7-8 (UI Grafana) deferidas pós-deploy: ACEITO.** O scaffold JSON em `docs/dashboards/grafana-epic1.json` tem os 4 painéis com queries OTel-conformantes (`http_server_duration_milliseconds_bucket`, etc.) e thresholds correctos; as queries estão também no runbook §6. Criar painéis na UI Grafana sem dados reais produziria gráficos vazios sem valor de validação — a deferência é pragmática, não ginástica.

**1 Issue minor de documentação** (DOC-001) — corrigido directamente neste gate via commit `chore(docs)`: o runbook §7 ainda sugeria `experimental.instrumentationHook: true` como resolução para "spans não aparecem no Grafana", contradizendo o §4 do mesmo runbook e a implementação real.

## Decisão sobre `experimental.instrumentationHook` — investigação definitiva

| Fonte | Citação | Conclusão |
|-------|---------|-----------|
| Next.js 15 release notes | "The instrumentation.js API is now stable as a new API for server lifecycle observability." | Estável, sem flag |
| Next.js 15 docs (`/api-reference/file-conventions/instrumentation`) | "Starting with Next.js 15, instrumentation.ts is detected automatically. If your config still contains experimental: { instrumentationHook: true }, remove it — it is ignored and clutters your config." | Activo por default |
| Sentry GH issue #13551 | Título: "[Next.js] Don't set `experimental.instrumentationHook` for Next.js >= 15" | Confirmação cross-vendor |
| TypeScript types `next` 15.5.15 | `experimental.instrumentationHook` não existe → `TS2353: Object literal may only specify known properties` | Manter o flag PARTE o build |

**Veredicto:** decisão @dev autónoma **CORRECTA**. Spans vão chegar ao Grafana sem flag. Não há ginástica nem workaround — é o comportamento documentado e suportado em Next 15+.
