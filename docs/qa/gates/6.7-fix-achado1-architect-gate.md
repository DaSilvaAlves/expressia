## Causa raiz (recap) e forma do fix

A 0020 (`accept_invite(p_token text)`) fazia `v_user_id := auth.uid()`. O `getDb()` runtime
liga via `DATABASE_URL` (pgbouncer 6543) como role `postgres` **sem** injectar
`request.jwt.claims` → `auth.uid()` = NULL → `raise AUTH_REQUIRED`, apesar de a sessão do
browser ser válida. Os db-test passavam porque o harness simula o JWT (`set_config('request.jwt.claims', …)`).
Fix idiomático e coerente com SEC-1 (isolamento app-enforced): passar a identidade
EXPLICITAMENTE como `p_user_id`, fornecida pelo handler a partir da sessão já validada
(`supabase.auth.getUser()`).

**Não deixa nada órfão a chamar a versão de 1 arg.** Grep em todo o código (`packages/`, `apps/`,
`*.sql`/`*.ts`): a única invocação **executável** é `route.ts:112` com 2 args. As restantes
ocorrências de `accept_invite(` são comentários (`0000_initial_schema.sql:209`,
`0001_rls_policies.sql:89,92`, `tenancy.ts:112,131`), o helper de teste (`accept_invite.test.ts:93,238`,
já 2 args) e os scripts de diagnóstico/comments. O `drop function if exists` é idempotente.

**Passar `p_user_id` arbitrário NÃO contorna nada.** O email-match valida sempre
`invite.email` vs o email do `p_user_id` lido de `auth.users` (`0022:96-108`). Um id de um
utilizador cujo email não bate com o convite → `INVITE_EMAIL_MISMATCH`. Provado pelo teste
`p_user_id diferente do convidado NÃO contorna email-match (segurança)`
(`accept_invite.test.ts:195-228`): o "atacante" recebe MISMATCH, **nenhuma** membership é
criada, e o convidado legítimo consegue aceitar. O handler, por construção, passa sempre o
utilizador autenticado (`user.id` de `getUser()`, `route.ts:109-112`), não input do cliente.

## Foco 2 — db-test (CRÍTICO): a 0022 validada contra Postgres real

`accept_invite.test.ts` = **9 testes**, todos verdes contra Postgres 16 (Testcontainers):

1. aceita convite válido → convidado vira membro (+ accepted_at/by preenchidos)
2. token inválido → `INVITE_NOT_FOUND`
3. convite expirado → `INVITE_EXPIRED`
4. email não corresponde → `INVITE_EMAIL_MISMATCH`
5. dupla aceitação → `INVITE_ALREADY_ACCEPTED`
6. limite plano Família=4 → 5º recusado `MEMBER_LIMIT_REACHED`
7. já membro → `ALREADY_MEMBER`
8. **(novo)** `p_user_id` diferente NÃO contorna email-match (atacante MISMATCH; sem membership)
9. **(novo)** `p_user_id` null → `AUTH_REQUIRED` (guarda defensiva)

O harness migrou para a nova assinatura: `callAccept` passa `p_user_id` explícito
(`accept_invite.test.ts:84-97`), e `asUser(...)` continua a correr como `authenticated`
(necessário para o GRANT), mas o claim já não é a fonte de verdade. Cobertura adequada para
o fix; os 2 testes adicionais provam exactamente a propriedade de segurança em causa.

## Foco 3 — Handler: mapeamento de erros PT-PT coerente

`route.ts:52-77` (`mapAcceptInviteError`) cobre 1:1 os códigos que a 0022 levanta:
`INVITE_NOT_FOUND`(404) · `INVITE_EXPIRED`(410) · `INVITE_ALREADY_ACCEPTED`(409) ·
`INVITE_EMAIL_MISMATCH`(403) · `ALREADY_MEMBER`(409) · `MEMBER_LIMIT_REACHED`(409) ·
`AUTH_REQUIRED`(401) · fallback `INTERNAL_ERROR`(500). `AUTH_REQUIRED` no caminho da função
agora só dispara pela guarda `p_user_id is null` (defesa em profundidade) — o caminho normal
de "sem sessão" é cortado antes, no handler (`route.ts:93-96`). O `user.id` passado é
sempre o da sessão validada. 8/8 mapeamentos cobertos por teste (`route.test.ts:72-90`).

  lexicográfica depois da 0020 (já em prod). Em prod, a 0020 existe; o `db:migrate` aplicará
