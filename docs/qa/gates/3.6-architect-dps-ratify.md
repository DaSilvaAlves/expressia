## 1. DPs HIGH — ratify explícito (novos DPs v1.1)

### DP-3.6.6 — Query design JOIN tasks↔tags (`json_agg` LEFT JOIN vs batch 2ª query)

**Recomendação @sm v1.1:** A — `json_agg(json_build_object(...) ORDER BY tags.name) FILTER (WHERE tags.id IS NOT NULL) AS tags` com LEFT JOIN task_tags + LEFT JOIN tags + GROUP BY tasks.id (1 query, idiomatic Postgres).
**Recomendação @po Pax v1.1:** [AUTO-DECISION → A] (justificação: TaskRow sem campo tags confirmado em `list-tasks.ts:32-50`; JOIN obrigatório; A é idiomatic + cardinalidade controlada).

**Verdict:** **APPROVED A — HIGH confidence técnica.** Recomendação unânime @sm + @po + @architect. Decisão técnica pura — não pertence a Eurico (sem implicação business/license).

**Razões técnicas (5 layers — pattern Story 3.4 DP-3.4.4 + Story 3.5 DP-3.5.1):**

1. **Cardinalidade controlada — `json_agg` com `GROUP BY` é O(n + m) onde n=tasks e m=task_tags links.** A query 1 round-trip produz exactamente N rows (uma por task), cada row contendo um JSON array de tags. Alternativa B (2ª query batch) força 2 round-trips: query 1 retorna N tasks + query 2 retorna M task_tags rows que precisam ser stitched in JavaScript via `Map<task_id, Tag[]>` — overhead de tempo de rede dobra (N=1 → N=2 RTT) + memória client-side. Para list-tasks com `limit: 50` (default) e fixture realista 1000 tasks × 5000 task_tags (5:1 média), o GROUP BY produz 50 rows com agregados; B produz 50 + ~250 task_tags = 300 rows a processar em userland. A é 6x mais leve em rows transferred.

2. **`FILTER (WHERE tags.id IS NOT NULL)` neutraliza o LEFT JOIN edge.** Tasks sem tags vêm com array vazio `[]` (não `[null]`). Sem o FILTER, `json_agg` produziria `[null]` para tasks sem tag — bug subtil que crasharia o frontend ao iterar `tags.map(tag => <TagBadge tag={tag}/>)`. Pattern idiomatic Postgres bem-conhecido (cross-confirmado em literature canónica e em pelo menos 2 helpers Drizzle no codebase Epic 1+2 — `listTasksHelper:124` `tag_id IN (select task_id from task_tags ...)` já usa sub-query similar).

3. **Index coverage existente é suficiente.** Schema confirmado `task_tags_household_idx` + `task_tags_tag_idx` (`packages/db/src/schema/tasks.ts:218-219`) + PK composta `(task_id, tag_id)` (linha 217) — todos os filtros do JOIN ficam servidos por index. `tags` table tem `tags_household_idx` (linha 188) + `tags_unique_name_per_household` (linha 190). EXPLAIN ANALYZE deve mostrar Index Scan + Bitmap Heap Scan para fixtures realistas (~1k tasks × 5k task_tags) bem dentro do budget AC10(d) 300ms p95.

4. **RLS é transparente para JOIN.** Policies `task_tags_select_member` + `tags_select_member` + `tasks_select_member` (já existentes) usam `is_household_member(household_id)` na coluna denormalizada de cada tabela. PG executa as 3 filtras em paralelo no plano JOIN — sem N+1 RLS overhead. Validado pattern Story 3.3 `listTasksHelper` que já usa LEFT-relation com `task_tags` em filter sub-query (`list-tasks.ts:123-126`) sem regressão.

5. **`DbShim.execute<T>` aceita SQL arbitrário via `sql\`...\`` template** (`list-tasks.ts:142-149` pattern). A extensão T6.0 é apenas: (a) novo SELECT enriquecido + (b) novo `GROUP BY tasks.id, ...` listando todas as columns da `SELECT` clause (PG requires explicit) + (c) `TaskRow` interface ganha `tags: Array<{ id: string; name: string; color: string }>`. Zero refactor estrutural — Story 3.3 G2 ("`TaskRow` interface vive aqui — single source-of-truth") aplica-se directamente.

**Trade-off honesto vs B (batch 2ª query):**

| Critério | A — `json_agg` LEFT JOIN | B — Batch 2ª query |
|----------|--------------------------|---------------------|
| Round-trips DB | 1 | 2 |
| SQL complexity | LEFT JOIN + GROUP BY + json_agg | 2 simple SELECTs |
| JavaScript complexity | Zero (PG retorna shape final) | `Map<task_id, Tag[]>` + merge loop |
| Rows transferred (fixture 1000×5000) | 50 (com aggregates) | 50 + ~250 = ~300 |
| Memory client-side | Constante | O(M) extra |
| Type-safety | `tags: Tag[]` directly em `TaskRow` | Two-step typing + merge type guard |
| Index utilization | Index Scan + Bitmap Heap (existing indexes) | Idem (mas 2x query plans) |
| EXPLAIN ANALYZE complexity | Único plan analisável | 2 plans separados |
| Future N filters (per-tag count, color filter etc) | Trivial extension | Requires more JS-side merging |

**Resultado:** A vence em 8/9 critérios; B só vence em "SQL initial complexity" (trade-off aceitável dado JavaScript ganha simplicity). **A é a escolha estrategicamente correcta.**

**Guidance notes (G1 — query optimization para @dev T6.0):**

| ID | Note | Why |
|----|------|-----|
| **G1.1** | **EXPLAIN ANALYZE obrigatório em fixture realista antes de marcar T6.0 done.** Comando: `EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) <query>`. Fixture mínima a montar (script `packages/db/src/scripts/seed-tags-fixture.ts` *opcional* — pode usar dados de teste existentes Story 3.5): ~1000 tasks + ~200 tags + ~5000 task_tags (média 5 tags por task). Validar: (a) Total execution time < 100ms (well within budget 300ms p95 AC10d), (b) zero Seq Scan em `task_tags` ou `tags`, (c) plan usa `task_tags_pkey` ou `task_tags_tag_idx` + Bitmap Heap Scan. Documentar output EXPLAIN no File List da story como comment no commit `feat(tasks): T6.0 listTasksHelper extension`. | Sem evidência empírica, p95 budget é assumption. Story 3.4 + 3.5 pre-flight Aria estabeleceram pattern de validar EXPLAIN antes de qualquer query JOIN nova. Esta é a primeira vez que `listTasksHelper` faz JOIN multi-row aggregation — risco baixo mas não zero. |
| **G1.2** | **`GROUP BY tasks.id, tasks.household_id, tasks.created_by_user_id, ..., tasks.updated_at`** (lista TODAS as colunas que aparecem no SELECT da `tasks` table). Postgres NÃO aceita `GROUP BY tasks.id` sozinho mesmo com PK quando o SELECT contém colunas não-agregadas — esse é um anti-pattern MySQL-only. Alternativa idiomatic se SELECT cresce: `SELECT tasks.*, json_agg(...) FROM tasks LEFT JOIN ... Escolha @dev — ambas funcionam em PG 16. | Pattern correcto evita 30 minutos perdidos a debugar "column must appear in the GROUP BY clause" error. Idempotent guidance — não muda performance. |
| **G1.3** | **`TaskRow.tags` Zod schema:** adicionar em `apps/web/src/lib/api-schemas/tasks.ts` (NÃO em tags.ts) `tags: z.array(TagSchema).optional().default([])` no `TaskListItemSchema` (ou equivalente). Re-export `TagSchema` de `tags.ts` para evitar circular import. Validar com `.default([])` para garantir que tasks sem tags vêm com array vazio (defesa em profundidade contra G1.1.b edge `[null]`). Adicionar test em `T8.5` ou novo file: "task com 0 tags retorna `tags: []`" + "task com 3 tags retorna `tags: [{...}, {...}, {...}]` ordered by name asc". | Type-safety end-to-end (DB → API → React props). Pattern Story 3.2 D-3.2.1 (extracted helpers + Zod) ratifies. |

**Risk note HIGH:** Nenhum risco material. Migração de schema = zero. Mudança de comportamento API `/api/tasks` = adicionar campo `tags` ao response (additive, backward-compat — clientes existentes ignoram novo campo).

### DP-3.6.7 — Endpoint extension `GET /api/tags?with_counts=true` (opt-in vs novo endpoint vs always-on)

**Recomendação @sm v1.1:** A — Query param opt-in `?with_counts=true` sem breaking change Story 3.2 (clientes sem param continuam sem `task_count`).
**Recomendação @po Pax v1.1:** [AUTO-DECISION → A] (justificação: count é UI requirement TagsManager AC3b, derivado, não FR PRD directo; pattern segue Story 3.2 `?status=`/`?priority=` opt-in).

**Verdict:** **APPROVED A — HIGH confidence técnica.** Recomendação unânime @sm + @po + @architect. C (sempre incluir `task_count`) está REJECT documented por @sm — Aria confirma a rejeição: alteraria response shape do `GET /api/tags` actualmente em produção pós-Story 3.2 (pushed e1954b4), forçando bumped Zod schema + retesting Story 3.2 tests + possíveis bugs em consumers Story 3.6 (TagPicker) que não precisam de `task_count` mas teriam o campo no payload.

**Razões técnicas (4 layers):**

1. **Backward compatibility 100% — additive change.** Story 3.2 `GET /api/tags` route handler (`apps/web/src/app/api/tags/route.ts:35-63` confirmado via Read) retorna `{ tags: TagRow[] }` onde `TagRow = { id, household_id, name, color, created_at, updated_at }`. Adicionar `?with_counts=true` que adiciona `task_count: number` apenas quando o param é truthy é o pattern menos invasivo possível. Story 3.2 tests existentes (`apps/web/src/app/api/tags/__tests__/...`) NÃO precisam ser alterados — continuam a validar o shape original sem `task_count`. Os novos tests da Story 3.6 (T3.3) validam o shape estendido apenas quando param activo.

2. **Pattern já estabelecido em Epic 3.** `GET /api/tasks?status=`, `?priority=`, `?kanban_column_id=`, `?assigned_to_user_id=`, `?project=`, `?due_date_from=`, `?due_date_to=`, `?tag_id=` (cross-confirmado em `list-tasks.ts:106-126`) — todos opt-in query params. Adicionar `?with_counts=true` ao `GET /api/tags` segue o **mesmo pattern de boolean opt-in** que outros endpoints Epic 3 (sem prior art directa para boolean count flag mas o princípio "opt-in via query string" é canónico). Pattern consistency reduz mental load de consumers.

3. **Constitution Article IV — [STORY-INVENT-DP-3.6.7] formalmente declarada.** Story v1.1 AC6 header (linha 257 confirmado via Read) inclui tag `[STORY-INVENT-DP-3.6.7]` + Constitution Checks Article IV linha 556 trace: "AC6 extensão `?with_counts` documentada como [STORY-INVENT-DP-3.6.7] com justificação explícita (UI requirement TagsManager AC3b — derivado, não inventado)". Aria confirma: a invenção é defensável porque (a) count é necessário para UX TagsManager (AC3b "count de tarefas associadas"), (b) PRD §FR12 cobre "listagem" mas não especifica metadata adicional — count é derivado para UX informativa, (c) `[STORY-INVENT-DP-3.6.7]` ratify gate é o mecanismo correcto Article IV. **Article IV: PASS com observance.**

4. **Subquery COUNT performance.** Query estendida: `SELECT tags.*, (SELECT COUNT(*)::int FROM task_tags WHERE task_tags.tag_id = tags.id AND task_tags.household_id = current_household_id()) AS task_count FROM tags ORDER BY name asc LIMIT 200`. Para household típico (`~50 tags × ~500 task_tags total`), Postgres executa 50 subqueries cada uma indexed via `task_tags_tag_idx` — `Total: ~50 × index seek ~0.5ms = ~25ms`. Para household power-user (`~200 tags × ~5000 task_tags`), `~200 × ~1ms = ~200ms` ainda dentro do budget AC10(d) 300ms p95. Validação empírica fica a cargo de T3.3 + EXPLAIN ANALYZE (G1.1 transferível para T3 — ver G2 below). Alternativa optimization se p95 ultrapassar 300ms: `GROUP BY tags.id` + `LEFT JOIN task_tags` (similar pattern a DP-3.6.6 mas inverso). **NÃO implementar prematuramente** — KISS, validar primeiro com subquery.

**Trade-off honesto vs B (novo endpoint `/api/tags/with-counts`) e C (always-on):**

| Critério | A — Opt-in param | B — Novo endpoint | C — Always-on |
|----------|------------------|-------------------|----------------|
| Backward compat | 100% | 100% (mantém antigo) | Quebra Story 3.2 |
| Duplicação código | Mínima (if-branch) | Alta (handler duplicado) | Mínima |
| Discoverability | Boa (query param documented) | Excelente (endpoint dedicated) | N/A |
| Pattern consistency Epic 3 | Alta (opt-in pattern uniforme) | Baixa (1 endpoint diferente) | N/A |
| TagPicker payload size | Mínimo (sem param) | Mínimo (chama endpoint original) | Sempre +N bytes por tag |
| Tests | 2 fluxos no mesmo handler | 2 endpoints separados | 1 fluxo (tag_count sempre presente) |
| HTTP cache headers | Same URL → same cache key OR varying by query (correcto) | Distinct URLs → cache key independente | Same |
| Article IV trace | [STORY-INVENT-DP-3.6.7] explícito | Idem | REJECT — breaking forçaria re-validate 3.2 |

**Resultado:** A vence em 6/8; B vence em discoverability + cache key clarity; C perde em backward compat + Article IV. **A é a escolha estrategicamente correcta** — pattern uniformity vence discoverability marginal.

**Guidance notes (G2 — endpoint extension para @dev T3):**

| ID | Note | Why |
|----|------|-----|
| **G2.1** | **Parse robusto de boolean query param.** `request.nextUrl.searchParams.get('with_counts') === 'true'` (string-comparison estrita). Aceitar apenas string literal `'true'` — rejeitar implicit truthy strings como `'1'`, `'yes'`, `'on'`. Justificação: pattern HTTP-canónico, evita surpresas (cliente passa `?with_counts=false` → param presente como string `'false'` → `Boolean('false')` é `true` JS, MAS `'false' === 'true'` é `false` — comportamento correcto). Adicionar test T3.3.4 (case 4 do test suite — `?with_counts=false` retorna sem `task_count`). | Defesa em profundidade contra clientes mal-comportados + clarity de contrato. |
| **G2.2** | **`current_household_id()` SQL helper já existe** (cross-confirmado em precedent migrations — pattern Story 1.3 RLS bootstrap). Subquery `WHERE task_tags.household_id = current_household_id()` é redundante com RLS (que já filtra automaticamente via JWT claims) MAS é defesa em profundidade + faz o plano de execução PG mais explícito (Bitmap Index Scan on task_tags_household_idx). Aria recomenda manter explicit como @sm propôs em AC6(a). | Cinto + suspensórios para multi-tenant. RLS é primeira linha de defesa; explicit filter é a segunda. Performance overhead = zero (mesmo index scan). |

**Risk note HIGH:** Nenhum risco material. Backward compat preservada. Article IV trace explícito.
