## Sumário executivo

Story 3.7 é **backend puro reuse-heavy** e a implementação reflecte-o com disciplina: ~3 ficheiros novos de produção (`rrule-helpers.ts` 223 LOC, `generate-recurring-tasks.ts` 283 LOC, migration 0013 59 linhas), 1 modify trivial (`route.ts` — uma linha de import + um elemento na array), 2 modificações de schema (1 enum value, 1 JSDoc fix), 1 runbook (148 linhas) e 33 testes Vitest novos. A arquitectura segue **exactamente** o template Story 2.8 (`cleanup-expired-reverse-ops`): cron Inngest nativo `0 3 * * *`, handler em `step.run`, `getServiceDb()` para RLS-bypass de job sistémico, errors re-thrown ao Inngest retry engine, observability via `childLogger` + `captureException`. A novidade arquitectural — expansão de recorrências timezone-aware — está bem isolada numa função pura (`expandRecurrence`) testável sem mocks, sem rede, sem Drizzle: separação de responsabilidades exemplar (a lógica RRULE/DST é testada exaustivamente em isolamento; o handler é testado com `expandRecurrence` mockado para controlar deterministicamente o nº de INSERTs).

A decisão arquitectural mais sensível desta gate é a **ratificação de [DEV-DECISION D-3.7.9]** — a correcção de um bug off-by-one genuíno na contagem de ocorrências. Não é um desvio cosmético: a Story (Dev Notes linha 202) descrevia `rule.after(now + horizonDays + 1)`. O @dev descobriu durante a implementação que, como as ocorrências são fixadas ao meio-dia Lisboa (≈12:00 UTC — escolha deliberada para evitar a janela DST ambígua 02:00→03:00), uma janela `between(now, horizonEnd)` com `horizonEnd` à meia-noite **excluiria a ocorrência do último dia**, devolvendo sistematicamente N−1 ocorrências (ex.: horizon=7 produziria 7 em vez das 8 esperadas pelo AC8 teste 1). A correcção — levar `horizonEnd` a `23:59:59.999` UTC do dia `now + horizonDays` — é a fix correcta. Verifiquei a lógica directamente em `rrule-helpers.ts:185` (`new Date(addDays(now, horizonDays).getTime() + (86_400_000 - 1))`) e confirmei que os 22 testes de `rrule-helpers.test.ts` validam genuinamente a fronteira do horizonte: teste (1) `daily horizon=7 → 8 ocorrências` (a fronteira inclusiva), teste (11) `endsOn dentro do horizonte → trunca` (a interacção `endsOn`/`horizon`), teste (14) `nextRunAfterHorizon` para `monthly horizon=90` partindo de `now=2026-05-20` → exactamente `2026-09-15` (a continuidade pós-horizonte sem salto nem repetição). Consequentemente `nextRunAfterHorizon = rule.after(horizonEnd, false)` (fronteira **exclusiva**) é a escolha correcta: garante que a primeira ocorrência *estritamente posterior* ao último instante do horizonte é apanhada sem ser saltada (um `after(horizonEnd+1day, true)` correria o risco de saltar uma ocorrência que caísse exactamente no dia seguinte). E `isExhausted = (nextRunAfterHorizon === null)` deriva correctamente: se a RRULE não tem nada para lá do horizonte — porque `endsOn` caiu dentro dele ou porque a frequência é finita — a recorrência está esgotada e o handler desactiva-a (`active = false`). **RATIFICADA — fix arquitecturalmente correcta, cobertura de testes genuína.**

A **mitigação DST [DEV-DECISION D-3.7.5 / R-3.7.2]** é real e verificável. `rrule-helpers.ts` converte cada `Date` UTC devolvido pela `rrule` para a data de calendário visível via `formatInTimeZone(d, 'Europe/Lisbon', 'yyyy-MM-dd')` (linha 192) — nunca `date.toISOString().slice(0,10)`. O `DTSTART` é fixado ao meio-dia Lisboa via `fromZonedTime(\`${isoDate}T12:00:00\`, TZ)` (linha 126), o que evita a janela ambígua das transições. Adicionalmente há um `Set<string> seen` (linhas 189-198) que deduplica datas — defesa contra o fall-back de Outubro onde duas instâncias UTC distintas poderiam mapear no mesmo dia de calendário. Os testes (12) e (13) validam explicitamente as transições reais de 2026: spring forward 2026-03-29 (último domingo de Março — relógio WET→WEST) confirma 3 dias consecutivos `28/29/30` sem buraco; fall back 2026-10-25 (WEST→WET) confirma `2026-10-25` aparece exactamente uma vez (`dates.filter(d => d === '2026-10-25').toHaveLength(1)`) e a sequência `24/25/26` é consecutiva. Corri os 22 testes localmente: **22/22 GREEN**. A mitigação não é teatro — cobre a fronteira genuína.

A **idempotência [R-3.7.1]** é sólida por construção. A migration 0013 cria `tasks_recurrence_id_due_date_unique ON tasks (recurrence_id, due_date) WHERE recurrence_id IS NOT NULL` — índice unique **parcial**, o que é a escolha correcta: tasks não-recorrentes (`recurrence_id NULL`) não são afectadas pela constraint, e como NULLs distintos não colidem num índice não-parcial isto evita qualquer ambiguidade. O handler usa `INSERT ... ON CONFLICT (recurrence_id, due_date) WHERE recurrence_id IS NOT NULL DO NOTHING RETURNING id` (`generate-recurring-tasks.ts:171-173`) — a cláusula `WHERE` do `ON CONFLICT` casa exactamente com a predicate do índice parcial (obrigatório no Postgres para o conflict target resolver para um índice parcial). `inserted.length > 0` discrimina geração de skip. O Inngest tem entrega *at-least-once* — o mesmo cron tick pode disparar 2× — e este desenho torna o rerun inofensivo. O teste AC9 (3) valida o cenário rerun: 91 ocorrências, todos os INSERTs devolvem array vazio (DO NOTHING disparou), `total_generated = 0`, `total_skipped = 91`, zero corrupção de estado. Corri o teste: **GREEN**.

O **audit log INSERT [AC6 / NFR9 / D-3.7.2]** respeita a imutabilidade. NFR9 bloqueia UPDATE/DELETE em `audit_log` mas **permite INSERT** — o gate 2.8 já verificou directamente a policy `audit_log_insert_member` em `0001_rls_policies.sql:569-573` (`with check (household_id is null or is_household_member(household_id))`), e o `household_id IS NULL` é explicitamente aceite, o que valida o INSERT cross-household agregado deste job. **1 row agregada por run** (não 1 por recorrência) é a granularidade correcta: NFR9 satisfeito com row + summary JSON contendo todos os counts + `run_id`, e poupa ~100× row growth no worst case — o detalhe per-recorrência vive nos logs Pino estruturados (Grafana), que é o sítio certo para debug. O bloco audit está envolto em `try/catch` que faz `log.warn` + `captureException({ tags: { job, phase: 'audit_log' } })` mas **não re-throws** — falha do audit não aborta o job (pattern Story 2.8 AC5). Isto é defensável: a geração de tarefas é a função core; o audit é compliance secundário. O teste AC9 (7) valida: audit INSERT rejeitado → handler conclui na mesma com return statement não-zero + warn registado.

A **whitelist OTel [AC7 / NFR12]** é limpa. Os 7 atributos do span `agent.recurrences.generate` são exclusivamente counts numéricos (`processed_count`, `generated_count`, `skipped_count`, `inactivated_count`, `horizon_days`, `duration_ms`) + `inngest.run_id` (UUID de run, não é PII). Zero `title` de task, zero `householdId`/`userId` raw, zero `recurrence.id` raw. O teste AC9 (8) afirma isto positivamente (`attrKeys` contém os 7) **e** negativamente (`not.toContain('household.id')`, `not.toContain('user.id')`, `attrKeys.some(k => k.includes('title')) === false`). Os logs Pino podem conter `recurrence_id` para debug — aceitável, `recurrence_id` é um UUID interno, não PII de utilizador, e os logs não são traces partilhados.

O **RLS via `getServiceDb()`** é justificado e consistente com o precedente. Um cron Inngest não tem JWT de utilizador no contexto, logo `current_household_id()` não está disponível e a RLS de `authenticated` não pode ser satisfeita. O job é sistémico e cross-household por natureza (processa recorrências de todos os agregados). Pattern idêntico ao `cleanup-expired-reverse-ops` (Story 2.8 D38) e análogo à GDPR purge mensal. `getServiceDb()` bypassa a RLS — seguro aqui porque o WHERE da query SELECT é puramente funcional (`active = true AND (next_run_on IS NULL OR next_run_on <= current_date)`), não há filtro por household que pudesse vazar dados, e cada INSERT em `tasks` herda o `household_id` correcto **da própria recorrência** (`r.household_id` — `generate-recurring-tasks.ts:165`), preservando o tenant boundary nos dados gerados.

A **migration 0013** é não-destrutiva e a sua aplicação diferida a @devops é aceitável. As 3 mudanças (ALTER TYPE ADD VALUE, CREATE UNIQUE INDEX parcial, COMMENT ON TABLE) usam `IF NOT EXISTS` / predicate parcial — idempotentes, re-run safe. `ALTER TYPE ... ADD VALUE` não altera queries existentes; o índice parcial só cobre rows com `recurrence_id` não-nulo e zero rows em produção têm esse campo populado (Story 3.7 é a primeira a popular). O comentário da migration nota correctamente que PG 16 suporta `ALTER TYPE ADD VALUE` dentro de transação e que o novo value não é usado na mesma transação que o adiciona (a migration apenas adiciona — zero INSERT). Diferir a aplicação real a @devops segue o pattern Story 3.8 D-3.8.1 (mockable-only + EB4 deferred) — aceitável porque a migration é idempotente e o runbook documenta explicitamente a aplicação como Step 1.

**Reuse Inngest sem regressão:** `route.ts` adiciona `generateRecurringTasks` à array `functions: [cleanupExpiredReverseOps, generateRecurringTasks]` — uma linha. O Inngest introspecciona via GET e regista ambas as funções. Build confirma `/api/inngest` permanece registado (rota pré-existente Story 2.8 — zero rotas novas). 510/510 testes web GREEN incluindo os 5 testes Story 2.8 de `cleanup-expired-reverse-ops` — sem regressão.

## Análise específica: as decisões críticas desta gate

### [DEV-DECISION D-3.7.9] ratification — off-by-one fix no horizonte de expansão

**Pergunta arquitectural:** A escolha @dev de levar `horizonEnd` ao fim do dia + usar `after(horizonEnd, false)` é uma correcção legítima de bug ou um desvio do desenho da story que requer escrutínio?

**Análise:** É correcção legítima de bug, não desvio. A raiz do problema é a **interacção entre duas decisões correctas**: (a) D-3.7.5 fixa as ocorrências ao meio-dia Lisboa (≈12:00 UTC) para evitar a janela DST ambígua — decisão sólida; (b) o desenho original da story computava a janela `between(now, now + horizonDays)` com o fim à meia-noite. A consequência é que a ocorrência do dia `now + horizonDays`, datada às 12:00, cai **fora** de uma janela que termina às 00:00 desse mesmo dia → N−1 ocorrências sistemático.

| Critério | `horizonEnd` à meia-noite (desenho original) | `horizonEnd` a 23:59:59.999 (escolha @dev) |
|----------|---------------------------------------------|---------------------------------------------|
| Contagem para `daily horizon=7` | 7 ocorrências (errado — AC8 teste 1 espera 8) | 8 ocorrências ✓ |
| Ocorrência do último dia (12:00) | excluída do `between` | incluída ✓ |
| `nextRunAfterHorizon` | `after(horizonEnd+1day, true)` — risco de saltar ocorrência do dia seguinte | `after(horizonEnd, false)` — primeira estritamente posterior, sem salto ✓ |
| `isExhausted` | derivação ambígua | `nextRun === null` — limpo ✓ |

**Verdict:** **PASS — RATIFICADA.** A fix é a correcta. Verifiquei `rrule-helpers.ts:185` (`horizonEnd = now + horizonDays + 86_399_999 ms`), `:203` (`after(horizonEnd, false)`), `:209` (`isExhausted = nextRunAfterHorizon === null`). Os 22 testes cobrem a fronteira genuinamente — teste (1) a contagem inclusiva, (11) a truncação por `endsOn` dentro do horizonte, (14) a continuidade pós-horizonte exacta (`monthly` → `2026-09-15`), (12)+(13) as transições DST reais. **Sem follow-up.**

### [DEV-DECISION D-3.7.10] ratification — `${JSON.stringify}::jsonb` vs `sql.json`

**Pergunta arquitectural:** O padrão `${JSON.stringify(state)}::jsonb` é seguro contra SQL injection?

**Análise:** Sim. A interpolação `${...}` dentro de um Drizzle `sql\`...\`` template **não é concatenação de string** — Drizzle trata cada valor interpolado como bind parameter (placeholder `$N` na query final, valor passado separadamente ao driver). `JSON.stringify(summary)` produz uma string; essa string entra como parâmetro; o `::jsonb` é um cast SQL aplicado ao parâmetro. Mesmo que o conteúdo do JSON contivesse aspas ou `;`, nunca seria interpretado como SQL — está do lado dos dados, não do código. É exactamente o padrão de `api-helpers/audit.ts:106-107` (Story 3.2, já em produção). Verifiquei independentemente que `sql.json` é `undefined` em `drizzle-orm@^0.34.0` — a alternativa mencionada na story genuinamente não existe nesta versão.

**Verdict:** **PASS — RATIFICADA.** Escolha correcta e consistente com o codebase. **Sem follow-up.**

### [DEV-DECISION D-3.7.8] ratification — assinatura `withSpan` 3-arg

**Verdict:** **PASS — RATIFICADA.** Verificação directa `packages/observability/src/tracer.ts:124` — `withSpan<T>(name, initialAttrs: DomainAttributes, fn)`. A story mostrava 2 args; a assinatura real exige 3. `initialAttrs = {}` é a escolha correcta — os counts só existem após o `step.run`, aplicados via `span.setAttribute`.
