## Sumário executivo

A Story 2.4 entrega exactamente o Estágio 1 da pipeline AI multi-intent prometido pela arquitectura §4.2: **um package source-only `@meu-jarvis/classifier` que invoca GPT-4o-mini via OpenAI structured outputs (`response_format: { type: 'json_schema', strict: true }`), valida output com `ClassificationSchema` Zod (1-5 intents, FR3 satisfeito), aplica language gate determinístico pré-LLM (poupa tokens em inputs non-PT-PT), e expõe uma classe `Classifier` injectável via `OpenAIClientLike` (interface estructural da Story 2.2 — sem acoplamento a SDK).** A disciplina "Foundation, not behaviour" estabelecida pelas Stories 2.2 e 2.3 é aqui rigorosamente respeitada: zero wiring com endpoint produção (fica para 2.6), zero integração com Planner/Executor (fica para 2.5), zero benchmark com prompts reais (fica para 2.10). EB2 (OpenAI DPA UE) **não bloqueou** a story porque toda a implementação é 100% mockable — `pnpm test` corre sem `OPENAI_API_KEY` e entrega 78 testes determinísticos (target ≥35; mapeados 46; **70% acima** do target mínimo).

A decisão arquitectural mais escrutinável (D11 — sanity-check Article IV via `fs.readFile + regex` sobre `packages/db/src/schema/agent.ts`, em vez de import directo) está **bem fundamentada e consistente com o pattern do monorepo**, replicando exactamente a abordagem aprovada na Story 2.2 (D1 `LLM_MODEL_VALUES` hardcoded + sanity test contra enum). A causa raiz é a mesma: TypeScript com `moduleResolution: Bundler` não resolve aliases `@/*` internos do `@meu-jarvis/db` quando outro package faz typecheck, e adicionar project references seria over-engineering para o ganho. A solução I/O em runtime é robusta: **o test FAILS deterministicamente se alguém alterar o enum DB sem actualizar `INTENT_VALUES`** (verificado neste gate via inspecção de `schemas.test.ts:32-43` que faz `readFileSync(DB_AGENT_SCHEMA_PATH, 'utf-8')` + regex `agentIntentEnum\s*=\s*pgEnum\(\s*'agent_intent'\s*,\s*\[([^\]]+)\]`). O único trade-off é fragilidade ao path absoluto se a estrutura do monorepo mudar — registado como NIT-002 não-bloqueante (failure mode é limpo, não silent).

A decisão D8 (max(5) intents — adicionada pelo @po na validation, AC3) é **defensável como guardrail anti-hallucination** sem fonte FR explícita, mas vale a pena consolidar no futuro. A Architecture §4.2:327 só especifica `min(1)` literal; o PRD Epic 2 AC1 ilustra cenário canónico com 2 intents simultâneas. Avaliei 3 alternativas:
- **Manter max(5)** (recomendado): bom equilíbrio prudência vs flexibilidade. PRD não fixa máximo, mas testar 5+ intents num único prompt é raro e o LLM pode hallucinar arrays excessivos quando incerto. **Validável em telemetry pós-2.5/2.11** (Story 2.11 dashboards mostrarão distribuição real de `classifier.intent_count`; se p99 < 4, max(5) está confortável).
- **max(10)**: permitir prompts mais densos. Risco de aumentar entropy de output (LLM gera 7-8 intents fracas em vez de 2-3 fortes). Sem evidência empírica que isto seja necessário.
- **Remover max (só min(1))**: literal Architecture. Risco real — uma única misclassification do LLM pode gerar 50+ "intents" de baixa confidence, degradando UX e custo de execução pelo Planner (Story 2.5).

**Verdict arquitectural:** PASS para max(5) + nota para Story 2.11 telemetry validation. Se telemetry mostrar p99 < 3 em produção, max(5) é over-cautious mas não-prejudicial. Se mostrar misclassifications a chocar o teto, escalar para max(10) com bumping de `CLASSIFIER_SYSTEM_PROMPT_VERSION` (snapshot test apanha).

A decisão D9/D10 (export de `OpenAIClientLike` + `sanitizeHint` + `mapOpenAIError` do `@meu-jarvis/agent`) modifica uma story Done (2.2) e merece análise dedicada. **Modificações são zero-comportamentais**: apenas tornam pública uma type e duas functions já existentes em `packages/agent/src/providers/openai.ts:42` e `packages/agent/src/errors.ts:298,335`. Verifiquei independentemente neste gate: `pnpm --filter @meu-jarvis/agent typecheck` e `test` passam **inalterados** pós-export (102 agent tests preservados). O contract público da 2.2 expandiu mas não quebrou — tudo que estava exportado continua exportado, e a Story 2.4 (que precisa destes símbolos por AC7 e AC10) consome-os de forma legítima. **Este é o pattern correcto de evolução de packages no monorepo**: re-exports zero-risk a packages a montante são preferíveis a duplicação local ou redefinições inconsistentes. Trade-off de modificar uma story Done é mais que compensado pela manutenção do princípio "single source of truth" no `@meu-jarvis/agent`.

### D9 — Export `OpenAIClientLike` de `@meu-jarvis/agent`

**Pergunta arquitectural:** Modificar uma story Done (2.2) para expor uma type interna é aceitável?

**Resposta:** **Sim, quando a modificação é zero-comportamental e satisfaz uma necessidade declarada por outra story.** A AC7 da Story 2.4 EXPLICITA `OpenAIClientLike é importado de @meu-jarvis/agent`. A type já existia em `packages/agent/src/providers/openai.ts:42-44` como `export interface` mas não estava re-exportada via barrel `index.ts`. **Verificação independente neste gate:**
- `pnpm --filter @meu-jarvis/agent typecheck` → exit 0 (102 tests preservados)

**Verdict:** PASS. Pattern correcto de evolução incremental do `@meu-jarvis/agent` à medida que mais consumers (Stories 2.4, 2.5, 2.6) precisam de constructs internos.

### D10 — Export `sanitizeHint` + `mapOpenAIError` de `@meu-jarvis/agent`

**Pergunta arquitectural:** Tornar uma function private em public é breaking change semântico?

**Resposta:** **Não é breaking change**: o universo de consumers que precisam de `sanitizeHint` é apenas o classifier (story 2.4) e potencialmente o Planner (story 2.5). `sanitizeHint` é uma function pura (`(raw: string) => string`) sem efeitos colaterais — exportá-la é low-risk. `mapOpenAIError` já era exportado de `errors.ts` mas não do barrel — adicionado para coerência (todos os mappers errors são públicos).

**Alternativa rejeitada (re-implementar `sanitizeHint` localmente):**
- Duplica regex `[A-Za-z0-9_-]{10,}` e `\d{9,}` em 2 packages — drift risk se Story 2.2 evoluir o pattern (ex: adicionar novo redaction class para CC/IBAN).
- Viola DRY com 4 linhas de código.

**Verdict:** PASS. Re-export é o pattern correcto para function purely shared.

### D11 — Article IV sanity-check via `fs.readFile + regex`

**Pergunta arquitectural:** I/O em test é aceitável vs project references no tsconfig?

**Resposta:** **Sim para este escopo.** Project references seria a solução TypeScript-canonical, mas requer:
- Adicionar `composite: true` em `packages/db/tsconfig.json`
- Adicionar `references: [{ path: '../db' }]` em todos os tsconfigs que importam `@meu-jarvis/db` (web, classifier, agent — futuro)
- `tsc --build` em vez de `tsc --noEmit` (mudança em scripts)
- Coordenação com turborepo task dependencies

Para uma única necessidade (sanity-check de 1 enum em 1 test), o overhead é desproporcional. A solução I/O custa 1 `readFileSync` síncrono + 1 regex match — execução < 5ms — uma única vez no setup do test file.

**Failure mode é claro:**
- Path move → erro "ENOENT: ...path/agent.ts"
- Regex desalinha (rename `agentIntentEnum` → `intentEnum`) → erro "agentIntentEnum não encontrado em <path>"

Não é silent failure. Documentação inline + Directive em commit message do `adc57d7`.

**Verdict:** PASS com NIT-002 (path absoluto frágil em refactor monorepo). Aceitável.
